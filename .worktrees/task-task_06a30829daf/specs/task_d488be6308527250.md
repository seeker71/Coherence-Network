# Spec: Invest Page — Garden Metaphor Over Spreadsheet Metrics

**Spec ID**: `task_d488be6308527250`  
**Task ID**: `task_d488be6308527250`  
**Status**: Draft (spec-only; implementation follows separate task)  
**Surface**: Web — [`/invest`](https://coherencycoin.com/invest)  
**Related**: Spec 157 (investment UX), Spec 116 (portfolio metrics), UX guidance in `AGENTS.md`

---

## Summary

Reframe the **Invest** page so contributors experience **nurturing living ideas** (a garden) instead of scanning a **financial mini-sheet** (value gap, estimated cost, ROI bar). Primary perception should be *growth potential* shown as a **sprout → tree** visual progression derived from existing idea scores. **Numeric detail stays available** (secondary: disclosure, tooltip, or compact “soil facts” row) so power users and auditors are not blocked.

**Non-goals for this spec**: Changing staking economics, ledger APIs, or how `value_gap` / `estimated_cost` are computed server-side — only **presentation, copy, and client-side visual mapping** unless an optional read-only aggregation endpoint is justified below.

---

## Problem Statement

Today (`web/app/invest/page.tsx`) each idea card exposes:

- Labels: **Value gap**, **Est. cost**, **ROI**
- A **horizontal ROI bar** capped at 20× visually

This reads like a spreadsheet or trading desk. It does not reinforce the product story (“your attention becomes working code”) and does not communicate *stage of life* of an idea as clearly as a **growth arc** could.

---

## Goals

1. **Language**: Replace spreadsheet-first labels with garden-first labels everywhere above the fold on `/invest`, while preserving semantic alignment with underlying fields.
2. **Growth visualization**: Replace the bare ROI progress bar with a **sprout-to-tree** (multi-stage) illustration whose stage is **deterministic from API fields** already returned by `GET /api/ideas`.
3. **Numbers secondary**: Show the emotional/visual layer first; place USD/CC numbers in a **secondary** pattern (smaller type, “details” toggle, or footnote row — exact pattern in Requirements).
4. **Proof over time**: Define how we will **know the metaphor works** and how **evidence tightens** as we ship iterations (metrics, qualitative checks, optional instrumentation).

---

## Requirements

### R1 — Copy: Garden-First Labels (Invest List)

On `/invest`, for each idea card:

| Current label | Underlying field(s) | New primary label (garden) | Notes |
|---------------|---------------------|----------------------------|-------|
| Value gap | `value_gap` | **Room to grow** (or **Untapped soil**) | Conveys potential without “trader” tone |
| Est. cost | `estimated_cost` | **Water needed** (or **Rain budget**) | Cost as nurture input, not brokerage fee |
| ROI (ratio + bar) | derived `value_gap / estimated_cost` (with guard) | **Growth strength** (visual stage + optional multiplier in secondary text) | No standalone “ROI” as the hero word |

**Acceptance**: With JavaScript enabled and disabled (SSR HTML), headings visible without hover must use the new vocabulary. Screen readers hear the new terms via visible text (not `aria-label` alone).

### R2 — Sprout-to-Tree Progression (Replaces ROI Bar)

Implement a **discrete stage control** (4–5 stages), not a single thin bar:

- **Stages** (example names; finalize in implementation): `Seed` → `Sprout` → `Sapling` → `Canopy` (optional fifth: `Old growth` for exceptional confidence + manifestation).
- **Mapping** must be **pure function** of server data, e.g. combine:

  - `manifestation_status` (lifecycle),
  - `confidence` or `free_energy_score` (vitality),
  - normalized **growth ratio** (same math as today’s ROI but fed into **stage buckets**, not a stock ticker).

Document the exact formula in code comments and in §Data Mapping below.

**Acceptance**: For two ideas with identical numeric inputs, the visual stage is identical; changing only `manifestation_status` can change stage without a full reload (client navigation).

### R3 — Numbers Secondary

- Default card view: **garden visual + short metaphor subtitle** (one line).
- **Figures** (`formatUsd(value_gap)`, `formatUsd(estimated_cost)`, ratio): shown in a **muted row**, smaller typography, or behind **“Soil facts”** / **“Measure”** expander (one pattern sitewide on this page).
- **Stake** CTA remains visible; do not hide staking behind the expander.

**Acceptance**: Lighthouse / manual check: primary content is readable without parsing currency strings.

### R4 — Balance Section Tone

`InvestBalanceSection.tsx` currently says **Your CC Balance**. Reframe to garden-adjacent but **clear**:

- Title: e.g. **Your watering can** or **Seeds available (CC)** — must still communicate **CC** in the string for clarity.
- Numeric CC value unchanged; `GET /api/contributions/ledger/{contributor_id}` unchanged.

### R5 — Empty and Error States

- **No ideas**: keep link to share ideas; optionally add one line of garden copy (“Plant the first idea”).
- **API failure** (empty list from failed fetch): show **non-deceptive** empty state or error — do not fake garden health. Spec: distinguish **“could not load garden”** vs **“garden is empty”** when feasible (HTTP status from fetch).

### R6 — Accessibility & Motion

- Sprout-to-tree art must have **text equivalent** (stage name + short description).
- Respect `prefers-reduced-motion`: static illustration or cross-fade only.

### R7 — Proof & Iteration (Is the Metaphor Working?)

Address the open question *“How do we improve this idea, show whether it is working yet, and make that proof clearer over time?”* with a **defined evidence ladder**:

| Phase | What we measure | Decision |
|-------|-----------------|----------|
| **Launch (MVP)** | Qualitative: 3–5 user-readable scenarios pass; stakeholder signoff; no increase in support confusion | Ship |
| **Short-term** | Web analytics: time-on-page `/invest`, scroll depth to “Soil facts”, click-through to `/ideas/{id}` vs baseline (if tracking exists) | If engagement drops, revisit copy |
| **Medium-term** | Optional event: `invest_growth_stage_viewed` with `{ idea_id, stage }` (privacy-preserving; no PII) | Validates that users see stages |
| **Long-term** | Correlate with stakes created post-view (API or ledger events) — **needs privacy/product review** | Only with explicit consent/spec |

**Improvement loop**: A/B or sequential releases — garden copy vs control — documented in `docs/` only if a future spec requires it (out of scope here).

### R8 — Internationalization (Future)

Copy is English-only for MVP; structure strings so extraction to i18n is possible (no logic embedded in JSX string concatenation beyond trivial).

## API Contract

### Existing (must continue to work — verification relies on these)

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/ideas?limit=60` | Portfolio list for `/invest` SSR fetch |
| `GET` | `/api/contributions/ledger/{contributor_id}` | CC balance (URL-encoded contributor id) |

**Response shape** (relevant fields from ideas list items): `id`, `name`, `manifestation_status`, `value_gap`, `estimated_cost`, `confidence`, `free_energy_score`, `actual_value`, `actual_cost` (others may exist; UI must tolerate omissions with safe defaults).

### New endpoints

**None required for MVP.** Optional follow-up (not in this spec’s acceptance): `GET /api/ideas/{idea_id}/invest-story` for curated copy — only if SSR needs richer narrative than list payload.

## Data Mapping (Normative for Implementers)

**Growth stage** = `f(manifestation_status, confidence | free_energy_score, growth_ratio)` where:

- `growth_ratio = value_gap / max(estimated_cost, ε)` with ε avoiding divide-by-zero (align with current `computeRoi` guard).
- Bucket thresholds and status weights **must be documented** in a single module (e.g. `web/lib/invest_growth_stage.ts`) with unit tests.

**Sorting**: Today ideas sort by ROI descending. Preserve **same sort order** for MVP (growth potential ranking) unless product explicitly changes — document if visual order differs from numeric order.

## Verification Scenarios

Each scenario is runnable against **production** (`https://coherencycoin.com`) or **local** (`NEXT_PUBLIC_API_URL`).

### Scenario 1 — Page loads and shows garden vocabulary (read)

- **Setup**: Network allows GET to public API; at least one idea exists in production (or use local API with seeded ideas).
- **Action**: `curl -sS -o /dev/null -w "%{http_code}" https://coherencycoin.com/invest`  
  For HTML content check: open `/invest` in a browser or `curl -sS https://coherencycoin.com/invest | head -c 8000` and search for old labels.
- **Expected**: HTTP **200**. Rendered/SSR HTML must **not** contain the old primary headings **“Value gap”**, **“Est. cost”**, or hero-level **“ROI”** as the main label pattern (garden labels present per R1). Stake CTA or equivalent remains.
- **Edge**: If API returns zero ideas, page shows empty garden state **without** crashing; **no** “Value gap” grid.

### Scenario 2 — Full read cycle for portfolio data (ideas list)

- **Setup**: API returns JSON for ideas.
- **Action**: `curl -sS "https://api.coherencycoin.com/api/ideas?limit=5" | head -c 4000`  
  Note `id` of first idea. Then `curl -sS -o /dev/null -w "%{http_code}" "https://coherencycoin.com/ideas/<idea_id>"`.
- **Expected**: Ideas endpoint returns **200** with JSON `ideas` array; idea detail page returns **200** (navigation from Invest works).
- **Edge**: `curl -sS -o /dev/null -w "%{http_code}" "https://api.coherencycoin.com/api/ideas?limit=0"` — server accepts or returns valid error; Invest page must not 500 if limit is ignored client-side.

### Scenario 3 — Ledger read + error handling (balance strip)

- **Setup**: Contributor id known or use nonsense id for error path.
- **Action**: `curl -sS -o /dev/null -w "%{http_code}" "https://api.coherencycoin.com/api/contributions/ledger/test-nonexistent-contributor-xyz"`  
  (Adjust path encoding if spaces.)
- **Expected**: **200** with JSON (possibly zero balance) **or** documented **404** — in either case, Invest page client must show **non-throwing** balance section (no uncaught exception in browser console for that response).
- **Edge**: Malformed contributor id in URL — API returns **4xx/422** if applicable; UI shows balance unavailable without breaking the page.

### Scenario 4 — Growth visual deterministic from data

- **Setup**: Two ideas with same `manifestation_status`, `value_gap`, `estimated_cost`, `confidence` in API response.
- **Action**: Compare rendered **stage** for both on `/invest` (visual or `data-testid` / aria in HTML).
- **Expected**: **Identical** growth stage for both (R2).
- **Edge**: `estimated_cost = 0` — mapping uses ε; **no** NaN or empty stage; no divide-by-zero in client.

### Scenario 5 — Reduced motion and accessibility

- **Setup**: Browser with `prefers-reduced-motion: reduce` enabled.
- **Action**: Load `/invest`; inspect growth visual (no infinite animation) / screen reader announces stage text.
- **Expected**: Motion reduced per R6; stage name visible in DOM for assistive tech.
- **Edge**: With CSS disabled, stage **text** still visible (progressive enhancement).

## Risks and Assumptions

- **Risk**: Garden metaphors may confuse users who want raw numbers — **Mitigation**: secondary “Soil facts” row and tooltips with exact USD/CC.
- **Risk**: Stage mapping feels arbitrary — **Mitigation**: document formula; tune thresholds with one A/B or PM review.
- **Assumption**: `GET /api/ideas` fields remain stable; if renamed, mapping module fails tests first.
- **Assumption**: Production API and web are deployed together enough that SSR fetch sees same schema as documented.
