# Spec: Homepage Readability — Text Contrast on Dark Background

**Spec ID**: task_f4a9da594102d8e0
**Status**: Draft
**Author**: product-manager agent
**Related**: specs/150-homepage-readability-contrast.md (primary implementation spec)

---

## Summary

New contributors landing on `https://coherencycoin.com/` encounter a hero section where the
primary body copy — the invitation prompt, live stats row, section descriptions, and footer tagline —
is barely legible against the dark warm-amber background. The layered `body::before` gradient bloom and
the opacity modifiers applied to Tailwind `text-foreground/*` classes combine to render secondary text
at effective opacity values as low as 0.70, well below the minimum readable threshold.

This spec defines **what must be true** after the fix, adds the open-question decisions missing from
spec 150, and establishes runnable verification scenarios. It does **not** introduce a light-mode
toggle (deferred) but does define the proof mechanism requested by the task.

---

## Goal

Make every piece of body copy on the homepage (`/`) readable without strain. Define a minimum opacity
floor of **0.85** for prose, stat labels, and interactive affordances, while keeping warm amber
headings and decorative glows at their current aesthetic strength.

---

## Problem Analysis

### Current state (pre-fix)

| Element | Tailwind class used | Effective opacity |
|---------|--------------------|--------------------|
| Hero sub-paragraph | `text-foreground/70` (inferred from glow wash) | ≈ 0.70 |
| Pulse stat labels | `text-foreground/80` | ≈ 0.80 |
| "How it works" step descriptions | `text-foreground/70` | ≈ 0.70 |
| Feed card secondary lines | `text-foreground/70` | ≈ 0.70 |
| Footer tagline / links | `text-foreground/70` | ≈ 0.70 |
| Idea submit form placeholder | browser default / unset | ≈ 0.50 |

The `body::before` fixed overlay adds three radial gradients with `mix-blend-mode: screen`, further
desaturating low-opacity text relative to the `--background` token (`hsl(24 22% 11%)`). The net
effect is text that appears dim even though the raw CSS token `--foreground: 38 32% 93%` is
nominally bright.

### Root cause

1. Too-low opacity modifiers on prose classes (`/70`, `/75`).
2. `body::before` bloom is strong enough to create a mid-tone wash that reduces perceived contrast
   of anything below full opacity.
3. No floor enforcement — any developer can accidentally add a `/60` class without CI failure.

---

## Requirements

### R1 — Body copy contrast floor

All homepage prose text (hero description, stat labels, "How it works" descriptions, resonance feed
secondary lines, footer links, closing quote, idea form helper text) **must** use a Tailwind opacity
modifier of **≥ 0.85** relative to `hsl(var(--foreground))`.

Acceptable classes: `text-foreground`, `text-foreground/85`, `text-foreground/90`, `text-foreground/95`.

Prohibited for body copy: `text-foreground/80` or lower on marketing/prose elements (error states,
ping animations, and chart accents are exempt).

### R2 — CSS token adjustments

The CSS design tokens in `web/app/globals.css` must be checked to ensure:

- `--foreground` lightness is **≥ 90%** in the dark theme (current: `38 32% 93%` — already adequate;
  do not lower it).
- `--muted-foreground` lightness is **≥ 85%** (current: `34 22% 90%` — already adequate).
- `body::before` radial gradients use **≤ 0.05 alpha** on the amber channel and **≤ 0.04** on the
  teal channel (current: `0.05` amber, `0.035` teal — verify these are not increased).

### R3 — Hero headline exemption

The `<h1>` element (`.hero-headline`) remains full-opacity with `text-shadow` for separation from the
ambient gradient. Its opacity must be **1.0** (i.e., no `/85` modifier on the headline itself).

### R4 — No API contract changes

All four API endpoints consumed by the homepage remain unchanged:

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/ideas` | Idea list + summary |
| GET | `/api/ideas/resonance?window_hours=72&limit=3` | Recent activity feed |
| GET | `/api/coherence/score` | Live coherence stat |
| GET | `/api/federation/nodes` | Node count badge |

### R5 — Regression guard

A pytest-based static analysis test at `api/tests/test_ui_readability.py` must:

1. Parse `web/app/page.tsx` and assert no prose element uses `text-foreground/80` or lower.
2. Parse `web/app/globals.css` and assert `--foreground` lightness stays ≥ 90%.
3. Assert the `body::before` alpha values do not exceed the maximum thresholds from R2.
4. Call `GET /api/coherence/score` and assert `0.0 ≤ score ≤ 1.0`.

## Open Questions — Decisions

### OQ1: Should we add a light mode toggle or just fix the dark mode contrast?

**Decision**: Fix dark-mode contrast only in this spec.

**Rationale**: The Coherence Network brand identity is intentionally warm-dark. A light mode toggle
requires a complete second color palette, system-preference detection (`prefers-color-scheme`),
persistence (localStorage or cookie), and a UI control in the header — estimated 3–5× the work of
the contrast fix. The contrast problem is a **P0 readability blocker**; light mode is a **P2 polish
feature**. Fix the blocker first; file a light-mode spec as a follow-up.

**Follow-up spec trigger**: If user research shows ≥ 20% of visitors prefer light mode or if a major
partner integration requires it, open a dedicated spec: `specs/xxx-light-mode-toggle.md`.

### OQ2: How can we improve this idea, show whether it is working, and make that proof clearer over time?

**Decision**: Three-layer proof strategy:

**Layer 1 — Static test (immediate, in CI)**

`api/tests/test_ui_readability.py` provides a machine-checkable contract. Every PR that touches
homepage CSS or JSX runs these tests. If opacity regresses, CI fails. This is the primary proof that
the fix exists and holds.

**Layer 2 — Live homepage stats (continuous, no extra work)**

The hero section already shows live stats pulled from the API: idea count, value created, node count,
and coherence score. These prove the system is running and the data pipeline feeds the UI. A reader
who can actually read these numbers is proof the readability fix works.

**Layer 3 — Contrast audit comment in code (persistent documentation)**

Add a single comment block in `web/app/globals.css` and `web/app/page.tsx` marking the opacity floor
decision with the spec ID (`task_f4a9da594102d8e0`). Future developers can grep for the spec ID to
understand why the floor exists. Example:

```css
/* [task_f4a9da594102d8e0] Body prose floor: opacity >= 0.85. Do not lower. */
```

**Making proof clearer over time** (future follow-ups, not in this spec):

- Add a `prefers-contrast: more` media query block that bumps all `text-foreground/*` to full
  opacity for users who require it (accessibility best practice, low effort).
- Add a Playwright/Chromium visual regression snapshot of the hero so future PRs show a diff.
- Track bounce rate on `/` in analytics — a meaningful drop after the contrast fix would be
  evidence of improved UX.

## Verification Scenarios

The reviewer will run these scenarios against production (`https://coherencycoin.com` /
`https://api.coherencycoin.com`) or a local stack (`http://localhost:3000` + `http://localhost:8000`).

### Scenario 1 — Full idea create → read → update cycle (proves API is live and homepage data flows)

**Setup**: API is reachable at `https://api.coherencycoin.com`. Use env var `API=https://api.coherencycoin.com`.

**Action**:

```bash
# Step 1: Create a test idea
curl -sS -w "\nHTTP %{http_code}\n" -X POST "$API/api/ideas" \
  -H "Content-Type: application/json" \
  -d '{"id":"contrast-verify-f4a9","name":"Contrast spec verification","description":"Spec task_f4a9da594102d8e0 test","potential_value":5,"estimated_cost":1,"confidence":0.8,"manifestation_status":"none"}'

# Step 2: Read it back
curl -sS -w "\nHTTP %{http_code}\n" "$API/api/ideas/contrast-verify-f4a9"

# Step 3: Update manifestation status (requires API key)
curl -sS -w "\nHTTP %{http_code}\n" -X PATCH "$API/api/ideas/contrast-verify-f4a9" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $COHERENCE_API_KEY" \
  -d '{"manifestation_status":"partial"}'

# Step 4: Verify list contains the idea
curl -sS "$API/api/ideas" | python3 -c "import sys,json; d=json.load(sys.stdin); ids=[i['id'] for i in d.get('ideas',[])]; print('FOUND' if 'contrast-verify-f4a9' in ids else 'MISSING')"
```

**Expected**:

- Step 1: HTTP 201 with JSON containing `"id":"contrast-verify-f4a9"`. If 409 (already exists from a
  prior run), Steps 2–4 must still succeed.
- Step 2: HTTP 200 with `"id":"contrast-verify-f4a9"` and matching `name`.
- Step 3: HTTP 200 with `"manifestation_status":"partial"`.
- Step 4: prints `FOUND`.

**Edge cases**:
- POST same `id` again → HTTP **409** (not 500, not 200)
- PATCH without `X-API-Key` → HTTP **401**
- GET `/api/ideas/nonexistent-xyz-f4a9` → HTTP **404** (not 500)

### Scenario 2 — Coherence score endpoint (displayed in hero stats)

**Setup**: None required.

**Action**:

```bash
curl -sS -w "\nHTTP %{http_code}\n" "https://api.coherencycoin.com/api/coherence/score"
```

**Expected**: HTTP **200**, JSON with:
- `"score"` — float in `[0.0, 1.0]`
- `"signals_with_data"` — non-negative integer
- `"total_signals"` — positive integer
- `"computed_at"` — ISO 8601 UTC string

**Edge cases**:
- If API is in maintenance and returns 503, the homepage must still render (client code tolerates null coherence).
- Invalid query parameters → 422 (not 500).

### Scenario 4 — Resonance feed error handling (homepage fallback path)

**Setup**: None required.

**Action**:

```bash
# Normal request
curl -sS -o /dev/null -w "%{http_code}" \
  "https://api.coherencycoin.com/api/ideas/resonance?window_hours=72&limit=3"

# Invalid parameter
curl -sS -o /dev/null -w "%{http_code}" \
  "https://api.coherencycoin.com/api/ideas/resonance?window_hours=-999&limit=3"
```

**Expected**:
- Normal: HTTP **200** with JSON array (may be empty — homepage handles empty by showing top ideas).
- Invalid `window_hours`: HTTP **422** with validation error body (not 500).

**Edge cases**:
- Empty array response → homepage shows `topIdeas` fallback section (not a blank area).
- Timeout: `fetchJsonOrNull` 5 000 ms timeout → page renders without the feed section.

---

### Scenario 5 — Visual browser spot-check (homepage readability confirmation)

**Setup**: Open `https://coherencycoin.com/` in a modern browser (Chromium or Safari). No browser
extensions that alter contrast/color.

**Action**: Without zooming or adjusting display settings:

1. Read the hero sub-paragraph ("A pattern you noticed...") aloud. Can you read it clearly?
2. Read the three pulse stats (ideas, value, nodes) labels.
3. Read the three "How it works" step descriptions.
4. Read the footer tagline and navigation links.
5. Read the idea submission textarea placeholder.

**Expected**: All five items are legible without straining. No element requires hovering to reveal
full-opacity text. The amber headline ("What idea are you holding?") remains visually prominent.

**Edge cases**:
- Headings must still feel warm/ambient, not clinical — if the fix over-brightens to pure white,
  aesthetic intent is broken (a new spec concern, not a readability regression).
- At 1x display scaling on a 1080p monitor, text must pass a simple "arm's-length readability" test.

## Risks and Assumptions

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| CDN caching masks the fix for returning visitors | Medium | Cloudflare cache purge on deploy; version-bust CSS asset |
| Brightening tokens affects non-homepage pages | Low | Test with `npm run build` + visual check of `/ideas`, `/resonance` |
| Over-brightening kills warm brand feel | Low | Use `/85`–`/90` range, not full white; keep amber accents at 100% |
| CI grep misses dynamically-constructed class names | Low | Complementary visual check in Scenario 5 |
