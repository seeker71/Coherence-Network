# Spec: UX Overhaul — Tabs, Mobile-First, Novice/Expert Modes

**Spec ID**: `task_5cb9d9ff27681827`
**Idea**: UX overhaul — tabs not scrolling, mobile-first, novice and expert modes
**Status**: draft
**Date**: 2026-03-28
**Author**: product-manager agent

---

## Summary

Coherence Network's web UI is currently a long-scroll experience with minimal mobile
support and no differentiation between novice contributors exploring ideas and expert
contributors who need raw JSON, UUIDs, and API links. This spec defines the complete
UX overhaul:

1. **Tab-based navigation** replaces long-scroll pages — every detail view becomes a
   set of named tabs so users see only what they need.
2. **Mobile-first layout** — bottom tab bar, swipeable cards, collapsible sections,
   responsive breakpoints.
3. **Expert / Novice modes** — expert shows IDs, JSON toggles, and API links; novice
   hides technical fields and shows guided tooltips with plain vocabulary.
4. **Persistent UI preferences** — stored per contributor via
   `GET/PUT/PATCH/DELETE /api/preferences/ui`.

The spec verifies that: (a) the API contract is respected, (b) the web pages render
correctly in both modes, and (c) the preference changes persist across page reloads.

---

## Purpose

Long-scroll pages lose context as content grows. Mobile users have no bottom navigation.
New contributors abandon ideas because they see raw UUIDs and JSON where they expect
plain language. Power users want direct API links to build integrations. A single
preference store with well-defined defaults lets both personas co-exist without
separate codepaths.

## Requirements

### R1 — Idea Detail: Tab Layout

- [ ] The idea detail page (`/ideas/[idea_id]`) MUST render a shadcn/ui `<Tabs>`
  component with exactly six tabs: **Overview | Specs | Tasks | Contributors | Edges | History**.
- [ ] Each tab MUST load only its own data (lazy or conditional fetch) — no full
  page re-render on tab switch.
- [ ] The active tab MUST be reflected in the URL query param `?tab=<name>` so deep
  links and browser-back work correctly.
- [ ] On mobile (viewport < 640 px) the tab labels MUST scroll horizontally without
  wrapping; long-scroll content within each tab MUST use collapsible sections.
- [ ] The last active tab MUST be persisted via `PATCH /api/preferences/ui` with
  `idea_detail_tab=<name>` and restored on next visit (when a `contributor_id` is
  available from session).

### R2 — Ideas List: View Modes

- [ ] The ideas list page (`/ideas`) MUST offer three view modes toggled via a view-mode
  switcher: **Cards | Table | Graph**.
- [ ] The selected view mode MUST be persisted in `ideas_view` via
  `PATCH /api/preferences/ui`.
- [ ] **Cards**: responsive card grid, swipeable on touch devices (horizontal swipe
  changes "focus" card, vertical scroll moves through the list).
- [ ] **Table**: sortable columns — Name, Status, Priority, CC Score, Last Updated;
  horizontal scroll on narrow viewports.
- [ ] **Graph**: reuses the existing `/graph` canvas (or embeds it as an iframe/widget);
  labelled nodes for ideas linked to the query result set.
- [ ] The active view mode is reflected in URL query param `?view=<mode>`.

### R3 — Primary Navigation

- [ ] The site header (`site_header.tsx`) MUST show primary tabs:
  **Ideas | Concepts | Contributors | News | Tasks**.
- [ ] Secondary items (Graph, Blog, Automation, Onboarding, etc.) MUST move into a
  dropdown/overflow menu (labelled "More") accessible from the primary nav.
- [ ] On mobile (< 640 px) the primary tabs MUST render as a **bottom tab bar** — a
  sticky `<nav>` at the bottom of the viewport with icon + label.
- [ ] The bottom tab bar is controlled by `nav_layout: "bottom_bar"` in UI preferences;
  desktop defaults to `nav_layout: "top"`.

### R4 — Mobile Layout

- [ ] Breakpoint strategy: mobile < 640 px, tablet 640–1024 px, desktop > 1024 px.
- [ ] `swipeable_cards: true` enables horizontal swipe for idea cards on touch devices;
  defaults to `false` (opt-in to avoid accidental swipes on non-touch desktop).
- [ ] `collapsible_sections: true` wraps content sections > 200 px tall in a
  collapsible (`<details>` or shadcn/ui Collapsible) to reduce initial scroll depth.
- [ ] All interactive elements (buttons, tabs, toggles) MUST meet WCAG 2.1 touch target
  minimum of 44 × 44 px on mobile.

### R5 — Expert Mode

- [ ] When `expert_mode: true`:
  - Show UUID / system IDs below human-readable names (dimmed, monospace).
  - Show a **JSON** toggle on idea detail, concept detail, and task detail pages; toggle
    reveals the raw API response in a syntax-highlighted `<pre>` block.
  - Show **API link** (e.g. `GET /api/ideas/<id>`) as a clickable badge on each resource.
  - Show `coherence_score` and `priority_score` numeric values (not just the humanised
    label).
- [ ] When `expert_mode: false` (novice/default):
  - UUIDs and system IDs are hidden.
  - JSON toggle is absent.
  - API links are absent.
  - `coherence_score` shows only the humanised label (e.g. "High").
  - Guided tooltips (`show_tooltips: true`) appear on hover/focus for jargon terms
    (e.g. "Coherence Score", "CC", "Edge").

### R6 — API: `/api/preferences/ui`

- [ ] `GET /api/preferences/ui?contributor_id=<id>` — returns full `UIPreferences`
  object; returns defaults for unknown contributor (HTTP 200, never 404).
- [ ] `PUT /api/preferences/ui` — upsert full preferences (body: `UIPreferences`).
- [ ] `PATCH /api/preferences/ui?contributor_id=<id>` — partial update; unknown fields
  ignored, invalid enum values return HTTP 422 with detail.
- [ ] `DELETE /api/preferences/ui?contributor_id=<id>` — reset to defaults (HTTP 204).
- [ ] The router MUST be registered in `api/app/main.py`; it is currently omitted.
- [ ] `UIPreferences.expert_mode=true` MUST set `show_tooltips=false` automatically
  (server-side enforcement on PUT/PATCH).

### R7 — Proof of Progress

- [ ] A `/api/preferences/ui/audit` endpoint (GET) returns a count of contributors who
  have preferences stored, broken down by `expert_mode` and `nav_layout` values; this
  lets us see adoption over time.
- [ ] The ideas list page logs a `view_mode_change` event to the existing analytics/
  audit log when the user switches view mode (Cards ↔ Table ↔ Graph).

## API Contract

### `GET /api/preferences/ui`

**Request**
- `contributor_id`: string (query, default `"anonymous"`)

**Response 200**
```json
{
  "contributor_id": "user-abc",
  "expert_mode": false,
  "nav_layout": "top",
  "ideas_view": "cards",
  "idea_detail_tab": "overview",
  "swipeable_cards": false,
  "collapsible_sections": true,
  "show_tooltips": true
}
```

**Behaviour**: always 200; returns defaults for unknown contributor.

---

### `PUT /api/preferences/ui`

**Request body** (full `UIPreferences`)
```json
{
  "contributor_id": "user-abc",
  "expert_mode": true,
  "nav_layout": "bottom_bar",
  "ideas_view": "table",
  "idea_detail_tab": "tasks",
  "swipeable_cards": true,
  "collapsible_sections": false,
  "show_tooltips": true
}
```

**Response 200** — returns the stored object.
**Server-side rule**: if `expert_mode=true` and `show_tooltips=true` in request body,
the stored and returned value of `show_tooltips` MUST be `false`.

**Response 422** — if `contributor_id` is blank/whitespace.

---

### `PATCH /api/preferences/ui`

**Request**
- `contributor_id`: string (query, required)
- Body: partial `UIPreferencesUpdate` (all fields optional)

```json
{ "ideas_view": "table" }
```

**Response 200** — returns merged full preferences.
**Response 422** — if `contributor_id` is blank, or if an enum field has an invalid
value (e.g. `"ideas_view": "kanban"` is not in `{cards, table, graph}`).

---

### `DELETE /api/preferences/ui`

**Request**
- `contributor_id`: string (query, default `"anonymous"`)

**Response 204** — no body. Subsequent GET returns defaults.

---

### `GET /api/preferences/ui/audit`

**Response 200**
```json
{
  "total_contributors": 42,
  "expert_mode_enabled": 7,
  "nav_layout_counts": { "top": 38, "bottom_bar": 4 },
  "ideas_view_counts": { "cards": 30, "table": 8, "graph": 4 }
}
```

## Data Model

```yaml
UIPreferences:
  properties:
    contributor_id:
      type: string
      required: true
      description: >
        Contributor identifier; "anonymous" is the default for unauthenticated
        sessions. In a DB-backed implementation this would be a FK to contributors.

    expert_mode:
      type: boolean
      default: false
      description: >
        When true: show UUIDs, raw JSON toggle, API links, numeric scores.
        When false (novice): hide technical fields, show tooltips, use plain vocabulary.

    nav_layout:
      type: enum
      values: [top, bottom_bar]
      default: top
      description: >
        top = horizontal header tabs (desktop default).
        bottom_bar = sticky bottom navigation (mobile default).

    ideas_view:
      type: enum
      values: [cards, table, graph]
      default: cards
      description: View mode for /ideas list page.

    idea_detail_tab:
      type: enum
      values: [overview, specs, tasks, contributors, edges, history]
      default: overview
      description: Last-active tab on idea detail page; restored on next visit.

    swipeable_cards:
      type: boolean
      default: false
      description: Enable horizontal swipe gesture on idea cards on touch devices.

    collapsible_sections:
      type: boolean
      default: true
      description: Wrap tall sections in collapsible accordion on mobile.

    show_tooltips:
      type: boolean
      default: true
      description: >
        Show guided tooltips for jargon terms.
        Automatically forced to false when expert_mode=true.

UIPreferencesUpdate:
  description: All fields optional; omitted fields are unchanged.
  properties: (same as UIPreferences minus contributor_id)
```

## Verification Scenarios

> These scenarios are the reviewer's contract. Each one must be runnable against
> production (`$API = https://api.coherencycoin.com`).

### Scenario 1 — Full Preferences CRUD cycle

**Setup**: No preferences stored for `contributor_id=ux-test-01`.

**Action 1 — GET defaults**:
```bash
curl -s "$API/api/preferences/ui?contributor_id=ux-test-01"
```
**Expected**: HTTP 200, body contains
`{"contributor_id":"ux-test-01","expert_mode":false,"nav_layout":"top","ideas_view":"cards","idea_detail_tab":"overview","show_tooltips":true}`.

**Action 2 — PUT upsert**:
```bash
curl -s -X PUT "$API/api/preferences/ui" \
  -H "Content-Type: application/json" \
  -d '{"contributor_id":"ux-test-01","expert_mode":false,"nav_layout":"bottom_bar","ideas_view":"table","idea_detail_tab":"tasks","swipeable_cards":true,"collapsible_sections":false,"show_tooltips":true}'
```
**Expected**: HTTP 200, `nav_layout="bottom_bar"`, `ideas_view="table"`.

**Action 3 — GET persisted**:
```bash
curl -s "$API/api/preferences/ui?contributor_id=ux-test-01"
```
**Expected**: HTTP 200, `ideas_view="table"`, `nav_layout="bottom_bar"` (persisted).

**Action 4 — PATCH partial**:
```bash
curl -s -X PATCH "$API/api/preferences/ui?contributor_id=ux-test-01" \
  -H "Content-Type: application/json" \
  -d '{"ideas_view":"graph"}'
```
**Expected**: HTTP 200, `ideas_view="graph"`, all other fields unchanged from Action 2.

**Action 5 — DELETE reset**:
```bash
curl -s -X DELETE "$API/api/preferences/ui?contributor_id=ux-test-01"
```
**Expected**: HTTP 204, no body.

**Action 6 — GET after delete**:
```bash
curl -s "$API/api/preferences/ui?contributor_id=ux-test-01"
```
**Expected**: HTTP 200, defaults (`ideas_view="cards"`, `nav_layout="top"`).

## Risks and Assumptions

| Risk | Severity | Mitigation |
|------|----------|------------|
| In-memory `_STORE` is lost on API restart | Medium | Acceptable for MVP; annotate with TODO for DB migration (PostgreSQL `contributor_preferences` table). |
| `contributor_id` is unauthenticated — any caller can read/write any user's prefs | High | Current design is intentional for anonymous contributors; full auth integration is a follow-up. Add `SECURITY NOTE` comment in router. |
| shadcn/ui Tabs not installed | Medium | Install command: `npx shadcn@latest add tabs collapsible tooltip`. CI must fail loudly if component is missing. |
| Bottom tab bar duplicates primary nav on large screens | Low | CSS media query: `display: none` on `>= 640px`; `site_header.tsx` primary nav hidden on `< 640px`. |
| Swipeable cards interfering with page scroll on touch | Medium | Use `touch-action: pan-y` on card list container; horizontal swipe only activates if gesture angle < 30° from horizontal. |
| URL `?tab=` param conflicts with existing query params | Low | Use `URLSearchParams` merge (not replace) when setting `?tab=`; existing params are preserved. |
| Graph view mode requires a loaded graph canvas — slow first paint | Medium | Show skeleton/spinner; lazy-load graph component; fallback to Cards if graph data endpoint times out. |

## Known Gaps and Follow-up Tasks

1. **DB persistence for preferences** — replace `_STORE` dict with PostgreSQL
   `contributor_preferences` table; add Alembic migration.
2. **Authentication-scoped preferences** — tie `contributor_id` to session/JWT so
   preferences roam across devices.
3. **Graph view on `/ideas`** — requires the graph canvas component to support
   idea-filtered node sets (currently renders the full graph).
4. **Swipe gesture library** — evaluate `react-swipeable` vs native pointer events;
   ensure a11y keyboard navigation parity.
5. **Tooltip content authoring** — per-term tooltip strings need a content registry
   (e.g. `web/lib/glossary.ts`) so they can be updated without component edits.
6. **Analytics events** — `view_mode_change` event spec needs a matching event schema
   in the audit log model.
7. **Dark mode compatibility** — expert mode JSON block and UUID labels need dark-mode
   colour tokens; add to Tailwind config.
8. **Playwright CI** — web tests must run in CI against a local dev server; add to
   `.github/workflows/web-ci.yml`.

## Open Questions — Resolved

### How can we improve this idea, show whether it is working, and make that proof clearer over time?

**Measurable adoption signals** (built into this spec):

| Signal | How to measure |
|--------|---------------|
| Preference store adoption | `GET /api/preferences/ui/audit` — `total_contributors` growing over time proves users are hitting preferences; `expert_mode_enabled` count proves expert mode is being used. |
| View mode preference | `ideas_view_counts` in audit endpoint shows which view mode contributors prefer; a growing `table` or `graph` count means users discovered alternatives to the default card view. |
| Tab navigation depth | `idea_detail_tab` distribution in audit data shows which tabs users reach; if `history` is always 0, that tab may be unnecessary or undiscoverable. |
| URL deep-links | `?tab=` and `?view=` params in server access logs (Traefik/Cloudflare) indicate whether contributors bookmark or share specific views. |

**Proof that it's working** (not just deployed):

- Run Scenario 1–4 against production after merge: all must pass.
- After 48 h of live traffic, run `GET /api/preferences/ui/audit` and confirm
  `total_contributors > 0` (real users saved preferences).
- Screenshot `/ideas/<known_id>?tab=tasks` in mobile viewport (375 px) confirming
  bottom tab bar and no horizontal overflow.
- Screenshot same page in expert mode confirming UUID and JSON toggle visibility.

**How to make proof clearer over time**:

- Add a `/api/preferences/ui/audit` badge to the project dashboard showing
  "N contributors using expert mode".
- Include `idea_detail_tab` distribution in the monthly contributor digest so the team
  can see which features drive engagement.
- Gate the "Graph" view behind a feature flag initially and log activation rate.
