# Spec: Geolocation awareness — nearby contributors, local ideas, regional news

**Spec ID / task:** `task_6313cc742c4d9739`  
**Related:** `specs/e165294430dc487b.md` (same feature line; this file is the deliverable for task `task_6313cc742c4d9739`)

## Summary

Coherence Network should treat **geography** as a discoverability and trust signal: contributors may share **coarse** location on profiles; the platform resolves human-readable places to coordinates using a **GeocodingService** borrowed from the Living Codex pattern (**OpenCage** primary, **Nominatim** secondary, explicit fallback and caching); clients filter **ideas**, **news**, and **agent tasks** by proximity; and the **web** surfaces nearby collaborators. Baseline APIs already implement city-level storage, haversine **nearby** search, and **keyword-based** local news resonance. This specification defines the **target contract**, **data model extensions**, **verification scenarios** runnable against production, and **observability** so operators can prove the feature works and improves over time.

## Purpose

- **Who benefits:** Contributors seeking local collaboration; readers of regional news; operators validating regional engagement.
- **Costs avoided:** Opaque or broken location features (mitigated by health endpoints, metrics, and explicit provider behavior).

## Living Codex borrow: GeocodingService

**Intent:** Reuse the architectural pattern from Living Codex: a thin adapter that:

1. **Primary:** Calls **OpenCage** when `OPENCAGE_API_KEY` is present, using their forward-geocoding API with structured responses.
2. **Secondary:** On rate limit, HTTP error, or missing key, falls back to **Nominatim** (`https://nominatim.openstreetmap.org`) with a **static User-Agent** identifying the Coherence Network deployment, honoring [Nominatim Usage Policy](https://operations.osmfoundation.org/policies/nominatim/) (no bulk unthrottled use; cache aggressively).
3. **Fallback:** If both fail, return a **typed error** (no silent wrong-city substitution).
4. **Output shape (conceptual):** `{ latitude, longitude, display_label, source: opencage|nominatim|cache, confidence?: float }`, normalized for consumers.

Implementation may be a **new** module `api/app/services/geocoding_service.py` (not a symlink into reference repos — references are read-only per project rules). Contract tests prove parity with the behavioral expectations above.

## Requirements

### Functional

- [ ] **R1 — GeocodingService:** OpenCage → Nominatim → error; normalized query cache (e.g. 24h TTL); no full street addresses stored (existing two-decimal rounding ~1 km).
- [ ] **R2 — Profile enrichment:** Contributor read/portfolio responses expose optional **city/region/country** when visibility allows; never leak precision beyond stored policy.
- [ ] **R3 — Ideas:** List or filter ideas by contributor geography via `near_lat`, `near_lon`, `radius_km` and/or `location_query` (resolved via GeocodingService).
- [ ] **R4 — News:** Keep `.../local`; add optional proximity path when article metadata supports it.
- [ ] **R5 — Tasks:** Agent task listing supports the same proximity parameters when assignee/creator contributor has eligible location.
- [ ] **R6 — Web:** `/explore/nearby` (or equivalent route under `web/app/`) uses browser geolocation (opt-in) or manual place search and calls `GET /api/nearby`.

### Non-functional

- [ ] **N1 — Privacy:** `visibility=private` excludes from public/nearby aggregates.
- [ ] **N2 — Rate limits:** Server-side throttling + cache on geocode queries.
- [ ] **N3 — Observability:** Health + metrics (see below).

## Data model

### Geocode result (new)

```yaml
GeocodeResult:
  query_normalized: string
  latitude: float
  longitude: float
  label: string
  source: enum [opencage, nominatim, cache]
  confidence: float  # 0.0–1.0 when provider supplies
```

### Contributor

Graph node `properties.geo_location` remains source of truth (`_GEO_KEY` in `geolocation_service.py`). API DTOs may add `location_summary` for lists.

### Cache entry

- Key: hash of normalized query + schema version
- Value: `GeocodeResult` + `cached_at` (ISO 8601 UTC)

## Verification Scenarios

Each scenario is intended to be run by a reviewer against **production** (`https://api.coherencycoin.com`) or staging with `$API` set.

### Scenario 1 — Full location create-read-update-delete cycle

- **Setup:** API running. Contributor `contrib-geo-test` exists (create via existing `POST /api/contributors` or graph flow if required). No stored location initially.
- **Action:**  
  `curl -sS -X PATCH "$API/api/contributors/contrib-geo-test/location" -H "Content-Type: application/json" -d '{"city":"Boulder","region":"CO","country":"US","latitude":40.02,"longitude":-105.27,"visibility":"public"}'`  
  Then: `curl -sS "$API/api/contributors/contrib-geo-test/location"`  
  Then: `curl -sS -X DELETE "$API/api/contributors/contrib-geo-test/location"`
- **Expected:** PATCH returns **200** with lat/lon rounded to two decimals; GET returns **200** with matching city/country and `visibility` `public`; DELETE returns **204**.
- **Edge:** PATCH with `latitude` **91** → **422** (validation). GET after DELETE → **404** with JSON `detail`, not **500**. POST duplicate location without PATCH — N/A (use PATCH for upsert).

### Scenario 2 — Nearby aggregate read (proves distance filtering)

- **Setup:** Two contributors: `near-prox` at ~(40.0, -105.0) and `far-prox` at ~(48.0, -122.0); at least one idea linked to `near-prox` per existing data model.
- **Action:** `curl -sS "$API/api/nearby?lat=40.015&lon=-105.28&radius_km=50&limit=20"`
- **Expected:** **200** JSON includes `near-prox` in `contributors` (or equivalent field names in `NearbyResult`); idea from `near-prox` appears in `ideas`; `far-prox` absent; distances consistent with haversine ordering where applicable.
- **Edge:** `lat=999` → **422**; `radius_km=-1` → **422**.

### Scenario 3 — Forward geocode (after `POST /api/geocode` ships)

- **Setup:** Staging with `OPENCAGE_API_KEY` or Nominatim-only fallback allowed.
- **Action:** `curl -sS -X POST "$API/api/geocode" -H "Content-Type: application/json" -d '{"query":"Boulder, Colorado, USA"}'`
- **Expected:** **200** with `latitude`, `longitude`, `label`, `source` ∈ `{opencage,nominatim,cache}`.
- **Edge:** `{"query":""}` → **422**; unresolvable query → contract-chosen **404** or **200** with explicit `no_match` — implementation must document and test one behavior.

### Scenario 4 — Ideas proximity filter (after implementation)

- **Setup:** As Scenario 2; idea tied to nearby contributor.
- **Action:** `curl -sS "$API/api/ideas?near_lat=40.015&near_lon=-105.28&radius_km=100"`
- **Expected:** Response lists include the nearby idea per filter contract (sorted or filtered — as implemented and documented).
- **Edge:** `near_lat` without `near_lon` → **422**.

### Scenario 5 — Health and provider failure behavior

- **Setup:** Environment with OpenCage key absent; Nominatim reachable.
- **Action:** `curl -sS "$API/api/health/geolocation"`
- **Expected:** **200** JSON: `opencage_configured: false`; no API keys in body; `nominatim_reachable` true or false with explicit reason string.
- **Edge:** Forced provider failure in test env — `POST /api/geocode` returns **503** (or documented error) with message, not **200** with zero coordinates.
