## Summary

Ship the **Coherence Network MCP server** as two first-class installable artifacts — **npm** (`coherence-mcp-server`) and **PyPI** (`coherence-mcp-server`) — with a **canonical `server.json`** manifest that satisfies the [Model Context Protocol registry schema](https://registry.modelcontextprotocol.io/schemas/server.json) and tooling such as **mcp-publisher** (and downstream listings: **Official MCP Registry**, **Smithery**, **Glama**, **MCP.so**, etc.). Today the repo already contains `mcp-server/package.json`, `mcp-server/server.json`, `mcp-server/pyproject.toml`, and `registry_discovery_service` checks; this spec locks **version alignment**, **publish automation**, **verification commands**, and **ongoing proof** so paid implementation work has a clear contract.

---

## Purpose

- **Users**: developers and agents who configure MCP clients need a **one-line install** from npm *or* PyPI without cloning the monorepo.
- **Discovery**: registries and search engines index **published package names + versions + manifest** — not raw Git paths.
- **Operators**: need **machine-checkable** evidence that packages exist, versions match, and **registry readiness** stays green in CI and production checks.

**Definition of “working” for this spec**: both registries return the expected package; local `npx`/`pipx` runs the server; `registry-submissions` shows MCP targets ready; manifest validates against the official JSON Schema.

## Requirements

- [ ] **`mcp-server/server.json`** remains the **single canonical MCP server manifest** for the Official MCP Registry pipeline, with `$schema` pointing at `https://registry.modelcontextprotocol.io/schemas/server.json`.
- [ ] **`packages` array** lists **both** install surfaces:
  - `registry: "npm"`, `name: "coherence-mcp-server"`, `version` aligned with `mcp-server/package.json`.
  - `registry: "pypi"`, `name: "coherence-mcp-server"`, `version` aligned with `mcp-server/pyproject.toml` `[project].version`.
- [ ] **Semantic versioning**: `server.json` `version`, npm `version`, and PyPI `version` are **equal** at release time (no silent drift like npm `0.3.1` vs server.json `0.2.0`).

- [ ] **PyPI package**: `mcp-server/pyproject.toml` defines `project.scripts` → `coherence-mcp-server`; wheel/sdist publish produces `pip install coherence-mcp-server` / `pipx run coherence-mcp-server` entry point that runs the same logical server (stdio MCP).
- [ ] **mcp-publisher / registry bots**: repository includes **documented** steps (Makefile target or `docs/` section referenced from `mcp-server/README.md`) to validate `server.json` and run publisher-compatible checks where applicable — exact CLI flags pinned in **Verification** below.
- [ ] **Discovery API contract** (no new endpoints required for MVP if existing endpoint suffices): **`GET /api/discovery/registry-submissions`** MUST return `submission_ready` for at least `modelcontextprotocol-registry`, `npm`, and `smithery` when manifests and files on disk pass validators in [`registry_discovery_service.py`](../api/app/services/registry_discovery_service.py).
- [ ] **Release automation** (implementation detail): CI or documented script performs **npm publish** and **PyPI publish** from `mcp-server/` (or monorepo root with `working-directory`), with **trusted publishing** or token-based auth per org policy — *scope in impl task, acceptance via published versions + verification commands*.

## Data Model

### `mcp-server/server.json` (canonical manifest)

- Must include: `name`, `version`, `description`, `repository`, `homepage`, `license`, `packages[]` with **npm + pypi** entries, `tools` (or reference to generated list), `$schema`.
- **Version sync rule**: `version` (top-level) equals the released semver for both packages in `packages[]`.

### `mcp-server/package.json`

- `name`: `coherence-mcp-server`
- `bin`: `coherence-mcp-server` → `index.mjs`
- `version`: must match `server.json` for releases.

### `mcp-server/pyproject.toml`

- `[project].name`: `coherence-mcp-server`
- `[project].version`: must match `server.json` for releases.
- `[project.scripts]`: `coherence-mcp-server` entry point implemented under `src/coherence_mcp_server/`.

### PyPI package layout (existing)

- Hatchling build; `src/coherence_mcp_server` package — implementation must not break this layout when publishing.

## Acceptance Criteria (product)

1. **Dual publish**: A tagged release produces **npm** and **PyPI** artifacts with the **same** semver and that semver is reflected in `server.json`.
2. **Manifest completeness**: `server.json` validates against the official MCP registry schema and lists npm + PyPI packages.
3. **Runtime**: `npx --yes coherence-mcp-server@<version>` and `pipx run coherence-mcp-server==<version>` (or equivalent) start the MCP server (stdio) with documented `COHERENCE_API_URL` / auth env vars.
4. **Readiness API**: `GET /api/discovery/registry-submissions` shows MCP-related targets as ready when validators pass.
5. **No orphan docs**: Root `README.md` or `mcp-server/README.md` links to npm and PyPI package pages.

## Verification Scenarios

### Scenario 1 — Full read path: published packages match manifest (production)

**Setup**: Release `vX.Y.Z` has been published to npm and PyPI; public API is up.

**Action**:

```bash
npm view coherence-mcp-server version --json
curl -sS https://pypi.org/pypi/coherence-mcp-server/json | jq -r .info.version
curl -sS "$API_URL/api/discovery/registry-submissions" | jq '.items[] | select(.registry_id=="npm" or .registry_id=="modelcontextprotocol-registry")'
```

**Expected**:

- npm JSON `version` equals PyPI `.info.version` and equals `mcp-server/server.json` `version` on tag `vX.Y.Z`.
- Registry submission items show `status` consistent with `submission_ready` for completed work (not `blocked` by missing files).

**Edge**: If PyPI package was never published, PyPI URL returns 404 — scenario **fails** until publish; document as pre-release gate.

---

### Scenario 2 — Create / update / read cycle: install and run (local or CI)

**Setup**: Clean environment with Node ≥18 and Python ≥3.10; `COHERENCE_API_URL=https://api.coherencycoin.com` (or mock) set.

**Action**:

```bash
cd mcp-server && npm ci && node --check index.mjs
npx --yes coherence-mcp-server@latest --help 2>/dev/null || true   # if CLI exposes help; else smoke: timeout 2 npx ...
pipx run coherence-mcp-server --help 2>/dev/null || python -m coherence_mcp_server 2>&1 | head -5
```

**Expected**: Process starts without import/syntax errors; MCP stdio server initializes (implementation may only log — **acceptance**: exit code 0 for `--version` or documented smoke).

**Edge**: **Bad input** — missing `COHERENCE_API_URL`: server exits with **non-zero** exit code and **stderr** message containing `COHERENCE` or `API` (documented string), not an unhandled stack trace.

---

### Scenario 3 — Error handling: manifest / schema failure

**Setup**: Branch with **invalid** `server.json` (remove `$schema` or break `packages`).

**Action**:

```bash
python3 -c "import json,sys; json.load(open('mcp-server/server.json'))"
# Plus optional: npx @modelcontextprotocol/inspector or registry validator if pinned in repo
```

**Expected**: CI or pre-publish script **fails**; `GET /api/discovery/registry-submissions` shows `modelcontextprotocol-registry` **not** `submission_ready`.

**Edge**: **Duplicate release** — republishing same version to npm/PyPI returns **409/400** from registry; release docs instruct bumping semver.

---

### Scenario 4 — Registry discovery service alignment (API proof)

**Setup**: Local API with repo mounted; no DB required for static file checks if service only reads disk.

**Action**:

```bash
curl -sS "http://localhost:8000/api/discovery/registry-submissions"
```

**Expected**: Response JSON includes `summary` and `items`; items for `registry_id` in `smithery`, `modelcontextprotocol-registry`, `npm` include `install_hint` containing `npx coherence-mcp-server` where applicable; validators pass when files are correct.

**Edge**: If `mcp-server/server.json` is missing — relevant targets **blocked** with actionable `notes` / detail fields (per existing model).

---

### Scenario 5 — Web / human proof (optional but recommended)

**Setup**: Browser or `curl`.

**Action**: Open `https://www.npmjs.com/package/coherence-mcp-server` and `https://pypi.org/project/coherence-mcp-server/`.

**Expected**: Pages exist; latest version matches GitHub release tag; README sections visible.

**Edge**: Private npm org — if package is public, pages must not 404; if scoped (`@scope/pkg`), URLs adjusted in spec appendix.

## Risks and Assumptions

- **Risk**: Version drift between `server.json`, npm, and PyPI causes registry rejection or user confusion. **Mitigation**: single release script bumps all three; CI fails on mismatch.
- **Risk**: PyPI name squatting if `coherence-mcp-server` is taken. **Mitigation**: verify PyPI name early; fallback name documented in follow-up.
- **Assumption**: Node and Python implementations remain feature-compatible for the same major version (shared tool catalog).
