# Spec - Consolidate `/automation`, `/usage`, `/remote-ops` into `/nodes` and `/pipeline`

**Task ID**: `task_0ac213ec7b80d275`  
**Author role**: product-manager  
**Date**: 2026-03-29  
**Status**: proposed  
**Related drafts**: `specs/task_588507d48375713f.md`, `specs/task_db97e8e02d420314.md`

## Summary

The current web surface splits operational information across `/automation`, `/usage`, `/remote-ops`, `/nodes`, and `/pipeline`. That overlap makes simple operator questions harder than they should be because users have to remember which page holds provider readiness, which page holds queue state, and which page contains remote control actions. This simplified retry keeps only the core requirement: define two canonical top-level destinations and treat the three older routes as legacy entry points.

The canonical model is simple:

- `/nodes` = infrastructure and control plane
- `/pipeline` = execution and usage plane

For this minimal scope, the requirement is not a full redesign or a widget-by-widget migration plan. The requirement is to make the route model unambiguous, move users toward the two canonical pages, and preserve the current backend contracts so implementation can stay small and low risk.

## Requirements

1. `/nodes` must be the canonical page for node and operator-control concerns. At minimum, it must remain the destination for federation or node visibility, provider readiness context, and remote-ops style control actions.
2. `/pipeline` must be the canonical page for execution concerns. At minimum, it must remain the destination for pipeline status, task activity, and usage or cost-oriented summaries.
3. The legacy routes must stop acting as first-class destinations and permanently redirect as follows:
   - `/automation` -> `/nodes`
   - `/remote-ops` -> `/nodes`
   - `/usage` -> `/pipeline`
4. Global navigation and shared page-context links must prefer `/nodes` and `/pipeline` and must no longer advertise `/automation`, `/usage`, or `/remote-ops` as primary destinations.
5. The implementation for this spec must stay minimal. It must reuse current data sources and page building blocks where possible, and it must not require a new information architecture beyond the two-page consolidation model.
6. If a legacy URL is requested with existing query parameters, the request must still land on the canonical destination instead of returning a broken or orphaned page. Query preservation is preferred but not required for this minimal pass.

## API Changes

No new API endpoints are required.

No existing backend route is renamed by this spec. The consolidation is a web-routing and page-ownership change that continues to rely on the current APIs, including the existing node, pipeline, and automation usage contracts. The implementation may continue using current endpoints such as:

- `GET /api/federation/nodes`
- `GET /api/federation/nodes/stats`
- `GET /api/automation/usage`
- `GET /api/automation/usage/readiness`
- `GET /api/agent/pipeline-status`
- `GET /api/agent/tasks`

If implementation later finds that a page needs additional aggregation or reshaping, that is a follow-up spec and not part of this minimal requirement.

## Data Model

No database, graph, or Pydantic model changes are required.

This spec does not introduce new persisted entities, new API payload contracts, or new schema fields. Existing frontend types may be reused or relocated as needed during implementation, but those refactors are outside the required product contract here.

## Verification Criteria

1. Requesting `/automation` returns a permanent redirect to `/nodes`.
2. Requesting `/remote-ops` returns a permanent redirect to `/nodes`.
3. Requesting `/usage` returns a permanent redirect to `/pipeline`.
4. `/nodes` still loads as the canonical infrastructure page and exposes the current node or control-plane concerns without requiring the legacy routes.
5. `/pipeline` still loads as the canonical execution page and exposes the current pipeline and usage-oriented concerns without requiring the legacy routes.
6. Shared navigation surfaces no longer present `/automation`, `/usage`, or `/remote-ops` as top-level destinations.

Suggested validation commands for implementation:

```bash
curl -sI https://coherencycoin.com/automation
curl -sI https://coherencycoin.com/remote-ops
curl -sI https://coherencycoin.com/usage
rg -n '"/automation"|"/usage"|"/remote-ops"' web/app web/components
```

## Risks

- The largest risk is over-scoping the work into a full UI redesign. This spec avoids that by limiting the requirement to canonical ownership, redirects, and navigation cleanup.
- Some sections currently shown on the legacy pages may still need careful placement on `/nodes` or `/pipeline`. The implementation should preserve the existing surfaces first and improve layout second.
- Bookmarked legacy URLs will continue to exist in the wild. Permanent redirects mitigate this, but incorrect destination mapping would create user confusion, so the redirect matrix must remain explicit and tested.
- `/remote-ops` could be argued as either control-plane or execution-plane content. This spec resolves that ambiguity by treating remote operations as a control-plane concern and routing it to `/nodes`.
