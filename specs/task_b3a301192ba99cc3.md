# Spec: SSE control channel — real-time steer, checkpoint, abort, ask-permission for native agent CLIs

**Spec ID / task:** `task_b3a301192ba99cc3`  
**Status:** Draft (product specification — implementation follows in a separate task card)  
**Related code (today):** `api/scripts/task_control_channel.py`, `api/scripts/local_runner.py` (integration), `api/app/routers/federation.py` (node SSE), `api/app/routers/task_activity_routes.py` (task activity + task SSE)

---

## Summary

Native provider CLIs (Claude Code, Codex, Cursor agent, Gemini CLI, etc.) are long-running and opaque: operators cannot see incremental intent, cannot redirect work mid-flight, and cannot stop gracefully without losing context. This specification defines an **SSE-backed control plane** that pairs every executing task with a **bidirectional bridge**:

1. **Inbound (network → agent):** While a task runs, the runner maintains an **SSE subscription** (per node) and forwards **control commands** into a **local command file** (JSONL) that the agent process is instructed to read on a short interval or after each tool round.
2. **Outbound (agent → network):** The agent (or a thin wrapper) **POSTs structured responses and heartbeats** to the API so dashboards, monitors, and humans see **live state**, not a five-minute black box.

Supported command verbs (normative minimum): **`checkpoint`**, **`steer`**, **`abort`**, **`ask`** (pause for human permission), **`report`** (status-only; may be echoed to activity without mutating task direction), and **`ping`** (liveness). Responses must be attributable to `task_id` and sequence where possible.

**Goal:** Every agent execution is **observable** (what it is doing, last heartbeat, last control ack) and **controllable** (steer/abort/checkpoint) in near real time — with explicit proof scenarios operators can run against production.

---

## Purpose

- **Operators** need to intervene when a task drifts, stalls, or needs approval without waiting for terminal failure.
- **The network** needs a single contract for “how a running task listens” so CLIs are interchangeable behind the same runner wrapper.
- **Audit and value lineage** improve when control events and responses are logged with timestamps and correlation IDs.

## Requirements

### Functional

- [ ] **R1 — SSE subscription on task start:** When the runner begins provider execution for a task, it starts a **background control channel** tied to `(node_id, task_id)` that connects to the node SSE stream and stays up until the provider process exits or the task is terminal.
- [ ] **R2 — Command file protocol:** Commands arriving from SSE are **appended** to `{task_work_dir}/.task-control` as **JSONL** (one JSON object per line). Initial line includes `type: connected` with `task_id` and timestamp. Shutdown writes `type: disconnected`.
- [ ] **R3 — Command vocabulary:** The runner forwards SSE payloads whose top-level `event_type` or `type` is one of: `checkpoint`, `steer`, `abort`, `ask`, `report`, `ping`. For generic envelopes, `type: command` with `payload.task_id` matching the active task is accepted (see **Event envelope** below).
- [ ] **R4 — Steer semantics:** `steer` carries a payload (e.g. `direction`, `constraints`, `priority`) that **supersedes or augments** the in-flight instruction set; the agent must acknowledge in the response channel (see R6).
- [ ] **R5 — Checkpoint / abort / ask:**  
  - `checkpoint` — persist progress to `.task-checkpoint.md` (or equivalent) and ack.  
  - `abort` — stop safely, persist partial state, exit with non-zero if required by runner contract.  
  - `ask` — pause until operator answers (payload contains `question` / `options`); response POST includes `answer` or `approved: bool`.
- [ ] **R6 — Response path:** Agent-side responses are appended to `{task_work_dir}/.task-response` as JSONL; the control channel **POSTs** each new line to **`POST /api/agent/tasks/{task_id}/activity`** with a distinguishable `event_type` (e.g. `control_response_*`) and full JSON in `data`.
- [ ] **R7 — Prompt injection:** The runner **injects** documented operator instructions (how often to poll control file, how to format responses) into the task prompt so **all** supported CLIs receive the same contract without per-vendor forks.
- [ ] **R8 — Idempotency:** Duplicate SSE deliveries (reconnect) must not corrupt state: agents **ignore duplicate command ids** when `command_id` is present; if absent, last-write-wins for same `type`+`timestamp` bucket is acceptable for MVP.

### Non-functional

- [ ] **N1 — Resilience:** SSE disconnects trigger **exponential backoff reconnect**; gaps are logged; no silent drop without log line.
- [ ] **N2 — Security:** Only **authenticated** callers may enqueue commands that reach a node stream (existing federation / auth model); task-scoped commands should include **`task_id`** validation on the API side when a dedicated enqueue endpoint is added (see API changes).
- [ ] **N3 — Performance:** Control file polling interval default ≤ **7 minutes** for human-scale steering; **≤ 30s** optional fast mode via env for high-touch tasks.

### Open question: Improving the idea, proving it works, and clearer proof over time

| Mechanism | Purpose |
|-----------|---------|
| **Control-plane metrics** | Counters: `control_commands_received`, `control_acks_posted`, `sse_reconnects`, `command_latency_ms` (histogram), exported via `GET /api/runtime/endpoints/summary` or dedicated metrics row — proves the path is live, not “best effort”. |
| **Task activity schema** | Standardize `event_type` prefixes (`control_in_*`, `control_out_*`, `heartbeat`) so monitors can alert when **heartbeat age** > threshold while task status is still `executing`. |
| **E2E proof script** | Repo script: start mock SSE publisher → assert lines in `.task-control` → agent stub POSTs activity → `GET /api/agent/tasks/{id}/stream` shows events (see Verification). |
| **Dashboards** | Web `/tasks` or `/pipeline`: column “last control ack” and “SSE connected” for running tasks — human-visible proof. |
| **Version field** | `control_channel_version` in activity payload — reviewers compare rollout across nodes. |

## API changes

### Existing endpoints (must remain stable; this spec builds on them)

| Method | Path | Role |
|--------|------|------|
| `GET` | `/api/federation/nodes/{node_id}/stream` | SSE: node receives messages/events; runner’s `TaskControlChannel` connects here (`Accept: text/event-stream`). |
| `POST` | `/api/federation/nodes/{node_id}/messages` | Send message to node; can target task via `payload` (broadcast with `to_node: null` uses `/api/federation/broadcast`). |
| `POST` | `/api/federation/broadcast` | Broadcast to all node SSE subscribers. |
| `POST` | `/api/agent/tasks/{task_id}/activity` | Ingest heartbeats, `control_response_*`, progress (`ActivityEvent`: `event_type`, `data`, optional `node_id`, `provider`). |
| `GET` | `/api/agent/tasks/{task_id}/events` | SSE: **task-level** stream of logged activity/events for observers (media type `text/event-stream`). |
| `GET` | `/api/agent/tasks/{task_id}/stream` | Snapshot/history of events for one task (non-SSE list in current implementation — verify name collision in implementation phase). |

### Proposed additions (implementation task)

| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/agent/tasks/{task_id}/control` | **Optional but recommended:** Authenticated enqueue of a control command **validated against** `task_id` and `node_id`; server pushes to that node’s SSE queue in a **canonical envelope** (reduces “guess the federation payload shape”). |
| `GET` | `/api/agent/tasks/{task_id}/control/status` | **Optional:** Last SSE connect time, last command id processed, heartbeat age — for `/tasks` UI. |

*If `POST .../control` is deferred, operators use federation message POST with a documented JSON envelope (below).*

### Event envelope (canonical)

**SSE `data:` line (JSON):**

```json
{
  "event_type": "steer",
  "task_id": "task_abc123",
  "command_id": "cmd_01HZ...",
  "payload": {
    "direction": "Prefer pytest over ad-hoc scripts.",
    "constraints": ["no new md files"]
  },
  "timestamp": "2026-03-28T12:00:00Z"
}
```

**Activity POST (agent → API) for a control ack:**

```json
{
  "event_type": "control_response_steer",
  "node_id": "node_worker_1",
  "provider": "claude",
  "data": {
    "command_id": "cmd_01HZ...",
    "ack": true,
    "summary": "Will prioritize pytest; noted constraints."
  }
}
```

## Data model

### Files (local, runner-managed)

| Path | Direction | Format |
|------|-----------|--------|
| `{task_work_dir}/.task-control` | Network → agent | JSONL; includes `connected`, command lines, `disconnected` |
| `{task_work_dir}/.task-response` | Agent → network | JSONL; drained by control channel and POSTed to activity |

### Logical entities

```yaml
ControlCommand:
  properties:
    event_type: { type: string, enum: [checkpoint, steer, abort, ask, report, ping, command] }
    task_id: { type: string }
    command_id: { type: string, description: optional idempotency key }
    payload: { type: object }
    timestamp: { type: string, format: date-time }

ControlResponse:
  properties:
    type: { type: string }
    task_id: { type: string }
    command_id: { type: string }
    ack: { type: boolean }
    data: { type: object }
```

## Acceptance criteria (product)

1. A running task **logs** at least one `connected` control event and periodic **heartbeat** or **control_response** activity when the agent obeys the prompt contract.
2. An operator can **inject** a `checkpoint` or `steer` and see a **correlated** activity row within one poll window.
3. **Abort** results in terminal task state with **partial artifacts** saved (checkpoint file present) and no silent hang.
4. **Duplicate** command ids do not produce duplicate side effects (or duplicate rows are clearly marked `duplicate: true`).

## Verification Scenarios

Scenarios below are **contract tests** for reviewers; they assume `API` = `https://api.coherencycoin.com` (or staging) and valid auth headers where required.

### Scenario 1 — Create task, observe SSE + activity CRUD cycle

- **Setup:** Node `NODE_ID` is registered and can open SSE; no prior activity rows for a new `task_id` (or use a freshly `POST`ed task).
- **Action:**  
  1. `curl -s -X POST "$API/api/agent/tasks" -H "Content-Type: application/json" -d '{"direction":"noop smoke","task_type":"impl"}'` → capture `task_id`.  
  2. `curl -s -N -H "Accept: text/event-stream" "$API/api/federation/nodes/NODE_ID/stream"` in background (or timeout 5s) → expect initial `event_type":"connected"`.  
  3. `curl -s -X POST "$API/api/agent/tasks/{task_id}/activity" -H "Content-Type: application/json" -d '{"event_type":"heartbeat","data":{"step":"sse-proof"}}'`  
  4. `curl -s "$API/api/agent/tasks/{task_id}/stream"` → JSON array includes an event where `event_type` is `heartbeat` and `data.step` is `sse-proof`.
- **Expected:** HTTP 201 on POST activity; GET stream returns the heartbeat event with matching `task_id`; SSE connected event present for the node stream.
- **Edge — bad input:** POST activity with **empty** `event_type` → **422** Unprocessable Entity (validation error), not 500.  
- **Edge — missing resource:** GET `/api/agent/tasks/nonexistent_task/stream` → **404** or empty documented behavior (must not 500).

### Scenario 2 — Control command delivery to `.task-control` (runner integration)

- **Setup:** Local or CI runner with `task_control_channel` enabled; task work dir writable; `TaskControlChannel` started for `(node_id, task_id)`.
- **Action:**  
  1. Publish a federation message or broadcast that matches the canonical envelope with `event_type":"checkpoint"` and the active `task_id`.  
  2. Read `{task_work_dir}/.task-control` after ≤ 30s.
- **Expected:** File contains a JSON line with `"type":"checkpoint"` or forwarded command shape; log line `CONTROL_CMD checkpoint → task ...` appears in runner logs.
- **Edge — duplicate:** Send same `command_id` twice → second line may appear but agent **must** ack once (or POST includes `duplicate: true`); no double checkpoint side effects in task state machine.

### Scenario 3 — Full steer cycle (network → file → POST ack)

- **Setup:** As Scenario 2; agent stub appends to `.task-response`: `{"type":"steer_ack","task_id":"...","command_id":"cmd_1"}`.
- **Action:** Emit `steer` with `command_id":"cmd_1"`; wait for poller; `curl -s "$API/api/agent/tasks/{task_id}/stream"` filtered for `control_response`.
- **Expected:** Activity contains `event_type` matching `control_response_*` and `data.command_id` == `cmd_1`.
- **Edge — wrong task_id:** Envelope with mismatched `task_id` → **no** line in `.task-control` for this task (filtered out).

### Scenario 4 — Abort and terminal state

- **Setup:** Task in `executing` with control channel active.
- **Action:** Send `abort` via SSE envelope; agent writes abort ack and exits; runner PATCHes task `failed` or `cancelled` per policy.
- **Expected:** `.task-checkpoint.md` exists OR last activity records `abort` completion; task status terminal within timeout; `GET /api/agent/tasks/{task_id}` shows non-running status.
- **Edge — abort without running task:** POST control for **completed** task → **409** or 404 with clear body (no crash).

### Scenario 5 — Error handling and SSE reconnect

- **Setup:** Simulate API restart or drop TCP to node SSE.
- **Action:** Observe runner logs for `CONTROL_SSE error` and reconnect; send `ping` after reconnect.
- **Expected:** Within **≤ 60s** SSE re-establishes; `ping` appears in `.task-control` or activity shows reconnect metric; no orphaned zombie provider without watchdog.
- **Edge — malformed SSE line:** Non-JSON `data:` line must **not** crash listener; skipped with at most debug log.

## Concurrency behavior

- **Control file:** JSONL append from one writer (control channel thread); agent **reads** and tracks byte offset or line count — avoid whole-file rewrite from agent.
- **Activity POST:** May arrive out of order; consumers use `timestamp` and `command_id` for ordering.

## Out of scope

- Replacing provider-specific CLIs with a custom runtime.
- Guaranteed sub-second steering latency (MVP is human-scale).
- Binary streaming of large artifacts over SSE.

---

## Risks and assumptions

- **Risk:** Federation SSE payload shapes drift from what `task_control_channel` expects → **mitigation:** canonical envelope + single `POST /control` endpoint.  
- **Assumption:** Operators have a valid path to publish to the node stream (auth + network).  
- **Assumption:** `POST /api/agent/tasks/{task_id}/activity` remains the single ingress for agent-emitted telemetry on public API.

## Failure / retry reflection

- **Failure mode:** SSE silent stall (no lines, no errors).  
- **Blind spot:** Proxy buffering despite `X-Accel-Buffering: no`.  
- **Next action:** Server-side keepalive comments + client read timeout + metric `sse_last_event_age_seconds`.

## Decision gates

- Whether control commands are **only** via federation messages vs. a **first-class** task control API (security + UX tradeoff).
- Whether **`ask`** requires a synchronous blocking wait in the runner or only agent-side pause (HITL policy).
