## Summary

Package the Coherence Network MCP server as both an npm package
(`coherence-mcp-server`) and a PyPI package (`coherence-mcp-server`) so that
the server appears on the **Official MCP Registry**, **Smithery**, **Glama**,
and all major discovery platforms. Provide a fully updated `server.json`
manifest compatible with `mcp-publisher` for automated registry submission.
Add CI/CD workflows that publish both packages on version tag, and expose an
API endpoint that verifies published versions against discovery platform
status so "is it working?" is answered with hard data, not hope.

## Requirements

### R1 — npm Package (Node.js)

1. `mcp-server/package.json` MUST have `"name": "coherence-mcp-server"` and a
   `"version"` that matches `mcp-server/server.json`.
2. The package MUST be installable without root: `npx coherence-mcp-server`
   starts the MCP server on stdio.
3. `package.json` MUST declare `"files": ["index.mjs", "README.md"]` (already
   done) and `"engines": {"node": ">=18.0.0"}`.
4. The package MUST be published to the public npm registry under the unscoped
   name `coherence-mcp-server`.
5. A `.npmignore` MUST exclude `*.md` template files, `pulsemcp.json`,
   `glama.json`, `server.json` (registry manifests should not ship in the
   tarball).

### R2 — PyPI Package (Python)

6. A new top-level directory `mcp-server-py/` MUST contain:
   - `pyproject.toml` — defines `coherence-mcp-server` PyPI package
   - `coherence_mcp_server/__init__.py` — package init
   - `coherence_mcp_server/__main__.py` — `python -m coherence_mcp_server`
     entry that delegates to `api/mcp_server.py`-equivalent logic
   - `coherence_mcp_server/server.py` — self-contained server (no `api/`
     dependency) using the `mcp` Python SDK, connecting to `COHERENCE_API_URL`
     via HTTP (same tool surface as the Node version)
   - `README.md` — symlink or copy of `mcp-server/README.md` adapted for Python
7. The package MUST be installable via:
   `pip install coherence-mcp-server` and launched via:
   `python -m coherence_mcp_server`
8. Entry point script `coherence-mcp-server` MUST be registered in
   `[project.scripts]`.
9. Dependencies: `mcp>=1.0.0`, `httpx>=0.28.0`.
10. Python >=3.10 required (uses `match` + modern type hints).

### R3 — server.json Manifest (mcp-publisher)

11. `mcp-server/server.json` MUST be updated to:
    - Sync `"version"` with `mcp-server/package.json` (currently `0.3.1`)
    - Add a second entry in `"packages"` for the PyPI package:
      ```json
      {
        "registry": "pypi",
        "name": "coherence-mcp-server",
        "version": "<same as npm version>"
      }
      ```
    - Include `"runtime_config"` with env var documentation:
      ```json
      {
        "runtime_config": {
          "env": {
            "COHERENCE_API_URL": {
              "description": "Base URL of the Coherence Network API",
              "default": "https://api.coherencycoin.com",
              "required": false
            },
            "COHERENCE_API_KEY": {
              "description": "Optional API key for write operations",
              "required": false
            }
          }
        }
      }
      ```
    - Ensure `"categories"`, `"tags"` are populated per registry requirements.

### R4 — CI/CD Publish Workflows

12. `.github/workflows/publish-npm.yml` MUST:
    - Trigger on push of tags matching `mcp-v*` (e.g., `mcp-v0.3.2`)
    - Run `npm publish` from `mcp-server/` using `NODE_AUTH_TOKEN` secret
    - Post-publish step: `curl` npm API to verify the version is live
    - Fail the job if the published version does not match the tag

13. `.github/workflows/publish-pypi.yml` MUST:
    - Trigger on push of tags matching `mcp-py-v*`
    - Use `pypa/gh-action-pypi-publish` with `PYPI_API_TOKEN` secret
    - Source: `mcp-server-py/` directory
    - Post-publish step: `pip install coherence-mcp-server==<version>` smoke test

14. A version-sync CI step MUST fail the PR if `server.json` version ≠
    `package.json` version. Add to `.github/workflows/publish-npm.yml` as a
    pre-publish check, or as a standalone `check-versions.yml`.

### R5 — API: Publish Status Endpoint

15. A new endpoint `GET /api/discovery/publish-status` MUST exist and return:
    ```json
    {
      "npm": {
        "package": "coherence-mcp-server",
        "latest_version": "0.3.1",
        "published_at": "2026-03-28T12:00:00Z",
        "weekly_downloads": 142,
        "listed": true
      },
      "pypi": {
        "package": "coherence-mcp-server",
        "latest_version": "0.3.1",
        "published_at": "2026-03-28T12:00:00Z",
        "monthly_downloads": 87,
        "listed": true
      },
      "registries": {
        "mcp_official": { "listed": true, "url": "https://registry.modelcontextprotocol.io/..." },
        "smithery": { "listed": false, "url": null },
        "glama": { "listed": true, "url": "https://glama.ai/mcp/servers/coherence-network" },
        "pulsemcp": { "listed": true, "url": "https://pulsemcp.com/servers/coherence-network" }
      },
      "server_json_version": "0.3.1",
      "npm_version_in_sync": true,
      "pypi_version_in_sync": true,
      "checked_at": "2026-03-28T15:30:00Z"
    }
    ```
16. The endpoint MUST fetch live data from:
    - npm: `https://registry.npmjs.org/coherence-mcp-server` (latest version,
      publish date)
    - npm downloads: `https://api.npmjs.org/downloads/point/last-week/coherence-mcp-server`
    - PyPI: `https://pypi.org/pypi/coherence-mcp-server/json` (latest version,
      upload time)
    - PyPI downloads: via `pypistats.org` API or `https://pypistats.org/api/packages/coherence-mcp-server/recent`
17. The endpoint MUST cache responses for 5 minutes to avoid hammering external
    APIs (use a module-level TTL cache or Redis if available).
18. If an external API is unreachable, the corresponding field MUST return
    `{"error": "upstream timeout", "listed": null}` — never 500.

### R6 — Discovery Platform Listing Verification

19. Smithery listing requires submitting to
    `https://smithery.ai/submit` (form or API). The spec requires that the
    submission is made and tracked. The `registries.smithery.listed` field in R5
    reflects actual listing status queried from Smithery's search API or HTML
    (fallback: `listed: null` if unverifiable programmatically).
20. Glama listing requires a PR to the `awesome-mcp-servers` repo (already
    tracked in Spec 180). The `registries.glama.listed` field is verified by
    checking `https://glama.ai/mcp/servers` for the package name.
21. Official MCP Registry (`registry.modelcontextprotocol.io`) requires a
    `server.json` that passes schema validation. The listing check queries the
    public search endpoint.

## Data Model

### New Pydantic Models (api/app/models/publish_status.py)

```python
class RegistryListingStatus(BaseModel):
    listed: bool | None          # None = could not determine
    url: str | None = None
    checked_at: datetime | None = None

class PackagePublishInfo(BaseModel):
    package: str
    latest_version: str | None
    published_at: datetime | None
    weekly_downloads: int | None = None   # npm
    monthly_downloads: int | None = None  # PyPI
    listed: bool
    error: str | None = None

class PublishStatusResponse(BaseModel):
    npm: PackagePublishInfo
    pypi: PackagePublishInfo
    registries: dict[str, RegistryListingStatus]
    server_json_version: str | None
    npm_version_in_sync: bool
    pypi_version_in_sync: bool
    checked_at: datetime
```

## API Changes

### New endpoint

```
GET /api/discovery/publish-status
```

**Response:** `PublishStatusResponse` (see Data Model above)

**Status codes:**
- `200 OK` — always returned even if upstream APIs fail (degraded data)
- `503` — only if the API itself is broken (not if npm/PyPI are slow)

**No auth required.** Public endpoint.

### Existing endpoint not changed

`GET /api/discovery/registry-submissions` remains unchanged. The new endpoint
supplements it with live publish data.

## Improvement Loop: Proving It Works Over Time

The open question "how do we show whether it is working and make that proof
clearer over time?" is answered by three mechanisms:

### 1. Live Publish Status Dashboard Widget (web)

A card on the `/settings` or `/nodes` page shows:
- npm latest version + weekly downloads
- PyPI latest version + monthly downloads
- Registry listing badges (MCP Official ✓/✗, Smithery ✓/✗, Glama ✓/✗)
- Last checked timestamp

Data comes from `GET /api/discovery/publish-status` polled every 5 minutes.

### 2. Download Count Trend in Resonance

Weekly npm downloads and monthly PyPI downloads are treated as external
_resonance signals_. They are appended to the resonance feed so that rising
install counts show up as positive energy events in the network activity view.

### 3. Version Mismatch Alert

If `server.json` version ≠ `package.json` version, `GET /api/health` downgrades
`status` to `degraded` with a specific reason code
(`"mcp_server_version_mismatch"`). This makes the gap impossible to ignore.

## Verification Scenarios

### Scenario 1 — npm Package Installation (Happy Path)

**Setup:** A clean shell with Node.js 18+, no prior install of `coherence-mcp-server`

**Action:**
```bash
npx coherence-mcp-server --help 2>&1 | head -5
```

**Expected result:** Process starts (or prints usage), no error about missing
modules. If `--help` is not implemented, the server starts and waits for MCP
stdio — verify via:
```bash
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' | npx coherence-mcp-server
```
**Expected:** JSON response containing `"tools"` array with ≥15 entries

**Edge case — stale npx cache:**
```bash
npx --yes coherence-mcp-server@0.3.1
```
Must install version 0.3.1 exactly. If npm is down, `npx` returns non-zero exit
and the test is skipped (not failed).

---

### Scenario 2 — PyPI Package Installation (Happy Path)

**Setup:** A clean Python 3.10+ virtualenv

**Action:**
```bash
pip install coherence-mcp-server
coherence-mcp-server --version  # or python -m coherence_mcp_server --version
```

**Expected result:** Version string matching PyPI latest (e.g. `0.3.1`)

**Then — tool list check:**
```bash
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' \
  | COHERENCE_API_URL=https://api.coherencycoin.com python -m coherence_mcp_server
```
**Expected:** JSON with `"tools"` array, ≥15 named tools

**Edge case — Python 3.9:**
`pip install coherence-mcp-server` must fail with a clear error
("Requires-Python >=3.10") not a cryptic import error.

---

### Scenario 3 — Publish Status API Endpoint

**Setup:** API running (local or production)

**Action:**
```bash
curl -s https://api.coherencycoin.com/api/discovery/publish-status | python3 -m json.tool
```

**Expected result:** JSON matching the schema in R5, including:
- `npm.latest_version` — non-null string (e.g. `"0.3.1"`)
- `npm.listed` — `true`
- `pypi.latest_version` — non-null string
- `npm_version_in_sync` — `true`
- `pypi_version_in_sync` — `true`
- `checked_at` — ISO 8601 timestamp within the last 5 minutes

**Edge case — external API unreachable:** Simulate by adding a bad hosts entry
for `registry.npmjs.org`. The endpoint MUST still return HTTP 200 with
`npm.error: "upstream timeout"` — never HTTP 500.

---

### Scenario 4 — server.json Version Sync Guard (CI)

**Setup:** Git branch where `mcp-server/server.json` has version `"0.2.0"` but
`mcp-server/package.json` has `"version": "0.3.1"`

**Action:** Push branch; the `check-versions.yml` CI workflow runs

**Expected result:** CI job fails with message:
```
ERROR: server.json version (0.2.0) does not match package.json version (0.3.1)
```
Exit code non-zero; PR cannot merge.

**Edge case — versions match:** Same workflow passes with exit 0 and message:
```
OK: server.json and package.json both at 0.3.1
```

---

### Scenario 5 — mcp-publisher server.json Dry Run

**Setup:** `mcp-publisher` installed locally (`npm install -g mcp-publisher`)

**Action:**
```bash
mcp-publisher validate mcp-server/server.json
```

**Expected result:** Exit 0, no errors. Output confirms both `npm` and `pypi`
package entries are present and schema-valid.

**Edge case — missing PyPI entry:** Remove `pypi` from `packages[]` in
`server.json`. `mcp-publisher validate` must exit non-zero or warn that
only one registry package is declared.

## Risks and Assumptions

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| npm package name `coherence-mcp-server` already claimed by another publisher | Low (check: `npm view coherence-mcp-server` before publish) | Use scoped `@coherence-network/mcp-server` as fallback |
| PyPI name conflict | Medium | Check `pypi.org/project/coherence-mcp-server` before create |
| Smithery auto-listing requires manual approval queue | High | Submit immediately; track in `registries.smithery.pending` field |
| `mcp` Python library API changes | Medium | Pin `mcp>=1.0.0,<2.0.0` |
| PyPI stats API rate limits | Low | Cache for 1 hour, not 5 minutes, for PyPI stats specifically |
| npm `npx` caches old version | Low | Document `npx --yes` flag in README |
| server.json schema changes at `registry.modelcontextprotocol.io` | Low | Track schema version in CI lint step |

**Assumptions:**
- `NODE_AUTH_TOKEN` and `PYPI_API_TOKEN` secrets are available in GitHub Actions
- The repo has access to push to npm under the unscoped `coherence-mcp-server` name
- `mcp-publisher` CLI tool is maintained and can validate `server.json`
- Smithery, Glama, PulseMCP all offer some form of machine-queryable listing status

## Known Gaps and Follow-up Tasks

1. **Scoped npm alternative** — If `coherence-mcp-server` is taken on npm,
   follow-up spec needed for `@coherence-network/mcp-server` + README update.

2. **PyPI download stats granularity** — `pypistats.org` only provides recent
   (last day/week/month) counts. Long-term trend requires our own timeseries
   recording (daily cron job writing to PostgreSQL). Not in scope for this spec.

3. **Smithery programmatic listing check** — Smithery does not expose a public
   search API. Current implementation will check via `fetch` + HTML parse of
   their search results page. Fragile. Follow-up: watch Smithery for official
   API and switch when available.

4. **Automated version bump** — A tool (script or CI) to bump all version
   references atomically (package.json, server.json, pyproject.toml) is
   desirable but out of scope here.

5. **MCP Official Registry PRs** — Submission to
   `registry.modelcontextprotocol.io` may require a PR to their GitHub repo.
   This spec authorises the submission; the PR itself is a follow-up task.

6. **Python server tool parity** — The PyPI `server.py` must expose exactly the
   same tool names and schemas as the Node.js `index.mjs`. Drift is a risk.
   A future spec should add a tool-surface parity test that runs both and diffs
   their `tools/list` responses.
