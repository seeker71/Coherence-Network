# Verification-First Task Execution

## Summary

The `failed-tasks` friction category is the highest-ROI signal in the body right now:
severity **high**, **296 events**, ~2.2M wasted minutes. The dominant failure shape is a
task that *reports* completion with no durable proof, or that hits a known blocker
(lint/test failure, missing config key, stale branch, flaky CI, missing tool) and dies
instead of unblocking. This spec adds a **proof gate** on the completion path and a
**blocker→unblock playbook**, so a cheap/fast executor that misses constraints is forced
to either produce proof or document a blocker — never to claim a hollow success.

## Purpose

Reduce repeated `failed-tasks` failures before running more CI-heavy iterations by making
proof a precondition of completion and by turning the five common blockers into exact,
copy-paste unblock recipes with their own acceptance proof. This optimizes first for
**trust** (a "complete" task is verifiable), then **clarity** (blockers self-describe their
fix), then **reuse** (one playbook serves every task type).

## Intent and System-Level Behavior

- **Intent first** — we optimize for *trust* in the completion signal, *clarity* of the
  next action when blocked, and *reuse* of one diagnostics/retry/playbook spine across all
  task types. Completion percentage is a side effect, not the target.
- **System-level lens** — today the runner marks a task complete on the executor's word and
  records a friction event only after the fact. After this change the running system's
  behavior shifts: `agent_execution_completion` calls the proof gate *before* writing
  terminal status. A claim without a proof artifact transitions to **retry** (bounded), and
  a genuine blocker is recorded as a structured friction event carrying the matching
  playbook key. Operators watching `GET /api/agent/friction` see fewer "failed, unknown"
  rows and more "blocked, playbook=X" rows that point straight at the fix.
- **Proof of meaning** — for the human operator this means the dashboard stops lying: a
  green task is one whose proof exists on disk or in an API response, and a red task names
  exactly what to run next. Less time spent re-deriving why a task died; the wasted-minutes
  number trends down because retries replace dead-ends.

## Approaches Considered

1. **Prompt-only discipline** — instruct executors to "show proof." Cheap, zero code, but
   the friction data is *generated by* executors that already ignore implicit constraints;
   a cheap model will skip it. Rejected: no enforcement.
2. **Hard external CI gate** — block every completion behind full CI. Strong proof, but
   CI-heavy (the exact thing the friction signal says to reduce) and flaky-CI becomes a new
   failure source. Rejected as the primary mechanism; CI stays the deep check, not the gate.
3. **In-process proof gate + unblock playbook (chosen)** — a deterministic gate on the
   completion path that accepts a *named* proof artifact (file path, command exit status, or
   API response field) and, on its absence, routes to bounded retry or records a structured
   blocker. **Chosen** because it enforces without adding CI load, reuses the existing
   diagnostics/retry/friction services, and degrades to an explicit blocker rather than a
   silent failure. Long-term tradeoff: a small amount of completion-path code to maintain in
   exchange for removing the single largest waste category at its root.

## Unblock Playbook

Each blocker has exact commands and an acceptance proof. Prefer the **spec-first** step when
the gap is a missing requirement rather than a mechanical failure.

- **lint/test failure** — `python3 -m pytest <path> -q` then `ruff check api/` /
  `npm run lint`. Acceptance proof: `0 failed` in pytest summary and lint exit 0.
  Spec-first: if the test encodes an unspecified behavior, update the spec's Requirements
  before changing code.
- **missing config key/secret** — read the file-backed config sources in order:
  `python3 -m json.tool ~/.coherence-network/config.json` and
  `python3 -m json.tool api/config/api.json`.
  Acceptance proof: the required key is present in one of those files, or the blocker names
  the CI/CD secret carrier explicitly; never echo secret values into git.
  Spec-first: add the required config key to `## Risks and Assumptions` if undocumented.
- **stale branch / rebase conflict** — `git fetch origin && git rebase origin/main`; on
  conflict `git status` then resolve, `git rebase --continue`. Acceptance proof:
  `git log --oneline -1 origin/main` reachable from HEAD and `git status` clean.
- **flaky CI / network error** — re-run once: `gh run rerun <id> --failed`; for local
  network, retry with backoff. Acceptance proof: second run green, or the error names a
  deterministic cause (not transient) → escalate as a blocker, do not loop.
- **missing tool dependency** — `pip install -r api/requirements.txt` /
  `npm ci`; verify with `python3 -c "import <mod>"` or `<tool> --version`. Acceptance
  proof: import/version command exits 0.

## Verification Scenarios

1. **Proof-less completion is blocked** — submit a task whose completion claim names no
   artifact. Expected: `evaluate_proof` returns `missing`; `agent_execution_completion`
   calls `agent_execution_retry` with `reason="missing_proof"`; status is not `complete`.
   Proof: test assertion + absence of a terminal `complete` write in the captured calls.
2. **Blocker is documented, not swallowed** — a task hits a missing-tool error. Expected: a
   friction event is recorded with `category="missing_proof"`-adjacent classification and a
   `playbook_key`, and `GET /api/agent/unblock/missing_tool` returns the fix. Proof: the
   recorded event JSON + the API response body.
3. **Valid proof passes cleanly** — a task that wrote `data/out.json` and captured exit 0.
   Expected: gate `pass`, status `complete`, no retry. Proof: pytest assertion on verdict
   `pass` and a single terminal-status write.

## Out of Scope

- Replacing or expanding CI itself — CI remains the deep check, not the per-step gate.
- Changing executor model selection, routing, or cost controls.
- New web UI surfaces; this spec is API + service + tests only.
- Retroactively reclassifying the 296 historical friction events.

## Failure Anticipation and Guardrails

- **Two-week degradation A — proof theatre.** Executors learn to write a trivial file
  (`touch proof.txt`) to satisfy the gate. Guardrail: the gate requires the artifact to be
  *referenced by the task's claimed output* (path appears in the diff or response), and a
  weekly friction query flags tasks whose proof file is < 1 line or untouched by the diff.
- **Two-week degradation B — retry storms.** Missing-proof routing loops if the attempt cap
  is mis-wired. Guardrail: reuse `agent_execution_retry`'s existing max-attempts; on cap,
  record a blocker friction event (terminal) rather than retrying. Detector: alert if any
  task exceeds the cap or if `failed-tasks` event rate rises after deploy.

## Maintainability

Use the body's own quality-awareness signals to keep drift low: treat
`agent_execution_completion.py` and `failed_task_diagnostics_service.py` as **hotspots**
(highest churn × failure linkage) and keep the proof-gate logic in its own service so the
hotspots gain only a single call site. Recommendation: after merge, watch
`GET /api/agent/friction` for the `failed-tasks` category — if the `missing_proof` slice
does not shrink within two weeks, the gate is mis-placed and should move earlier in the
completion path. Keep the playbook table as data, not branching code, so adding a blocker is
a row edit, not a control-flow change.

## Risks and Assumptions

- Assumes `agent_execution_completion` is the single terminal-status write path; if multiple
  paths exist, the gate must wrap each (verify via `grep` before patching).
- Assumes `agent_execution_retry` exposes a callable honoring a max-attempt cap; reuse it
  rather than adding a counter.
- Risk: an over-strict gate blocks legitimate completions whose proof is implicit. Mitigated
  by accepting three proof kinds (file, command, api_field) and by defaulting to retry, not
  hard-fail, on the first miss.
- Assumes application configuration is file-backed through `~/.coherence-network/config.json`
  and `api/config/api.json`; CI/CD secrets remain runner-managed and are named by carrier,
  not value.
