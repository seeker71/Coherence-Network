# Spec: Consolidate Navigation — /nodes + /pipeline (Remove /automation, /usage, /remote-ops)

**Idea ID**: task_b605232289da6daf
**Status**: Proposed
**Author**: product-manager agent
**Date**: 2026-03-28
**Priority**: High — reduces cognitive overhead for all users

---

## Summary

Four pages currently show overlapping data:

| Page | Primary Content |
|------|----------------|
| `/automation` | Provider readiness, provider exec stats, node network stats, fleet capabilities |
| `/usage` | Provider usage, daily runtime summary, tool attention, friction signals, view performance |
| `/remote-ops` | Queue/pipeline controls, deployment/uptime status, task dispatch |
| `/nodes` | Federation node list, health/status, messaging |
| `/pipeline` | Running tasks, completed tasks, streaks, provider performance |

The consolidation collapses these into two clear, opinionated pages:

1. **`/nodes`** — Everything about *nodes*: list, health, providers per node, system metrics, remote messaging, deployment uptime.
2. **`/pipeline`** — Everything about *task execution*: queue, running, completed, streaks, provider performance, dispatch controls.

`/automation`, `/usage`, and `/remote-ops` are removed and replaced with HTTP 301 redirects.

---

## Goal

**Single mental model**: A user monitoring the network visits `/nodes` for "what is running" and `/pipeline` for "what is being done." No need to correlate three tabs to answer one question.

**Success signal**: A user who previously visited `/automation`, `/usage`, or `/remote-ops` is seamlessly redirected and finds the relevant data in the destination page within one scroll.

---

## Motivation

### Pain points today

- **Duplicate provider stats**: `/automation` and `/usage` both render `GET /api/providers/stats`. Users see the same table in two places.
- **Split queue visibility**: `/remote-ops` shows the queue; `/pipeline` also shows running tasks. Neither gives the full picture alone.
- **Node health fragmentation**: `/automation` shows node network stats; `/nodes` shows the federation list. Operators need two tabs open to assess a node.
- **Dispatch controls buried**: `/remote-ops` has task dispatch, but `/pipeline` is where users watch results. The action and its feedback are separated.

### Design principle

> "Show me what's alive, show me what's running."
> `/nodes` answers "alive"; `/pipeline` answers "running."

## Requirements

### REQ-1: `/nodes` Page Enhancement

The existing `/nodes` page MUST absorb the following content from removed pages:

| Content | Source | Placement |
|---------|--------|-----------|
| Provider readiness table (configured/missing env) | `/automation` → `GET /api/providers/readiness` | New "Provider Readiness" section |
| Node network stats (per-node provider success rates) | `/automation` → `GET /api/automation/network-stats` | Inline in each node card |
| Fleet capabilities summary (total CPUs, executors) | `/automation` → `GET /api/automation/fleet-capabilities` | Fleet summary header card |
| Deployment/uptime status | `/remote-ops` → `GET /api/health`, `/api/proxy/health` | New "Infrastructure Health" section |
| Node-level system metrics (CPU %, memory %, disk) | Already in `/nodes` | No change |
| Messaging / MessageForm | Already in `/nodes` | No change |

### REQ-2: `/pipeline` Page Enhancement

The existing `/pipeline` page MUST absorb the following content:

| Content | Source | Placement |
|---------|--------|-----------|
| Provider exec stats (success rates, last-5, duration) | `/automation` + `/usage` → `GET /api/providers/stats` | New "Provider Performance" tab or section |
| Usage alerts (providers near limit) | `/automation` → `GET /api/automation/usage-alerts` | Alert banner at top of pipeline page |
| Daily runtime summary (tasks/day, cost/day) | `/usage` → `GET /api/tasks/summary` | Summary row in pipeline header |
| Queue / pending tasks table | `/remote-ops` → `GET /api/tasks?status=pending` | Already present; consolidate |
| Task dispatch controls (executor, model, run) | `/remote-ops` | Collapsible "Dispatch" panel |
| Tool attention / top-tools signals | `/usage` | New "Signals" section (low priority, below fold) |
| Friction signals | `/usage` | New "Signals" section |
| View performance metrics | `/usage` | Move to `/pipeline` Signals or remove |

### REQ-3: Redirects

All three removed routes MUST redirect:

| Old Path | Redirect Target | HTTP Status |
|----------|-----------------|-------------|
| `/automation` | `/nodes` | 301 Permanent |
| `/usage` | `/pipeline` | 301 Permanent |
| `/remote-ops` | `/pipeline` | 301 Permanent |

Redirects are implemented via Next.js `redirects()` in `next.config.ts` (or `.js`).

### REQ-4: Navigation Updates

All in-app navigation links referencing `/automation`, `/usage`, or `/remote-ops` MUST be updated:

- Main layout nav (if present)
- Home page / dashboard quick-links
- Internal `<Link>` references within any page component
- Any breadcrumb or sidebar items

### REQ-5: Page Files Removal

After redirects are confirmed working:

- `web/app/automation/` directory removed
- `web/app/usage/` directory removed
- `web/app/remote-ops/` directory removed

### REQ-6: No Data Regression

Every data display that existed on the three removed pages MUST appear on either `/nodes` or `/pipeline`. Nothing is silently dropped.

## API Endpoints Required

All existing API endpoints remain unchanged. No new backend endpoints are needed. The consolidation is purely a frontend restructuring.

Endpoints consumed by the new pages:

**`/nodes`**:
- `GET /api/nodes` — federation node list
- `GET /api/automation/network-stats` — per-node provider stats
- `GET /api/automation/fleet-capabilities` — fleet hardware summary
- `GET /api/providers/readiness` — provider config status
- `GET /api/health` — API health
- `GET /api/proxy/health` — deploy/proxy health
- `POST /api/nodes/{node_id}/messages` — node messaging

**`/pipeline`**:
- `GET /api/tasks?status=pending` — pending queue
- `GET /api/tasks?status=running` — active tasks
- `GET /api/tasks?status=completed` — recent completed
- `GET /api/providers/stats` — provider exec performance
- `GET /api/automation/usage-alerts` — usage limit alerts
- `GET /api/tasks/summary` — daily summary
- `POST /api/tasks` — task dispatch (from controls panel)
- `POST /api/tasks/{id}/run` — force-run a pending task

## Implementation Plan

### Phase 1 — Redirects (1 file change)
Add to `next.config.ts`:
```ts
async redirects() {
  return [
    { source: '/automation', destination: '/nodes', permanent: true },
    { source: '/usage', destination: '/pipeline', permanent: true },
    { source: '/remote-ops', destination: '/pipeline', permanent: true },
  ];
}
```

### Phase 2 — `/nodes` absorbs automation + infra health content
1. Add `FleetSummaryCard` component (fleet capabilities)
2. Add `ProviderReadinessSection` component
3. Add `InfraHealthSection` (deployment/uptime from remote-ops)
4. Wire into `web/app/nodes/page.tsx`

### Phase 3 — `/pipeline` absorbs usage + remote-ops content
1. Add `ProviderPerformanceSection` component (from provider stats)
2. Add `UsageAlertsBar` component (alert banner)
3. Move `DispatchControls` from remote-ops into pipeline page (collapsible)

### Phase 4 — Navigation cleanup
1. Update all `<Link href="/automation">`, `<Link href="/usage">`, `<Link href="/remote-ops">` across the codebase
2. Update home/dashboard page quick-links

### Phase 5 — File deletion
1. Remove `web/app/automation/`
2. Remove `web/app/usage/`
3. Remove `web/app/remote-ops/`

### Scenario 1: Redirects work correctly

**Setup**: Production site deployed with redirects configured in `next.config.ts`.

**Action**:
```bash
curl -sI https://coherencycoin.com/automation | grep -E "HTTP|Location"
curl -sI https://coherencycoin.com/usage | grep -E "HTTP|Location"
curl -sI https://coherencycoin.com/remote-ops | grep -E "HTTP|Location"
```

**Expected**:
```
HTTP/2 301
location: /nodes

HTTP/2 301
location: /pipeline

HTTP/2 301
location: /pipeline
```

**Edge case**: Browser navigating to `/automation` arrives at `/nodes` without error page. URL bar shows `/nodes`.

### Scenario 2: /nodes contains provider readiness data

**Setup**: API running, at least one provider configured.

**Action**:
```bash
# Verify API responds
curl -s https://api.coherencycoin.com/api/providers/readiness | jq '.providers | length'

# Verify /nodes page renders (no 404, no 500)
curl -sI https://coherencycoin.com/nodes | grep "HTTP"
```

**Expected**:
- API returns JSON with `providers` array, count > 0
- `/nodes` returns HTTP 200
- Browser visit to `/nodes` shows "Provider Readiness" section with at least one provider row showing `configured: true/false`

**Edge case**: If `GET /api/providers/readiness` returns 500, the `/nodes` page still loads (section shows "unavailable" gracefully rather than crashing the page).

### Scenario 3: /pipeline contains dispatch controls and queue

**Setup**: At least one task in `pending` state exists, or can be created.

**Action**:
```bash

curl -s "https://api.coherencycoin.com/api/tasks?status=pending&limit=5" | jq '.tasks | length'

# Create a test task to appear in queue
curl -s -X POST https://api.coherencycoin.com/api/tasks \
```

**Expected**:
- Queue API returns tasks list (may be 0 if queue is empty)
- New task creates successfully, returns task ID
- `/pipeline` returns HTTP 200
- Browser visit shows: pending queue, dispatch controls panel (collapsible), provider performance section

**Edge case**: If `GET /api/providers/stats` returns empty/null, provider performance section shows "No data yet" rather than crashing.

### Scenario 5: Provider stats visible on /pipeline (not duplicated)

**Setup**: API running with recorded provider executions.

**Action**:
```bash

curl -s https://api.coherencycoin.com/api/providers/stats | jq '.summary'
```

## Risks and Assumptions

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Users have bookmarked `/automation`, `/usage`, `/remote-ops` | Medium | 301 redirects handle silently |
| `/nodes` page becomes too long / overwhelming | Medium | Use collapsible sections; fleet summary folded by default |
| `/pipeline` with dispatch + queue + provider stats becomes cluttered | High | Tab or accordion layout; dispatch controls hidden behind "New Task" toggle |
| Some data in `/usage` is not mirrored anywhere (e.g. friction signals) | Low | Audit during implementation — spec requires no data regression |
| API endpoints used by removed pages are not consumed elsewhere and get removed | Low | Backend endpoints stay; only frontend files change |
| `next.config.ts` redirect syntax differs between Next.js versions | Low | Verify with `next --version`; use `permanent: true` in `redirects()` array |
| Dispatch controls require AGENT_EXECUTE_TOKEN gate currently only on /remote-ops | Medium | Carry the token check into the /pipeline DispatchPanel |

## Known Gaps and Follow-up Tasks

- **`/usage` friction/quality signals**: These are unique to `/usage`. They should move to `/pipeline` Signals section. If they prove low-value, they may be removed in a follow-up PR rather than carried forward.
- **Mobile layout**: The enriched `/nodes` and `/pipeline` pages will be longer. A follow-up spec should validate mobile layout with the UX tab/mobile spec (`specs/ux-tabs-mobile-expert-novice`).
- **Metrics for success**: Add a page-view counter (or rely on Cloudflare analytics) to confirm `/automation`, `/usage`, `/remote-ops` traffic drops to zero after 2 weeks, validating that all users have migrated.
- **Stale sitemap**: `public/sitemap.xml` (if present) must remove the three old paths.
- **SEO**: 301 redirects are sufficient; search engines will update their index automatically.
- **Access control**: `/remote-ops` had an execute token gate. The dispatch controls moving to `/pipeline` must carry the same token check (env: `AGENT_EXECUTE_TOKEN`).
