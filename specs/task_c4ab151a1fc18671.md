# Specification: Universal Node + Edge Primitives with Typed Relationships

## Summary

Coherence Network models intelligence as a **fractal graph**: every durable thing—ideas, concepts, services, contributors, specs, implementations, domains—is a **Node** with a stable identity, metadata, and a **lifecycle state** (Ice / Water / Gas). Relationships between nodes are **directed, typed Edges** drawn from a small, versioned vocabulary. This spec defines the **data layer contract** that makes recursive decomposition, lineage, and resonance scoring possible without ad-hoc tables per entity type.

**Goal:** Provide one graph substrate (nodes + edges + lifecycle + typed predicates) so projections (REST, UI, CLI, reports) are thin adapters over the same atoms.

**Non-goals (this spec):** Full Neo4j/Cypher exposure to clients; automatic migration of every legacy resource in one release; ML-based edge inference (may emit suggested edges later).

---

## Purpose

Without universal primitives, each feature adds parallel schemas (“ideas table”, “specs table”, “links table”), breaking traversal, attribution, and cross-cutting queries. A single node/edge model with explicit lifecycle and edge types keeps **tiny deltas** (git-like patches on graph facts), supports **deterministic projections** (OpenAPI from the same meta-model), and lets operators prove value through **measurable graph health** (coverage, depth, contradiction detection), not anecdotal UI checks.

---

## Open Questions — Resolved in This Spec

### 1. What is the minimal set of edge types that covers meaningful relationships without being overwhelming?

**Answer:** Start from a **closed enum of 8 core types** that map to how work actually flows in OSS intelligence graphs, plus **one generic escape hatch**:

| `edge_type` | Meaning | Typical use |
|-------------|---------|-------------|
| `depends_on` | Hard prerequisite; removal/blocking | Service → DB; spec → upstream spec |
| `implements` | Realizes or satisfies | Code → spec; task → idea |
| `extends` | Adds surface without replacing | Domain B extends Domain A |
| `contradicts` | Incompatible claims or requirements | Spec A vs Spec B |
| `inspires` | Soft upstream influence (non-blocking) | Idea → idea; article → concept |
| `analogous_to` | Structural similarity, not identity | Pattern ↔ pattern |
| `parent_of` | Containment / hierarchy (tree edges) | Epic → story; domain → subdomain |
| `relates_to` | **Generic** association when no finer predicate fits; lowest semantic precision |

**Rationale:** Eight named predicates cover dependency, realization, hierarchy, conflict, influence, similarity, and catch-all. This stays below cognitive overload (~7±2) while avoiding a proliferation of near-duplicates (“`similar_to`”, “`like`”, “`see_also`” collapse into `relates_to` or `analogous_to` by convention documented in edge `metadata`).

**Direction conventions:** All types are **directed**. For symmetric notions (`analogous_to`, `contradicts`), store **one canonical direction** (lexicographically smaller `from_node_id` first) and enforce idempotency on the pair + type; APIs may mirror-read as undirected for UX.

**Extensibility:** New first-class types require a **spec revision** and a bump to `edge_types_version` (see Data model). Until then, use `relates_to` with `metadata.predicate_hint` for experimental predicates.

### 2. How can we improve this idea, show whether it is working, and make proof clearer over time?

**Improvements baked into the design:**

- **Proof metrics (operational, not vanity):** Expose counters and samples via API: nodes per `kind`, edges per `edge_type`, distribution of lifecycle states, mean shortest path depth from “root” domain nodes, count of `contradicts` edges unresolved by a later `extends` or `implements` chain (heuristic flags).
- **Time series:** Snapshot daily totals (`graph_stats` or metrics endpoint) so trend lines show adoption, not just point-in-time.
- **Contractual verification:** Pytest + production `curl` scenarios (below) gate releases; failing scenarios block deploy.
- **UX (future phase):** `/graph` explorer shows subset of nodes/edges with filters; not required for this spec’s acceptance but listed so reviewers know the surface area.

**Clarity over time:** Each release notes include `edge_types_version`, migration notes, and **one** “graph health” number (e.g. fraction of ideas with ≥1 `implements` edge to a spec or code node).

## Requirements

- [ ] **R1 — Node primitive:** Every first-class entity can be represented as a node with: `id`, `kind` (enum string, e.g. `idea`, `spec`, `implementation`, `service`, `contributor`, `concept`, `domain`), `lifecycle` ∈ {`ice`, `water`, `gas`}, `title`/`label`, `metadata` (object), `created_at`, `updated_at`.
- [ ] **R2 — Edge primitive:** Edges are directed, uniquely identified, with `from_node_id`, `to_node_id`, `edge_type` ∈ allowed set (§ Open Questions), optional `metadata`, `created_at`.
- [ ] **R3 — Lifecycle semantics:** **Ice** = persisted, canonical atom; **Water** = materialized view / working copy / draft projection; **Gas** = ephemeral or derived-only (not durable as source of truth). APIs must reject illegal transitions where listed (e.g. `gas` → `ice` only via explicit promote operation if we add one).
- [ ] **R4 — Registry:** `GET /api/graph/edge-types` returns the canonical list, JSON Schema or OpenAPI fragment for `edge_type`, and `version`.
- [ ] **R5 — CRUD + list:** HTTP API supports create/read/update for nodes, create/delete for edges, and list/filter with pagination (see API changes).
- [ ] **R6 — Validation:** Reject unknown `edge_type`; reject edges whose endpoints do not exist; optional: `kind` pairing rules in a later spec (v1 may allow any kind pair with `relates_to`).
- [ ] **R7 — Observability:** Implementation exposes at least one metrics or summary endpoint counting nodes/edges by type (may be combined with existing health/metrics patterns in the API).

## API Changes

Base path: **`/api/graph`** (namespaced to avoid collision with `/api/ideas`, `/api/federation/nodes`, etc.).

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/graph/edge-types` | List allowed `edge_type` values and `version`. |
| `POST` | `/api/graph/nodes` | Create node; body includes `kind`, `lifecycle`, `label`, optional `metadata`. |
| `GET` | `/api/graph/nodes/{node_id}` | Fetch node by id. |
| `PATCH` | `/api/graph/nodes/{node_id}` | Partial update (`label`, `lifecycle`, `metadata`). |
| `GET` | `/api/graph/nodes` | List/filter: `kind`, `lifecycle`, `q` (substring on label), `limit`, `cursor`. |
| `POST` | `/api/graph/edges` | Create edge: `from_node_id`, `to_node_id`, `edge_type`, optional `metadata`. |
| `GET` | `/api/graph/edges/{edge_id}` | Fetch edge. |
| `DELETE` | `/api/graph/edges/{edge_id}` | Remove edge. |
| `GET` | `/api/graph/nodes/{node_id}/edges` | Query param `direction`: `out` \| `in` \| `both` (default `out`). |
| `GET` | `/api/graph/stats` | Aggregate counts: nodes by `kind`, edges by `edge_type`, nodes by `lifecycle`. |

**Response shape:** All JSON responses use Pydantic models; errors follow FastAPI conventions (`422` validation, `404` missing resource, `409` conflict when applicable).

**Web (informational for roadmap):** **`/graph`** — explorer page listing nodes and edges with filters (implementation may follow in a separate web spec).

**CLI (informational):** Prefer **`cc graph`** subcommands when the Coherence CLI gains graph support, e.g. `cc graph node create`, `cc graph edge add`, `cc graph stats`. Until implemented, verification uses **`curl`** against `$API`.

## Data Model

```yaml
Node:
  id: { type: string, format: uuid }
  kind:
    type: string
    enum:
      - idea
      - concept
      - service
      - contributor
      - spec
      - implementation
      - domain
      # extensible with spec bump
  lifecycle:
    type: string
    enum: [ice, water, gas]
  label: { type: string, maxLength: 512 }
  metadata: { type: object, additionalProperties: true }
  created_at: { type: string, format: date-time }
  updated_at: { type: string, format: date-time }

Edge:
  id: { type: string, format: uuid }
  from_node_id: { type: string, format: uuid }
  to_node_id: { type: string, format: uuid }
  edge_type:
    type: string
    enum:
      - depends_on
      - implements
      - extends
      - contradicts
      - inspires
      - analogous_to
      - parent_of
      - relates_to
  metadata: { type: object, additionalProperties: true }
  created_at: { type: string, format: date-time }

EdgeTypeRegistry:
  version: { type: string, example: "2026.03.1" }
  types:
    - name: depends_on
      directed: true
      description: "Hard prerequisite or blocking dependency"

GraphStats:
  nodes_by_kind: { type: object, additionalProperties: integer }
  nodes_by_lifecycle: { type: object, additionalProperties: integer }
  edges_by_type: { type: object, additionalProperties: integer }
```

**Storage:** Implementation may use PostgreSQL (relational), Neo4j, or a hybrid; this spec does not mandate the store—it mandates **behavior and API**. Migrations must preserve stable `id` for nodes and edges.

## Verification Scenarios

**Environment:** `API=https://api.coherencycoin.com` (production reviewer run). Local: `API=http://localhost:8000`. Replace `$API` below.

### Scenario 1: Node create — read — update (full cycle)

- **Setup:** No graph node with label `Verification Node Alpha` exists yet (or use a fresh UUID in `metadata.run_id` for idempotency tracking).
- **Action:**
  ```bash
  N=$(curl -sS -X POST "$API/api/graph/nodes" -H "Content-Type: application/json" \
    -d '{"kind":"concept","lifecycle":"ice","label":"Verification Node Alpha","metadata":{"run":"proof-1"}}')
  echo "$N" | jq -e '.id,.lifecycle,.kind'
  ID=$(echo "$N" | jq -r '.id')
  curl -sS "$API/api/graph/nodes/$ID" | jq -e '.label=="Verification Node Alpha"'
  curl -sS -X PATCH "$API/api/graph/nodes/$ID" -H "Content-Type: application/json" \
    -d '{"lifecycle":"water"}' | jq -e '.lifecycle=="water"'
  curl -sS "$API/api/graph/nodes?kind=concept&limit=5" | jq --arg id "$ID" '.items[] | select(.id==$id) | .lifecycle'
  ```
- **Expected result:** `POST` returns **201** with JSON containing a UUID `id`, `kind`=`concept`, `lifecycle`=`ice`. `GET /api/graph/nodes/{id}` returns the same node. `PATCH` returns **200** with `lifecycle`=`water`. List response includes the node with updated lifecycle (shape may be `{ "items": [...] }` or equivalent documented list envelope).
- **Edge case:** `PATCH` with invalid `lifecycle` value (e.g. `"plasma"`) returns **422** with validation detail, not **500**.

### Scenario 2: Typed edge create — read — delete + stats

- **Setup:** Two existing node ids `N1` and `N2` (from Scenario 1 or two POSTs creating `spec` and `implementation` nodes).
- **Action:**
  ```bash
  E=$(curl -sS -X POST "$API/api/graph/edges" -H "Content-Type: application/json" \
    -d "{\"from_node_id\":\"$N1\",\"to_node_id\":\"$N2\",\"edge_type\":\"implements\",\"metadata\":{\"proof\":true}}")
  echo "$E" | jq -e '.id,.edge_type'
  EID=$(echo "$E" | jq -r '.id')
  curl -sS "$API/api/graph/edges/$EID" | jq -e '.edge_type=="implements"'
  curl -sS "$API/api/graph/nodes/$N1/edges?direction=out" | jq --arg e "$EID" '.[] | select(.id==$e) | .edge_type'
  curl -sS "$API/api/graph/stats" | jq -e '.edges_by_type.implements >= 1'
  curl -sS -X DELETE "$API/api/graph/edges/$EID" -w "\n%{http_code}\n"
  curl -sS "$API/api/graph/edges/$EID"
  ```
- **Expected result:** `POST` returns **201** with `edge_type`=`implements`. `GET` edge returns same. `GET .../edges` includes the new edge. `GET /api/graph/stats` shows non-zero `implements` count while edge exists. `DELETE` returns **204** or **200**. Subsequent `GET` on the edge returns **404**.
- **Edge case:** `POST` edge with non-existent `from_node_id` returns **404** or **422** with explicit message referencing missing node (must not return **500**).

### Scenario 3: Edge type registry

- **Setup:** None.
- **Action:** `curl -sS "$API/api/graph/edge-types" | jq -e '.version, (.types|length) >= 8'`
- **Expected result:** **200** JSON including `version` string and a list/array of type definitions covering at least the eight core types plus `relates_to`.
- **Edge case:** Response must be stable JSON (no HTML error page). If unauthenticated deploy blocks writes, **GET** registry must still succeed for reviewer (read-only public).

### Scenario 4: Invalid edge type (error handling)

- **Setup:** Valid node ids `N1`, `N2`.
- **Action:**
  ```bash
  curl -sS -o /dev/stderr -w "%{http_code}" -X POST "$API/api/graph/edges" \
    -H "Content-Type: application/json" \
    -d "{\"from_node_id\":\"$N1\",\"to_node_id\":\"$N2\",\"edge_type\":\"likes\"}"
  ```
- **Expected result:** HTTP **422** (unknown enum) or **400** with machine-readable `detail`; body mentions unknown `edge_type`.
- **Edge case:** Duplicate `POST` with identical `(from_node_id, to_node_id, edge_type)` either returns **409** or idempotent **200/201** with same `id`—behavior must be documented in implementation and tested; spec prefers **409** for accidental duplicates.

### Scenario 5: Missing node (error handling)

- **Setup:** Random UUID not present: `BAD=00000000-0000-4000-8000-000000000001` (v4 format).
- **Action:** `curl -sS -w "\n%{http_code}\n" "$API/api/graph/nodes/$BAD"`
- **Expected result:** HTTP **404**, JSON `detail` consistent with other API 404s.
- **Edge case:** Malformed UUID in path returns **422**, not **500**.

## Concurrency Behavior

- **Reads:** Safe under concurrent reads; list endpoints use cursor pagination.
- **Writes:** Define last-write-wins on `PATCH` for nodes unless ETags added later.
- **Edges:** Creating the same edge twice: prefer **409** or deterministic idempotent return; must be covered by tests.
