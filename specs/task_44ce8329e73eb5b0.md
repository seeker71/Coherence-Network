# Spec: SSE Control Channel — Real-Time Steer, Checkpoint, Abort, Ask-Permission for Native Agent CLIs

**Spec ID**: task_44ce8329e73eb5b0
**Status**: draft
**Date**: 2026-03-29

---

## Summary

Native agent CLIs (claude, codex, cursor agent, gemini) today run as black boxes: once launched, there is no way to steer them, request a checkpoint, or abort them gracefully without killing the process. This spec introduces an **SSE Control Channel** — a persistent, bidirectional control plane built on Server-Sent Events (SSE) that lets the runner and operators inject real-time commands (`checkpoint`, `steer`, `abort`, `ask`, `report`) into a running agent process, and receive structured responses back via `POST /api/agent/tasks/{task_id}/control-ack`.

The control channel makes every agent observable and steerable at any moment without changing the agent's core execution model. Agents receive commands via a **command file** that the SSE listener thread writes to, and agents respond by posting structured acknowledgements back through the API.

---

## Problem Statement

Current state:
- Agents run for 5–60 minutes with no real-time visibility beyond log tails.
- "Steer" today means waiting for task failure, then manually editing context and re-running.
- No mechanism to request a checkpoint mid-task (critical for long-running impls that may fail at step 9 of 10).
- "Abort" kills the OS process — no graceful cleanup, no partial result save.
- User-permission gates (`ask`) require out-of-band communication (Telegram, manual PATCH).

Target state with this spec:
- Operator types `cc steer task_abc123 "focus on failing tests in api/tests/"` and the running agent receives and acknowledges the steer within 2 seconds.
- Operator requests `cc checkpoint task_abc123` and within 5 seconds the agent writes `.task-checkpoint.md` and replies "checkpoint saved".
- Operator sends `cc abort task_abc123` and the agent commits in-progress work, posts a final status update, and exits cleanly within 10 seconds.
- Operator receives an `ask` event when the agent needs permission to perform a destructive action; the operator replies and the agent unpauses.

## Requirements

### R1 — SSE Control Stream Endpoint
The API must expose a **control stream** endpoint:

```
GET /api/agent/tasks/{task_id}/control-stream
```

- Returns `text/event-stream` (SSE) response.
- Streams `ControlCommand` events as they are enqueued.
- Stays open until task reaches a terminal state (`completed`, `failed`, `aborted`) or client disconnects.
- Sends a `keepalive` comment every 30 seconds to prevent proxy timeouts.
- Authentication: requires `X-Node-Token` header OR `?token=` query param (same auth as other agent endpoints).

### R2 — Control Command Submission
Operators and other nodes can submit commands via:

```
POST /api/agent/tasks/{task_id}/control
```

Accepted command types: `checkpoint`, `steer`, `abort`, `ask`, `report`.

Each command is queued and delivered to the SSE subscriber (the runner) atomically. If no subscriber is connected, commands are queued for up to 5 minutes and delivered when the runner reconnects.

### R3 — Runner SSE Listener Thread
The `agent_runner.py` runner must spawn an **SSE listener thread** at task start that:

1. Connects to `GET /api/agent/tasks/{task_id}/control-stream`.
2. Parses incoming `ControlCommand` JSON events.
3. Writes commands to a **command file** at `api/logs/task_{task_id}.cmd.json` (one JSON line per command, append mode).
4. Sets a **command event** (`threading.Event`) to wake the agent's main loop.
5. Reconnects automatically with exponential backoff (1s, 2s, 4s, max 30s) if the stream disconnects.

### R4 — Agent Command File Protocol
The agent process reads its command file on each loop iteration (or when the command event is set). Commands are processed in order:

| Command | Agent Action |
|---------|-------------|
| `checkpoint` | Flush progress to `.task-checkpoint.md`, POST ack with `{"status":"saved","path":".task-checkpoint.md"}` |
| `steer` | Append steer instructions to task context, POST ack with `{"status":"steering_applied","instruction":"<text>"}` |
| `abort` | Complete current atomic action, commit in-progress work, POST ack with `{"status":"aborting"}`, then exit with code 0 |
| `ask` | Pause execution, POST pending ack with `{"status":"paused","question":"<text>"}`, wait for `reply` command |
| `report` | POST current status including last step, file count, elapsed time |

### R5 — Control Acknowledgement Endpoint
Agents post responses back via:

```
POST /api/agent/tasks/{task_id}/control-ack
```

This endpoint stores acknowledgements in the activity log with `event_type = "control_ack"` and notifies any waiting `ask` poll.

### R6 — CC CLI Commands
The `cc` CLI must expose the following control surface:

```bash
cc checkpoint <task_id>                    # request checkpoint
cc steer <task_id> "<instruction>"         # steer with new direction
cc abort <task_id>                         # graceful abort
cc ask <task_id> "<question>"              # request user permission pause
cc control-status <task_id>               # show pending/sent commands and acks
```

### R7 — Command Queuing and Delivery Guarantee
- Commands are stored in a lightweight in-memory queue per task (backed by SQLite for persistence across API restarts, max 100 commands per task).
- Commands older than 5 minutes that have not been delivered are expired and marked `expired`.
- `GET /api/agent/tasks/{task_id}/control` returns the full command history with delivery status.

### R8 — Observability
- All control commands are logged as activity events (`event_type = "control_command"`) so they appear in `cc nodes` task views.
- The `GET /api/agent/tasks/{task_id}/stream` endpoint already returns all activity events including control acks — no additional endpoint needed for read-back.
- A new `GET /api/agent/tasks/{task_id}/control` endpoint returns command queue state (pending, delivered, acked, expired).

### R9 — Safety Guardrails
- `abort` command cannot be undone; the endpoint requires `{"confirm": true}` in the body.
- `steer` commands are appended, not replaced — a full steer history is preserved.
- If the runner is not connected to the control stream (no SSE subscriber), submitting an `abort` or `ask` returns `409 Conflict` with body `{"error":"no_subscriber","message":"Runner not connected to control stream for this task"}`.
- Maximum 1 pending `ask` command per task at a time; submitting a second returns `409`.

### R10 — Proof and Observability Improvement
To make it clear the control channel is working:

1. `GET /api/agent/tasks/{task_id}/control` returns a `channel_connected` boolean (true when SSE subscriber is active).
2. Runner logs `[SSE-CTRL] connected` and `[SSE-CTRL] received: <cmd>` at INFO level.
3. The web task detail page shows a "Control" panel with: connected status, command history, and a steer input box.
4. After every `checkpoint` ack, the `.task-checkpoint.md` file diff is stored in the activity log as `event_type = "checkpoint_saved"` so it's auditable.

## API Contract

### `GET /api/agent/tasks/{task_id}/control-stream`

**Headers**: `X-Node-Token: <token>` or `?token=<token>`

**Response 200** (SSE stream):
```
Content-Type: text/event-stream

data: {"event_type":"connected","task_id":"task_abc123","timestamp":"2026-03-29T10:00:00Z"}

: keepalive

data: {"event_type":"control_command","command_id":"cmd_001","command":"steer","payload":{"instruction":"focus on failing tests"},"timestamp":"2026-03-29T10:01:00Z"}

data: {"event_type":"end","task_id":"task_abc123","reason":"completed"}
```

**Response 404**: Task not found.
**Response 410**: Task already in terminal state (stream would be empty).

---

### `POST /api/agent/tasks/{task_id}/control`

**Request body**:
```json
{
  "command": "steer",
  "payload": {
    "instruction": "Focus on the failing test in api/tests/test_concepts.py"
  },
  "requester": "operator"
}
```

For `abort`:
```json
{
  "command": "abort",
  "payload": {"reason": "budget exceeded"},
  "confirm": true,
  "requester": "operator"
}
```

**Response 201**:
```json
{
  "command_id": "cmd_a1b2c3",
  "task_id": "task_abc123",
  "command": "steer",
  "status": "queued",
  "queued_at": "2026-03-29T10:01:00Z",
  "channel_connected": true
}
```

**Response 409**: No subscriber connected (for `abort`/`ask`), or duplicate `ask`.
**Response 422**: Invalid command type or missing required fields.
**Response 404**: Task not found.

---

### `POST /api/agent/tasks/{task_id}/control-ack`

**Request body**:
```json
{
  "command_id": "cmd_a1b2c3",

  "data": {

```

**Response 201**:
```json
{
  "ack_id": "ack_x1y2z3",
  "command_id": "cmd_a1b2c3",
  "recorded_at": "2026-03-29T10:01:05Z"
}
```

## Data Model

### `TaskControlCommand` (SQLite / in-memory)

```yaml
command_id: string (UUID)
task_id: string
command: enum[checkpoint, steer, abort, ask, report]
payload: JSON
requester: string
status: enum[queued, delivered, acked, expired, failed]
queued_at: datetime UTC
delivered_at: datetime UTC | null
acked_at: datetime UTC | null
expires_at: datetime UTC  # queued_at + 5 minutes
```

### `TaskControlAck` (activity log entry)

```yaml
event_type: "control_ack"
task_id: string
data:
  command_id: string
  command: string
  status: string
  data: JSON
```

## Runner Integration Detail

```python
# In agent_runner.py — spawned at task execution start

class SSEControlListener(threading.Thread):
    """Listens to /api/agent/tasks/{task_id}/control-stream and writes commands to command file."""

    def __init__(self, task_id: str, cmd_file: Path, event: threading.Event):
        super().__init__(daemon=True, name=f"sse-ctrl-{task_id[:8]}")
        self.task_id = task_id
        self.cmd_file = cmd_file
        self.stop_flag = threading.Event()
        self.command_event = event

    def run(self):
        backoff = 1
        while not self.stop_flag.is_set():
            try:
                self._connect_and_read()
                backoff = 1  # reset on clean disconnect
            except Exception as e:
                logger.warning(f"[SSE-CTRL] disconnected ({e}), retry in {backoff}s")
                time.sleep(backoff)
                backoff = min(backoff * 2, 30)

    def _connect_and_read(self):
        url = f"{BASE}/api/agent/tasks/{self.task_id}/control-stream"
        logger.info("[SSE-CTRL] connected")
        with httpx.stream("GET", url, timeout=None, headers={"X-Node-Token": NODE_TOKEN}) as r:
            for line in r.iter_lines():
                if self.stop_flag.is_set():
                    return
                if line.startswith("data:"):
                    data = json.loads(line[5:].strip())
                    if data.get("event_type") == "control_command":
                        logger.info(f"[SSE-CTRL] received: {data['command']}")
                        with open(self.cmd_file, "a") as f:
                            f.write(json.dumps(data) + "\n")
                        self.command_event.set()

# Usage in _run_task():
cmd_file = Path(LOG_DIR) / f"task_{task_id}.cmd.json"
cmd_event = threading.Event()
sse_listener = SSEControlListener(task_id, cmd_file, cmd_event)
sse_listener.start()

# In the execution loop, check cmd_event periodically:
if cmd_event.is_set():
    cmd_event.clear()
    _process_pending_commands(task_id, cmd_file)
```

## Verification Scenarios

### Scenario 1 — Steer a Running Task

**Setup**: Task `task_abc123` is `running`, runner is connected to control stream.

**Action**:
```bash
curl -s -X POST $API/api/agent/tasks/task_abc123/control \
  -H "Content-Type: application/json" \
  -d '{"command":"steer","payload":{"instruction":"Focus on failing tests only"},"requester":"operator"}'
```

**Expected result**:
- HTTP 201, body contains `"status":"queued"` and `"channel_connected":true`.
- Within 2 seconds: `GET $API/api/agent/tasks/task_abc123/control` shows command with `"status":"delivered"`.
- Within 5 seconds: status shows `"status":"acked"` and `"ack_data"` contains `"applied_at_step"`.

**Edge case**: If no runner is connected (channel_connected=false), POST returns 201 but status is `"queued"` not `"delivered"`. Command is held for up to 5 minutes. After 5 minutes it becomes `"expired"`.

---

### Scenario 2 — Checkpoint a Running Task

**Setup**: Task `task_abc123` is `running`. `.task-checkpoint.md` does not exist yet.

**Action**:
```bash
curl -s -X POST $API/api/agent/tasks/task_abc123/control \
  -H "Content-Type: application/json" \
  -d '{"command":"checkpoint","payload":{},"requester":"operator"}'
```

**Expected result**:
- HTTP 201 with `"command":"checkpoint"`, `"status":"queued"`.
- Within 5 seconds: `.task-checkpoint.md` exists and contains current progress.
- `GET $API/api/agent/tasks/task_abc123/stream` contains an event with `event_type="checkpoint_saved"` and the checkpoint diff in `data`.

**Edge case**: Sending 3 `checkpoint` commands rapidly should queue all 3. Each is processed in order. The third ack arrives within 15 seconds of the first submission. No checkpoints are lost.

---

### Scenario 3 — Graceful Abort

**Setup**: Task `task_abc123` is `running`. Runner has created 3 files in the worktree but has not committed.

**Action**:
```bash
curl -s -X POST $API/api/agent/tasks/task_abc123/control \
  -H "Content-Type: application/json" \
  -d '{"command":"abort","payload":{"reason":"budget exceeded"},"confirm":true,"requester":"operator"}'
```

**Expected result**:
- HTTP 201 with `"command":"abort"`, `"status":"queued"`.
- Within 10 seconds: runner commits in-progress files, posts control-ack with `"status":"aborting"`.
- Task transitions to `"aborted"` status in `GET $API/api/agent/tasks/task_abc123`.
- The 3 created files are in a commit on the worktree branch (work is not lost).
- SSE stream sends `{"event_type":"end","reason":"aborted"}` and closes.

**Edge case**: `abort` without `"confirm":true` returns HTTP 422 with error `"confirm_required"`. Task is NOT aborted.

---

### Scenario 4 — Ask-Permission Pause and Reply

**Setup**: Task `task_abc123` is `running`. Agent has reached a step requiring user permission for a destructive action.

**Agent action** (runner sends internally):
```bash
curl -s -X POST $API/api/agent/tasks/task_abc123/control \
  -H "Content-Type: application/json" \
  -d '{"command":"ask","payload":{"question":"May I drop the test database and recreate it?"},"requester":"runner"}'
```

**Expected result**:
- Task transitions to `"awaiting_permission"` sub-state visible in `GET $API/api/agent/tasks/task_abc123`.
- `GET $API/api/agent/tasks/task_abc123/control` shows `"status":"delivered"` for the ask command.
- Operator sends reply: `cc ask task_abc123 "yes, proceed"` (which posts `control-ack` with `{"command":"ask","status":"replied","data":{"answer":"yes, proceed"}}`).
- Agent unpauses within 2 seconds of the reply.
- Task returns to `"running"` sub-state.

**Edge case**: Submitting a second `ask` while one is already pending returns HTTP 409 with `{"error":"ask_pending","message":"A pending ask already exists for this task"}`.

---

### Scenario 5 — No Subscriber Error Handling

**Setup**: Task `task_abc123` is `running` but the runner disconnected from the control stream (e.g., network failure). `channel_connected = false`.

**Action**:
```bash
curl -s -X POST $API/api/agent/tasks/task_abc123/control \
  -H "Content-Type: application/json" \
  -d '{"command":"abort","payload":{"reason":"test"},"confirm":true,"requester":"operator"}'
```

**Expected result**:
- HTTP 409 with `{"error":"no_subscriber","message":"Runner not connected to control stream for this task"}`.
- Task is NOT aborted.
- The abort command is NOT queued (abort requires live connection).

**Edge case**: `steer` and `checkpoint` ARE queued even without a subscriber (they are non-destructive and can be delivered when the runner reconnects). Only `abort` and pending `ask` require a live subscriber.

## Risks and Assumptions

| Risk | Mitigation |
|------|------------|
| Runner cannot maintain persistent SSE connection across load balancers / proxies | Use `X-Accel-Buffering: no` + 30s keepalive; document `AGENT_SSE_TIMEOUT` env override |
| Command file grows unbounded for long tasks | Rotate command file after 1000 lines; archive to `task_{id}.cmd.archive.json` |
| SSE listener thread leaks if runner crashes | Thread is daemon=True; OS reclaims. Listener state is ephemeral — reconnection re-establishes state from command queue |
| Multiple runners claim the same task | Existing claim semantics unchanged; control stream is per-task-id, not per-runner |
| Agent ignores command file (black-box providers) | For fully opaque providers, the runner wraps the process and injects steer as updated context on next retry; abort sends SIGTERM then SIGKILL |
| ask-pause deadlock if operator never replies | ask commands expire after `ASK_TIMEOUT` (default 10 minutes); task transitions to `failed` with reason `ask_timeout` |

## Known Gaps and Follow-up Tasks

- **Web UI control panel**: Task detail page should show channel_connected status, command history, and a steer input. (Follow-up: spec 183+)
- **claude native integration**: Claude CLI supports `--stream-output` and context injection. For full steer support, test whether context file injection triggers mid-session re-routing.
- **codex integration**: OpenAI Codex CLI may not support mid-execution context injection; stub for now, implement proper support in follow-up.
- **Multi-node broadcast**: When multiple nodes pick up the same task (rare), all should receive the abort command. Add `broadcast: true` option to control endpoint.
- **Rate limiting**: Protect control endpoint from flooding (max 10 commands/minute per task).
- **CLI cc control-status output**: Needs pretty-print table showing command history with timestamps and delivery status.

## Open Questions — Addressed

**Q: How can we improve this idea, show whether it is working, and make that proof clearer over time?**

**Evidence trail strategy**:

1. **Operational proof** — `GET /api/agent/tasks/{task_id}/control` shows a command history with timestamps for `queued → delivered → acked`. This is machine-readable and can be polled from CI or the web dashboard. Zero acked commands = feature is not working. N acked commands = feature is working for N interactions.

2. **Activity log integration** — Every control command and ack appears in `GET /api/agent/tasks/{task_id}/stream` as a timestamped event. The existing nodes dashboard already renders activity streams, so control interactions become first-class observable events.

3. **Latency metric** — For each acked command, store `delivered_at - queued_at` (queue latency) and `acked_at - delivered_at` (agent response latency). Expose via `GET /api/agent/tasks/{task_id}/control` in a `latency_ms` field. Target: queue latency < 500ms, agent response < 5s.

4. **Integration test in CI** — `test_agent_control.py` uses a mock runner that subscribes to the control stream, receives a steer command, and posts an ack. CI passes only if the round-trip completes in under 5 seconds. This is a runnable proof that the channel works end-to-end.

5. **Improvement path** — Start with in-memory queue (no persistence). Once we have 10+ successful round-trips in staging, add SQLite persistence for cross-restart delivery. Once persistence is stable, add the web UI control panel. Proof accumulates incrementally: latency histogram, acked command count, and CI green = channel is healthy.
