## Summary

Coherence Network should be discoverable where developers look for MCP servers and agent skills: **Smithery**, **Glama** (via `punkpeye/awesome-mcp-servers` pull request), **PulseMCP**, **mcp.so**, **skills.sh**, and **askill.sh** — **six** surfaces, exceeding the “5+ registries” bar. This spec defines the **product outcome** (listed everywhere that matters, with evidence of installs/downloads), ties it to **concrete API contracts** for readiness and stats, and specifies **verification scenarios** a reviewer can run against production.

Success is not “we submitted forms once.” Success is **provable**: repository assets validate as submission-ready, external registries show non-zero engagement where APIs exist, and the platform exposes those signals so the team can see trends over time.

---

## Requirements

1. **Coverage:** Inventory and operational checklist MUST include all six targets: `smithery`, `glama`, `pulsemcp`, `mcp-so`, `skills-sh`, `askill-sh`.
2. **Readiness API:** `GET /api/discovery/registry-submissions` MUST return one item per target with `registry_id`, `status`, `missing_files`, and a summary flag indicating when at least five registries are asset-complete (see `180-mcp-skill-registry-submission.md` for `core_requirement_met` semantics).
3. **Stats API (post-implementation):** `GET /api/discovery/registry-stats` MUST return per-registry install/download/star fields where data is obtainable, with explicit `source` (`live` | `cached` | `unavailable`) so “no data” is not confused with “zero installs.”
4. **Unified view (post-implementation):** `GET /api/discovery/registry-dashboard` MUST merge readiness + stats in one payload for dashboards and CLI summaries.
5. **Tracking:** Install/download counts MUST be persisted or cached with timestamps so week-over-week comparison is possible (initially file cache per 180; optional DB time series as follow-up).
6. **Documentation:** Human submission steps (PR URLs, form links, last submission date) MUST live in-repo (`mcp-server/README.md`, `skills/coherence-network/SKILL.md`, and/or `docs/registry-submissions.json` inventory) and be reflected in API `notes` where applicable.

Aligned with `RegistrySubmissionInventory` / per-item models in `api/app/models/registry_discovery.py` — each registry row includes `registry_id`, `registry_name`, `category` (`mcp` | `skill`), `status`, `missing_files`, `install_hint`.

### Stats and dashboard (planned)

See Spec 180: `RegistryStats`, `RegistryStatsSummary`, `RegistryStatsList`, `RegistryDashboard`, `RegistryDashboardItem` — includes `install_count`, `download_count`, `star_count`, `fetched_at`, `source`, `error`.

### Ideas (milestones)

Use existing `Idea` / `IdeaWithScore` models: store tags such as `registry`, `smithery`, `submitted` and body text with links to PRs and registry listing URLs for auditability.

## How We Improve the Idea, Show It Is Working, and Make Proof Clearer Over Time

| Layer | What “working” means | How we make it clearer |
|-------|----------------------|-------------------------|
| **Assets** | Required files exist in-repo | `/api/discovery/registry-submissions` → `submission_ready` and `core_requirement_met` |
| **Listing** | PRs merged or packages live on Smithery / MCP.so / skills directories | Manual `notes` + links on idea; future: `notes` populated from `docs/registry-submissions.json` |
| **Demand** | Non-null installs/downloads or stars on trackable registries | `/api/discovery/registry-stats` with `source: live` |
| **Trend** | Counts increase week over week | Weekly `?refresh=true` snapshots; optional PostgreSQL table for sparklines |
| **Narrative** | Stakeholders see one screen | `/api/discovery/registry-dashboard` (then web `/discovery`) |

**Open questions (to resolve during implementation):**

1. **mcp.so / skills.sh / askill.sh:** Often no public stats API — do we accept “listed” as proof via URL in `notes`, or require a screenshot hash in-repo?
2. **Glama:** Star count on GitHub repo is a proxy, not installs — is that an acceptable KPI until Glama exposes installs?
3. **Single source of truth:** Should `docs/registry-submissions.json` become the canonical list of external URLs, or only API-built inventory?

## Verification Scenarios

### Scenario 1 — Readiness inventory lists six registries (read)

**Setup:** API deployed with registry discovery service enabled (production or local with `uvicorn`).

**Action:**

```bash
curl -s https://api.coherencycoin.com/api/discovery/registry-submissions | python3 -c "import sys,json; d=json.load(sys.stdin); print(sorted([i['registry_id'] for i in d['items']]))"
```

**Expected:** A list containing exactly these six strings: `askill-sh`, `glama`, `mcp-so`, `pulsemcp`, `skills-sh`, `smithery` (order may vary).

**Edge:** If a deployment lags behind Spec 180, fewer than six IDs return — **fails** until deployment catches up.

---

### Scenario 2 — Full create–read–update cycle for a discovery milestone idea

**Setup:** Valid API base URL and auth if required for mutating ideas (use public API if mutations are open; otherwise test token).

**Action (create):**

```bash
curl -s -X POST https://api.coherencycoin.com/api/ideas \
  -H "Content-Type: application/json" \
  -d '{"title":"Registry: Smithery submission","description":"Track Smithery listing and install counts","tags":["registry","smithery","discovery"]}'
```

**Expected:** HTTP `201`, JSON includes `id` (e.g. `idea-...`).

**Action (read):**

```bash
IDEA_ID=<id from create>
curl -s "https://api.coherencycoin.com/api/ideas/${IDEA_ID}"
```

**Expected:** HTTP `200`, same `title`, `tags` contain `smithery`.

**Action (update):**

```bash
curl -s -X PATCH "https://api.coherencycoin.com/api/ideas/${IDEA_ID}" \
  -H "Content-Type: application/json" \
  -d '{"description":"Smithery live: https://smithery.ai/server/..."}'
```

**Expected:** HTTP `200`, `description` reflects the new text.

**Edge:** `PATCH` with invalid `idea_id` → HTTP `404` (not `500`).

---

### Scenario 3 — Stats endpoint behavior and error handling (post-implementation)

**Setup:** Endpoints from Spec 180 deployed.

**Action:**

```bash
curl -s -o /dev/null -w "%{http_code}" "https://api.coherencycoin.com/api/discovery/registry-stats?registry_id=smithery"
```

**Expected:** HTTP `200`. Body includes `items` with `registry_id` `smithery`, `source` in `live`, `cached`, or `unavailable`.

**Edge — bad filter:**

```bash
curl -s "https://api.coherencycoin.com/api/discovery/registry-stats?registry_id=not-a-real-registry"
```

**Expected:** HTTP `200` with empty filter result **or** HTTP `422` with validation detail — **not** HTTP `500`. (Exact behavior must match implementation; document in OpenAPI.)

---

### Scenario 4 — Dashboard degrades gracefully when upstream stats fail

**Setup:** Spec 180 dashboard route deployed.

**Action:**

```bash
curl -s -w "\n%{http_code}" https://api.coherencycoin.com/api/discovery/registry-dashboard | tail -1
```

**Expected:** Last line is `200` even when all stat sources are `unavailable`.

**Edge:** Submission section still reflects `missing_assets` vs `submission_ready` from disk checks.

---

### Scenario 5 — Core requirement flag matches asset reality

**Setup:** Known repo state — all MCP and skill assets present per Spec 180 checklist.

**Action:**

```bash
curl -s https://api.coherencycoin.com/api/discovery/registry-submissions | python3 -c "import sys,json; d=json.load(sys.stdin); print(d['summary'].get('core_requirement_met'), d['summary'].get('submission_ready_count', d['summary']))"
```

**Expected:** `True` and a count ≥ `5` when at least five registries are `submission_ready`.

**Edge:** Remove one required file in a dev branch → `core_requirement_met` becomes `False` and `missing_files` non-empty for affected registries.

## Risks and Assumptions

| Risk | Impact | Mitigation |
|------|--------|------------|
| Third-party registry APIs change | Broken stats | `source: unavailable`, cached last-known values |
| PRs to awesome lists delayed | No “live” proof | Track PR URL on idea; `submission_ready` still shows our side is done |
| No install API for skill directories | Incomplete numeric proof | Use listing URL + manual monthly check until APIs exist |
| Scope creep | PM spec becomes implementation | Keep file list authoritative in Spec 180 |
