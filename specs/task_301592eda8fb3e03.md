## Summary

Coherence Network exposes rich agent surfaces (MCP tools, Cursor skills, CLI workflows), but **discoverability is gated by repository familiarity**. Third-party **MCP registries** and **skill directories** are where practitioners search for integrations. This spec defines a repeatable program to **register** the Coherence Network MCP server and the canonical **Coherence Network skill** on **at least five** independent discovery surfaces, and to **track install and download counts** so we can prove adoption and iterate.

### Goal

1. **Publish** the MCP server and skill to: **Smithery**, **Glama** (via contribution to `awesome-mcp-servers` or Glama’s listing flow), **PulseMCP**, **mcp.so**, **skills.sh**, and **askill.sh** (six targets — exceeds the “5+” bar with one redundancy buffer).
2. **Instrument** discovery: capture baseline and ongoing **install / download / clone** signals per registry where APIs or dashboards exist; where they do not, capture **screenshot or export evidence** on a fixed cadence.
3. **Centralize proof** in-repo and (when implemented) via API so reviewers and operators can answer: *Is distribution working? Is it improving week over week?*

### Open Questions (Addressed in This Spec)

| Question | Resolution in this spec |
|----------|---------------------------|
| How can we improve this idea over time? | Versioned **distribution playbook** (`docs/DISTRIBUTION-MCP-SKILLS.md`), quarterly **registry landscape review**, and **A/B** short descriptions per registry when allowed. |
| How do we show whether it is working yet? | **Verification Scenarios** below + **metrics snapshot** (automated or manual) with pass/fail gates; “working” = listings live + non-zero signal or confirmed indexing within SLA. |
| How do we make proof clearer over time? | Single **source of truth JSON** + optional **GET** endpoint; **trend** fields (`week_ago`, `delta`); **evidence URLs** per registry. |

---

## Purpose

Without registry presence, the network’s agent integrations are invisible to developers who search “MCP coherence” or “Cursor skill graph” on aggregators. Listing on multiple registries reduces single-point dependency (one directory delisting should not erase discoverability). Measurable install/download counts turn advocacy into **falsifiable** operations: if counts stay flat while tasks grow, distribution messaging or packaging must change.

## Requirements

### R1 — Registry coverage (minimum five distinct ecosystems)

The program MUST deliver **verified listings** on all of the following **six** targets (five plus buffer):

| # | Registry / channel | Asset | Mechanism (typical) |
|---|-------------------|-------|---------------------|
| 1 | **Smithery** | MCP server | Smithery package manifest + publish flow |
| 2 | **Glama** | MCP server | PR to [awesome-mcp-servers](https://github.com/punkpeye/awesome-mcp-servers) and/or Glama UI listing per current process |
| 3 | **PulseMCP** | MCP server | Submission per PulseMCP guidelines |
| 4 | **mcp.so** | MCP server | Directory submission / claim |
| 5 | **skills.sh** | Cursor skill | Skill metadata + registry submission |
| 6 | **askill.sh** | Cursor skill | Skill metadata + registry submission |

**Coherence Network skill** refers to the packaged skill under `.cursor/skills/` (or successor path documented in `AGENTS.md` / `CLAUDE.md`) with a stable **name** and **description** suitable for directories.

**MCP server** refers to the published MCP entrypoint documented for this repo (Dockerfile or `stdio` command string, env vars, tool list) — exact command copied into each registry’s required format.

### R2 — Single source of truth for status and metrics

- **Authoritative file** (implementation phase): `docs/distribution/registry_state.json`  
  - Schema includes: `registries[]` with `id`, `display_name`, `asset_type` (`mcp` \| `skill` \| `both`), `listing_url`, `submitted_at`, `verified_live_at`, `last_checked_at`, `status` (`draft` \| `submitted` \| `live` \| `rejected` \| `delisted`), `metrics` (`installs`, `downloads`, `stars`, `clones` — nullable per platform).
- **Optional API** (recommended): `GET /api/distribution/registry-state` returns the same payload as the JSON file (read-through from deployed artifact or DB).

### R3 — Tracking install / download counts

- **Where APIs exist**: automate fetch (scheduled job or manual script) and write into `registry_state.json` with `captured_at` timestamps.
- **Where APIs do not exist**: manual **monthly** capture with `evidence_url` (screenshot stored in `docs/distribution/evidence/YYYY-MM/` — filenames only, no large binaries in git if policy forbids; prefer URLs to vendor dashboards).
- **Definition of “track”**: every `live` registry row MUST have either a numeric metric **or** a documented `metrics_unavailable_reason` and a `last_manual_proof_at` date.

### R4 — Naming and consistency

- **Public name**: `Coherence Network` (or exact string approved in `docs/DISTRIBUTION-MCP-SKILLS.md`).
- **MCP identifier**: stable slug (e.g. `coherence-network`) across Smithery, mcp.so, PulseMCP, Glama listing.
- **Skill identifier**: match `.cursor/skills/` folder name and SKILL.md title.

### R5 — Security and compliance

- No secrets in registry listings; environment variables documented as **names only**.
- Link to **license** and **privacy** summary as required by each platform.

## API Changes (Proposed)

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/distribution/registry-state` | Returns `registry_state` document (Pydantic model), HTTP 200. |
| `GET` | `/api/health` | Unchanged; used in scenarios to confirm deployment includes new routes. |

**Response 200** (`GET /api/distribution/registry-state`):

```json
  "schema_version": "1.0.0",
  "updated_at": "2026-03-28T12:00:00Z",
  "registries": [
    {
      "id": "smithery",
      "display_name": "Smithery",
      "asset_types": ["mcp"],
      "listing_url": "https://smithery.ai/server/...",
      "status": "live",
      "metrics": { "installs": 42, "downloads": null },
      "metrics_captured_at": "2026-03-28T00:00:00Z",
      "evidence_url": null
    }
  ]

```

**Response 404** (optional flag): if feature disabled in env — only if product decision requires; default is 200 with empty `registries` during rollout.

*N/A for pure documentation phase*: until implementation lands, verification uses the JSON file path and public URLs.

## Verification Scenarios

### Scenario 1 — Full create–read–update cycle for registry state (file or API)

- **Setup**: Branch includes `docs/distribution/registry_state.json` with `schema_version` `1.0.0` and at least one registry `id: "smithery"` in `draft`.
- **Action**: Update the Smithery row to `status: live`, set `listing_url` to the real Smithery URL, set `verified_live_at` to current UTC ISO timestamp; save. If API exists: `curl -sS "https://api.coherencycoin.com/api/distribution/registry-state" | jq '.registries[] | select(.id=="smithery")'`
- **Expected**: File on disk (or API JSON) shows `status: "live"` and `listing_url` matches `https://smithery.ai/` prefix pattern for server pages; `updated_at` is newer than prior commit.
- **Edge (bad input)**: Insert invalid `status: "bogus"` in a test branch — validation MUST reject (implementation: Pydantic 422 or CI schema check); restore valid enum before merge.
- **Edge (missing resource)**: If API enabled but file missing — `GET /api/distribution/registry-state` returns 200 with `registries: []` and `updated_at` reflecting build time **or** 503 with explicit `detail` (document which behavior is chosen in implementation).

### Scenario 2 — Health and deployment proof

- **Setup**: Production deploy includes distribution router (if API in scope).
- **Action**: `curl -sS -o /dev/null -w "%{http_code}" "https://api.coherencycoin.com/api/health"`
- **Expected**: HTTP `200` and body contains `"status": "ok"` (or project-standard health shape).
- **Edge**: Wrong host — expect connection failure, not 500 from app (proves DNS/ TLS, not app logic).

### Scenario 3 — Metrics capture fidelity

- **Setup**: One registry exposes a public counter (e.g. Smithery install count visible on page).
- **Action**: Manual or script: compare `registry_state.json` → `registries[].metrics.installs` to dashboard; re-run after 7 days.
- **Expected**: `metrics_captured_at` advances; value is within plausible range (± rounding); `delta` documented in playbook if automated.
- **Edge (duplicate request)**: Running capture script twice same day does not corrupt JSON — second run preserves `metrics_captured_at` or uses idempotent merge (implementation detail).

### Scenario 4 — Skill registry listing (skills.sh / askill.sh)

- **Setup**: Skill package exists at repo path documented in playbook.
- **Action**: Open `listing_url` for skills.sh and askill.sh entries in browser; confirm title contains **Coherence Network** and description matches SKILL.md first paragraph.
- **Expected**: HTTP 200 on listing pages; no broken install command.
- **Edge**: Delisted — `status` becomes `delisted`, playbook “reinstatement” section followed; **Expected**: new ticket or PR reference in `registry_state.json` notes field (optional extension: `notes` string).

### Scenario 5 — Glama / awesome-mcp-servers traceability

- **Setup**: PR merged to `punkpeye/awesome-mcp-servers` (or Glama-equivalent proof).
- **Action**: `curl -sS "https://raw.githubusercontent.com/punkpeye/awesome-mcp-servers/main/README.md" | grep -i "coherence"` (or follow PR URL from `evidence_url`).
- **Expected**: Line mentions Coherence Network MCP with link to repo or Smithery.
- **Edge**: Upstream rename — `listing_url` updated within one release cycle; old URL returns 404 — playbook documents check.

## Risks and Assumptions

- **Assumption**: Each registry accepts open-source listings without paid placement; if not, **scope** reduces to “maximum achievable free listings” with explicit `rejected` rows.
- **Risk**: Vendor metrics APIs change — mitigate with versioned fetcher script and `metrics_unavailable_reason`.
- **Risk**: `awesome-mcp-servers` PR backlog — mitigate by opening PR early and linking PR URL in `registry_state` while `status: submitted`.

- Automate metrics polling per vendor when tokens/API keys are available; until then manual monthly capture.
- Optional public web page `/distribution` for non-admin stakeholders — not required for MVP.

## Out of Scope

- Paid promotion / featured slots on registries.
- Localization of skill text beyond English for v1.
