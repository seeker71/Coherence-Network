## Summary

Ship the Coherence Network MCP server as **installable, versioned packages** on **npm** and **PyPI**, with a **typed `server.json` manifest** compatible with the **Official MCP Registry** schema and **mcp-publisher** workflows, so the server appears on **registry.modelcontextprotocol.io**, **Smithery**, **Glama** (`awesome-mcp-servers`), **MCP.so**, and similar discovery surfaces.

The repository already contains:

- **Node implementation:** `mcp-server/package.json`, `mcp-server/index.mjs`, `mcp-server/README.md` — `npx coherence-mcp-server`.
- **Manifest:** `mcp-server/server.json` — `$schema` from `registry.modelcontextprotocol.io`, `packages[]` with npm registry entry.
- **Python implementation (stdio):** `api/mcp_server.py` — `python -m api.mcp_server` when run from a checkout with `api/` on `PYTHONPATH`.
- **Readiness API:** `GET /api/discovery/registry-submissions` — validates on-disk assets for npm, Smithery, MCP Registry, etc. (`api/app/services/registry_discovery_service.py`).

**Gap this spec closes:** formal **release engineering** (version alignment, CI publish, PyPI surface, manifest updates for dual packages), **operational proof** that installs work and versions match registries, and **longitudinal metrics** so “is it working?” is answerable with machine-checkable evidence.

## Requirements

### Functional

1. **Single source of truth for version** — One release process bumps **npm `package.json`**, **`server.json` `version`**, **`server.json` `packages[].version`**, and **PyPI `pyproject.toml` / `__version__`** together (script or documented checklist; CI guard recommended).
2. **npm package** — `npm publish` (or trusted automation) produces a package whose `bin` matches `registry_discovery_service._npm_package_ready()` expectations (`coherence-mcp-server` → `index.mjs`).
3. **PyPI package** — Wheel + sdist on PyPI; runtime dependencies include the official Python **`mcp`** SDK and whatever minimal shared code is required to run the same tool surface as `api/mcp_server.py` (either package the `app.services.mcp_tool_registry` graph inside the wheel or depend on a published `coherence-api` slice — **decision in implementation**, must not duplicate divergent tool definitions).
4. **server.json for mcp-publisher** — Compliant `server.json` at `mcp-server/server.json` with:
   - `$schema`: `https://registry.modelcontextprotocol.io/schemas/server.json`
   - `name`, `version`, `description`, `repository`, `license`, `packages[]` including **`registry`: `npm`** and **`registry`: `pypi`** (or schema-supported identifier — align with current MCP registry docs at publish time).
5. **MCP Registry / Smithery / Glama** — Follow each platform’s current submission checklist (PR, CLI, or dashboard); link PR URLs / listing IDs in `README.md` or `docs/` as required by project conventions.

### Non-functional

- **MIT license** consistent with existing `mcp-server/package.json`.
- **No secrets** in packages; API keys remain env-based (`COHERENCE_API_URL`, `COHERENCE_API_KEY`).

## Verification Scenarios

Concrete scenarios a reviewer can run against **production** and **public registries** after implementation.

### Scenario 1 — npm: resolve, install, list tools (full read path)

- **Setup:** Released version `V` on npm (`coherence-mcp-server`). Clean machine or temp directory with Node ≥ 18.
- **Action:**  
  `npm view coherence-mcp-server version`  
  `npx --yes coherence-mcp-server@V --help` *(or MCP client invocation per README if no `--help`)*  
  Use any MCP inspector / client to connect via stdio and send `tools/list`.
- **Expected:** npm shows version `V`; process starts without import errors; `tools/list` returns non-empty tool array including names matching `mcp-server/server.json` `tools[]` (allow prefix conventions documented in README).
- **Edge:** Invalid Node version (< engines) — process exits with clear error (non-zero) and message; missing `COHERENCE_API_URL` — defaults to `https://api.coherencycoin.com` and health-backed tools still behave per README.

### Scenario 2 — PyPI: install, run console script, smoke tool call

- **Setup:** Released `coherence-mcp-server` `V` on PyPI. Python 3.11+ venv.
- **Action:**  
  `pip install coherence-mcp-server==V`  
  Run published entry point (documented in PyPI readme), e.g. `coherence-mcp-server` or `python -m coherence_mcp_server`, with stdio MCP handshake; call one read-only tool (e.g. list specs or health-dependent tool).
- **Expected:** Package installs without compiling fragile native deps; MCP session initializes; at least one tool returns structured JSON without 500 from Coherence API when API is up.
- **Edge:** Wrong Python version — `pip` fails with `Requires-Python` message; **conflicting package name** — document disambiguation from unrelated PyPI projects if any.

### Scenario 3 — Registry API: submission readiness (create-read-update cycle via repo state)

- **Setup:** Repo `main` contains `mcp-server/server.json` and `mcp-server/package.json` with versions aligned to published `V`.
- **Action:**  
  `curl -sS https://api.coherencycoin.com/api/discovery/registry-submissions | jq .`  
  Inspect `items[]` for `registry_id` `npm`, `smithery`, `modelcontextprotocol-registry`.
- **Expected:** Each relevant item has `status` == `submission_ready` (or project-defined equivalent), `missing_files` empty, `install_hint` contains `npx coherence-mcp-server`.
- **Edge:** Deliberately delete `mcp-server/server.json` locally — inventory shows `missing_files` including that path; restoring file returns to ready (proves detector works).

### Scenario 4 — External registry JSON: version parity (error handling)

- **Setup:** Published versions on npm and PyPI.
- **Action:**  
  `curl -sS https://registry.npmjs.org/coherence-mcp-server | jq -r '.["dist-tags"].latest'`  
  `curl -sS https://pypi.org/pypi/coherence-mcp-server/json | jq -r '.info.version'`  
  Compare to `version` in `mcp-server/server.json` on `main`.
- **Expected:** All three match **release tag** policy (exact match for same release train).
- **Edge:** Pre-release tag on npm (`next`) — document policy: either ignore `dist-tags.next` in parity check or include in automation; **unpublished package** — curl returns 404; CI should fail closed before claiming success.

### Scenario 5 — API health for HTTP-backed MCP tools (bad input / API down)

- **Setup:** Node MCP server running with default API base.
- **Action:**  
  `curl -sS -o /dev/null -w "%{http_code}" https://api.coherencycoin.com/api/health`  
  Invoke a tool with **invalid arguments** (e.g. unknown idea id) via MCP `tools/call`.
- **Expected:** Health returns `200` when platform is healthy; tool returns **structured error** in tool payload (not process crash), e.g. JSON with `error` field.
- **Edge:** Simulate wrong `COHERENCE_API_URL` — tools return API error object, not uncaught exception in MCP server.

## Risks and Assumptions

- **Risk:** npm and PyPI **name squatting** — verify ownership of `coherence-mcp-server` on both ecosystems before first publish.
- **Risk:** **Dual codepaths** (Node vs Python) diverge — mitigate with shared OpenAPI-derived tool list or contract tests.
- **Assumption:** Coherence API at `https://api.coherencycoin.com` remains the default public base for smoke tests.
- **Assumption:** Official MCP registry `packages[]` schema accepts multiple registries; **validate at publish time** against live schema.
