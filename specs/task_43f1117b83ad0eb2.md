## Summary

**Goal:** Maximize discoverability of the Coherence Network **MCP server** and **portable skills** by submitting them to **at least five** independent public registries, and make **install/download traction measurable** so the org can prove the program is working and improving over time.

**Primary targets (named in the task):**

| # | Surface | Kind | Typical submission path |
|---|---------|------|-------------------------|
| 1 | **Smithery** | MCP | Package listing + manifest (often via `server.json` / npm alignment) |
| 2 | **Glama** | MCP | PR to [`punkpeye/awesome-mcp-servers`](https://github.com/punkpeye/awesome-mcp-servers) (or equivalent curated list) |
| 3 | **PulseMCP** | MCP | Catalog PR / registry descriptor |
| 4 | **mcp.so** | MCP | Web submission + canonical repo artifacts |
| 5 | **skills.sh** | Skill | Directory PR or submission flow for `skills/coherence-network/` |
| 6 | **askill.sh** | Skill | Secondary skill index using the same portable skill package |

**Related in-repo surfaces (already tracked in inventory):** The live API inventory in `docs/registry-submissions.json` also tracks **npm**, **Model Context Protocol Registry**, **ClawHub**, **Glama**, **PulseMCP**, **skills.sh**, and **askill.sh**. The **Smithery** and **mcp.so** rows should be added there when submission work begins so a single endpoint reflects every target.

**Outcome:** A reviewer can (1) confirm assets and proofs exist via the API, (2) confirm external listings/PRs via URLs recorded as evidence, and (3) after the stats layer from spec 180 ships, confirm numeric adoption via dedicated endpoints.

---

## Purpose

OSS tools die in obscurity. Registries are the **search indexes** for MCP servers and skills. Submitting Coherence Network artifacts to **five or more** registries reduces friction for integrators and agents, and—when combined with metrics—gives a **leading indicator** that distribution strategy works. This spec ties human submission work to **machine-checkable readiness** and **time-bounded proof** (counts, stars, or cached snapshots).

- [ ] **R8 — Stats (spec 180):** After implementation, `GET /api/discovery/registry-stats` and `GET /api/discovery/registry-dashboard` provide install/download counts where upstream APIs exist; others return `source: "unavailable"` without 500s.

## API Contract

### Existing (must remain stable for this program)

#### `GET /api/discovery/registry-submissions`

**Response 200:** `RegistrySubmissionInventory` — `summary` + `items[]` with `registry_id`, `registry_name`, `category`, `status`, `missing_files`, `proof_url`, `proof_path`, `notes`, etc.

**Proof:** This is the **authoritative readiness** view for “are we ready to claim submission?”

### Planned (spec 180 — not all may be deployed yet)

#### `GET /api/discovery/registry-stats`

Query: `?refresh=true`, `?registry_id=<id>`  
**Response 200:** Per-registry install/download counts and `source`: `live` | `cached` | `unavailable`.

#### `GET /api/discovery/registry-dashboard`

**Response 200:** Merged submission inventory + stats for one-glance proof.

**Note for reviewers:** If `registry-stats` returns **404** on production, the stats portion of this program is **not yet shipped**; scenarios below state expected behavior in both cases.

## Open Questions (Task-Mandated) — Resolved in This Spec

### How can we improve this idea?

1. **Single canonical skill + MCP package** — Avoid N forks; all registries point at the same npm package and skill directory.
2. **Evidence-first JSON** — Every external action has `proof_url` or committed `proof_path`.
3. **Tiered metrics** — Tier A: live API counts (Smithery, PulseMCP). Tier B: GitHub stars for repo linked from Glama. Tier C: manual “last verified” dates for form-based registries.
4. **Automation follow-up** — Spec 180’s “PR creation script” reduces human drag for Glama/Pulse.

### How do we show whether it is working **yet**?

- **Now:** `submission_ready_count`, `core_requirement_met`, and absence of `missing_files` on target rows.
- **After publish:** Non-zero npm/PyPI downloads (related npm/PyPI spec), non-zero Smithery/Pulse stats when APIs respond.
- **Qualitative:** Merged PR links in `proof_url`.

### How do we make proof **clearer over time**?

- Weekly `registry-stats?refresh=true` (or cron) storing **timestamped** snapshots → trend line.
- Dashboard page (spec 180 follow-up: `/discovery`) for non-technical stakeholders.
- **Definition of Done for “working”:** `core_requirement_met === true` **and** (Tier A: any `install_count > 0` on a live source **or** Tier C: dated manual verification within 30 days for form-only registries).

## Verification Scenarios

### Scenario 1 — Readiness inventory lists all tracked registries (read path)

**Setup:** API deployed with current `docs/registry-submissions.json`.  
**Action:**

```bash
curl -sS "https://api.coherencycoin.com/api/discovery/registry-submissions" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(sorted([i['registry_id'] for i in d['items']]))"
```

**Expected:** JSON array of `registry_id` values including at minimum `glama`, `pulsemcp`, `skills-sh`, `askill-sh`; after inventory update, also `smithery` and a row for `mcp-so` (or equivalent id used in JSON). HTTP **200**.  
**Edge (bad input):** `GET /api/discovery/registry-submissions/foo` → **404** (no such route) or router 404, not **500**.  
**Edge (missing file):** If a required file is deleted locally, `missing_files` is non-empty and `status` is `missing_assets` for that row.

---

### Scenario 2 — Core requirement flag matches business rule (aggregate proof)

**Setup:** Same as Scenario 1.  
**Action:**

```bash
curl -sS "https://api.coherencycoin.com/api/discovery/registry-submissions" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); s=d['summary']; print(s['core_requirement_met'], s['submission_ready_count'], s['categories'])"
```

**Expected:** `core_requirement_met` is **true** only if `submission_ready_count >= 5` **and** `categories` includes at least **2** `mcp` and **2** `skill` ready items (matches code). Values are **booleans/integers**, not null.  
**Edge:** If only MCP targets are ready, `core_requirement_met` is **false** even if five MCP rows exist.

---

### Scenario 3 — Full create–read–update cycle for inventory evidence (human + API readback)

**Setup:** Branch with permission to edit `docs/registry-submissions.json`.  
**Action:**

1. **Create:** Add or update one item’s `proof_url` to a stable URL (e.g. merged PR).
2. **Read:** `curl -sS "$API/api/discovery/registry-submissions"` and locate that `registry_id`; confirm `proof_url` matches.
3. **Update:** Change `notes` to include submission date; re-`curl` and confirm updated text.

**Expected:** Each read returns **200** and the edited fields round-trip.  
**Edge (duplicate semantics):** Two items with the same `proof_url` are **allowed** (shared proof); API must not dedupe-merge rows into one.

---

### Scenario 4 — Stats endpoint: success and upstream failure (error handling)

**Setup:** Spec 180 deployed with `registry_stats_service`.  
**Action:**

```bash
curl -sS -o /tmp/rs.json -w "%{http_code}" "https://api.coherencycoin.com/api/discovery/registry-stats"
```

**Expected:** HTTP **200**; body parses as JSON; each item has `source` in `live`/`cached`/`unavailable`.  
**Edge:** When Smithery is unreachable, Smithery row has `source: "unavailable"` or cached values, `error` optional; **HTTP still 200**, never **500** for upstream outage.  
**If endpoint not deployed:** HTTP **404** — reviewer records “stats phase pending” and Scenario 4 is N/A until deploy.

---

### Scenario 5 — Dashboard merges inventory + stats (single-call proof)

**Setup:** Spec 180 dashboard route live.  
**Action:**

```bash
curl -sS "https://api.coherencycoin.com/api/discovery/registry-dashboard" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print('submission' in str(d).lower() or 'items' in d)"
```

**Expected:** HTTP **200**; response includes per-registry combined view; when stats fail, submission section still populated.  
**Edge:** Malformed query `?registry_id=<script>` — ignored or validated; **200** or **422**, never unhandled **500**.
