## Summary

The **Idea Lens Engine** lets the same underlying idea be *viewed* and *ingested* through multiple explicit **worldview lenses** (belief-system filters). News and ideas are not one-size-fits-all: a headline that reads as opportunity to one community reads as threat to another. This capability defines APIs, data, and UX so that **every idea is viewable through any registered lens**, with optional **point-of-view filters** on ingestion, **caching**, **resonance** against a contributor profile, and **measurable proof** that perspective-bridging is happening (not just that endpoints return 200).

This document is the **payable verification contract** for the feature. Implementation details and file lists remain aligned with spec **181**; this task spec adds **runnable verification scenarios** and clarity on **how we know it is working** over time.

---

## Purpose

- **Problem**: Semantic drift across worldviews hides shared intent; single-framed ingestion reduces trust and reach.  
- **Outcome**: A first-class **lens** abstraction, **translation** artifacts per `(idea, lens)`, and **ROI-style metrics** so operators and reviewers can prove adoption and bridge-quality in production.  
- **Open question — “How can we improve this idea, show whether it is working yet, and make that proof clearer over time?”**  
  - **Improve**: Iteration uses `GET /api/lenses/roi` (and future dashboards) to prioritize lenses with low engagement or high `avg_resonance_delta` variance.  
  - **Working yet**: Binary checks — lenses listed, translations generated with `spec_ref: "spec-181"`, non-zero `total_translations_generated` in realistic usage.  
  - **Clearer over time**: Same endpoints return **monotonic or trendable counters** (totals, rates, timestamps on translations) so week-over-week comparisons do not rely on vague UI impressions.

## Requirements

### Functional

1. **Lens registry**: Clients can list and fetch lens definitions; invalid IDs yield **404**, not **500**.  
2. **Per-idea translation**: For a given idea and lens, the API returns an **IdeaTranslation** (cached or generated) including `spec_ref: "spec-181"` and `source_hash` for invalidation.  
3. **Create / read / update cycle**: Operators can **create** a custom lens (`POST /api/lenses`), **read** it (`GET /api/lenses/{lens_id}`), and **update** translation material for an idea by **forcing regeneration** (`POST /api/ideas/{idea_id}/translations/{lens_id}` with `force_regenerate`).  
4. **News / ingestion (POV filters)**: Pipeline or UI can request alternative framings; MVP contract includes **documented** `GET /api/news/{article_id}/translations/{lens_id}` as a **follow-up** dependency when the news router is live (see spec 181). Ingestion filters are **opt-in** toggles (“view through my lens”), not silent rewrites of source text.  
5. **Resonance**: When `contributor_id` is supplied, `resonance_delta` is in **[-1.0, 1.0]** or **null** if profile missing (graceful degradation).  
6. **Builtin lenses**: Six seeded archetypes: `libertarian`, `engineer`, `institutionalist`, `entrepreneur`, `spiritual`, `systemic`.  
7. **Validation**: `archetype_axes` values outside **[0.0, 1.0]** on lens create → **422**.

### Non-functional

- **Cache key**: `(idea_id, lens_id, source_hash)`; hash mismatch forces regeneration.  
- **Rate limiting**: Documented follow-up for abuse prevention; MVP must not return **500** on malformed client input.

## API Changes

Base path: `/api` (production: `https://api.coherencycoin.com/api`).

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/lenses` | List registered worldview lenses |
| `GET` | `/api/lenses/{lens_id}` | Get one lens |
| `POST` | `/api/lenses` | Create a custom lens (operator); duplicate `lens_id` → **409** |
| `GET` | `/api/ideas/{idea_id}/translations` | List cached translations for an idea |
| `GET` | `/api/ideas/{idea_id}/translations/{lens_id}` | Get or generate one translation (`contributor_id` optional query) |
| `POST` | `/api/ideas/{idea_id}/translations/{lens_id}` | Body: `TranslationRequest`; `force_regenerate: true` refreshes cache |
| `GET` | `/api/lenses/roi` | Aggregate metrics (`spec_ref: "spec-181"`) |

**Web (user-facing)**

| Path | Purpose |
|------|---------|
| `/ideas/{idea_id}` | Idea detail with **lens selector** and translated callout |
| `/lenses` | Lens browser / gallery (post-MVP acceptable as phased) |

**CLI (convenience — follow-up)**

| Command | Purpose |
|---------|---------|
| `cc lenses list` | List lenses (wraps `GET /api/lenses`) |
| `cc lenses translate --idea <id> --lens <id>` | Fetch translation (wraps GET) |

CLI is **not** blocking for MVP; API scenarios below are authoritative.

## Data Model

Aligned with spec 181:

- **`WorldviewLens`**: `lens_id`, `name`, `description`, `archetype_axes` (floats in [0,1]), optional `framing_template`, `is_builtin`, timestamps.  
- **`IdeaTranslation`**: `idea_id`, `lens_id`, `original_name`, `translated_summary`, `emphasis[]`, `risk_framing`, `opportunity_framing`, optional `resonance_delta`, `cached`, `generated_at`, `source_hash`, `spec_ref`.  
- **`TranslationRequest`**: `force_regenerate`, optional `contributor_id`.

Persistence: PostgreSQL tables as in spec 181 (`lens_repo`); ROI aggregates from translation events / stored rows.

## Verification Scenarios

Each scenario is independently runnable. Replace `$API` with `https://api.coherencycoin.com` (or local `http://localhost:8000` during development).

### Scenario 1 — Full create–read–update cycle (lens + translation)

- **Setup**: API is up; at least one idea exists (`GET /api/ideas?limit=1`). Note `idea_id`.  
- **Action (create lens)**:  
  `curl -sS -w "\n%{http_code}\n" -X POST "$API/api/lenses" -H "Content-Type: application/json" -d '{"lens_id":"contract-lens-5005632","name":"Contract Lens","description":"PM verification lens","archetype_axes":{"pragmatic":0.7}}'`  
- **Expected**: HTTP **201**, JSON includes `"lens_id":"contract-lens-5005632"`.  
- **Action (read lens)**:  
  `curl -sS -w "\n%{http_code}\n" "$API/api/lenses/contract-lens-5005632"`  
- **Expected**: HTTP **200**, same `lens_id` and `archetype_axes.pragmatic` ≈ `0.7`.  
- **Action (read translation — create cache)**:  
  `curl -sS -w "\n%{http_code}\n" "$API/api/ideas/$IDEA_ID/translations/contract-lens-5005632"`  
- **Expected**: HTTP **200**, `translated_summary` is a non-empty string, `"spec_ref":"spec-181"`, `"source_hash"` present.  
- **Action (update via force regenerate)**:  
  `curl -sS -w "\n%{http_code}\n" -X POST "$API/api/ideas/$IDEA_ID/translations/contract-lens-5005632" -H "Content-Type: application/json" -d '{"force_regenerate":true}'`  
- **Expected**: HTTP **200**, new `generated_at` ≥ previous (or strictly newer timestamp), `cached` may be false then true on subsequent GET.  
- **Edge**: `GET $API/api/lenses/contract-lens-5005632` after deleting data (if test reset) returns **404** with JSON `detail`, not **500**.

### Scenario 2 — List builtins and prove registry integrity

- **Setup**: Fresh session; no auth headers required for public read endpoints (per deployment policy).  
- **Action**:  
  `curl -sS "$API/api/lenses" | jq '{total: .total, ids: [.lenses[].lens_id]}'`  
- **Expected**: `total` ≥ **6**; array `ids` contains all of: `libertarian`, `engineer`, `institutionalist`, `entrepreneur`, `spiritual`, `systemic`.  
- **Edge**: `GET $API/api/lenses/` with trailing slash behavior matches router (301/308 or 200); must not **500**.

### Scenario 3 — Error handling: bad axis range (422)

- **Setup**: None.  
- **Action**:  
  `curl -sS -w "\n%{http_code}\n" -X POST "$API/api/lenses" -H "Content-Type: application/json" -d '{"lens_id":"bad-axis","name":"Bad","description":"x","archetype_axes":{"x":1.5}}'`  
- **Expected**: HTTP **422**, body documents validation failure for axis out of range.  
- **Edge**: Empty body `POST /api/lenses` → **422** (not **500**).

### Scenario 4 — Error handling: missing lens and duplicate create

- **Setup**: From Scenario 1, `contract-lens-5005632` may already exist.  
- **Action (missing translation target)**:  
  `curl -sS -w "\n%{http_code}\n" "$API/api/ideas/$IDEA_ID/translations/nonexistent_lens_zzzz"`  
- **Expected**: HTTP **404**, `detail` explains unknown lens or translation.  
- **Action (duplicate lens)**:  
  `curl -sS -w "\n%{http_code}\n" -X POST "$API/api/lenses" -H "Content-Type: application/json" -d '{"lens_id":"contract-lens-5005632","name":"Dup","description":"dup","archetype_axes":{"pragmatic":0.5}}'`  
- **Expected**: HTTP **409** if lens exists; if first run created it, second POST must **409**, not **200**.  
- **Edge**: Duplicate `POST` translation with `force_regenerate: false` is idempotent (200, same cache) — acceptable; must not create duplicate DB rows without constraint handling.

### Scenario 5 — ROI proof metrics (working over time)

- **Setup**: After at least one successful translation in Scenario 1.  
- **Action**:  
  `curl -sS "$API/api/lenses/roi" | jq '{total_translations_generated, cross_lens_engagement_rate, spec_ref}'`  
- **Expected**: `spec_ref` is `"spec-181"`; `total_translations_generated` is a non-negative integer ≥ **1** after usage; `cross_lens_engagement_rate` between **0.0** and **1.0** (inclusive) or documented null semantics.  
- **Edge**: Cold deploy with zero usage may return zeros; reviewer accepts **0** only if no translation has ever run — after Scenario 1, must be **≥ 1**.

## Risks and Assumptions

| Risk | Mitigation |
|------|------------|
| LLM distorts meaning | Always show **original** idea first; label output as lens-filtered; `source_hash` ties translation to idea revision. |
| Metric gaming | Combine rates with **unique contributors** and **cross_lens** pairs (future table). |
| Latency | Cache + optional async (202) for heavy regen — per spec 181. |
| News route not shipped | Document dependency; POV filter on news is **follow-up** until `GET /api/news/...` exists. |

**Assumption**: Contributor profiles (spec 169) exist for full `resonance_delta`; otherwise **null**.

## Failure/Retry Reflection

- **Failure mode**: Translation timeout from LLM provider.  
- **Blind spot**: User sees spinner forever.  
- **Next action**: Return **503** with retry-after or **202** + job id (follow-up).
