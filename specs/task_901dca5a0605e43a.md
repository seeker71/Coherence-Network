# Spec: Accessible Ontology — Non-Technical Contributors Extend It Naturally

**Task / tracking ID:** `task_901dca5a0605e43a`  
**Related stub:** `api/app/routers/accessible_ontology.py` (mounted at `/api`, tag `ontology`)

## Summary

Non-technical contributors should extend the shared ontology **without learning graph theory**. They describe an idea in **plain language**, optionally tag **domains they already know** (e.g. governance, UX, security). The system **proposes where the idea sits** in existing structure—near known concepts, under a parent idea, or in a **new region** when nothing fits—and **infers relationships** from context, contributions, and resonance signals. **Technical peers** continue to use **graph-native surfaces**; **everyone else** uses **gardens, cards, and conversations** that hide nodes/edges vocabulary while still writing to the same underlying node/edge model.

This spec defines the **product contract**, **API and web surfaces**, **data responsibilities**, and **verification scenarios** so implementation can proceed in small, test-backed slices.

## Purpose

- Reduce the **cognitive tax** of contributing structured knowledge so portfolio and ontology growth are not gated on Neo4j or formal taxonomy training.
- Preserve **one lifecycle** (atoms → graph) while offering **two projections**: graph (power) vs garden/card (accessibility).
- Make **placement and inference auditable**: every suggestion is explainable and revisable.

## Requirements

- [ ] A contributor can submit **natural-language text** (and optional **domain tags**) and receive **ranked placement suggestions** that reference existing **ideas** and/or **ontology concepts** without manual graph editing.
- [ ] The system persists the contributor’s **chosen placement** (or explicit “new branch”) on the **idea** or **concept extension** record so downstream resonance and traceability stay consistent.
- [ ] **Relationship inference** runs as an explicit pipeline step (batch or async), using **context** (text), **contribution history**, and **resonance/coherence** metrics; inferred edges are **labeled with provenance** (auto vs confirmed).
- [ ] **Role- or capability-based UI routing**: default experience is **garden / card / thread**; users with graph capability (or opt-in) see **/graph**-style views. Same APIs; different presentation.
- [ ] **Duplicate and near-duplicate** submissions are detected before creating redundant nodes; user can **merge** or **link** instead of fork.
- [ ] **OpenAPI-visible** endpoints under `/api` for all placement and suggestion flows used by web and automation.

## Improving the Idea, Proving It Works, and Clarifying Proof Over Time

**Product iteration**

- Start with **suggestion + confirm** before full auto-linking; add **“why here?”** copy on every suggestion (top matching terms, parent idea, nearest concept IDs).
- Run periodic **usability sessions** with non-technical contributors using only garden/card flows; track **time-to-first-valid-placement** and **edit rate after placement**.

**Whether it is working (measurable)**

- **Placement acceptance rate**: % of suggestions where the user picks top-1 vs edits vs rejects (target thresholds defined in implementation phase).
- **Time to placement**: median seconds from draft text to confirmed placement.
- **Graph health**: inferred edges **confirmed / rejected / pending** counts; stale pending beyond SLA.
- **Duplication**: count of prevented duplicate ideas vs forced merges.

**Proof that gets clearer over time**

- Versioned **golden sets**: fixed NL inputs with expected top-k anchors; CI runs suggestion ranking regression tests.
- **Dashboards**: `/api` metrics endpoints (see below) feeding **web** summary widgets and operator alerts.
- **Audit trail**: immutable log of placement decisions and later merges (supports trust and attribution).

## API Changes (Contract)

All new routes are served from the existing **`accessible_ontology`** router (`prefix=/api`, tag `ontology`) unless merged into `ideas` or `concepts` with explicit deprecation notes. Names below are canonical for this spec.

| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/ontology/placements/suggest` | Body: natural language, optional `domain_tags[]`, optional `idea_id` (draft). Returns ranked suggestions with scores and human-readable rationales. |
| `POST` | `/api/ontology/placements/confirm` | Body: `idea_id` or new idea payload + chosen `anchor` (concept id(s), parent idea id, or `new_region` descriptor). Persists placement. |
| `GET` | `/api/ontology/placements/{idea_id}` | Returns current placement, anchors, provenance, and last inference run. |
| `PATCH` | `/api/ontology/placements/{idea_id}` | Updates placement or marks inferred edges as confirmed/rejected. |
| `POST` | `/api/ontology/inference/run` | Operator or automation: run inference for idea id(s) or scope; returns job id / summary. |
| `GET` | `/api/ontology/metrics/summary` | Aggregated KPIs: acceptance rates, pending inferences, duplicate prevention counts (for dashboards and Verification). |

**Existing endpoints to integrate (no breaking changes)**

- `POST /api/ideas` — may accept optional `natural_language_summary` and `domain_tags` passthrough to suggestion pipeline (fields additive).
- `GET/POST /api/concepts` — remains source of truth for formal concepts; placements may reference `concept_id` values from this list.
- `GET /api/ideas`, `GET /api/ideas/{id}` — responses include placement fields once implemented.

**Errors**

- `400` — empty or unusable natural language; invalid tag format.
- `404` — `idea_id` or anchor not found.
- `409` — conflicting placement update; duplicate idea detected (body includes merge candidates).
- `422` — validation errors from Pydantic.

### Example: `POST /api/ontology/placements/suggest`

**Request**

```json
{
  "text": "We should make onboarding feel less like a form and more like a short guided tour.",
  "domain_tags": ["ux", "onboarding"],
  "idea_id": null
}
```

**Response 200**

```json
{
  "suggestions": [
    {
      "rank": 1,
      "anchor_type": "idea",
      "anchor_id": "168-identity-driven-onboarding-tofu",
      "score": 0.82,
      "rationale": "Strong overlap with onboarding and identity framing in existing idea title and description."
    },
    {
      "rank": 2,
      "anchor_type": "concept",
      "anchor_id": "user-journey",
      "score": 0.61,
      "rationale": "Concept search match on journey and guided flow language."
    }
  ],
  "duplicate_candidates": []
}
```

## Data Model

**PlacementRecord** (logical; persisted on idea graph or relational store per implementation)

```yaml
PlacementRecord:
  idea_id: { type: string, required: true }
  anchor_type: { enum: [idea, concept, composite, new_region] }
  anchor_ids: { type: array of string, default: [] }
  domain_tags_normalized: { type: array of string }
  provenance: { enum: [user_confirmed, imported, inferred] }
  inference_status: { enum: [pending, complete, failed] }
  last_suggestion_snapshot_id: { type: string, optional }
  created_at: { type: datetime, utc: true }
  updated_at: { type: datetime, utc: true }
```

**InferredEdgeProposal** (for resonance/context pipeline)

```yaml
InferredEdgeProposal:
  id: { type: string }
  source_id: { type: string }
  target_id: { type: string }
  relationship_type: { type: string }
  confidence: { type: number, min: 0, max: 1 }
  provenance: { type: string }
  status: { enum: [pending, accepted, rejected] }
```

## Acceptance Criteria

1. **Suggest + confirm loop** is documented, implemented, and tested: NL in → ranked anchors → persisted placement retrievable by `GET`.
2. **Inference** produces auditable proposals with confidence and does not silently overwrite user-confirmed edges.
3. **Web** exposes a non-graph primary path for reviewing placement on an idea (card UI) while **graph** remains available for technical users.
4. **Metrics endpoint** returns counters needed to track acceptance and duplication prevention.
5. **Verification Scenarios** below are executable against staging/production with listed commands.

## Verification Scenarios

### Scenario 1 — Full suggest → confirm → read cycle

- **Setup:** Authenticated API client (or local API) with empty or known test idea namespace; at least one existing idea or concept seed present for anchoring.
- **Action:**
  1. `curl -s -X POST "$API/api/ontology/placements/suggest" -H "Content-Type: application/json" -d '{"text":"Guided onboarding that respects user identity","domain_tags":["ux","onboarding"]}'`
  2. `curl -s -X POST "$API/api/ideas" -H "Content-Type: application/json" -d '{"name":"Test placement NL","description":"..."}'` → capture `idea_id`
  3. `curl -s -X POST "$API/api/ontology/placements/confirm" -H "Content-Type: application/json" -d "{\"idea_id\":\"<idea_id>\",\"anchor_type\":\"idea\",\"anchor_ids\":[\"<anchor_from_step_1>\"]}"`
  4. `curl -s "$API/api/ontology/placements/<idea_id>"`
- **Expected:** Step 1 returns `200` with `suggestions` non-empty and each item has `anchor_type`, `anchor_id`, `score`, `rationale`. Step 3 returns `200` with persisted placement. Step 4 returns the same anchors and `provenance: user_confirmed`.
- **Edge (bad input):** Suggest with `{"text":""}` → `400` with detail that text is required.
- **Edge (missing resource):** Confirm with nonexistent `idea_id` → `404`.

### Scenario 2 — Idea create with domain tags integrates with suggestions

- **Setup:** API running; valid minimal idea create payload.
- **Action:** `curl -s -X POST "$API/api/ideas" -H "Content-Type: application/json" -d '{"name":"Tagged idea","description":"Plain language body","domain_tags":["security","api"]}'` then `curl -s "$API/api/ideas/<returned_id>"`
- **Expected:** `201` then `200`; response includes normalized `domain_tags` and any placement fields once implemented; tags match Spec 129 normalization rules if that spec is merged.
- **Edge (duplicate):** POST a second idea with same fingerprint as an existing one (per dedup rules) → `409` with `duplicate_candidates` populated, not a silent second node.

### Scenario 3 — Metrics prove the feature is observable

- **Setup:** At least one completed placement in the environment.
- **Action:** `curl -s "$API/api/ontology/metrics/summary"`
- **Expected:** `200` JSON including keys such as `placement_confirmed_total`, `suggestion_top1_acceptance_rate`, `inference_pending_count` (exact schema fixed at implementation).
- **Edge:** Cold environment → zeros or explicit `insufficient_data` flag, not `500`.

### Scenario 4 — Technical vs non-technical web surfaces

- **Setup:** Web deployed; browser or `curl -sI` for routes.
- **Action:** Open `https://<host>/ideas/<idea_id>` (card view) and `https://<host>/graph` (graph view). If `/garden` shipped: open `/garden`.
- **Expected:** Ideas page renders placement summary without requiring graph vocabulary; graph page still loads node/edge visualization for power users. No 404 on required routes that are in scope for the release.
- **Edge:** Unauthenticated access follows existing site policy (redirect or public read-only), not 500.

### Scenario 5 — Inference run and edge rejection

- **Setup:** Idea with `inference_status: pending`.
- **Action:** `curl -s -X POST "$API/api/ontology/inference/run" -H "Content-Type: application/json" -d '{"idea_ids":["<idea_id>"]}'` then `PATCH /api/ontology/placements/<idea_id>` with body rejecting a specific inferred edge id.
- **Expected:** Run returns `200`/`202` with job or result summary; PATCH updates edge status to `rejected`; subsequent GET does not show that edge as active.
- **Edge:** Run with invalid `idea_ids` → `400` or `422`, not unhandled exception.

## Concurrency Behavior

- **Suggest** is read-mostly and idempotent for the same text snapshot.
- **Confirm** must be **atomic** per idea (optimistic versioning or ETag recommended in implementation follow-up).
- **Inference** jobs must not block HTTP health; use background worker pattern where available.

## Out of Scope (this spec version)

- Replacing the full Neo4j schema or migrating all concepts to a new store.
- Full natural-language **generative** rewriting of contributor text (only placement/inference as defined).
- Legal/compliance review of user-generated ontology content (follow standard UGC policies).

## Risks and Assumptions

- **Risk:** Suggestion quality too low → user distrust. **Mitigation:** Always show rationale; start with conservative thresholds; golden-set regression tests.
- **Risk:** Inference creates noisy edges. **Mitigation:** Pending state + explicit accept/reject; decay low-confidence proposals.
- **Assumption:** Ideas and concepts remain the two primary anchor families; if a third grows, generalize `anchor_type` enum.

## Known Gaps and Follow-up Tasks

- Define **embedding / search** stack for semantic match (existing search vs new index) in implementation spec.
- Define **authorization** matrix for `inference/run` and metrics (operator-only vs contributor).
- **Garden** page may ship as Phase 2 behind feature flag if MVP is API-only.

## Failure/Retry Reflection

- **Failure mode:** Inference job backlog grows. **Blind spot:** underestimating async worker capacity. **Next action:** expose `inference_pending_count` alert and cap batch size.

## Decision Gates

- Product sign-off on **default UI** (ideas vs garden first).
- Engineering sign-off on **single source of truth** for placement storage (idea record vs side table).
