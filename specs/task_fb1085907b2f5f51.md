## Summary

The Coherence Network MCP server (`mcp-server/`) is functionally complete but
not yet published to the public package registries that discovery platforms
depend on. This spec defines what must be built so that:

1. `npx coherence-mcp-server` works for any user worldwide.
2. `pip install coherence-mcp-server` works for Python users who prefer the
   PyPI ecosystem.
3. The server appears on the **Official MCP Registry**
   (registry.modelcontextprotocol.io), **Smithery**, **Glama**, and every
   other major MCP discovery platform.
4. A `server.json` manifest is present and correct for `mcp-publisher`
   (Anthropic's submission helper), so CI can auto-submit registry entries.
5. Proof of working is measurable: install counts, version badges, and a
   live-endpoint integration test all update automatically after each release.

**Relationship to Spec 180:** Spec 180 defines registry tracking (API
endpoints that watch submission readiness and download counts). This spec
defines the **build and release pipeline** that makes the packages installable
in the first place. Both are required; neither replaces the other.

```bash
npm publish --dry-run --access public
```
Must complete without errors.

### R2 — PyPI Package (Python)

A Python package named `coherence-mcp-server` must be created at
`mcp-server-py/`. It wraps the Node.js server with a thin Python shim that:

1. Checks for Node.js ≥ 18 on the host.
2. Runs `npx coherence-mcp-server` as a subprocess, forwarding stdio.
3. Provides an `mcp_server` console_scripts entry point.

**Directory layout:**

```
mcp-server-py/
  pyproject.toml           ← package metadata, build backend = hatchling
  README.md                ← copied/adapted from mcp-server/README.md
  src/
    coherence_mcp_server/
      __init__.py          ← version = "0.3.1"
      __main__.py          ← entry point: exec npx coherence-mcp-server
      _node_check.py       ← checks node >= 18, raises helpful error if missing
```

`pyproject.toml` structure:

```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "coherence-mcp-server"
version = "0.3.1"
description = "MCP server for the Coherence Network — browse ideas, trace value chains, record contributions."
readme = "README.md"
license = {text = "MIT"}
requires-python = ">=3.9"
keywords = ["mcp", "model-context-protocol", "coherence", "ai-agent", "ideas"]
classifiers = [
  "Development Status :: 4 - Beta",
  "Intended Audience :: Developers",
  "License :: OSI Approved :: MIT License",
  "Programming Language :: Python :: 3",
  "Topic :: Scientific/Engineering :: Artificial Intelligence",
]
dependencies = []   # zero Python deps — delegates to Node.js

[project.urls]
Homepage = "https://coherencycoin.com"

Documentation = "https://github.com/seeker71/Coherence-Network/blob/main/mcp-server/README.md"

[project.scripts]
coherence-mcp-server = "coherence_mcp_server.__main__:main"
```

**`__main__.py`:**

```python
import os
import subprocess
import sys

def main():
    from coherence_mcp_server._node_check import ensure_node
    ensure_node()
    env = os.environ.copy()
    # Forward COHERENCE_API_URL and COHERENCE_API_KEY if set
    result = subprocess.run(
        ["npx", "--yes", "coherence-mcp-server@latest"],
        stdin=sys.stdin,
        stdout=sys.stdout,
        stderr=sys.stderr,
        env=env,
    )
    sys.exit(result.returncode)

if __name__ == "__main__":
    main()
```

**`_node_check.py`:**

```python
import shutil
import subprocess
import sys


def ensure_node():
    node = shutil.which("node")
    if not node:
        print(
            "ERROR: Node.js is required to run coherence-mcp-server. "
            "Install it from https://nodejs.org/ (v18 or later).",
            file=sys.stderr,
        )
        sys.exit(1)
    result = subprocess.run(
        ["node", "--version"], capture_output=True, text=True
    )
    version_str = result.stdout.strip().lstrip("v")
    major = int(version_str.split(".")[0])
    if major < 18:
        print(
            f"ERROR: Node.js v18+ required. Found: {result.stdout.strip()}. "
            "Upgrade at https://nodejs.org/",
            file=sys.stderr,
        )
        sys.exit(1)
```

### R4 — GitHub Actions Publish Workflow

File: `.github/workflows/publish-mcp.yml`

Trigger: `on: push: tags: ['v*.*.*']`

Jobs:

**Job 1: `publish-npm`**
1. `actions/checkout`
2. `actions/setup-node` with `registry-url: https://registry.npmjs.org`
3. `cd mcp-server && npm ci`
4. `npm publish --access public`
   - Secret: `NPM_TOKEN`

**Job 2: `publish-pypi`**
1. `actions/checkout`
2. `actions/setup-python`
3. `pip install hatch`
4. `cd mcp-server-py && hatch build`
5. `hatch publish` (or `twine upload dist/*`)
   - Secrets: `PYPI_TOKEN` (API token scoped to `coherence-mcp-server`)

**Job 3: `integration-test`** (runs after both publish jobs succeed)
1. Install freshly published packages:
   ```bash
   npm install -g coherence-mcp-server@latest
   pip install coherence-mcp-server==<version>
   ```
2. Run smoke test (see Verification Scenarios §5).

### R5 — mcp-publisher Submission

`mcp-publisher` is Anthropic's CLI for submitting servers to the Official MCP
Registry. After the first npm publish succeeds:

```bash
npx @anthropic-ai/mcp-publisher submit \
  --manifest mcp-server/server.json \
  --token $MCP_REGISTRY_TOKEN
```

This step is manual on first submission and automated on subsequent releases
via the publish workflow.

The `server.json` must pass `mcp-publisher validate` locally before the CI
step runs.

### R7 — How We Know It Is Working (Proof Infrastructure)

The question "is it working?" must be answerable with a single API call at any
time, not just at release time.

**API endpoint:** `GET /api/discovery/registry-stats` (from Spec 180)
extended to include:

- `npm_published_version`: latest version on npmjs.com (polled via
  `registry.npmjs.org/coherence-mcp-server/latest`)
- `pypi_published_version`: latest version on pypi.org (polled via
  `pypi.org/pypi/coherence-mcp-server/json`)
- `npm_weekly_downloads`: from `api.npmjs.org/downloads/point/last-week/coherence-mcp-server`
- `mcp_registry_listed`: boolean, polled from `registry.modelcontextprotocol.io`

**New API endpoint:** `GET /api/discovery/publish-status`

```json
{
  "npm": {
    "published": true,
    "version": "0.3.1",
    "weekly_downloads": 47,
    "installable": true,
    "last_checked": "2026-03-28T20:00:00Z"
  },
  "pypi": {
    "published": true,
    "version": "0.3.1",
    "monthly_downloads": 12,
    "installable": true,
    "last_checked": "2026-03-28T20:00:00Z"
  },
  "mcp_registry": {
    "listed": false,
    "submission_status": "pending",
    "last_checked": "2026-03-28T20:00:00Z"
  },
  "overall_status": "partial",
  "proof_available": true
}
```

`overall_status` values:
- `"not_published"` — neither npm nor PyPI published
- `"partial"` — one package published
- `"published"` — both packages published
- `"discoverable"` — published AND listed in ≥ 1 official registry

**Polling cadence:** Background task runs every 6 hours and writes results to
`.cache/publish_status/current.json`. The endpoint reads from cache; `?refresh=true`
forces live fetch.

## Data Model

### `PackagePublishInfo`

```python
class PackagePublishInfo(BaseModel):
    published: bool
    version: str | None = None
    weekly_downloads: int | None = None
    monthly_downloads: int | None = None
    installable: bool = False
    last_checked: datetime | None = None
    error: str | None = None
```

### `MCPRegistryInfo`

```python
class MCPRegistryInfo(BaseModel):
    listed: bool
    submission_status: str  # "not_submitted" | "pending" | "listed" | "rejected"
    registry_url: str | None = None
    last_checked: datetime | None = None
```

### `PublishStatus`

```python
class PublishStatus(BaseModel):
    npm: PackagePublishInfo
    pypi: PackagePublishInfo
    mcp_registry: MCPRegistryInfo
    overall_status: str  # "not_published" | "partial" | "published" | "discoverable"
    proof_available: bool  # True when at least one live count > 0
    last_updated: datetime
```

## API Changes

### New endpoint

```
GET /api/discovery/publish-status
```

Query params:
- `?refresh=true` — bypass cache and fetch live data from npm, PyPI, MCP Registry

Returns `PublishStatus`.

| State | HTTP | Behavior |
|-------|------|---------|
| Packages not published | 200 | `published: false`, `overall_status: "not_published"` |
| npm published, PyPI pending | 200 | `overall_status: "partial"` |
| Both published | 200 | `overall_status: "published"` |
| Both published + registry listed | 200 | `overall_status: "discoverable"` |
| Upstream API unreachable | 200 | Cached values returned; `error` field set |
| No cache + upstream down | 200 | `published: false`, `error: "upstream unavailable"` |

## Risks and Assumptions

| Risk | Mitigation |
|------|-----------|
| npm token may expire or be misconfigured | Use granular npm automation token scoped to `coherence-mcp-server`; store as `NPM_TOKEN` GitHub secret |

| PyPI package name squatting | Same risk; pre-check with `pip index versions coherence-mcp-server` |
| MCP Registry schema changes | Pin `$schema` URL; validate in CI using `mcp-publisher validate` before submit |

## Verification

### How to Improve This Idea Over Time

The open question "how can we improve this idea and make proof clearer over time"
is answered by the `proof_available` field in `PublishStatus`:

- **Phase 1 (now):** `proof_available = false` — packages published but zero
  downloads yet. The endpoint still returns 200 with version numbers, proving
  the publish pipeline worked.
- **Phase 2 (days 2–7):** First external installs arrive. `npm.weekly_downloads > 0`.
  `proof_available = true`. The fact is now externally validated.
- **Phase 3 (week 2+):** Time-series table in PostgreSQL stores weekly snapshots.
  `/discovery/publish` dashboard shows a sparkline. Growth is visible and
  attributable.
- **Phase 4 (month 2+):** MCP Registry lists the server.
  `mcp_registry.listed = true`. `overall_status = "discoverable"`. This is the
  strongest proof: an independent third party has validated the server.

Each phase is machine-readable via `GET /api/discovery/publish-status` and
human-readable via the `/discovery` dashboard (Spec 183 follow-up).
