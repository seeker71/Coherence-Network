## Summary

The same proposition lands differently depending on worldview: a decentralized network reads as freedom to a libertarian, efficiency to an engineer, institutional risk to a regulator, and opportunity to an entrepreneur. Coherence Network should help people **see how their perspective connects to others** without erasing or replacing the underlying idea ‚Äî **reframing**, not rewriting truth claims.

Today the API already exposes **conceptual translation** (not machine translation of language) via `GET /api/ideas/{idea_id}/translate?view=<lens>` with a fixed enum of lenses (`scientific`, `economic`, `spiritual`, `artistic`, `philosophical`) implemented in `translate_service.py`. This spec defines the **next product slice**: user- and community-defined **worldview lenses**, **news ingestion with point-of-view (POV) filters**, **consistent availability of any stored idea through any registered lens**, and **observable proof** that the capability is used and improving (metrics, audits, and reviewer-verifiable scenarios).

---

## Purpose

- Reduce cross-community misunderstanding by making implicit frames explicit.
- Support discovery: users find ideas via the vocabulary they already use (political, professional, cultural).
- Align with ‚ÄúEverything is a Node‚Äù: lenses, POV filters, and translation artifacts are durable nodes with edges to ideas and ingest items.
- Give operators and reviewers **independent evidence** that the feature is live and healthy.

---

## Requirements

### R1 ‚Äî Lens catalog (extensible)

1. **System lenses**: Preserve and document the existing `TranslateLens` values as the default system set; they remain valid for all verification scenarios that reference them.
2. **Custom lenses**: Contributors can define additional lenses with `id`, `display_name`, `description`, optional `keyword_hints[]`, and optional `political_or_value_alignment` metadata (free text or controlled tags ‚Äî see Decision Gates).
3. **Stable IDs**: Lens IDs are URL-safe slugs; renames change display only.

### R2 ‚Äî Every idea viewable through every lens

1. For any `idea_id` that exists, `GET /api/ideas/{idea_id}/translate?view=<lens_id>` returns a **200** with a structured payload when `lens_id` is a known system or custom lens.
2. If the lens is unknown, return **422** with a clear error (not 500).
3. Response includes at minimum: `idea_id`, `lens_id`, `lens_label`, `summary` (human framing), `bridging_concepts[]` (when ontology data exists), and `lens_metadata_version` (for cache busting and audits).

### R3 ‚Äî News ingestion with POV filters

1. Ingested news items (or external links) are stored with normalized text and optional source tags.
2. Users can attach a **default POV lens** to a feed or saved query so the **indexing/ranking step boosts** items whose language overlaps lens keyword hints (transparent scoring, not black-box reranking).
3. API exposes list/filter by `pov_lens_id` for ingested items.

### R4 ‚Äî Proof and improvement over time (answers the open question)

1. **Usage metrics** (aggregated): counts of translate calls per lens per day; top ideas by translate views; error rate (404/422/5xx).
2. **Quality proxies** (non-blocking): optional explicit feedback `helpful` / `not helpful` on a translation response (rate-limited per user/session).
3. **Audit trail**: append-only events `worldview_translate_viewed` with `{ idea_id, lens_id, schema_version }` for dashboards (PII-minimal).
4. **Public evidence**: production URL paths documented in Verification Scenarios; health checks include translate route smoke.

### R5 ‚Äî Web UX (minimal)

1. Idea detail page exposes a **lens selector**; changing the lens refetches translation (or uses cached client state with explicit refresh).
2. News/explore page exposes **POV filter** when ingest is enabled for the deployment.

## API changes (target contract)

Existing (baseline ‚Äî must remain compatible unless this spec is explicitly superseded):

| Method | Path | Notes |
|--------|------|--------|
| GET | `/api/ideas/{idea_id}/translate?view=<lens>` | Today: `TranslateLens` enum |

Proposed additions (implementation task ‚Äî not required to merge this spec alone):

### `GET /api/worldviews/lenses`

Returns `{ "lenses": [ { "id", "display_name", "kind": "system"|"custom", "description" } ] }`.

### `POST /api/worldviews/lenses`

Creates a custom lens (auth + validation). **201** with body; duplicate `id` ‚Üí **409**.

### `GET /api/ideas/{idea_id}/translate`

Query: `view` (lens id, string). Extends current behavior to resolve custom lenses from catalog.

### `GET /api/news/items` (or existing ingest list route)

Query: `pov_lens_id` optional ‚Äî filters/ranks per R3.

### `POST /api/metrics/events/worldview-translate` (optional internal)

Body: minimal event for R4; or reuse generic analytics pipeline if present.

**Deprecation policy**: If `view` must remain enum-only short term, add parallel `lens_id` query param and document migration; prefer single `view` accepting any registered lens id once catalog ships.

- `display_name`

- `kind`: `system` | `custom`

### Edges

- `(Idea)-[:TRANSLATED_THROUGH]->(WorldviewLens)` ‚Äî logical association for analytics (materialized or derived).

## Verification Scenarios

Each scenario is runnable against production when `API=https://api.coherencycoin.com/api` and a valid `idea_id` exists (use `GET /api/ideas?limit=1` to discover one).

### Scenario A ‚Äî Baseline system lens translation

- **Setup**: API healthy; at least one public idea exists (`idea_id` from list).
- **Action**:  
  `curl -sS "$API/ideas/$IDEA_ID/translate?view=scientific"`  
  (replace `$API` and `$IDEA_ID`).
- **Expected**: HTTP **200**; JSON contains non-empty `summary` (or equivalent top-level framing field per implementation); includes reference to `scientific` lens; no stack trace in body.
- **Edge (bad lens)**:  
  `curl -sS -o /dev/null -w "%{http_code}" "$API/ideas/$IDEA_ID/translate?view=not_a_real_lens_xyz"`  
  ‚Üí **422** (preferred) or documented **400**; must not be **500**.
- **Edge (missing idea)**:  
  `curl -sS -o /dev/null -w "%{http_code}" "$API/ideas/does-not-exist-idea-id-00000/translate?view=scientific"`  
  ‚Üí **404**.

### Scenario B ‚Äî Lens list discoverability

- **Setup**: After `GET /api/worldviews/lenses` ships; if not yet deployed, skip with note ‚Äúblocked on implementation‚Äù and fall back to documented enum in OpenAPI/spec.
- **Action**: `curl -sS "$API/worldviews/lenses" | jq '.lenses | length'`
- **Expected**: count ‚â• 5 when system lenses are listed (scientific, economic, spiritual, artistic, philosophical).
- **Edge**: Unauthenticated `POST` to create custom lens returns **401** or **403** if auth required.

### Scenario C ‚Äî Cross-lens consistency (same idea, two lenses)

- **Setup**: Same `idea_id` as Scenario A.
- **Action**:  
  `curl -sS "$API/ideas/$IDEA_ID/translate?view=economic"`  
  `curl -sS "$API/ideas/$IDEA_ID/translate?view=spiritual"`
- **Expected**: Both **200**; summaries **differ** in text (not byte-identical) OR documented equality exception when idea text is empty (explicit in response).
- **Edge**: If `description` is empty, response still **200** with lens-appropriate generic framing (per `translate_service` behavior).

### Scenario D ‚Äî News POV filter (when ingest enabled)

- **Setup**: At least one ingested news item exists in the deployment; feature flag on if applicable.
- **Action**:  
  `curl -sS "$API/news/items?pov_lens_id=economic&limit=5"`  
  (exact path per implementation; may be `ingest`, `feed`, or `articles`.)
- **Expected**: HTTP **200**; array response; each item has `id` and source metadata; ordering reflects POV boost (documented tie-break: `ingested_at desc`).
- **Edge**: `pov_lens_id` unknown ‚Üí **422** or empty list with warning field ‚Äî behavior must be documented and stable.

### Scenario E ‚Äî Proof metrics (working yet?)

- **Setup**: Translate endpoint used at least once in last 7 days.
- **Action**: Operator runs documented query or `curl -sS "$API/metrics/worldviews?window=7d"` (or Grafana/dashboard URL from runbook).
- **Expected**: Non-zero `translate_invocations` or equivalent; includes breakdown by `lens_id`.
- **Edge**: Zero traffic ‚Üí dashboard shows **0** (not missing series); alerts optional, not required for MVP.

## Concurrency behavior

- Lens catalog reads are safe for concurrent access; writes (custom lens) are infrequent ‚Äî last-write-wins acceptable for MVP.
- Translate is read-only; idempotent for same inputs.

---

## Out of scope

- LLM-generated paraphrase of full idea text in user‚Äôs political dialect (expensive, safety-sensitive) ‚Äî future spec.
- Automatic inference of user‚Äôs worldview without consent ‚Äî excluded; use explicit POV and spec 169 beliefs.
- Full multilingual localization ‚Äî separate i18n spec.

## Risks and assumptions

1. **Political sensitivity**: Labels like ‚Äúlibertarian‚Äù vs ‚Äúinstitutional‚Äù can antagonize users. Mitigation: lenses describe **epistemic stance and vocabulary**, not pejoratives; moderation on custom lens names.
2. **Assumption**: Ontology/concept graph remains available for bridging concepts; if empty, translations degrade to generic lens text (already partially true).
3. **Metrics privacy**: Aggregates only; no per-user worldview tracking without opt-in.
