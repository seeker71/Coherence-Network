# Spec: Resonance alive empty state with ambient breathing

## Goal

Ensure the Resonance heartbeat surface never feels dead when activity is empty by replacing the flat empty state with an organic breathing state that shows:

- a soft living animation (rings, particle drift, gentle wave),
- the last known pulse timestamp when available,
- warm invitation copy that guides users to contribute.

## Files to Modify

- `web/app/resonance/page.tsx`
- `web/app/globals.css`
- `api/tests/test_resonance_alive_empty_state.py`

## Scope

- In scope:
  - Empty-state UX on the Resonance page (`/resonance`) when no recent resonance activity is returned.
  - Timestamp affordance for "last known pulse".
  - API/UI contract tests in `pytest` that validate endpoint behavior and source-level rendering contract.
- Out of scope:
  - New API endpoints.
  - Changes to authentication, orchestration, or non-resonance pages.

## API / Web / CLI Contract

- API endpoint (must exist): `GET /api/ideas/resonance`
- API endpoint (must exist): `GET /api/ideas`
- Web page (must exist): `/resonance`
- CLI command (verification only): `cc dif verify --language python --file api/tests/test_resonance_alive_empty_state.py --json`

## Acceptance Criteria

1. When Resonance activity is empty and there are no fallback ideas, the `/resonance` empty state renders breathing ambient visuals (rings + particles + wave) and warm invitation copy.
2. The empty state surfaces "Last known pulse" with either formatted timestamp text or a graceful "No pulse recorded yet" fallback.
3. Existing resonance API behavior remains valid (`GET /api/ideas/resonance` returns `200` with a JSON array, including empty arrays).
4. API contract supports heartbeat freshness proof over time via `last_activity_at` values in resonance items.
5. New tests in `api/tests/test_resonance_alive_empty_state.py` are runnable with pytest and pass.

## Open Questions and Improvement Path

How to make proof clearer over time:

- Add a rolling heartbeat KPI panel on `/resonance` (for example: pulses in last 1h/24h/7d) derived from `last_activity_at`.
- Persist "last known pulse" in a small server-side projection so it is available even when transient feeds return empty.
- Track empty-state dwell and conversion (`viewed_empty_state` -> `created_idea`) to validate whether warm invitation copy increases engagement.
- Add visual health states (calm / active / high-resonance) tied to deterministic thresholds so users can verify network vitality at a glance.

## Verification Scenarios

### 1) Resonance endpoint empty-state safety
- Setup: Fresh test storage with no ideas.
- Action: `curl -sS "http://localhost:8000/api/ideas/resonance?window_hours=72&limit=5"`
- Expected result: HTTP `200`; response body is `[]` (JSON array, not error object).
- Edge case: `window_hours=0` returns `422` validation error (bad input is rejected).

### 2) Full create-read-update cycle produces pulse evidence
- Setup: API running with writable local store.
- Action:
  1. `POST /api/ideas` create idea `alive-empty-proof`.
  2. `GET /api/ideas/alive-empty-proof` confirm persisted.
  3. `POST /api/ideas/alive-empty-proof/questions` add question.
  4. `POST /api/ideas/alive-empty-proof/questions/answer` answer question.
  5. `GET /api/ideas/resonance?window_hours=72&limit=10`.
- Expected result: Resonance response includes `idea_id="alive-empty-proof"` with non-empty `last_activity_at`.
- Edge case: answering a non-existent question returns `404` (missing resource path).

### 3) Resonance web empty state is alive, not flat
- Setup: `/resonance` renders with no resonance items and no fallback ideas.
- Action: Open `/resonance` in browser.
- Expected result: Breathing ambient empty-state visuals are visible (ring pulse, particle drift, and gentle wave), plus warm invitation text.
- Edge case: if no timestamp is available, UI shows `No pulse recorded yet` instead of broken/blank value.

### 4) Last known pulse presentation contract
- Setup: Resonance page source and API contract available.
- Action: Verify source contains pulse label + formatter logic and API returns `last_activity_at` for active ideas.
- Expected result: Label `Last known pulse` is present; formatter path exists; API field `last_activity_at` is string timestamp when activity exists.
- Edge case: malformed or missing timestamp input falls back to neutral text rather than throwing.

### 5) Error handling and bounds contract
- Setup: API running.
- Action:
  - `GET /api/ideas/resonance?limit=101`
  - `GET /api/ideas/resonance?window_hours=721`
- Expected result: both return `422` due to query bound violations.
- Edge case: valid boundary values (`limit=100`, `window_hours=720`) return `200`.
