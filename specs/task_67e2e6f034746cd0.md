# Spec: Smart Reap — Diagnose Stuck Tasks, Capture Partial Work, Resume from Checkpoint

**Idea / task ID**: `task_67e2e6f034746cd0`  
**Related specs**: `specs/113-failed-task-diagnostics-contract.md` (error classification), `specs/136-data-driven-timeout-dashboard.md` (timeout observability), `specs/083-task-claim-tracking-and-roi-dedupe.md` (task lineage)  
**Primary implementation anchor (current behavior)**: `api/scripts/local_runner.py` — `_reap_stale_tasks()` (age-based reap, partial log tail, checkpoint/patch capture, capped retries per idea+phase)

## Summary

Today the runner’s reaper treats long-running tasks as uniformly stale: after a wall-clock threshold it marks tasks `timed_out`, appends a short diagnosis string, and may seed a retry task. That is better than silent failure but still **blind** relative to ground truth: it does not distinguish a healthy-but-slow provider from a dead runner, does not quantify partial output, does not extend deadlines when work is clearly progressing, and does not maintain a **durable per-idea reap history** strong enough to stop infinite blind retries or to surface “this idea needs a human” early.

This spec defines **Smart Reap**: a diagnosis-first, evidence-backed timeout policy that (1) consults runner and process state before reaping, (2) extends timeouts when work is active but slow, (3) captures stderr and partial artifacts when the executor crashed, (4) writes structured diagnosis into **task context** (not only `output` text), (5) creates **resume tasks** when partial output exceeds a threshold, with an explicit continuation prompt, and (6) tracks **reap history per idea** to flag chronic timeouts and halt blind automation.

## Purpose

Operators and pipeline monitors need to know why a task stopped—whether the runner died, the provider is slow, the process crashed, or the task was orphaned. Downstream automation must not burn quota retrying the same under-specified idea forever. Executors should resume from real partial work expressed in logs and checkpoints, not from a generic retry stub.

## Requirements

- [ ] **R1 — Liveness** — Before reaping, resolve claiming runner, registry online status, and provider process running when available; persist structured `reap_decision` fields.
- [ ] **R2 — Extension** — When runner is alive and progress signals exist, extend effective deadline up to configured caps instead of immediate `timed_out`.
- [ ] **R3 — Capture** — On crash or stall, attach bounded log tail, stderr excerpt, checkpoint, and optional patch path into `context.reap_capture`.
- [ ] **R4 — Context diagnosis** — Merge `reap_diagnosis` (provider, `error_class`, partial ratio, timestamps) into task context on reap.
- [ ] **R5 — Resume task** — If partial output >20% (or material checkpoint/patch), create resume task with required continuation direction template and `seed_source=smart_reap_resume`.
- [ ] **R6 — Idea history** — Track reap events per idea; emit friction and stop blind retries when thresholds exceeded.
- [ ] **R7 — Observability** — Expose counters or `GET /api/agent/reap/summary` so operators can prove Smart Reap behavior without log diving.

### R1 — Pre-reap liveness and ownership checks

Before transitioning a candidate “stale” task to `timed_out` (or any terminal reap state), the system MUST resolve:

- **Which runner claimed the task** (runner id, host, pid if known, last heartbeat age).
- **Whether that runner is still considered online** per registry lease semantics (see `POST /api/agent/runners/heartbeat`, `GET /api/agent/runners`).
- **Whether the provider subprocess / tool process is still running** on the runner host when that signal exists (local runner: PID check or equivalent; remote runners: best-effort signal documented in implementation).

**Acceptance**: Reap decisions are logged with a structured `reap_decision` record that includes `runner_online`, `runner_last_seen_ago_sec`, `provider_process_running`, and `candidate_reason` (e.g. `age_exceeded`, `runner_dead`, `orphan_running`).

### R2 — Extend timeout instead of kill when work is active but slow

If the runner is alive, the provider process is running, and **progress signals** indicate ongoing work (non-stale log growth, recent `POST /api/agent/tasks/{task_id}/activity` events, or increasing partial output size), the system MUST **extend** the effective deadline (bump `context.reap_extension_until` or equivalent) instead of reaping immediately.

Caps: maximum extensions per task and maximum total wall time MUST be configurable (env or settings) to prevent infinite extension.

**Acceptance**: Slow-but-healthy tasks are not marked `timed_out` solely for crossing the initial threshold; they receive at least one extension when progress signals are present.

### R3 — Capture stderr / partial output on crash

If the provider process is **not** running and the runner is dead or idle with the task still `running`, the reap path MUST attach:

- Tail of **task log on disk** (existing convention: `task_{task_id}.log` under runner log dir).
- **Stderr capture** if separately available from the executor wrapper.
- **Checkpoint file** (e.g. `.task-checkpoint.md` in worktree) and optional **git patch** path when present (already partially implemented).

**Acceptance**: `context.reap_capture` contains `log_tail`, `stderr_excerpt`, `checkpoint_excerpt`, `patch_path`, each bounded by max length (e.g. 8–32 KiB total) with truncation metadata.

### R4 — Structured diagnosis in task context

On reap, PATCH payload MUST merge into `context` (in addition to human-readable `output` / `error_summary`):

- `reap_diagnosis.provider` — resolved provider id (from context / route_decision).
- `reap_diagnosis.error_class` — enum: `slow_progress_extended`, `runner_dead`, `provider_exited`, `provider_stalled`, `orphan_task`, `unknown`.
- `reap_diagnosis.partial_output_ratio` — float 0.0–1.0 vs **expected output heuristic** (see Data model).
- `reap_diagnosis.has_partial_output` — boolean.
- `reap_diagnosis.captured_at` — ISO 8601 UTC.

This aligns with spec 113 categories where possible; new classes extend the taxonomy for reap-specific states.

### R5 — Resume task when partial output is material (>20%)

If `partial_output_ratio > 0.2` **or** checkpoint/patch exists with non-trivial content, create a **new** pending task (same `task_type`, same `idea_id` linkage) with:

- `context.resume_of_task_id` = prior task id  
- `context.partial_work_attachment` = bounded text (log + checkpoint summary)  
- `seed_source` = `smart_reap_resume` (distinct from `reaper_retry`)

**Direction text (required template)**:

> Previous attempt produced this partial work [attached]. Continue from where it left off.

The bracketed attachment MUST be populated from `partial_work_attachment` (not an empty placeholder).

### R6 — Per-idea reap history and human attention flag

For each `idea_id`, persist rolling metrics:

- `reap_events` count (total and last 24h / 7d).
- `consecutive_timeout_or_reap` streak.
- `last_reap_at`, `last_error_class`.

When the same idea crosses thresholds (default: **≥3 reap/timeout events in 7d** or **≥2 consecutive** with no successful phase completion), the system MUST:

- Set `context.idea_attention_required = true` on new tasks **or** PATCH the idea with a machine-readable flag (see API changes).
- Emit `POST /api/friction/events` with `block_type` e.g. `repeated_reap_idea` (distinct from existing `repeated_timeout` where semantics differ).
- **Stop** auto-seeding new retries for that idea until a human clears the flag (or lowers severity).

### R7 — Observability / proof hooks

Expose metrics or a read API so operators can answer: “Is Smart Reap working?” without reading raw logs.

- Counters: `smart_reap_extended`, `smart_reap_killed`, `smart_reap_resume_created`, `smart_reap_idea_flagged`.
- Optional: `GET /api/agent/reap/summary` (see API changes).

## API Changes

### Existing endpoints (behavior extensions)

| Method | Path | Change |
|--------|------|--------|
| `PATCH` | `/api/agent/tasks/{task_id}` | When status → `timed_out` from reap, server MUST merge `context.reap_diagnosis` and `context.reap_capture` (schema below). Idempotent PATCH with same diagnosis hash should not duplicate friction events. |
| `POST` | `/api/agent/tasks` | Resume tasks created by smart reap MUST include `context.seed_source=smart_reap_resume` and `context.resume_of_task_id`. |
| `POST` | `/api/friction/events` | New `block_type` values: `repeated_reap_idea`, optional `smart_reap_extension_exhausted`. |
| `GET` | `/api/agent/tasks?status=running` | No schema break; clients use for verification. |
| `POST` | `/api/agent/runners/heartbeat` | Runners SHOULD send `active_task_id`, `active_run_id`, and optional `provider_pid` when running a task (additive JSON fields, backward compatible). |
| `GET` | `/api/agent/runners` | Used in verification to confirm runner online before/after reap. |
| `POST` | `/api/agent/tasks/{task_id}/activity` | Progress signals for extension decisions (event types e.g. `agent_heartbeat`, `tool_progress`). |
| `PATCH` | `/api/ideas/{idea_id}` | May set `manifestation_status` or a new optional field `pipeline_attention` / `reap_hold` (see Data model) when idea is flagged. |

### New or optional endpoints

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/agent/reap/summary` | Returns aggregate counts, top ideas by reap frequency, last 24h extensions vs kills. Auth: same as other agent admin routes. |
| `GET` | `/api/ideas/{idea_id}/reap-history` | Returns reap timeline for the idea (events, error classes, linked task ids). |

If a dedicated read API is deferred, the spec still requires **persisted** `context` on tasks so `GET /api/agent/tasks/{id}` proves behavior.

## Data Model

### Task `context` extensions (JSON, additive)

```yaml
reap_diagnosis:
  provider: string
  error_class: enum
  partial_output_ratio: float  # 0.0 - 1.0
  has_partial_output: bool
  captured_at: string          # ISO 8601 UTC
  runner_id: string
  runner_online: bool
  provider_process_running: bool

reap_capture:
  log_tail: string
  stderr_excerpt: string
  checkpoint_excerpt: string
  patch_path: string | null
  truncation: { log_chars: int, stderr_chars: int }

reap_extension:
  count: int
  extended_until: string       # ISO 8601 UTC
  reason: string

resume_of_task_id: string       # set on resume tasks
partial_work_attachment: string # bounded blob for direction injection

idea_reap_history:             # denormalized snapshot on task or fetched via idea API
  total_reaps: int
  window_7d_reaps: int
  consecutive_failures: int
  attention_required: bool
```

### Partial output ratio heuristic

`partial_output_ratio` MUST be defined deterministically in implementation, for example:

- `min(len(log_tail), len(expected_direction)) / max(len(expected_direction), 1)` with clamp, **or**
- Token/line-count estimate vs `direction` length and task-type baseline.

Document the exact formula in code comments and in this spec’s implementation PR.

### Idea record (optional columns or JSON)

- `reap_hold: bool` — when true, seeders skip auto task creation for this idea.
- `last_reap_summary: Optional[str]` — short operator-facing string.

## Verification Scenarios

### Scenario 1 — Full cycle: create task, observe extension, complete (happy path)

- **Setup**: API running; runner registered via `POST /api/agent/runners/heartbeat` with `status=running`, `active_task_id=task_A`, `lease_seconds=300`. Task `task_A` exists with `status=running`, `context.idea_id=idea_test`, age under threshold initially.
- **Action**: Simulate progress: `curl -s -X POST "$API/api/agent/tasks/task_A/activity" -H "Content-Type: application/json" -d '{"event_type":"tool_progress","node_id":"n1","provider":"codex","data":{"bytes_written":1024}}'`. Advance time or lower threshold in test env so reaper evaluates `task_A` as “old” but with fresh activity.
- **Expected**: Task remains `running` OR `context.reap_extension.count` increments; HTTP `PATCH` to `timed_out` does **not** occur without extension exhaustion. `GET /api/agent/tasks/task_A` shows `context.reap_extension.extended_until` when extension path used.
- **Edge**: If activity stops and provider PID is gone, task eventually `timed_out` with `error_class=provider_exited` or `runner_dead`, not `slow_progress_extended`.

### Scenario 2 — Error handling: dead runner + orphan task

- **Setup**: `task_B` in `running`, runner heartbeat stale (no heartbeat for > lease + grace). Registry lists runner offline via `GET /api/agent/runners` (default excludes stale).
- **Action**: Trigger reap job (or call internal function in tests).
- **Expected**: `PATCH` result `status=timed_out`, `error_category` consistent with diagnostics spec, `context.reap_diagnosis.runner_online=false`, `context.reap_diagnosis.error_class` in `{runner_dead, orphan_task}`.
- **Edge**: Double reap request must not create duplicate resume tasks (idempotent guard on `resume_of_task_id`).

### Scenario 3 — Partial work resume prompt

- **Setup**: Task log on disk exceeds 20% heuristic vs expected direction; checkpoint file exists in worktree path used by runner tests (`tmp_path`).
- **Action**: Run reap path.
- **Expected**: New task `task_C` from `POST /api/agent/tasks` with `context.seed_source=smart_reap_resume`, `context.resume_of_task_id=task_prior`, and `direction` containing verbatim: `Previous attempt produced this partial work [attached]. Continue from where it left off.` with non-empty attachment (not literal `[attached]` only).
- **Edge**: If partial ratio ≤ 0.2 and no checkpoint, no resume task OR resume only when explicit patch present (per R5 — document branch in implementation).

### Scenario 4 — Per-idea history and friction (create → read → flag)

- **Setup**: Idea `idea_flap` with synthetic history of 3 timeout/reap events in 7d (via test fixture or PATCHed context on prior tasks).
- **Action**: `curl -s "$API/api/ideas/idea_flap"` then trigger smart reap threshold.
- **Expected**: `POST /api/friction/events` payload retrievable or logged; `GET /api/ideas/idea_flap` shows hold/attention field OR new tasks include `context.idea_attention_required=true`. `GET /api/agent/tasks?status=pending` does not grow unlimited retries for same idea.
- **Edge**: Human clears hold via `PATCH /api/ideas/idea_flap` with `reap_hold=false` — seeding resumes.

### Scenario 5 — Read API proof for operators

- **Setup**: Production or staging with Smart Reap deployed.
- **Action**: `curl -s "$API/api/agent/reap/summary"` **or** `curl -s "$API/api/agent/tasks?status=timed_out&limit=5"` and inspect first task JSON.
- **Expected**: Response includes non-zero `smart_reap_*` counters **or** each task shows `context.reap_diagnosis` object with `error_class` and `captured_at`.
- **Edge**: Unauthorized request returns `401`/`403` (not `500`).

## Risks and Assumptions

- **Assumption**: Runner registry heartbeats are authoritative for “runner alive.” Misconfigured runners may appear dead while working — mitigated by PID + log progress signals.
- **Risk**: PID checks are OS-specific and may not apply to containerized remote runners — implementation MUST degrade gracefully to log/activity-based signals only.
- **Risk**: Extending timeouts can starve queue — cap extensions and surface `smart_reap_extension_exhausted` friction.
- **Assumption**: “Expected output” for ratio is heuristic; wrong baseline may over/under-resume — tune per `task_type`.

## Out of Scope

- Changing provider billing, model routing, or executor command templates (see `agent_service_executor.py` policy).
- Full web dashboard UI for reap timelines (optional later; API-first in this spec).
- Backfilling historical `timed_out` tasks with new diagnosis fields (follow-up migration).

## Known Gaps and Follow-up Tasks

- Define exact maximum sizes for `partial_work_attachment` to respect model context limits.
- Cross-region runner clock skew: use server time for all `captured_at`.
- Web UI surfacing (`/tasks` page) for reap diagnosis — optional follow-up spec.

## Open Questions — Improving the Idea, Proof, and Clarity Over Time

1. **Success metrics**: Track rolling **reap→success rate** per idea and globally; dashboard compares before/after Smart Reap deploy (tie to `GET /api/agent/reap/summary`).
2. **Proof clarity**: Publish a short **operator runbook** snippet: three curls (runner list, task get, friction list) that demonstrate a healthy vs unhealthy reap.
3. **A/B or feature flag**: `SMART_REAP_ENABLED=1` per environment to compare timeout rates without blocking rollout.
4. **Human-in-the-loop**: When `attention_required`, auto-create a single **review** task instead of impl retry — ties to `task_type=review` pipeline.

## Failure / Retry Reflection

- **Failure mode**: Extension logic too aggressive → queue stalls. **Next action**: tighten progress signal definition, lower max extensions.
- **Failure mode**: Resume tasks duplicate work. **Next action**: dedupe by `resume_of_task_id` + `task_fingerprint`.

## Decision Gates

- Product approval on thresholds (20% partial, 3 events / 7d) before enabling in production automation.
- Security review if reap captures may include secrets from logs — apply redaction filters.
