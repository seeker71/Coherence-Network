## Summary

The Coherence Network currently detects "resonance" between ideas by comparing sets of
tokens extracted from text fields (name, description, tags, interfaces). This is keyword
matching — an idea about *symbiosis* resonates with an idea about *microservices* only if
they share literal words. They won't.

True concept resonance is **structural**: two ideas resonate when they occupy analogous
positions in the graph — when they solve the same class of problem in different domains,
even if they share no vocabulary. A food-web's *keystone species* and a software system's
*API gateway* both act as high-betweenness bridges that, if removed, fragment their graphs.
They are structurally resonant. The words never overlap.

This spec defines the data model, scoring algorithm, API endpoints, proof surface, and
verification contract for **graph-structural concept resonance** — a system that surfaces
cross-domain connections automatically, without human curation, by reading the shape of the
graph itself.

## Motivation

### The keyword trap

The existing `GET /api/ideas/{id}/concept-resonance` endpoint uses
`_idea_concept_tokens()` and `_idea_domain_tokens()` to extract word-level tokens, then
scores by Jaccard overlap (`shared_concepts / union_concepts`). This has a fundamental
ceiling: it can only surface connections the author already expressed in their language.

A biologist writes about *mutualistic symbiosis*. A software engineer writes about
*microservice mesh*. The systems are structurally identical — multiple autonomous agents
providing services to each other with no central orchestrator, mutual dependency, emergent
resilience — but the token sets are disjoint. The existing engine returns `resonance_score:
0.0`.

### Structural similarity is the signal

The ontology assigns each concept a position in a graph. Every node has:
- **Edge types** — which relationship types connect it (causes, enables, analogous_to,
  depends_on, etc.)
- **Neighborhood topology** — the degree distribution and types of its k-hop neighbors
- **Domain annotation** — which ontology domain the concept belongs to
- **Role profile** — whether it acts as hub, bridge, leaf, or cycle node

Two concepts that share none of the same words can share the same structural fingerprint:
same role, same edge-type distribution, same neighborhood topology relative to their
domain. When that fingerprint matches across domains, the connection is real and useful —
it tells contributors "this insight from biology already solved the problem you are working
on in software."

### How the ontology grows organically

When structural resonance surfaces a connection, it creates two effects:

1. **Immediate value**: the engineer discovers a biological pattern they hadn't considered.
2. **Graph enrichment**: the system records the connection as an `analogous_to` edge between
   the two concepts, strengthening the ontology for all future resonance queries.

The ontology does not need a curator. It grows by usage: every resonance event is a
potential edge addition. Over time, the graph acquires a dense cross-domain resonance layer
built from machine-detected structural similarities, validated when contributors act on them.

## Goals

1. **Structural fingerprint computation** — compute a fingerprint for each concept/idea node
   based on: edge-type distribution, k-hop neighbor type distribution, betweenness
   centrality bucket, and domain annotation.
2. **Structural resonance scoring** — score candidate pairs by fingerprint similarity (not
   token overlap). Score = weighted combination of role match, edge-profile cosine
   similarity, and domain novelty bonus.
3. **Cross-domain gate** — only surface matches where source and candidate belong to
   different ontology domains. Same-domain structural similarities exist but are not the
   goal of this spec.
4. **Resonance edge creation** — when a match exceeds a confidence threshold, automatically
   propose an `analogous_to` edge between the two concepts. The edge enters a `proposed`
   state; it does not auto-commit without contributor confirmation.
5. **Proof surface** — `GET /api/resonance/proof` returns aggregate evidence that the
   engine is working: match counts, domain pair coverage, top-scoring pairs, edge creation
   rate over time.
6. **Deprecation of pure keyword fallback** — the existing token-overlap scorer becomes the
   fallback when structural data is unavailable (new or uncategorized ideas), not the
   primary scorer.
7. **CLI integration** — `cc resonate <idea-id>` returns structural matches with
   explanation of why they resonate.

## Architecture

### Data model additions

#### `ConceptFingerprint` (in-memory, computed on demand, cached)

```python
class ConceptFingerprint(BaseModel):
    concept_id: str
    domain: str                          # ontology domain tag
    role: str                            # hub | bridge | leaf | cycle
    edge_type_distribution: dict[str, float]  # normalized counts per edge type
    neighbor_type_distribution: dict[str, float]  # normalized node-type counts in 2-hop
    betweenness_bucket: int              # 0=periphery, 1=low, 2=medium, 3=hub
    computed_at: str                     # ISO 8601 UTC
```

Role classification:
- **hub**: degree > 75th percentile in its domain subgraph
- **bridge**: high betweenness, moderate degree (critical path node)
- **leaf**: degree ≤ 1 (terminal node)
- **cycle**: part of a detected cycle in its subgraph

#### `StructuralResonanceMatch` (response model)

```python
class StructuralResonanceMatch(BaseModel):
    source_id: str
    candidate_id: str
    candidate_name: str
    structural_score: float              # 0.0–1.0
    role_match: bool                     # same structural role?
    edge_profile_similarity: float       # cosine sim of edge-type distributions
    domain_novelty: float                # how different are the domains?
    source_domain: str
    candidate_domain: str
    explanation: str                     # human-readable: "both act as bridges..."
    proposed_edge_id: str | None         # set if edge was auto-proposed
    scorer_version: str                  # "structural-v1" or "keyword-fallback"
```

#### `ResonanceProof` (aggregate evidence)

```python
class ResonanceProof(BaseModel):
    total_pairs_evaluated: int
    total_matches_found: int
    total_proposed_edges: int
    domain_pairs_covered: list[dict]     # [{"source": "biology", "target": "software", "count": 12}]
    top_matches: list[StructuralResonanceMatch]  # top 5 by score
    match_rate_7d: float                 # matches / evaluations over last 7 days
    engine_status: str                   # "structural" | "keyword-fallback" | "hybrid"
    last_evaluation_at: str             # ISO 8601 UTC
```

### Scoring algorithm

```
score(A, B) =
    w_role  * role_match_bonus(A, B)       # 0.0 or 0.35
  + w_edge  * cosine(A.edge_type_dist, B.edge_type_dist)  # 0.0–0.40
  + w_nhbd  * cosine(A.neighbor_type_dist, B.neighbor_type_dist)  # 0.0–0.20
  + w_novel * domain_novelty(A, B)        # 0.0–0.15

where:
  role_match_bonus = 0.35 if A.role == B.role else 0.0
  domain_novelty   = 1.0 if domains are disjoint else 0.0
  w_role=1.0, w_edge=1.0, w_nhbd=1.0, w_novel=1.0 (then normalize to 0–1)
```

The cross-domain gate: if `source_domain == candidate_domain`, skip (score = 0.0,
not returned).

Minimum score threshold: `0.15` (configurable via query param).

### Fallback behavior

If a concept/idea has fewer than 2 edges in the graph (insufficient structural data),
the scorer falls back to the existing keyword-token approach and annotates the response
with `scorer_version: "keyword-fallback"`. This prevents silent degradation for new ideas.

## Proof Contract — How We Know It's Working

The system must be able to answer: *"Is this actually producing structural matches, or just
returning empty lists?"*

### Proof signals

1. **Match rate** — `total_matches_found / total_pairs_evaluated > 0.01` (at least 1%
   of evaluated pairs produce a match). If this falls to 0, the engine is broken.

2. **Cross-domain coverage** — at least 3 distinct domain pairs appear in
   `domain_pairs_covered`. If all matches are within one domain, the cross-domain gate is
   broken.

3. **Proposed edge creation rate** — `total_proposed_edges > 0` after 50+ evaluations.
   If no edges are ever proposed, the confidence threshold may be too high or data too sparse.

4. **Scorer version distribution** — `GET /api/resonance/proof` shows both
   `"structural"` and `"keyword-fallback"` matches in the result set (proves the hybrid
   works, not just one path).

5. **Fingerprint stability** — calling `GET /api/resonance/fingerprint/{id}` twice on the
   same idea returns identical `role` and `edge_type_distribution` values (fingerprints
   are deterministic).

## Implementation Notes

### Fingerprint computation

Fingerprints are computed from the SQLite graph (nodes + edges tables). The algorithm:

1. Fetch all edges where `from_node_id = concept_id` OR `to_node_id = concept_id`.
2. Count edge types → normalize to probability distribution.
3. Fetch 2-hop neighbors (via one intermediate node) → count their node types → normalize.
4. Compute degree: if degree > 75th percentile of same-domain nodes → role=hub.
   If betweenness (approximated as ratio of shortest paths through this node) > threshold → role=bridge.
   If degree ≤ 1 → role=leaf. Else → role=cycle (default).
5. Domain = first tag that matches a known ontology domain list
   (`["biology","software","physics","economics","social","mathematics","language","philosophy"]`).

For performance: fingerprints are computed on first request and cached in-process (TTL=300s).
The `POST /api/resonance/evaluate` call precomputes all fingerprints as a side effect.

### Cosine similarity

```python
def cosine_sim(a: dict[str, float], b: dict[str, float]) -> float:
    keys = set(a) | set(b)
    if not keys:
        return 0.0
    va = [a.get(k, 0.0) for k in keys]
    vb = [b.get(k, 0.0) for k in keys]
    dot = sum(x * y for x, y in zip(va, vb))
    norm_a = sum(x ** 2 for x in va) ** 0.5
    norm_b = sum(x ** 2 for x in vb) ** 0.5
    if norm_a == 0 or norm_b == 0:
        return 0.0
    return dot / (norm_a * norm_b)
```

### Proposed edge creation

When `structural_score >= 0.60` AND `cross_domain=True`:
- Call `graph_service.create_edge()` with `type="analogous_to"`, `created_by="resonance-engine"`,
  `properties={"proposed": True, "structural_score": score, "auto": True}`.
- The edge is created in `proposed` state (the `proposed` property = True).
- Contributors can confirm by removing the `proposed` flag via `PATCH /api/graph/edges/{id}`.

## Risks and Assumptions

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Graph too sparse to compute meaningful fingerprints | High (early stage) | Keyword fallback activated automatically; proof endpoint shows engine_status |
| Cosine similarity biased toward ideas with same edge types but different semantics | Medium | Domain novelty bonus rewards cross-domain matches; role gate filters same-domain |
| Proposed edge spam if threshold too low | Medium | Default threshold 0.60 is conservative; `proposed=True` flag requires human confirmation |
| Performance: O(n²) pair evaluation | Medium | Bounded by `limit` param; `evaluate` endpoint uses early-exit at 500ms |
| Betweenness centrality approximation inaccurate on small graphs | Low | Exact betweenness for graphs < 500 nodes; approximate for larger (future) |

## Known Gaps and Follow-up Tasks

- **Semantic embedding integration** (Phase 2): combine structural fingerprint with
  sentence-transformer embeddings for ideas with rich text but sparse edges.
- **User-confirmable proposed edges**: UI for contributors to accept/reject proposed
  `analogous_to` edges. Currently only API-level.
- **Resonance history**: track which pairs have been evaluated over time to compute
  `match_rate_7d` accurately. Currently approximated.
- **Domain vocabulary expansion**: the hardcoded 8-domain list needs to grow. Should be
  loaded from the ontology axes (53 axes already defined in `concept_service`).
- **CLI command `cc resonate`**: full implementation with human-readable output.
  This spec defines the API contract; CLI implementation is a follow-up task.
- **Feedback loop**: when a user acts on a resonance suggestion (forks an idea, creates a
  contribution), mark the resonance match as `validated=True` for proof tracking.
