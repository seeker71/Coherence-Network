# Spec 162: Identity-Driven Onboarding — `cc setup` Auto-Creates Contributor + API Key

**Idea ID**: `identity-driven-onboarding`
**Task ID**: `task_957a8a7e00501874`

**Author**: product-manager agent (task_957a8a7e00501874)

## Summary

New contributors cannot participate in Coherence Network (record ideas, vote, earn CC) without
a personal API key. Currently the only path is manual: know the dev API key, curl the right
endpoint, copy-paste the key into `~/.coherence-network/keys.json`. This is a silent onboarding
blocker — most first-time users silently fail rather than ask for help.

This spec defines **`cc setup`**: an interactive 7-step terminal wizard that asks your name,
links one identity from 40+ providers, auto-creates a contributor record, generates a personal
API key scoped to that contributor, and writes it into `~/.coherence-network/keys.json`. All
future `cc` commands then use this key automatically. No shared dev-key, no manual setup.

---

## Problem Statement

### Current Pain Points

| Symptom | Root cause |
|---------|-----------|
| `cc contribute` fails with "No API key" | No key in `keys.json`, no setup flow |
| `cc ideas` works (read-only) but `cc share` fails silently | Write routes require key; new contributors have none |
| Governance votes rejected | No contributor record linked to key |
| Contributors credited as `"system"` | Shared dev-key does not carry identity |

### Who This Affects

- **First-time local contributors**: running `cc` for the first time after cloning the repo
- **Web users**: clicking "Connect" on coherencycoin.com should trigger an equivalent web flow
- **CI agents**: new agent nodes need programmatic onboarding via the same API endpoints

## Requirements

### R1 — `cc setup` CLI Command

- **R1.1** Running `cc setup` when no API key exists in `~/.coherence-network/keys.json` launches
  the interactive wizard.
- **R1.2** Running `cc setup` when a valid key already exists must prompt: `"You already have a
  key for <contributor_id>. Re-run setup? [y/N]"` and abort unless the user confirms.
- **R1.3** Any `cc` command that requires authentication (`cc contribute`, `cc share`, `cc stake`,
  `cc vote`) and finds no key must print: `"No API key found. Run: cc setup"` and exit with
  code 1 — never silently fail.

### R2 — Interactive Wizard Steps

The wizard must execute these steps in order, with clear prompts:

**Step 1 — Name**
- Prompt: `"What is your contributor name? (letters, numbers, hyphens, max 32 chars)"`
- Validate: `^[a-zA-Z0-9][a-zA-Z0-9\-]{0,31}$`
- Error on blank or invalid: `"Invalid name. Use letters, numbers, hyphens only."`

**Step 2 — Identity provider selection**
- Prompt: `"Link an identity [github / email / ethereum / other]:"`
- Show numbered list of categories: Social, Dev, Crypto/Web3, Professional
- Accept: provider key (e.g. `github`) or provider number
- Required: at least one identity must be selected
- Providers available: all 40+ from `GET /api/identity/providers`

**Step 3 — Identity handle input**
- Prompt is provider-specific: e.g. `"GitHub username:"`, `"Email address:"`, `"ETH address (0x...):"`
- Validate format per provider: GitHub regex `^[a-zA-Z0-9\-]{1,39}$`, email via RFC 5322
  pattern, Ethereum `^0x[0-9a-fA-F]{40}$`

**Step 4 — Verification**
- For MVP: **trust-on-first-use (TOFU)** — no live proof required. Identity is linked with
  `verified: false`. Verification can be upgraded later via `POST /api/auth/verify/challenge`.
- Show: `"Identity linked (unverified). You can verify ownership later with: cc verify"`
- Future: GitHub OAuth (`can_oauth: true`) upgrades to `verified: true` automatically.

**Step 5 — Auto-create contributor record**
- Call `POST /api/contributors` with `{name: <name>, contributor_type: "HUMAN"}`
- Idempotent: if contributor already exists (409), continue — do not abort.
- Show: `"Contributor record created: <name>"`

**Step 6 — Auto-generate personal API key**
- Call `POST /api/auth/keys` with `{contributor_id, provider, provider_id}`
- API generates key with scopes: `["own:read", "own:write", "contribute", "stake", "vote"]`
- Key is returned **once** — the server only stores the hash.
- Show: `"API key generated: cc_<name>_<hex16>"`

**Step 7 — Persist key to disk**
- Write key to `~/.coherence-network/keys.json`:
  ```json
  {
    "api_key": "cc_<name>_<hex16>",
    "contributor_id": "<name>",
    "provider": "<provider>",
    "provider_id": "<handle>",
    "created_at": "<ISO8601>"
  }
  ```
- Create `~/.coherence-network/` directory if missing. Set file mode `0o600`.
- Show: `"Key saved to ~/.coherence-network/keys.json"`
- Show final summary: `"Setup complete! You are now: <name> (via <provider>:<handle>). Run 'cc status' to verify."`

### R3 — Automatic Key Discovery

- **R3.1** All `cc` commands must load `api_key` from `~/.coherence-network/keys.json` if present,
  and include it as `X-API-Key: <key>` header on all API requests.
- **R3.2** Legacy `CC_API_KEY` environment variable remains supported as override (higher priority
  than file). Env var should not be removed — agents use it.
- **R3.3** The key loading order: `CC_API_KEY` env var → `keys.json` → anonymous (read-only).

### R4 — API Endpoints (backend)

All endpoints in this spec already exist or must be confirmed functional:

- **R4.1** `GET /api/identity/providers` — returns 40+ providers grouped by category ✓ (exists)
- **R4.2** `POST /api/identity/link` — links an identity to a contributor ✓ (exists)
- **R4.3** `POST /api/auth/keys` — generates a personal API key ✓ (exists in `auth_keys.py`)
- **R4.4** `GET /api/auth/whoami` — returns contributor info for the current key ✓ (exists)
- **R4.5** `POST /api/auth/verify/challenge` — creates a verification challenge ✓ (exists)
- **R4.6** `POST /api/auth/verify/proof` — submits proof to mark identity verified ✓ (exists)
- **R4.7** `POST /api/contributors` — creates a contributor record (must return 409 on duplicate,
  not 500)

### R5 — Key Scope Enforcement

- **R5.1** The API key scopes are: `own:read`, `own:write`, `contribute`, `stake`, `vote`
- **R5.2** `own:write` scope means: the key owner can edit/delete their own ideas, contributions,
  and identity links — not other contributors' records.
- **R5.3** Role-based expansion (admin, moderator) is out of scope for this spec. Current MVP
  uses contributor-scoped CRUD only.
- **R5.4** The `/api/auth/whoami` endpoint must reflect the actual scopes attached to the key.

### R6 — `cc verify` Command (post-setup upgrade)

- **R6.1** `cc verify --provider github` fetches a challenge token, prints instructions for
  creating a gist, waits for user input (the gist URL), then submits proof.
- **R6.2** On success: updates `keys.json` to add `verified: true` and prints `"GitHub identity
  verified!"`.
- **R6.3** This is optional post-setup — setup does not block on verification.

---

## Decision: Trust-on-First-Use (TOFU) for MVP

**Yes, TOFU is the right approach for MVP.** Rationale:

1. The bottleneck today is onboarding friction, not identity fraud. Requiring OAuth adds
   redirect flows, env var configuration (client_id/secret), and a browser dependency — all of
   which block contributors who are working in headless/SSH environments.
2. TOFU with `verified: false` still creates a unique contributor record tied to an identity
   handle. This is sufficient for attribution and governance (CC balance is earned, not claimed).
3. GitHub OAuth (`can_oauth: true` in provider registry) is already scaffolded and can be added
   as an optional upgrade path without changing the core flow.
4. Ethereum signature verification (`can_verify: true`) works natively in CLI with no browser —
   suitable as a first "strong verification" option.

**TOFU → OAuth upgrade path**: After MVP, `cc verify --provider github` will do a device-flow
OAuth (no browser redirect needed) to upgrade `verified: false` to `verified: true`.

## Data Model

### `~/.coherence-network/keys.json` (local, client-side)

```json
{
  "api_key": "cc_alice_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
  "contributor_id": "alice",
  "provider": "github",
  "provider_id": "alice-gh",
  "created_at": "2026-03-28T06:00:00Z",
  "verified": false
}
```

### API Key Store (server-side, `auth_keys.py` `_KEY_STORE`)

Currently in-memory. Production must persist to PostgreSQL `contributor_keys` table:

```yaml
contributor_keys:
  id: { type: uuid, pk: true }
  contributor_id: { type: string, fk: contributors.id }
  key_hash: { type: string, unique: true }  # sha256 of raw key
  provider: { type: string }
  provider_id: { type: string }
  scopes: { type: json }  # ["own:read", "own:write", "contribute", "stake", "vote"]
  created_at: { type: timestamptz }
  revoked_at: { type: timestamptz, nullable: true }
```

**Gap**: The current `_KEY_STORE` in `auth_keys.py` is in-memory and resets on API restart.
This means keys are lost on deploy. Follow-up task: persist to PostgreSQL.

## API Contract

### `POST /api/auth/keys`

**Request**
```json
{
  "contributor_id": "alice",
  "provider": "github",
  "provider_id": "alice-gh"
}
```

**Response 201**
```json
{
  "api_key": "cc_alice_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
  "contributor_id": "alice",
  "created_at": "2026-03-28T06:00:00Z",
  "scopes": ["own:read", "own:write", "contribute", "stake", "vote"]
}
```

**Response 400** — identity link failed
```json
{ "detail": "Could not link identity: <reason>" }
```

### `GET /api/auth/whoami`

**Header**: `X-API-Key: cc_alice_...`

**Response 200 (authenticated)**
```json
{
  "authenticated": true,
  "contributor_id": "alice",
  "provider": "github",
  "provider_id": "alice-gh",
  "created_at": "2026-03-28T06:00:00Z",
  "scopes": ["own:read", "own:write", "contribute", "stake", "vote"]
}
```

**Response 200 (unauthenticated)**
```json
{
  "authenticated": false,
  "message": "No API key provided. Run: cc setup"
}
```

### `GET /api/identity/providers`

**Response 200**
```json
{
  "categories": {
    "Social": [{"key": "github", "label": "GitHub", "placeholder": "username", ...}],
    "Dev": [...],
    "Crypto / Web3": [...],
    "Professional": [...]
  }
}
```

## Verification Scenarios

Each scenario is executable against production at `https://api.coherencycoin.com`.

### Scenario 1 — Happy Path: New Contributor Onboards via GitHub TOFU

**Setup**: No existing contributor named `alice-test-{timestamp}`. No API key in keys.json.

**Action (simulated non-interactive)**:
```bash
# Step 5: Create contributor
curl -s -X POST https://api.coherencycoin.com/api/contributors \
  -H "Content-Type: application/json" \
  -d '{"name":"alice-test-1234","contributor_type":"HUMAN"}'

# Step 6: Generate API key
curl -s -X POST https://api.coherencycoin.com/api/auth/keys \
  -H "Content-Type: application/json" \
  -d '{"contributor_id":"alice-test-1234","provider":"github","provider_id":"alice-gh-test"}'
```

**Expected**:
- Contributor creation returns HTTP 201 with `{"id": "alice-test-1234", ...}`
- Key generation returns HTTP 201 with `{"api_key": "cc_alice-test-1234_<hex16>", "scopes": ["own:read","own:write","contribute","stake","vote"]}`
- Key format matches regex `^cc_[a-zA-Z0-9\-]+_[0-9a-f]{32}$`

**Then**:
```bash
# Step: Verify key works
curl -s https://api.coherencycoin.com/api/auth/whoami \
  -H "X-API-Key: cc_alice-test-1234_<returned_key>"
```
**Expected**: `{"authenticated": true, "contributor_id": "alice-test-1234", ...}`

## Concurrency Behavior

- **Key generation**: `secrets.token_hex(16)` produces a unique key with 2^128 entropy — collision probability is negligible; no locking required.
- **Contributor creation**: `POST /api/contributors` must be idempotent — return 409 on duplicate, never 500.
- **Key store**: Once migrated to PostgreSQL, use `INSERT ... ON CONFLICT DO NOTHING` for the key hash.

## Risks and Assumptions

- **Risk**: `_KEY_STORE` in `auth_keys.py` is in-memory and resets on API restart. Keys generated
  before deploy are lost. **Mitigation**: for MVP, document this limitation clearly; add
  PostgreSQL persistence as next immediate follow-up task.
- **Risk**: `POST /api/contributors` may return 500 instead of 409 on duplicate.
  **Mitigation**: `auth_keys.py` already wraps this call with `except Exception: pass`, so
  key generation still succeeds even if contributor creation fails. Fix the 409 behavior in
  the contributors router separately.
- **Assumption**: `auth_keys.router` is included in `api/app/main.py`. If not, all `/api/auth/*`
  endpoints return 404 — verify by hitting `GET /api/auth/whoami` before any other testing.
- **Assumption**: `~/.coherence-network/` directory exists or `cc setup` creates it. Code must
  handle `FileNotFoundError` and create the directory with `mkdir(parents=True)`.
- **Risk**: User runs `cc setup` in a CI environment without a TTY. **Mitigation**: detect
  `sys.stdin.isatty()` and fall back to `--name`, `--provider`, `--provider-id` CLI flags for
  non-interactive use.

## Failure/Retry Reflection

- **Failure mode**: User runs `cc contribute` before setup — command crashes with unhelpful KeyError
  - **Blind spot**: Assumed `keys.json` always exists; no graceful missing-key path
  - **Next action**: Add `if not api_key: print("No key. Run: cc setup"); sys.exit(1)` guard in `cc.py`

- **Failure mode**: `POST /api/auth/keys` returns 500 because contributor doesn't exist yet
  - **Blind spot**: Key endpoint tries to create contributor but contributor service raises on conflict
  - **Next action**: Wrap contributor creation in try/except in `auth_keys.py` (already done)

- **Failure mode**: Keys lost after deploy because `_KEY_STORE` is in-memory
  - **Blind spot**: Assumed server-side store was persistent
  - **Next action**: Persist to PostgreSQL in the DB migration follow-up task
