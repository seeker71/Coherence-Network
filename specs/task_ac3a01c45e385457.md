# Spec: Submit Coherence Network to 5+ MCP and Skill Registries for Discovery

## Purpose

Coherence Network exposes an MCP server and a suite of agent skills that allow external AI agents,
tools, and users to discover, install, and compose its capabilities. Currently these assets are not
listed in any public registry, so the project is invisible to the broader MCP/agent-skills ecosystem.
This spec describes registering the Coherence Network MCP server and skill on every major registry
(Smithery, Glama/awesome-mcp-servers, PulseMCP, mcp.so, skills.sh, and askill.sh), adding API
endpoints to track install/download counts returned from those registries, and establishing a
lightweight dashboard panel so that growth is visible over time.

## Background

The Model Context Protocol (MCP) ecosystem has a growing set of directories where AI builders
discover servers to integrate. Skill registries (skills.sh, askill.sh) serve a parallel role for
Claude Code / agent skill files. Being listed early maximises organic discovery, attribution, and
trust signals (stars, installs, reviews) that compound over time. This work is a one-time
registration effort combined with ongoing telemetry collection so we can prove the registrations
are driving real usage.

## Requirements

- [ ] **R1** — Coherence Network MCP server package is published to npm under
      `@coherence-network/mcp-server` (or equivalent) with a conformant `package.json` and
      `mcp.json` / `smithery.yaml` manifest so automated registry crawlers can index it.
- [ ] **R2** — A pull request is submitted to `punkpeye/awesome-mcp-servers` (Glama source list)
      that adds Coherence Network to the appropriate category.
- [ ] **R3** — The Coherence Network MCP server is submitted to Smithery
      (`smithery.ai/submit`), PulseMCP (`pulsemcp.com/submit`), and `mcp.so` via their respective
      submission forms or GitHub-based intake flows.
- [ ] **R4** — The Coherence Network skill file (`coherence-network.md` or equivalent) is
      submitted to `skills.sh` and `askill.sh` via their respective submission processes.
- [ ] **R5** — A new API endpoint `GET /api/registry/stats` returns current install/download counts
      scraped or pulled from each registry, keyed by registry name, with a `last_fetched_at`
      timestamp.
- [ ] **R6** — A scheduled background job (APScheduler or equivalent) refreshes registry stats
      every 6 hours and stores results in the PostgreSQL `registry_stats` table.
- [ ] **R7** — The web dashboard (`/registry`) displays a table of registries, links, current
      install counts, and a sparkline of 30-day trend (populated once data accumulates).
- [ ] **R8** — Each registry listing includes a canonical back-link to `https://coherencycoin.com`
      and the GitHub repo so download/install events are attributable.

**Response 503** (all scrapers failed)
```json

```

### `POST /api/registry/stats/refresh`

Triggers an immediate (synchronous) refresh of all registry stats. Admin-only (requires
`X-Admin-Key` header matching env var `ADMIN_KEY`).

**Response 200**
```json
{ "refreshed": 6, "errors": 0, "duration_ms": 1842 }
```

**Response 401** (missing/invalid admin key)
```json
{ "detail": "Unauthorized" }
```

### `GET /api/registry/submissions`

Returns the submission log tracking which registries have been submitted to, with dates and
receipt/PR URLs.

**Response 200**
```json
{
  "submissions": [
    {
      "registry": "smithery",
      "submitted_at": "2026-04-01T10:00:00Z",
      "status": "listed",
      "receipt_url": "https://smithery.ai/server/@coherence-network/mcp-server",
      "notes": "Auto-indexed after npm publish"
    },
    {
      "registry": "glama",
      "submitted_at": "2026-04-02T09:00:00Z",
      "status": "listed",
      "receipt_url": "https://github.com/punkpeye/awesome-mcp-servers/pull/XXXX",
      "notes": "PR merged"
    }
  ]
}
```

## Data Model

```yaml
registry_stats:
  table: registry_stats
  columns:
    id: { type: SERIAL, primary_key: true }
    registry: { type: VARCHAR(64), not_null: true }
    installs: { type: INTEGER, nullable: true }
    stars: { type: INTEGER, nullable: true }
    status: { type: VARCHAR(32), not_null: true, default: "unknown" }
    url: { type: TEXT, nullable: true }
    fetched_at: { type: TIMESTAMPTZ, not_null: true }
  indexes:
    - columns: [registry, fetched_at]
      unique: false

registry_submissions:
  table: registry_submissions
  columns:
    id: { type: SERIAL, primary_key: true }
    registry: { type: VARCHAR(64), not_null: true, unique: true }
    submitted_at: { type: TIMESTAMPTZ, not_null: true }
    status: { type: VARCHAR(32), not_null: true }
    receipt_url: { type: TEXT, nullable: true }
    notes: { type: TEXT, nullable: true }
    updated_at: { type: TIMESTAMPTZ, not_null: true }
```

```markdown
## Usage

Set the API base URL:
export COHERENCE_API=https://api.coherencycoin.com

## Commands

### Search concepts
curl -s "$COHERENCE_API/api/concepts?q=<query>"

### Get idea by ID
curl -s "$COHERENCE_API/api/ideas/<id>"

### Create idea
curl -s -X POST "$COHERENCE_API/api/ideas" \
  -H "Content-Type: application/json" \
  -d '{"title":"<title>","body":"<body>"}'

### Get coherence score
curl -s "$COHERENCE_API/api/coherence/score?a=<id1>&b=<id2>"

### Spawn agent task
curl -s -X POST "$COHERENCE_API/api/agent/tasks" \
  -H "Content-Type: application/json" \
  -d '{"type":"spec","payload":{"title":"<task>"}}'
```

## Verification Scenarios

### Scenario 1 — Registry stats endpoint returns all 6 registries

**Setup:** Service is deployed; background job has run at least once (or test data is seeded).

**Action:**
```bash
curl -s https://api.coherencycoin.com/api/registry/stats
```

**Expected result:**
- HTTP 200
- Response body contains `"stats"` array with exactly 6 entries
- Each entry has keys: `registry`, `url`, `last_fetched_at`, `status`
- `total_installs` field is present and is an integer ≥ 0
- `updated_at` is a valid ISO 8601 timestamp

**Edge case:** If all scrapers fail (network outage), response is HTTP 503 with `{ "detail": "Registry stats temporarily unavailable" }` — never a 500.

---

### Scenario 2 — Admin refresh endpoint updates stats

**Setup:** Service is running with `ADMIN_KEY=test-admin-key` env var set.

**Action:**
```bash
curl -s -X POST https://api.coherencycoin.com/api/registry/stats/refresh \
  -H "X-Admin-Key: test-admin-key"
```

**Expected result:**
- HTTP 200
- Response body: `{ "refreshed": <integer 1-6>, "errors": <integer ≥ 0>, "duration_ms": <integer> }`
- `refreshed + errors == 6`
- Immediately after: `GET /api/registry/stats` returns `last_fetched_at` timestamps within the last 60 seconds

**Edge case:** Call with wrong admin key → HTTP 401 `{ "detail": "Unauthorized" }`.

---

### Scenario 3 — Submissions log is accurate and queryable

**Setup:** At least one registry submission has been recorded in `registry/submissions.json` and seeded into the DB.

**Action:**
```bash
curl -s https://api.coherencycoin.com/api/registry/submissions
```

**Expected result:**
- HTTP 200
- `"submissions"` array contains ≥ 1 entry
- Each entry has: `registry`, `submitted_at`, `status`, `receipt_url`
- `submitted_at` is a valid ISO 8601 timestamp
- `status` is one of: `"pending"`, `"listed"`, `"rejected"`, `"not_submitted"`

**Edge case:** No submissions recorded yet → HTTP 200 with `{ "submissions": [] }` — not 404.

---

### Scenario 4 — Web dashboard renders registry table

**Setup:** Web service is deployed; API `/registry/stats` returns valid data.

**Action:** Navigate to `https://coherencycoin.com/registry` in a browser (or via curl for build verification).

**Expected result:**
- Page returns HTTP 200
- Page contains table with columns: Registry, Status, Installs, Stars, Last Updated
- Each of the 6 registries appears as a row
- "Listed" entries have a clickable link to the registry URL

**Edge case:** If API returns 503, page shows "Registry stats temporarily unavailable" error state — does not crash with unhandled error.

---

### Scenario 5 — Smithery manifest is valid and discoverable

**Setup:** `registry/smithery.yaml` exists in the repository root (or `registry/` subdirectory).

**Action:**
```bash
cat registry/smithery.yaml | python3 -c "import sys, yaml; d = yaml.safe_load(sys.stdin); assert 'name' in d and 'tools' in d and len(d['tools']) >= 1; print('VALID')"
```

**Expected result:** Prints `VALID` — the manifest parses as valid YAML and contains required fields `name` and `tools`.

**Edge case:** Missing `tools` key → assertion error printed, CI fails; spec is not satisfied until the manifest passes this check.

## Concurrency Behavior

- **Read operations** (`GET /api/registry/stats`): Always served from cache (DB row); safe for concurrent access.
- **Write operations** (background refresh job): Single scheduled job; APScheduler `max_instances=1` prevents overlapping runs.
- **Admin refresh**: Synchronous; concurrent calls may produce overlapping DB writes. Last write wins — acceptable for a low-frequency admin operation.

## Out of Scope

- Building a fully custom npm MCP server package (only manifests/metadata files needed for registry listing in Phase 1).
- Automated PR submission (human submits PR; spec only requires the draft file and receipt tracking).
- Paid registry tiers or promoted listings.
- Real-time install webhook ingestion from registries.

## Risks and Assumptions

- **Risk**: Registries may change their submission processes or scraping endpoints after this spec is written. Mitigation: store submission evidence in `registry/submissions.json` so manual re-submission is easy; add a `status: stale` flag when scraping fails for >24h.
- **Risk**: Smithery may require a published npm package before auto-indexing. Assumption: publishing even a thin wrapper package with the correct `mcp.json` manifest is sufficient. If false, a minimal npm package must be created (follow-up task).
- **Risk**: Some registries (skills.sh, askill.sh) may be nascent with low traffic, delivering low signal. Mitigation: track them anyway for first-mover advantage; weight Smithery and Glama more heavily in the dashboard summary.
- **Assumption**: The Coherence Network GitHub repository is public and accessible at the time of submission.
- **Assumption**: `ADMIN_KEY` env var is already set in the production Docker Compose environment.

## Failure/Retry Reflection

- **Failure mode**: Registry scraper returns 429 (rate limited).
  - **Blind spot**: Assumed public endpoints are freely scrapeable without auth.
  - **Next action**: Add exponential backoff; if rate-limited for >1h, fall back to last cached value and set `status: stale`.

- **Failure mode**: `awesome-mcp-servers` PR is not merged for weeks.
  - **Blind spot**: PR review velocity for that repo is unpredictable.
  - **Next action**: Track PR URL in `registry/submissions.json`; follow up after 2 weeks; if rejected, document reason and resubmit with corrections.

- **Failure mode**: npm publish fails due to package name conflict.
  - **Blind spot**: `@coherence-network` npm org may not be claimed yet.
  - **Next action**: Claim the npm org immediately; use `coherence-network-mcp` as fallback package name.

## How This Proves It Is Working

The spec answers the open question *"how do we show whether this is working?"* with three
compounding proof layers:

1. **Submission receipts** — `registry/submissions.json` and `GET /api/registry/submissions`
   provide immutable evidence that each submission was made, with date and receipt URL.
   Even before any traffic, this proves the work was done.

2. **Install/download counters** — `GET /api/registry/stats` returns the current numbers from each
   registry. Numbers above zero prove real users are installing. Numbers growing over time
   (visible in the `/registry` dashboard) prove the registrations are driving compounding
   discovery.

3. **Trend dashboard** — The `/registry` web page with 30-day sparklines makes the growth (or
   stagnation) unambiguous to any stakeholder. Zero growth is also a signal — it prompts
   investigation into whether a listing is stale, miscategorised, or missing a description
   improvement.

This three-layer model (receipt → counter → trend) means the proof gets clearer over time: on day 1
you have receipts; on day 7 you have first install counts; on day 30+ you have trend data.
