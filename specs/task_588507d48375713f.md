## Summary

Four web surfaces—`/automation`, `/usage`, `/remote-ops`, and `/nodes`—currently overlap in purpose: operators see **provider health**, **node federation**, **usage/cost**, **task queue**, and **remote controls** spread across multiple routes. That fragmentation increases cognitive load and makes it hard to answer “is the network healthy?” or “what is executing right now?” without opening several tabs.

This specification defines a **two-surface mental model**:

1. **`/nodes`** — **Infrastructure and control plane**: federated node list, health, system metrics, provider readiness, provider execution statistics, node messaging, and **remote control** (execute token, dispatch, pick-up).
2. **`/pipeline`** — **Execution plane**: task queue, running and completed work, streaks, friction/quality summaries, provider performance, runtime cost, and daily operational summaries.

The legacy routes **`/automation`**, **`/usage`**, and **`/remote-ops`** are **removed as first-class pages** and replaced with **HTTP 301** redirects into `/nodes` or `/pipeline`, plus updated global navigation so users are not offered redundant top-level items.

**Open question addressed — “How do we improve this idea, show whether it is working, and make proof clearer over time?”**  
See **§ Observable proof and evolution** below: versioned freshness chips, section-level `generated_at` display, trend-friendly aggregates (streaks, success rates), and a lightweight **“consolidation health”** checklist on each page (redirect OK, APIs 200, last full refresh time). Over time, add **snapshot comparison** (same metrics week-over-week) using existing automation snapshot APIs where available—without blocking the first implementation.

---

## Purpose

Reduce navigation noise and duplicate concepts; align the product with a **nodes vs pipeline** story that matches how the API is already organized (federation + agent/automation). Prevent operators from missing incidents because information was on a page they do not habitually open.

---

## Requirements

### Functional

1. **Single infrastructure home**: All federation node list, node health/metrics, provider readiness, consolidated provider exec stats, and remote-control actions **must** be reachable from **`/nodes`** without requiring `/automation` or `/remote-ops`.
2. **Single execution home**: Queue depth, running tasks, completed history, streaks, provider performance, usage/daily summary, and runtime cost views **must** be reachable from **`/pipeline`** without requiring `/usage`.
3. **Legacy URLs**: Requests to **`/automation`**, **`/usage`**, and **`/remote-ops`** **must** respond with **301 Moved Permanently** to the targets in **§ Redirect matrix** (implemented via Next.js `redirects()` in `web/next.config.ts` or equivalent).
4. **No dead internal links**: No rendered link in `web/app` or shared components **should** point to the removed paths (grep-clean as a release gate).
5. **No information loss**: Every user-visible widget or section that existed on the removed pages **must** either appear on `/nodes` or `/pipeline`, or have an explicit **Known gap** entry with a follow-up task ID.

### Non-functional

6. **Proof of liveness**: Both `/nodes` and `/pipeline` **must** show **when data was last refreshed** (relative time + optional absolute ISO timestamp from API payloads where present).
7. **Error containment**: If one backing API fails (e.g. federation 500), the page **must** degrade with a **section-level** error state, not a full white-screen crash.

## Redirect matrix (canonical)

| Legacy path        | Destination   | Rationale |
|-------------------|---------------|-----------|
| `/automation`     | `/nodes`      | Provider readiness, garden/automation visualization, node-adjacent stats. |
| `/usage`          | `/pipeline`   | Usage, daily summary, costs, friction—execution and outcomes. |
| `/remote-ops`     | `/nodes`      | Queue visibility may also appear on `/pipeline`, but **operator controls** and dispatch are **node/control-plane** actions per product definition; users bookmarking “ops” land where controls live. **Note:** If the current branch temporarily maps `/remote-ops` → `/pipeline`, implementation **must** converge to this matrix or the spec must be amended with recorded UX rationale. |

## Acceptance criteria

1. Hitting **`/automation`**, **`/usage`**, **`/remote-ops`** in production returns **301** with `Location` matching **§ Redirect matrix**.
2. **`/nodes`** presents: federation nodes, provider readiness + exec stats (as today’s automation page provided), remote control, messaging—without dependency on legacy routes.
3. **`/pipeline`** presents: queue/running/completed views, streaks, provider performance, daily summary / costs—without dependency on legacy routes.
4. Site navigation does not advertise removed routes as primary destinations.
5. Section-level loading/error states behave as specified under **Requirements**.

## Observable proof and evolution (answers the open question)

1. **Immediate proof (ship gate)**  
   - Each primary section shows **“Updated Xm ago”** derived from fetch time or API `generated_at` / `timestamp` fields.  
   - A compact **API status row** on each page (green/yellow/red) for the **critical** calls (`/api/federation/nodes`, `/api/agent/pipeline-status`, `/api/automation/usage/readiness`).

2. **Proof over days/weeks**  
   - Reuse **automation snapshots** or daily summary fields to show **trend arrows** (success rate up/down, friction down) where data exists.  
   - Optional follow-up: **exportable JSON** of the same summary operators see (no new backend if served from existing endpoints).

3. **Proof the consolidation itself works**  
   - **Redirect monitor**: periodic smoke (CI or scheduled workflow) hits the three legacy URLs and asserts 301 + Location.  
   - **Link crawler**: fail if any production HTML contains `href="/automation"` etc.

4. **Improving the idea**  
   - After launch, collect **time-on-task** informally: “find queue depth” and “dispatch a task” should require **one** page each. If not, iterate with tabs or sticky sub-nav **inside** `/nodes` and `/pipeline` only.

## Verification Scenarios

### Scenario 1 — Legacy URLs redirect (contract smoke)

**Setup**: Production or staging front door is reachable (`https://coherencycoin.com` or staging equivalent).

**Action**:
```bash
curl -sI "https://coherencycoin.com/automation" | tr -d '\r' | grep -iE '^(HTTP/|location:)'
curl -sI "https://coherencycoin.com/usage" | tr -d '\r' | grep -iE '^(HTTP/|location:)'
curl -sI "https://coherencycoin.com/remote-ops" | tr -d '\r' | grep -iE '^(HTTP/|location:)'
```

**Expected result**:  
- Each response is **HTTP/2 301** or **HTTP/1.1 301** (or 308 if intentionally configured—document if so).  
- `location:` is an **absolute or relative** URL matching **§ Redirect matrix** (path `/nodes` or `/pipeline` as specified).

**Edge case**: Request with query string, e.g. `GET /usage?page=2&page_size=20` — **must** still redirect to `/pipeline` (query may be preserved or dropped; **must not** 404 or 500).

---

### Scenario 2 — Full read cycle for pipeline data (API → page)

**Setup**: API returns 200 for authenticated/public reads as today.

**Action**:
```bash
API="https://api.coherencycoin.com"
curl -sS "$API/api/agent/pipeline-status" | head -c 400
curl -sS "$API/api/agent/tasks?page=1&page_size=5" | head -c 400
```

Then in browser: open **`/pipeline`**, confirm queue/running sections populate from the same logical data (counts consistent with JSON, not necessarily byte-identical).

**Expected result**: JSON fields are non-error; page shows **non-crashing** sections with either data or explicit empty states.

**Edge case**: `GET /api/agent/tasks?page=99999` returns empty list with 200—UI shows “no tasks” not an exception.

---

### Scenario 3 — Full read cycle for node + automation readiness

**Setup**: At least zero or more nodes registered (empty fleet is valid).

**Action**:
```bash
API="https://api.coherencycoin.com"
curl -sS "$API/api/federation/nodes" | head -c 400
curl -sS "$API/api/automation/usage/readiness" | head -c 400
```

Browser: open **`/nodes`**, confirm federation list and readiness panel render.

**Expected result**: Pages load **200**; readiness panel shows configured/missing-env states or a clear “unavailable” card if fetch fails.

**Edge case**: Simulated API failure (staging mock)—section shows error banner; rest of page still interactive.

---

### Scenario 4 — Create task (remote control) with error handling

**Setup**: Valid execute token available in operator environment (staging).

**Action**: On **`/nodes`**, submit **Create task** with type `spec` and valid executor. Then submit once with **invalid token**.

**Expected result**:  
- Valid: **2xx** from `POST /api/agent/tasks`, inline success with `task_id` in UI or message body.  
- Invalid: **403/401** from API; UI shows **human-readable** error string, no unhandled exception overlay.

**Edge case**: Duplicate rapid clicks “Create” — either disabled while in-flight or idempotent UI feedback; must not create duplicate tasks unintentionally (best-effort client debounce).

---

### Scenario 5 — Bad navigation / missing resource

**Setup**: None.

**Action**: Open `https://coherencycoin.com/automation/foo` (if unsupported) or any **non-configured** subpath under legacy routes per Next behavior.

**Expected result**: **404** for unknown paths **or** same redirect rule as parent—**must not** 500. Document actual Next.js behavior in implementation notes.

**Edge**: `GET /nodes/not-a-real-subpage` — standard app 404, not blank document.
