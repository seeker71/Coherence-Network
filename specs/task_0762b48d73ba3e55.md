# Spec — Publish Coherence Network MCP Server to npm and PyPI

**ID:** `task_0762b48d73ba3e55`  
**Status:** draft (ready for implementation planning)  
**Priority:** high  
**Category:** distribution / developer experience / MCP ecosystem  
**Related:** `specs/180-mcp-skill-registry-submission.md` (registry submissions and metrics; this spec covers **installable packages** and **`server.json` for mcp-publisher**)

---

## Summary

Package the existing Coherence Network Model Context Protocol (MCP) server (`api/mcp_server.py`, tools from `app/services/mcp_tool_registry.py`) as **two first-class installable artifacts**:

1. **PyPI** — a Python distribution (wheel + sdist) so users can `pip install` and run via `python -m …` or a console script entry point.
2. **npm** — a Node package that wraps the same runtime (stdio MCP) using a `bin` script that invokes the published Python package **or** a documented `npx` flow, so the server appears in Node-centric discovery flows and can be referenced from `package.json` / Smithery / tooling that expects an npm coordinate.

Include a **`server.json`** manifest at the repository root (or under `mcp-server/`, per publisher docs) compatible with **mcp-publisher** and registries that ingest it (e.g. Smithery, MCP.so patterns), and align metadata with **Official MCP Registry**, **Glama** (via `awesome-mcp-servers` and listing requirements), and other discovery platforms.

**Problem solved:** Today, consumers must clone the monorepo and run `python api/mcp_server.py` with a correct `cwd` and `PYTHONPATH` (see `skills/coherence-network/mcp-config.json`). Published packages remove friction, enable semver, and make “does this MCP exist and install?” verifiable with standard package managers.

**Proof over time (open question):** Version alignment between npm and PyPI, automated smoke tests in CI, download counts from PyPI/npm APIs, and linkage to `GET /api/discovery/registry-submissions` (spec 180) so “listed + installable + healthy” is visible in one place.

## Requirements

### Functional

- [ ] **R1** PyPI artifact exposes the MCP server so that after install, a documented command starts stdio MCP (e.g. `coherence-network-mcp` or `python -m coherence_network_mcp`).
- [ ] **R2** The packaged server loads the same tool surface as production intent: tools from `mcp_tool_registry` (or a single packaged copy of that module tree), with `COHERENCE_API_BASE` defaulting to `https://api.coherencycoin.com` when unset.
- [ ] **R3** npm package ships `package.json` with `name`, `version`, `description`, `license`, `repository`, `keywords` (`mcp`, `coherence-network`, `model-context-protocol`), and a `bin` that invokes the Python entry point when Python is available, **or** documents a `npx` + `pip install` bootstrap; implementation must choose one primary story and test it.
- [ ] **R4** Root-level (or `mcp-server/server.json`) **`server.json`** includes at minimum: server name, version, description, transport (`stdio`), command/args for running after install, and environment variable schema hints as required by mcp-publisher.
- [ ] **R5** Versioning: PyPI and npm versions stay in sync for each release (single source of truth in CI, e.g. bump one file that feeds both, or release automation that fails if skewed).

### Non-functional

- [ ] **R6** Package size and dependency set are minimized (no full FastAPI stack in the MCP-only wheel unless required by shared code; if the repo shares code with `coherence-api`, document whether the MCP package depends on `coherence-api` extras or vendors a thin client).
- [ ] **R7** Security: no secrets in packages; env-based configuration only; README warns about pointing at production vs self-hosted API.

## Verification Scenarios

### Scenario 1 — PyPI install and MCP tool list (happy path)

- **Setup:** Fresh Python 3.11+ venv; no repo clone; only `pip install` from TestPyPI or PyPI after release.
- **Action:**  
  `python -m venv .venv && source .venv/bin/activate`  
  `pip install coherence-network-mcp==<published_version>`  
  `COHERENCE_API_BASE=https://api.coherencycoin.com coherence-network-mcp`  
  Use an MCP inspector or test harness to send `initialize` then `tools/list` over stdio (or run the project’s `pytest` integration test if provided).
- **Expected:** `tools/list` returns at least one tool (e.g. `browse_ideas`) with valid JSON Schema in `inputSchema`; no import error from `mcp` or missing package modules.
- **Edge:** Unset `COHERENCE_API_BASE` — server still starts and uses documented default base URL; if hub is down, tool calls return structured error JSON from handlers, not a process crash.

### Scenario 2 — npm / npx discoverability path

- **Setup:** Machine with Node 20+ and Python available (as per README requirement).
- **Action:**  
  `npm view @coherence-network/mcp-server version` (or chosen package name)  
  `npx @coherence-network/mcp-server@<version> --help` or documented stdio start  
  Configure Cursor/Codex `mcpServers` block using the published README snippet (command + args only, no repo path).
- **Expected:** npm shows the published version; starting the server matches PyPI behavior (same tool list).
- **Edge:** User has only Node but no Python — documented error message and install link; exit code non-zero.

### Scenario 3 — `server.json` and mcp-publisher validation

- **Setup:** Repo checkout at release tag; `server.json` committed.
- **Action:** Run the registry’s documented validate command, e.g. `npx @modelcontextprotocol/publisher validate ./server.json` *(exact CLI TBD at implementation)*.
- **Expected:** Validator exits 0; `name` and `version` match the release tag.
- **Edge:** Invalid JSON or wrong transport — validator exits non-zero; CI fails.

### Scenario 4 — Full external read cycle via tool (CRUD where “update” is N/A)

- **Setup:** Packaged MCP running against production API (`COHERENCE_API_BASE=https://api.coherencycoin.com`).
- **Action:** `tools/call` `browse_ideas` with `{"limit": 1}` then `get_idea` with a real `idea_id` from the first response.
- **Expected:** First call returns HTTP-backed JSON with a list; second returns the same idea or a clear not-found payload for invalid id.
- **Edge:** `get_idea` with nonsense id — response contains `error` or `not found` message, not a stack trace.

### Scenario 5 — Error handling and bad input

- **Setup:** Same as Scenario 4.
- **Action:** `tools/call` with `name` that does not exist, e.g. `{"name": "__nonexistent_tool__", "arguments": {}}`.
- **Expected:** MCP returns text content with JSON `{"error": "Unknown tool: __nonexistent_tool__"}` (consistent with current `api/mcp_server.py` behavior).
- **Edge:** Malformed JSON arguments — handler or MCP layer returns structured error, process stays alive.

## Risks and Assumptions

| Risk | Mitigation |
|------|------------|
| Monorepo code sharing pulls entire `coherence-api` into the MCP wheel | Split a thin `coherence_network_mcp` package with minimal deps or optional extras. |
| npm-on-Windows path issues for spawning Python | Document and CI-test Windows + macOS + Linux; use `python -m` entry where possible. |
| Registry schema drift | Pin validator version in CI; re-run before each release. |
| PyPI name squatting | Reserve names early; use org scope on npm. |

**Assumption:** The `mcp` Python SDK is an explicit dependency with a version range tested in CI.

## Open Questions — Improving the Idea, Proof, and Clarity Over Time

1. **Single version source:** Use one `VERSION` file or `git tag` as the only bump surface for PyPI + npm + `server.json` to eliminate drift.
2. **“Working yet” signal:** Publish a **compatibility matrix** (MCP protocol version, Python versions, last green CI run) in the README and optionally on the marketing site.

5. **Ecosystem:** After npm/PyPI publish, file PRs to **Glama** / **awesome-mcp-servers** using the **released** coordinates (not monorepo paths), cross-linking spec 180 inventory.
