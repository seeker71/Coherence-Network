# Spec: Consolidate /automation, /usage, /remote-ops into /nodes and /pipeline

**Task ID**: task_7de286c308d78f97
**Type**: UI Consolidation / Navigation Redesign
**Status**: Draft
**Author**: product-manager agent
**Date**: 2026-03-28

---

## Summary

Four web pages currently display overlapping data about the Coherence Network's operational
state: `/automation`, `/usage`, `/remote-ops`, and `/nodes`. The duplication creates navigation
confusion and forces users to cross-reference multiple pages to get a complete operational picture.

This spec defines a consolidation into two clear, purpose-driven pages:

| Destination | Mental model | What moves there |
|---|---|---|
| **/nodes** | "Everything about the machines" | Federation list, node health, provider capabilities, remote control, deployment uptime, messaging |
| **/pipeline** | "Everything about the work" | Queue, running tasks, completed tasks, streaks, provider performance, capacity alerts, daily summary |

`/automation`, `/usage`, and `/remote-ops` are **removed** and replaced with HTTP 308 permanent
redirects to the appropriate destination page.

---

## Motivation

### Current state (problem)

- **4 pages, 1 conceptual domain**: all four pages describe the same runtime operational system.
- **Navigation burden**: users who want "is the system healthy?" must check multiple pages.
- **Data duplication**:
  - Provider execution stats appear in both `/automation` (Provider Execution Stats section) and `/usage` (Provider Health section).
  - Node list with capabilities appears in both `/automation` (Federation Network section) and `/nodes` (Registered Nodes section).
  - Queue/pipeline status appears in both `/remote-ops` (QueuePipelineSection) and `/pipeline` (What Is Happening Now).
  - Provider readiness/validation appears in `/automation` only, but rightfully belongs with provider performance in `/pipeline`.
- **Confusing mental models**: "Automation" implies scheduling or triggers, not provider health. "Remote Ops" implies a console, not a monitoring view.

### Desired state (goal)

- **2 pages, 2 mental models**: nodes = machines; pipeline = work.
- A user who asks "is my node healthy?" goes to `/nodes`.
- A user who asks "what tasks are running?" goes to `/pipeline`.
- Navigation bar shrinks from 4 entries to 2 for this domain.
- Old URLs continue to work via permanent redirects (no broken bookmarks).

## Page Definitions

### /nodes — Everything About the Machines

**Purpose**: Understand the current health, configuration, and control of every node in the federation.

**Sections** (in display order):

1. **Node Summary Stats** *(from existing `/nodes`)*
   Aggregate counts: total nodes, active nodes, nodes needing attention.
   Source: `GET /api/federation/nodes`

2. **Registered Nodes List** *(from existing `/nodes`)*
   Per-node cards showing: node_id, hostname, OS, status badge (active/stale/offline),
   last_seen_at, provider list, capabilities (executors, CPU, memory, GPU).
   Source: `GET /api/federation/nodes`

3. **Provider Capabilities per Node** *(merged from `/automation` — "Federation Node Capability Discovery")*
   Which providers each node supports, which models are available, hardware details.
   Source: `GET /api/federation/nodes` (capabilities.models_by_executor)

4. **Federation Provider Stats by Node** *(merged from `/automation` — "Federation Network — Provider Stats by Node")*
   Table showing per-node provider success rates, with mobile-responsive stacked cards.
   Source: `GET /api/agent/usage` or equivalent node-scoped endpoint

5. **Provider Readiness / Validation Contract** *(merged from `/automation`)*
   Which providers are configured and ready, blocking issues, missing env vars.
   Source: `GET /api/automation/usage/readiness`, `GET /api/automation/usage/provider-validation`

6. **Deployment & Uptime** *(moved from `/remote-ops`)*
   Health check status, proxy status, deploy status.
   Source: `GET /api/health`, deploy metadata endpoints

7. **Remote Control** *(moved from `/remote-ops`)*
   Execute token input, executor selector, task type selector, model override, force-paid toggle.
   Execute buttons (run codex, dispatch task).
   Source: `POST /api/agent/run`, `POST /api/agent/pipeline-status`

8. **Node Messaging** *(from existing `/nodes`)*
   Send a message to a specific node.
   Source: `POST /api/federation/nodes/{node_id}/messages`

---

### /pipeline — Everything About the Work

**Purpose**: Understand what tasks are running, queued, or completed, and how well providers are performing.

**Sections** (in display order):

1. **What Is Happening Now** *(from existing `/pipeline`)*
   Banner showing live count of executing tasks, pending tasks, completed today.
   Source: `GET /api/agent/pipeline-status`

2. **Stats Grid** *(from existing `/pipeline`)*
   Key metrics: tasks completed, success rate, streak, providers used.

3. **Executing Right Now** *(from existing `/pipeline`)*
   Live list of currently running tasks with node assignment, provider, start time.

4. **Queue** *(from existing `/pipeline` + `/remote-ops` QueuePipelineSection)*
   Pending tasks table with type, priority, created_at, estimated start.

5. **Completed Tasks / Activity Stream** *(from existing `/pipeline`)*
   Recent task completions with outcome (success/failure/timeout), duration, provider.

6. **Provider Performance** *(merged from `/automation` — "Provider Execution Stats" + `/usage` "Provider Health")*
   Unified table: total runs, success rate, last-5 rate, avg duration, blocked flag.
   Source: `GET /api/providers/stats`

7. **Capacity Alerts** *(merged from `/automation`)*
   Alert list for providers near usage limits.
   Source: `GET /api/automation/usage/alerts`

8. **Usage Limit Coverage** *(merged from `/automation`)*
   Which providers have limit metrics, coverage ratio.
   Source: `GET /api/automation/usage`

9. **Daily Summary** *(merged from `/usage`)*
   Friction score, quality awareness, host runner status, tool usage counters, recovery streak,
   view performance stats, runtime cost.
   Sources: `/api/agent/usage`, `/api/agent/pipeline-status`

10. **Provider Usage Snapshots** *(merged from `/automation` — "Provider Usage")*
    Per-provider snapshot: cost_usd, capacity_tasks_per_day, usage_remaining.
    Source: `GET /api/automation/usage`

## API Endpoints (No Changes Required)

This consolidation is purely a frontend routing/layout change. All existing API endpoints
remain unchanged. The following endpoints are consumed by the new pages:

**Used by /nodes:**
- `GET /api/federation/nodes` — node list with capabilities and provider stats
- `GET /api/automation/usage/readiness` — provider readiness check
- `GET /api/automation/usage/provider-validation` — provider validation contract
- `GET /api/agent/usage` — node-level provider stats
- `GET /api/health` — deployment health
- `POST /api/federation/nodes/{node_id}/messages` — node messaging
- `POST /api/agent/run` (execute, protected) — task dispatch

**Used by /pipeline:**
- `GET /api/agent/pipeline-status` — queue, running, completed tasks
- `GET /api/providers/stats` — provider execution stats
- `GET /api/automation/usage/alerts` — capacity alerts
- `GET /api/automation/usage` — usage snapshots and limit coverage
- `GET /api/agent/usage` — daily summary, friction, quality awareness

## Redirect Behaviour

| Old URL | New URL | HTTP Status |
|---|---|---|
| `/automation` | `/nodes` | 308 Permanent |
| `/automation/*` (any subpath) | `/nodes` | 308 Permanent |
| `/usage` | `/pipeline` | 308 Permanent |
| `/remote-ops` | `/nodes` | 308 Permanent |

Implementation: Next.js `redirect()` in page component, or `next.config.js` `redirects` array (permanent: true).

Rationale for redirect targets:
- `/automation` → `/nodes`: The majority of automation data is about provider and node state.
- `/usage` → `/pipeline`: Usage data is about work done (tasks, cost, performance).
- `/remote-ops` → `/nodes`: Remote control is node management.

## Verification Scenarios

### Scenario 1: /automation redirects to /nodes

**Setup**: Current production with `/automation` page existing
**Action**:
```bash
curl -sI https://coherencycoin.com/automation
```
**Expected result**: HTTP 308, `Location: /nodes` header present
**Then**: Browser navigation to `https://coherencycoin.com/automation` lands on `/nodes` page without 404
**Edge case**: `curl -sI https://coherencycoin.com/automation/anything` also returns 308 to `/nodes`

## Risks and Assumptions

| Risk | Mitigation |
|---|---|
| Sections from `/automation` (908 lines) may have complex shared state between sections | Migrate section-by-section; audit for shared data fetching before removing the page |
| `use-remote-ops.ts` is a client-side hook used by multiple sections | Move to `lib/use-remote-ops.ts` before deleting `/remote-ops/` directory |
| Old URLs may be embedded in documentation, Discord, or external links | 308 permanent redirects handle bookmarks; do a grep of docs for hardcoded paths |
| `/automation` page has 908 lines — migration risk of missing something | Write a section inventory checklist before starting; verify each section appears in new pages |
| Daily summary sections in `/usage` have complex section components in `usage/sections/` | Move these components to `web/components/pipeline/` or keep them in a shared location |
| Two pages now share provider stats data — both call `GET /api/providers/stats` | Accept double calls; consider a shared SWR/React Query key if performance is a concern in the future |

## Known Gaps and Follow-up Tasks

- **Search/filter on /nodes**: The node list will grow as more nodes join the federation. A filter by status or provider may be needed (out of scope here).
- **Deep links**: Once consolidation is done, consider anchored links (e.g., `/pipeline#provider-performance`) so existing internal cross-references can still reach specific sections.
- **Mobile layout for /nodes**: The remote control section from `/remote-ops` was a full client-component page; integrating it into the server-rendered `/nodes` page may require a `"use client"` boundary to be added.
- **Analytics**: Measure page visit drops on old URLs post-redirect to confirm users are following the new paths.
- **/pipeline already partially covers nodes**: The `pipeline/page.tsx` has a "Nodes" subsection (line 408). After consolidation, this subsection should be removed from `/pipeline` to keep the separation clean.
