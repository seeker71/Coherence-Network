## Summary

The same idea lands differently depending on the listener’s belief system and worldview: a decentralized network reads as **freedom** to a libertarian, **efficiency** to an engineer, **threat** to an institution, **opportunity** to an entrepreneur, and so on. Coherence Network should **translate ideas through multiple lenses** without replacing or falsifying the canonical idea—so people can see **how their perspective connects to others** and reduce unnecessary conflict from framing mismatch.

This task’s spec extends that intent to **news ingestion with point-of-view (POV) filters**: every ingested item and every **idea** should be **viewable through any registered worldview lens**, with **cached, inspectable translations**, **engagement and resonance metrics**, and **independently verifiable evidence** that the capability exists in production.

---

## Purpose

- **Bridge perspectives**: Make cross-worldview collaboration possible by surfacing parallel, good-faith restatements of the same underlying idea.
- **Preserve truth posture**: Original idea text remains authoritative; lens outputs are labeled as interpretive framing, not replacements.
- **Operationalize “is it working?”**: Define measurable signals (API counters, engagement rates, scenario checks) that improve over time and can be audited by any third party.

---

## Requirements

### Functional

1. **Worldview lenses registry**: System exposes a stable list of lenses (built-in plus optional custom), each with `lens_id`, human-readable name, description, and structured `archetype_axes` (values in `[0.0, 1.0]`).
2. **Per-idea translation**: For any existing `idea_id`, client can request `GET /api/ideas/{idea_id}/translations/{lens_id}` and receive an `IdeaTranslation` payload including `translated_summary`, emphasis tags, risk/opportunity framing, `source_hash`, `spec_ref`, and optional `resonance_delta` when `contributor_id` is supplied.
3. **Cache correctness**: Translations are keyed on `(idea_id, lens_id, source_hash)`; changing idea content invalidates or regenerates as defined in implementation spec 181.
4. **Force regeneration**: `POST` to the same translation resource with `force_regenerate: true` recomputes and updates cache (subject to auth/rate limits as implemented).
5. **News POV (MVP contract)**: Ingested news articles can be viewed through a lens via a dedicated translation surface (e.g. `GET /api/news/{article_id}/translations/{lens_id}`) **or** an equivalent documented path if news IDs are not yet stable—**the implementing task must pick one contract and test it**; this spec treats news lensing as **first-class** alongside ideas.
6. **Web UX (MVP)**: Idea detail page exposes a lens selector; user can switch lenses and see translated framing alongside the original.
7. **Proof API**: A read-only aggregate endpoint (e.g. `GET /api/lenses/roi`) returns counters and rates that answer “is perspective bridging happening?” without privileged dashboard access.

### Non-functional

- Invalid `lens_id` → **404**, never **500** for well-formed IDs.
- Duplicate lens registration → **409**.
- Out-of-range axis values on lens create → **422**.
- All public responses use Pydantic models; dates ISO 8601 UTC; coherence-like scores remain in documented ranges where applicable.

## Data Model (Summary)

Aligned with spec 181:

- **`WorldviewLens`**: `lens_id`, `name`, `description`, `archetype_axes`, optional `framing_template`, `is_builtin`, timestamps.
- **`IdeaTranslation`**: `idea_id`, `lens_id`, `original_name`, `translated_summary`, `emphasis[]`, `risk_framing`, `opportunity_framing`, optional `resonance_delta`, `cached`, `generated_at`, `source_hash`, `spec_ref`.
- **`NewsTranslation`** (or shared `Translation` with `subject_type`): same conceptual fields keyed by `article_id` where news is in scope.

Persistence: PostgreSQL tables backing lenses and translation cache; in-memory/dev fallback must still satisfy tests listed in spec 181.

## Improving the Idea and Proving It Works (Open Question)

**How we improve the idea**

- Treat lenses as **versioned configuration** (builtin + curated community lenses) with changelog, not one-off prompts.
- Add **side-by-side compare** (original vs two lenses) before adding more lenses—quality over quantity.
- Instrument **explicit user actions**: lens tab clicks, “view through my lens” toggles on news, and time-on-lens—not only server-side generation counts.

**How we show whether it is working yet**

- **Primary**: `GET /api/lenses/roi` exposes `total_translations_generated`, `unique_ideas_translated`, `cross_lens_engagement_rate`, `avg_resonance_delta`, `spec_ref`.
- **Secondary**: Web UI shows lens usage on idea pages (event logs or analytics pipeline—follow-up if not in MVP).
- **Tertiary**: Periodic **verification scenarios** (below) run against `https://api.coherencycoin.com` in CI or release checklist.

**How proof becomes clearer over time**

- Publish a **stable `spec_ref`** (e.g. `spec-181`) in every translation response so third parties can correlate behavior to the spec version.
- Add **time-series export** (follow-up): daily snapshots of ROI counters to `docs/system_audit/` or metrics backend—out of MVP if not listed in implementation task.

## Verification Scenarios

Each scenario is intended to be run by a reviewer against **production** (`API=https://api.coherencycoin.com`). Replace `$API` accordingly. Scenarios assume spec 181 is implemented and deployed.

### Scenario 1 — Lens registry is live and built-ins exist

- **Setup**: None; public API reachable.
- **Action**: `curl -sS "$API/api/lenses" | jq '{total: .total, first: .lenses[0].lens_id}'`
- **Expected result**: HTTP 200; JSON includes `total` ≥ 6; `first` is a non-empty string matching a builtin such as `libertarian` or `engineer` (exact ordering may vary; `total` must be ≥ 6).
- **Edge case**: `curl -sS -o /dev/null -w "%{http_code}" "$API/api/lenses/builtin_that_never_exists_zzzz"` → **404** (not 500).

### Scenario 2 — Idea translation returns structured payload with `spec_ref`

- **Setup**: At least one idea exists (`GET /api/ideas?limit=1` returns a non-empty list).
- **Action**:
  ```bash
  IDEA_ID=$(curl -sS "$API/api/ideas?limit=1" | jq -r '.ideas[0].id')
  curl -sS "$API/api/ideas/$IDEA_ID/translations/engineer" | jq '{spec_ref, translated_summary: (.translated_summary|length), source_hash: (.source_hash|length)}'
  ```
- **Expected result**: HTTP 200; `spec_ref` is `"spec-181"` (or current documented ref); `translated_summary` length > 20; `source_hash` non-empty string.
- **Edge case**: `curl -sS "$API/api/ideas/$IDEA_ID/translations/invalid_lens_slug_xyz"` → **404** with JSON `detail` present (not empty 500 body).

### Scenario 3 — Duplicate lens creation conflicts

- **Setup**: No lens `id` collision; use a unique test slug per run if operators re-run tests.
- **Action**:
  ```bash
  SLUG="pm-verify-$(date +%s)"
  curl -sS -w "\n%{http_code}\n" -X POST "$API/api/lenses" -H "Content-Type: application/json" \
    -d "{\"lens_id\":\"$SLUG\",\"name\":\"PM Verify\",\"description\":\"test\",\"archetype_axes\":{\"pragmatic\":0.7}}"
  curl -sS -w "\n%{http_code}\n" -X POST "$API/api/lenses" -H "Content-Type: application/json" \
    -d "{\"lens_id\":\"$SLUG\",\"name\":\"PM Verify Dup\",\"description\":\"dup\",\"archetype_axes\":{\"pragmatic\":0.7}}"
  ```
- **Expected result**: First response ends with `201`; second ends with `409`.
- **Edge case**: POST with `archetype_axes` value `1.5` → **422** with validation detail mentioning out-of-range.

### Scenario 4 — ROI endpoint proves activity and is stable JSON

- **Setup**: System has served at least one translation (may be zero before launch; reviewer re-runs after smoke translation).
- **Action**: `curl -sS "$API/api/lenses/roi" | jq '{total_translations_generated, cross_lens_engagement_rate, spec_ref}'`
- **Expected result**: HTTP 200; all three fields present; `spec_ref` matches documented value; `total_translations_generated` is a non-negative integer; `cross_lens_engagement_rate` is a number in `[0.0, 1.0]` (or documented nullable rule—if null, spec implementation must document why).
- **Edge case**: Repeated GET within 1 minute returns consistent typing (no field type flapping between number and string).

### Scenario 5 — Unknown idea yields controlled client error

- **Setup**: Pick an `idea_id` that does not exist, e.g. `nonexistent-idea-pm-verify`.
- **Action**: `curl -sS -w "\n%{http_code}\n" "$API/api/ideas/nonexistent-idea-pm-verify/translations/engineer"`
- **Expected result**: HTTP **404** (preferred) or **422** if API consistently uses validation for unknown IDs—**must not** be **500**; response body includes machine-readable `detail`.
- **Edge case**: Empty `idea_id` path segment or malformed URL returns **404** or **422**, never **500**.

## Risks and Assumptions

- **Risk**: LLM outputs drift or misrepresent the idea. **Mitigation**: Always show original first; label outputs as lens-filtered; store `source_hash` for audit.
- **Risk**: Lens labels politicize the product. **Mitigation**: Describe lenses as epistemic/engineering stances; allow neutral renaming; moderation for custom lenses.
- **Risk**: Cost/latency of generation. **Mitigation**: Cache; rate limits; async generation where needed.
- **Assumption**: Contributor profiles (spec 169) exist for `resonance_delta`; otherwise field is `null`.
- **Assumption**: News pipeline may lag ideas; news endpoint may ship in a follow-up task with the same POV semantics.
