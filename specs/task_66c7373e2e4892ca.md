# Spec 170: Split Review — Phase Health, Proof of Improvement, and Public Failure Handling

**Spec ID**: task_66c7373e2e4892ca
**Spec Number**: 170
**Status**: Draft
**Date**: 2026-03-28
**Depends on**: Spec 159 (code-review → deploy → verify-production phase split)
**Depended on by**: Spec 157 (Investment UX — validated status), Spec 139 (pipeline auto-advance)

---

## Purpose

Spec 159 defined the three-phase split (code-review → deploy → verify-production) and its failure modes. This spec answers the remaining open question: **How do we know the split is working? How do we make that proof clearer and more persistent over time?**

It specifies:

1. A **phase health dashboard endpoint** (`GET /api/pipeline/phase-health`) with historical trend data showing phase pass rates week-over-week — the primary proof signal.
2. A **grace handling contract** for public failures: when `verify-production` fails, the system must not silently swallow the failure. It must create a hotfix task, set `regression` status, and broadcast in a way that is visible to operators without requiring SSH access.
3. **Concrete SLOs** (Service Level Objectives) for each phase so the system can be declared "working" or "degraded" without ambiguity.
4. A **phase history timeline** field on each idea, letting reviewers see the exact sequence of phase completions for any idea at a glance.
5. **Automated weekly summary** emitted as a `cc` broadcast each Monday morning to force operator attention on health trends.

---

## Summary

The current pipeline has a 60–80% review failure rate because deployment and verification are conflated with code quality review. Spec 159 splits the pipeline into three discrete phases. This spec focuses on making the split's effectiveness **observable, measurable, and provably improving** — not just mechanically correct.

Without a clear proof mechanism, operators have no way to know whether the split is helping. The metrics from `GET /api/pipeline/phase-health` will be the canonical answer to "is this working?" If `code-review.pass_rate >= 0.80` and `deploy.pass_rate >= 0.85` and the number of ideas stuck at `reviewing` with no active task is 0, the split is working. If not, there is a specific phase to investigate.

A public failure (verify-production failing on a live feature) is a live incident. This spec adds clear escalation paths: hotfix task creation, `regression` status on the idea, and a broadcast visible in `cc inbox` for all nodes — so the failure is not silent.

## Problem Statement

### What Spec 159 Left Unresolved

Spec 159 defines the mechanics of the split but leaves open:

1. **Proof gap**: `GET /api/pipeline/status` now returns `phase_stats`, but these are point-in-time counts with no trend. An operator cannot answer "has the pass rate improved since we deployed Spec 159?" without querying the database directly.

2. **Silent regression risk**: When `verify-production` fails, Spec 159 requires a hotfix task. But if the task queue is not being processed (e.g., all agent slots are busy), the regression may sit undetected for hours. There is no broadcast, no status page degradation.

3. **Idea phase history gap**: Given an idea ID, there is no way to see "this idea went through code-review at 14:32, deploy at 14:40, verify failed at 14:45" without reading raw task records. Debugging is slow.

4. **SLO ambiguity**: The spec says "pass rate drops below 20%" as a failure threshold, but 20% is not a useful SLO. This spec replaces that with actionable, named SLO levels: `healthy`, `degraded`, `critical`.

## Requirements

### R1 — Phase Health Endpoint

**`GET /api/pipeline/phase-health`** must return a response conforming to:

```json
{
  "generated_at": "2026-03-28T14:00:00Z",
  "slo_status": "healthy",   // "healthy" | "degraded" | "critical"
  "phases": {
    "code-review": {
      "slo_status": "healthy",
      "window_7d": {
        "completed": 24,
        "passed": 21,
        "failed": 3,
        "pass_rate": 0.875,
        "avg_retries": 0.3
      },
      "window_30d": {
        "completed": 80,
        "passed": 66,
        "failed": 14,
        "pass_rate": 0.825,
        "avg_retries": 0.4
      },
      "trend": "improving"   // "improving" | "stable" | "degrading"
    },
    "deploy": {
      "slo_status": "healthy",
      "window_7d": { ... },
      "window_30d": { ... },
      "trend": "stable"
    },
    "verify-production": {
      "slo_status": "degraded",
      "window_7d": { ... },
      "window_30d": { ... },
      "trend": "degrading"
    }
  },
  "orphaned_reviewing_count": 0,
  "active_regressions": [],
  "hotfix_tasks_created_7d": 1
}
```

**SLO thresholds** (per phase):

| Status | code-review pass_rate | deploy pass_rate | verify-production pass_rate |
|--------|----------------------|-----------------|----------------------------|
| healthy | ≥ 0.80 | ≥ 0.85 | ≥ 0.75 |
| degraded | 0.60 – 0.79 | 0.70 – 0.84 | 0.50 – 0.74 |
| critical | < 0.60 | < 0.70 | < 0.50 |

The top-level `slo_status` is the worst of the three per-phase statuses.

**Trend calculation**: Compare `window_7d.pass_rate` against the prior 7 days (days -14 to -7). If the rate improved by > 5pp, report `improving`. If it dropped > 5pp, report `degrading`. Otherwise `stable`.

### R2 — Idea Phase Timeline

`GET /api/ideas/{idea_id}` must include a `phase_timeline` field:

```json
{
  "id": "abc123",
  "manifestation_status": "validated",
  "phase_timeline": [
    {"phase": "spec",               "completed_at": "2026-03-20T10:00:00Z", "outcome": "passed"},
    {"phase": "impl",               "completed_at": "2026-03-21T11:00:00Z", "outcome": "passed"},
    {"phase": "test",               "completed_at": "2026-03-22T09:30:00Z", "outcome": "passed"},
    {"phase": "code-review",        "completed_at": "2026-03-23T14:32:00Z", "outcome": "passed"},
    {"phase": "deploy",             "completed_at": "2026-03-23T14:40:00Z", "outcome": "passed"},
    {"phase": "verify-production",  "completed_at": "2026-03-23T14:45:00Z", "outcome": "passed"}
  ]
}
```

`outcome` is one of `passed`, `failed`, `retried` (passed on retry), `skipped`.

This field is derived from the `agent_tasks` table — no new schema needed beyond what Spec 159 requires.

### R3 — Regression Broadcast Contract

When `verify-production` fails for any idea:

1. The hotfix `AgentTask` is created (Spec 159 R4).
2. **Within 60 seconds**: broadcast via `cc msg broadcast "REGRESSION: <idea_name> (<idea_id>) — verify-production FAILED. Hotfix task <hotfix_task_id> created at priority=urgent. Check GET /api/ideas/<idea_id> for details."`.
3. **The `active_regressions` array** in `GET /api/pipeline/phase-health` must list the idea ID until the hotfix verify-production passes.
4. The API health endpoint (`GET /api/health`) must return HTTP **200** with a `"regressions": <count>` field — NOT a non-200 status (regressions are a pipeline state, not an infrastructure outage; changing health to 5xx would break load balancers).

### R4 — Weekly Health Summary

Every Monday at 09:00 UTC, the pipeline pulse service must emit:

```
cc msg broadcast "Weekly Phase Health: code-review 87% (+5pp), deploy 91% (+2pp), verify 78% (-3pp). SLO: healthy. 0 orphaned ideas. 1 active regression."
```

This is implemented as a cron entry in the runner (or as a scheduled task, not a new service). The broadcast is the proof that the operator review loop is closed.

### R5 — Orphaned Idea Detection

An "orphaned reviewing idea" is any idea with `manifestation_status IN ('reviewing', 'deployed')` and no `agent_task` in `pending` or `in_progress` state associated with it.

The phase health endpoint must count these under `orphaned_reviewing_count`. If this count is > 0, `slo_status` for the overall system is degraded or critical.

A background task (run every 15 minutes) must detect orphaned ideas and create a new phase task to unblock them. This prevents the 60–80% false-failure rate from returning via neglected ideas.

### R6 — Backward Compatibility

- The old `review` task type maps to `_NEXT_PHASE: None` and is not changed.
- Existing ideas with `manifestation_status = validated` (set before Spec 159 was implemented) are not retroactively changed to `regression`.
- The `phase_timeline` field is empty (`[]`) for ideas that have no recorded phase tasks in the new structure.

## Data Model

No new tables are required. All data is derived from `agent_tasks`:

```sql
-- phase_timeline query (illustrative)
SELECT task_type AS phase, completed_at,
  CASE WHEN status = 'completed' AND output LIKE '%_PASSED%' THEN 'passed'
       WHEN status = 'completed' AND output LIKE '%_FAILED%' THEN 'failed'
       WHEN status = 'failed' THEN 'failed'
       ELSE 'skipped' END AS outcome
FROM agent_tasks
WHERE idea_id = :idea_id
  AND task_type IN ('spec', 'impl', 'test', 'code-review', 'deploy', 'verify-production')
ORDER BY completed_at ASC;
```

The `regression` manifestation status (added in Spec 159) is the only schema change required. Alembic migration: add `'regression'` to the `manifestation_status` enum.

## Verification Scenarios

### Scenario 1 — phase-health endpoint returns structured SLO data

**Setup**: Seed 10 code-review tasks: 8 completed with `CODE_REVIEW_PASSED`, 2 with `CODE_REVIEW_FAILED`. Seed 8 deploy tasks: 7 `DEPLOY_PASSED`, 1 `DEPLOY_FAILED`. All completed within the last 7 days.

**Action**:
```bash
curl -s https://api.coherencycoin.com/api/pipeline/phase-health | jq '.'
```

**Expected result**:
```json
{
  "slo_status": "healthy",
  "phases": {
    "code-review": {
      "slo_status": "healthy",
      "window_7d": {"completed": 10, "passed": 8, "failed": 2, "pass_rate": 0.80}
    },
    "deploy": {
      "slo_status": "healthy",
      "window_7d": {"completed": 8, "passed": 7, "failed": 1, "pass_rate": 0.875}
    }
  }
}
```

**Edge case**: `GET /api/pipeline/phase-health` with zero tasks in database returns `"pass_rate": null` (not `0`) for all phases and `"slo_status": "healthy"` (null pass rate is not degraded — it's unknown).

### Scenario 2 — verify-production failure triggers regression + broadcast

**Setup**: An idea `idea_abc` with `manifestation_status = deployed` and one `verify-production` task in-progress.

**Action**:
```bash
API=https://api.coherencycoin.com

# Simulate verify failure
curl -s -X PATCH "$API/api/agent/tasks/<verify_task_id>" \
  -H "Content-Type: application/json" \
  -d '{"status":"completed","output":"VERIFY_FAILED: GET /api/ideas returned 404 in production"}'

# Wait 10 seconds, then check idea status
sleep 10
curl -s "$API/api/ideas/idea_abc" | jq '{manifestation_status, phase_timeline: .phase_timeline[-1]}'
```

**Expected result**:
```json
{
  "manifestation_status": "regression",
  "phase_timeline": {
    "phase": "verify-production",
    "outcome": "failed"
  }
}
```

Then:
```bash
# Check a hotfix task was created
curl -s "$API/api/agent/tasks?limit=5" | jq '.tasks[] | select(.context.hotfix == true) | {id, priority, task_type}'
```
Expected: One task with `priority = "urgent"`, `task_type = "impl"`, `context.hotfix = true`.

Then:
```bash
# Check active_regressions in phase-health
curl -s "$API/api/pipeline/phase-health" | jq '.active_regressions'
```
Expected: `["idea_abc"]` (idea appears in regression list).

**Edge case**: If `verify-production` output contains `VERIFY_PASSED`, `manifestation_status` becomes `validated` and `active_regressions` does NOT include the idea.

## How We Know It's Working (Proof Contract)

The implementation is **proven working** when ALL of the following are measurably true:

| Metric | Target | Measured via |
|--------|--------|-------------|
| code-review pass rate (7d) | ≥ 80% | `GET /api/pipeline/phase-health` `.phases.code-review.window_7d.pass_rate` |
| deploy pass rate (7d) | ≥ 85% | `GET /api/pipeline/phase-health` `.phases.deploy.window_7d.pass_rate` |
| orphaned reviewing ideas | 0 | `GET /api/pipeline/phase-health` `.orphaned_reviewing_count` |
| ideas validated = verify passed | 100% | Every idea with `manifestation_status=validated` has a `verify-production: passed` entry in `phase_timeline` |
| regression broadcast lag | ≤ 60s | Log timestamp of hotfix task creation vs. verify-production failure time |

**The primary proof signal** is the weekly Monday broadcast (`cc msg broadcast "Weekly Phase Health: ..."`). If this broadcast starts appearing with `code-review ≥ 80%` and `orphaned: 0`, the split is working. If either metric is below threshold, there is a specific phase and failure mode to investigate — no ambiguity.

**Proof over time**: The 7-day and 30-day windows in `/api/pipeline/phase-health` provide a natural before/after comparison. The 30-day window will span the deployment of Spec 159; the 7-day window will reflect the new split behavior. When the 7-day pass rate is meaningfully higher than the 30-day rate, the improvement is proven.

## Risks and Assumptions

- **Risk**: The `regression` enum value requires an Alembic migration. If deployed before the migration runs, existing ideas will fail to deserialize. Mitigation: migration must ship in the same PR as the model change.
- **Risk**: Phase health queries (7d, 30d) against `agent_tasks` may be slow if the table is large. Mitigation: add index on `(task_type, completed_at)` and cache the phase-health response for 5 minutes.
- **Assumption**: The Monday broadcast requires the runner to be alive and processing tasks at 09:00 UTC. If the runner is down, the broadcast is skipped. This is acceptable — the endpoint itself is always available for manual querying.
- **Risk**: `active_regressions` list could grow unbounded if hotfix tasks are never resolved. Mitigation: list only ideas with `manifestation_status = 'regression'` — once the hotfix verify-production passes, the status changes and the idea drops off the list automatically.
- **Assumption**: All spec files include a `Verification Scenarios` section with runnable curl commands. Ideas without this section produce `VERIFY_FAILED` output. The `phase_timeline` will capture this as a failed verify entry, making the spec quality gap visible.

## Known Gaps and Follow-up Tasks

- **Phase health caching**: The 5-minute cache on phase-health is not specified here. If performance is an issue, a follow-up spec should add explicit caching with a `Cache-Control` header.
- **Telegram/notification integration**: The regression broadcast goes to `cc inbox` only. A follow-up should pipe `REGRESSION` broadcasts to a Telegram alert (Spec 003 pattern) for out-of-band notification.
- **Historical phase health snapshots**: The current spec derives all stats from live queries. A follow-up should persist daily snapshots of phase health to enable 90-day trend charts in the web UI.
- **verify-production retry on transient failure**: This spec inherits Spec 159's "retry budget: 0" for verify-production. A follow-up may add a single transient-error retry (e.g., network timeout vs. actual 404) with a 5-minute delay.
