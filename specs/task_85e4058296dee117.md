## Summary

Coherence Network ships an MCP server (`coherence-mcp-server`) and portable skills (`skills/coherence-network/`, `.cursor/skills/`). **Discovery** depends on listing those assets on public registries so developers and agents can find and install them without reading this monorepo first.

This specification defines the **submission program** targeting **at least six** external surfaces: **Smithery**, **Glama** (via `punkpeye/awesome-mcp-servers` pull request), **PulseMCP**, **MCP.so** (`mcp.so`), **skills.sh**, and **askill.sh**. It also requires a **durable way to track install and download signals** (where APIs exist) and to **show whether the program is working** with evidence that gets **clearer over time**—not one-off screenshots.

The repository already exposes machine-readable readiness via **`GET /api/discovery/registry-submissions`**, backed by **`docs/registry-submissions.json`** and **`api/app/services/registry_discovery_service.py`**. Inventory rows must stay aligned with real submission work (PR URLs, npm pages, Smithery listings). **Spec 180** describes the next engineering increment: **`GET /api/discovery/registry-stats`** and **`GET /api/discovery/registry-dashboard`** for automated count ingestion and a merged view.

## Purpose

| Stakeholder | Benefit |
|-------------|---------|
| OSS contributors | Find the MCP server and skills through familiar indexes (Smithery, Glama lists, skill directories). |
| Operators | Prove adoption with numbers and trends, not anecdotes. |
| Maintainers | Single source of truth for “what was submitted where” and what assets must exist before tagging a release. |

Failure mode without this: excellent code remains invisible; agent tooling ecosystems route users to competitors with better listings.

## Requirements

### R1 — Registry coverage (minimum five, target six+)

- [ ] **R1a** — The submission inventory includes **Smithery**, **Glama (awesome-mcp-servers)**, **PulseMCP**, **MCP.so**, **skills.sh**, and **askill.sh** as first-class rows in `docs/registry-submissions.json` (or documented alias rows that map to the same assets if a registry is reached via another name).
- [ ] **R1b** — **At least five** rows achieve **`submission_ready`** simultaneously, per existing rules in `build_registry_submission_inventory()` (≥5 ready, ≥2 MCP category, ≥2 skill category).
- [ ] **R1c** — Each row has a non-empty **`install_hint`**, **`proof_note`**, and either **`proof_url`** or **`proof_path`** pointing at verifiable evidence (merged PR, package page, or committed manifest).

### R2 — Human submission actions (outbound)

- [ ] **R2a** — **Smithery:** Register or claim the package so **`mcp-server/server.json`** (and npm package name) resolve in Smithery’s directory; record the public listing URL in **`proof_url`**.
- [ ] **R2b** — **Glama:** Open (or reference) a merged PR to **`punkpeye/awesome-mcp-servers`**; store PR or commit link in **`proof_url`** or **`notes`**.
- [ ] **R2c** — **PulseMCP:** Submit via their catalog process; record listing or PR URL.
- [ ] **R2d** — **MCP.so:** Complete their submission flow; record listing URL.
- [ ] **R2e** — **skills.sh** and **askill.sh:** Submit the **`skills/coherence-network/`** tree per each site’s format; record listing URLs when live.

### R3 — Artifact and documentation hygiene

- [ ] **R3a** — `mcp-server/package.json`, `mcp-server/server.json`, `mcp-server/README.md`, and registry-specific files already referenced in **`docs/registry-submissions.json`** (`glama.json`, `pulsemcp.json`, etc.) stay valid and versioned with releases.
- [ ] **R3b** — Top-level **`README.md`** documents install paths: `npx coherence-mcp-server` and skill install (e.g. ClawHub / OpenClaw) where applicable.
- [ ] **R3c** — **`docs/shared/ecosystem-table.md`** (if present) stays consistent with registry links for reviewer spot-checks.

### R4 — Measurement and proof (working yet? clearer over time?)

- [ ] **R4a** — **Short term:** `GET /api/discovery/registry-submissions` must show **`summary.core_requirement_met: true`** when the repo and proofs are in good shape (binary “we are allowed to claim readiness”).
- [ ] **R4b** — **Medium term (Spec 180):** Implement **`GET /api/discovery/registry-stats`** to pull **install/download** counts from **Smithery** and **PulseMCP** where APIs exist; mark **`unavailable`** where no API exists (`mcp.so`, skills indexes until APIs appear).
- [ ] **R4c** — **Trend:** Store periodic snapshots (file cache first; PostgreSQL time series later—see Spec 180 Known Gaps) so week-over-week **`total_installs`** can increase—**the strongest proof** the idea works.
- [ ] **R4d** — **Dashboard:** **`GET /api/discovery/registry-dashboard`** (Spec 180) merges readiness + stats for a single reviewer curl.

### R5 — Open questions (must be addressed in implementation notes)

1. **How can we improve this idea?**  
   - Add **npm download weekly** and **GitHub dependency/dependent count** as secondary signals where registry APIs are thin.  
   - Publish a **one-page “discovery checklist”** in `docs/` linked from README for repeatability each release.  
   - Automate **PR scaffolding** to Glama/PulseMCP when manifests change (Spec 182 follow-on).

2. **How do we show whether it is working yet?**  
   - **Binary:** `core_requirement_met` + non-null **`proof_url`** for each target registry.  
   - **Quantitative:** non-zero **`install_count`** / **`download_count`** on at least one **live** stats source after Spec 180.  
   - **Qualitative:** merged external PRs or listings linked from inventory.

3. **How do we make proof clearer over time?**  
   - Timestamped stats (`fetched_at`) and optional **`?refresh=true`** for forced refresh.  
   - Weekly cron or manual “snapshot” discipline recorded in **`docs/system_audit/`** only if a spec explicitly requires it (avoid scope creep here).  
   - Web **`/discovery`** panel (Spec 180 follow-on) for non-technical stakeholders.

## API Contract

### Existing (must remain stable for verification)

#### `GET /api/discovery/registry-submissions`

**Response 200:** `RegistrySubmissionInventory` (`api/app/models/registry_discovery.py`)

- **`summary.target_count`** — number of inventory rows.  
- **`summary.submission_ready_count`** — rows with no missing files and valid proof.  
- **`summary.core_requirement_met`** — boolean; **true** only when ≥5 ready, ≥2 MCP, ≥2 skill.  
- **`items[]`** — per-registry status, `missing_files`, `proof_url`, `proof_path`.

**Response 5xx:** Should not occur for malformed JSON inventory; service returns empty items if file missing (implementation detail—tests define contract).

### Planned (Spec 180 — not blocking this spec’s acceptance of submission work)

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/discovery/registry-stats` | Per-registry install/download counts, cache metadata |
| `GET` | `/api/discovery/registry-dashboard` | Merged readiness + stats |

### Web / CLI (optional follow-on)

| Surface | Purpose |
|---------|---------|
| Web page `/discovery` (future) | Renders dashboard JSON for humans |
| `cc` CLI | No new command required for MVP; optional `cc discovery` if spec 148 extended |

## Verification Scenarios

### Scenario 1 — Readiness inventory lists six target registries

**Setup:** API deployed with current `docs/registry-submissions.json` containing rows for Smithery (or equivalent MCP directory row), Glama, PulseMCP, MCP.so, skills.sh, askill.sh.

**Action:**
```bash
curl -sS "https://api.coherencycoin.com/api/discovery/registry-submissions" | python3 -c "import sys,json; d=json.load(sys.stdin); print(sorted({i['registry_id'] for i in d['items']}))"
```

**Expected:** The printed set **includes** at least: `glama`, `pulsemcp`, `skills-sh`, `askill-sh`, and identifiers for **Smithery** and **MCP.so** (e.g. `smithery`, `mcp-so`, or documented aliases in `notes`—reviewer checks human-readable `registry_name` matches the task list).

**Edge — corrupt inventory file:** If `docs/registry-submissions.json` is invalid JSON, the service must not crash the process; **`items`** may be empty and **`summary.target_count`** is **0** (exact behavior covered by existing tests—implementation must match tests).

---

### Scenario 2 — Core requirement: full “create → read → update” cycle via inventory + GET

**Setup:** Local API or staging with writable repo checkout.

**Action (create):** Add a new registry row with `registry_id: "zzz-test-registry"`, `category: "mcp"`, minimal valid fields, **`proof_path`** pointing to an existing file, **`required_files`** listing only files that exist.

**Action (read):**
```bash
curl -sS "http://localhost:8000/api/discovery/registry-submissions" | python3 -c "import sys,json; d=json.load(sys.stdin); print(any(i['registry_id']=='zzz-test-registry' for i in d['items']))"
```

**Expected:** Prints **`True`**.

**Action (update):** Remove `proof_path` and `proof_url` so no proof exists; save file.

**Action (read again):** Same curl; record shows **`status`** **`missing_assets`** (or equivalent: missing proof) and **`summary.submission_ready_count`** decreases accordingly.

**Edge — duplicate `registry_id`:** Two rows with the same `registry_id` are **discouraged**; if present, both appear in list and reviewers must reconcile manually—implementation should not silently dedupe (document as operational risk).

---

### Scenario 3 — Proof surfaces: npm and Glama packets

**Setup:** `mcp-server/package.json` exists; npm listing exists if `proof_url` points to npm.

**Action:**
```bash
curl -sS "https://api.coherencycoin.com/api/discovery/registry-submissions" | python3 -c "import sys,json; d=json.load(sys.stdin);\
for i in d['items']:\
  print(i['registry_id'], i['status'], i.get('proof_url') or i.get('proof_path'))"
```

**Expected:** For **`npm`** (or equivalent) row, **`proof_url`** contains `npmjs.com` when published. For **`glama`**, **`proof_path`** or **`proof_url`** references **`mcp-server/glama.json`** or merged PR.

**Edge — missing `mcp-server/glama.json`:** `glama` row has **`missing_files`** including **`mcp-server/glama.json`** and **`status`** **`missing_assets`**.

---

### Scenario 4 — Error handling: bad category and missing resource

**Setup:** Unit/integration test client (see `api/tests/test_registry_discovery_api.py`).

**Action:** POST is **not** defined for this resource—**attempts to POST** `/api/discovery/registry-submissions` must return **405 Method Not Allowed** (or **404** if router does not mount POST—reviewer must see **not 500**).

**Expected:** No server error **500** for unsupported methods.

**Edge — GET with invalid query:** `GET /api/discovery/registry-submissions?foo=bar` returns **200** and ignores unknown params (forward-compatible) **or** **422** if strict validation is added—**must be documented**; default is **200** with full inventory.

---

### Scenario 5 — Stats endpoint (Spec 180) proves adoption

**Setup:** Spec 180 implemented; Smithery stats API reachable.

**Action:**
```bash
curl -sS "https://api.coherencycoin.com/api/discovery/registry-stats?registry_id=smithery"
```

**Expected:** **200** JSON with **`source`** in `live` \| `cached` \| `unavailable`, numeric **`install_count`** when available, **`fetched_at`** ISO timestamp.

**Edge — upstream timeout:** Smithery returns **200** response body with **`source: unavailable`**, **`error`** non-null, **`install_count: null`**—**not** HTTP 500 from our API.

## Risks and Assumptions

| Risk | Mitigation |
|------|------------|
| Third-party registry rejects or delays listing | Keep proofs in-repo; **`notes`** capture ticket/PR state; do not fake **`proof_url`**. |
| Stats APIs change or rate-limit | Cache + **`unavailable`** fallback (Spec 180). |
| Duplicate work vs Spec 180 | This task spec is the **program**; 180 is the **stats/dashboard** engineering spec—implement in order. |
| Confusion between **npm**, **Smithery**, and **official MCP registry** | Inventory **`registry_name`** and **`notes`** must disambiguate; align IDs in JSON as listings go live. |
