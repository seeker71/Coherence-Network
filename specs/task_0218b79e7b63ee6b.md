## Summary

Register the Coherence Network MCP server and skill on **six target registries**:
Smithery, Glama (`awesome-mcp-servers` PR), PulseMCP, `mcp.so`, `skills.sh`, and
`askill.sh`. Track install and download counts per registry so that growth in
discoverability is measurable and **provable over time**.

[Spec 180](./180-mcp-skill-registry-submission.md) established the data model and
API skeleton. This document is the **product-manager contract** that defines:

1. The **proof contract** — what "working" looks like concretely.
2. The **improvement loop** — how evidence accumulates and becomes clearer each week.
3. The **full verification scenarios** a reviewer runs against production to confirm
   the feature is shipped and functioning.
4. Remaining gaps and the path to close them.

## Open Question Resolution: How Do We Show It's Working?

### Problem
The original idea lacked a mechanism to distinguish "we submitted" from "people are
actually finding and installing it." A ticket can be closed by submitting a PR;
actual discovery requires external signals.

### Solution: Three-Layer Proof

#### Layer 1 — Asset Readiness (immediate, automated)
`GET /api/discovery/registry-submissions` answers: **"Did we do our half?"**

- All required manifest files exist and are valid.
- `summary.core_requirement_met = true` when ≥ 5 registries are `submission_ready`.
- Checkable in CI on every merge; verifiable in production via one `curl`.

#### Layer 2 — Live Count Signals (days–weeks, external)
`GET /api/discovery/registry-stats` answers: **"Is anyone discovering us?"**

- Smithery and PulseMCP expose public stats APIs.
- When `install_count > 0` for either, external discovery is confirmed.
- Cache files carry `fetched_at` timestamps, enabling trend reconstruction
  without a dedicated time-series store.

#### Layer 3 — Trend Accumulation (weeks–months, automated)
A weekly `?refresh=true` call (cron or operator habit) returns a snapshot with a
newer `fetched_at`. Comparing successive snapshots shows whether install counts are
rising. The first non-zero `total_installs` is the clearest external milestone.

#### Dashboard Clarity
`GET /api/discovery/registry-dashboard` collapses Layers 1 and 2 into a single
human-readable JSON: registry → `submission_ready` / `missing_assets` → `install_count`.
Any reviewer can run one `curl` command and see the full picture.

## API Contract

Base path: `/api/discovery` (FastAPI prefix).

### Existing (unchanged contract)

```
GET /api/discovery/registry-submissions
→ RegistrySubmissionInventory
```

Returns per-registry asset-readiness check. Must include all six registries. No
change to response schema.

### New endpoints

```
GET /api/discovery/registry-stats
→ RegistryStatsList
```

Returns install/download counts per registry. Query params:
- `?refresh=true` — bypass 24-h cache and fetch live
- `?registry_id={id}` — filter to one registry

```
GET /api/discovery/registry-dashboard
→ RegistryDashboard
```

Merges submission inventory with stats. Always HTTP 200 even if all stats fail
(graceful degradation).

## Data Model Summary

Full definitions in spec-180. Key types used in verification:

```python
class RegistryStatSource(str, Enum):
    LIVE = "live"
    CACHED = "cached"
    UNAVAILABLE = "unavailable"

class RegistryStats(BaseModel):
    registry_id: str
    registry_name: str
    install_count: int | None = None
    download_count: int | None = None
    star_count: int | None = None
    fetched_at: datetime | None = None
    source: RegistryStatSource
    error: str | None = None

class RegistryStatsSummary(BaseModel):
    total_installs: int
    total_downloads: int
    registries_with_counts: int
    registries_unavailable: int
    last_updated: datetime | None

class RegistryDashboardItem(BaseModel):
    registry_id: str
    registry_name: str
    category: str                          # "mcp" or "skill"
    status: RegistrySubmissionStatus       # "submission_ready" | "missing_assets"
    missing_files: list[str]
    install_count: int | None
    download_count: int | None
    stat_source: RegistryStatSource
```

## Stats Cache Design

```
.cache/registry_stats/
  smithery.json     { "install_count": N, "fetched_at": "2026-03-28T10:00:00Z" }
  pulsemcp.json     { "install_count": N, "fetched_at": "..." }
  glama.json        { "star_count": N, "fetched_at": "..." }
```

- **TTL**: 24 hours
- **On cache miss** or `?refresh=true`: fetch live, write cache file
- **On fetch failure**: return last cached value (`source: "cached"`); if no cache
  exists, return `source: "unavailable"` with `error` field populated
- Cache directory: `.cache/registry_stats/` — already covered by `.cache/` in
  `.gitignore`

## Verification Scenarios

**Base URL**: `API=https://api.coherencycoin.com`

### Scenario 1 — All six registries appear in submission inventory

**Setup**: API deployed with extended `_TARGETS` list (all six registries).

**Action**:
```bash
curl -s $API/api/discovery/registry-submissions | \
  python3 -c "import sys,json; d=json.load(sys.stdin); ids=[i['registry_id'] for i in d['items']]; print(sorted(ids))"
```

**Expected**:
```
['askill-sh', 'glama', 'mcp-so', 'pulsemcp', 'skills-sh', 'smithery']
```

**Then** — verify total count:
```bash
curl -s $API/api/discovery/registry-submissions | \
  python3 -c "import sys,json; d=json.load(sys.stdin); print(d['summary']['total_count'])"
```
**Expected**: `6`

**Edge — missing asset**: Remove `mcp-server/server.json` from deployed repo,
restart API, re-run. Expected: `smithery` item has `status: "missing_assets"` and
`missing_files` list contains `"mcp-server/server.json"`. HTTP 200.

---

### Scenario 2 — Core requirement met when all assets present (create-read cycle)

**Setup**: All required files exist in the deployed repo.

**Action**:
```bash
curl -s $API/api/discovery/registry-submissions | \
  python3 -c "
import sys, json
d = json.load(sys.stdin)
s = d['summary']
print('core_requirement_met:', s['core_requirement_met'])
print('submission_ready_count:', s['submission_ready_count'])
for item in d['items']:
    print(item['registry_id'], item['status'])
"
```

**Expected**:
```
core_requirement_met: True
submission_ready_count: 6
smithery submission_ready
glama submission_ready
pulsemcp submission_ready
mcp-so submission_ready
skills-sh submission_ready
askill-sh submission_ready
```

**Edge — partial assets**: Delete only `skills/coherence-network/SKILL.md`.
Re-run. Expected: `skills-sh` and `askill-sh` both report `missing_assets`.
`submission_ready_count` drops to 4. `core_requirement_met` is `false`.

---

### Scenario 3 — Stats endpoint returns correct source labels per registry

**Setup**: Fresh deployment; no cache files exist in `.cache/registry_stats/`.

**Action**:
```bash
curl -s "$API/api/discovery/registry-stats" | \
  python3 -c "
import sys, json
d = json.load(sys.stdin)
for i in d['items']:
    print(i['registry_id'], i['source'], i['install_count'])
print('total_installs:', d['summary']['total_installs'])
"
```

**Expected**:
- `smithery` — `source` is `live` or `unavailable` (depends on upstream)
- `pulsemcp` — `source` is `live` or `unavailable`
- `glama` — `source` is `live` (GitHub API) or `unavailable`
- `mcp-so`, `skills-sh`, `askill-sh` — `source: "unavailable"`, `install_count: null`
- HTTP 200 in all cases; no 500

**Edge — Smithery API times out**: Smithery item returns `source: "unavailable"`,
`error` field non-empty (e.g. `"upstream timeout"`). All other items unaffected.
HTTP 200 still returned.

---

### Scenario 4 — Cache refresh forces live fetch; stale cache auto-refreshes

**Setup**: Cache file `.cache/registry_stats/smithery.json` exists with data from
at least 1 hour ago.

**Action (no refresh flag)**:
```bash
curl -s "$API/api/discovery/registry-stats" | \
  python3 -c "import sys,json; d=json.load(sys.stdin); s=[i for i in d['items'] if i['registry_id']=='smithery'][0]; print(s['source'], s['fetched_at'])"
```
**Expected**: `cached <timestamp-from-file>`

**Action (force refresh)**:
```bash
curl -s "$API/api/discovery/registry-stats?refresh=true" | \
  python3 -c "import sys,json; d=json.load(sys.stdin); s=[i for i in d['items'] if i['registry_id']=='smithery'][0]; print(s['source'])"
```
**Expected**: `live` — cache bypassed and overwritten.

**Edge — cache file older than 24 h**: Auto-triggers live fetch on next regular
call (no `?refresh=true`). `source` becomes `live`.

---

### Scenario 5 — Dashboard merges both datasets; graceful on total stats failure

**Setup**: At least one registry is `submission_ready`; stats may or may not be
available.

**Action**:
```bash
curl -s "$API/api/discovery/registry-dashboard" | \
  python3 -c "
import sys, json
d = json.load(sys.stdin)
print('core_requirement_met:', d['submission_summary']['core_requirement_met'])
for i in d['items']:
    print(i['registry_id'], i['status'], 'installs:', i['install_count'], 'src:', i['stat_source'])
"
```

**Expected**: One line per registry. `core_requirement_met` matches the value
from `/registry-submissions`. Each item carries both `status` (submission side)
and `install_count` / `stat_source` (stats side). HTTP 200.

**Edge — stats service completely fails**: All `install_count` fields are `null`,
`stat_source` is `"unavailable"`. Dashboard HTTP 200 with submission fields intact.

## Improvement Loop (Proof Gets Clearer Over Time)

| Week | Signal | Source |
|------|--------|--------|
| Week 0 | Assets committed, PRs submitted | `submission_ready_count >= 5` |
| Week 1 | PRs merged to registries | Manual `notes` field in inventory |
| Week 2+ | First `install_count > 0` | `GET /registry-stats` — Smithery or PulseMCP |
| Month 1 | Trend: `total_installs` rising week-over-week | Compare `?refresh=true` snapshots |
| Month 3 | Follow-up spec: PostgreSQL time series + `/discovery` web dashboard | Spec 181 |

The key insight: **the API becomes the audit trail**. Every `?refresh=true` call is
a timestamped observation. Without additional infra, operators can reconstruct a
growth curve from cache file history or periodic log output.

## Risks and Assumptions

| Risk | Mitigation |
|------|-----------|
| Smithery/PulseMCP stat APIs may change schema | Parse defensively; unknown fields ignored; `source: "unavailable"` fallback |
| GitHub rate limits on star-count fetch | Unauthenticated GitHub API: 60 req/h — sufficient for weekly refresh |
| Registry PRs may be delayed or rejected | Asset readiness is tracked independently; `notes` field records PR URL once submitted |
| `mcp.so`, `skills.sh`, `askill.sh` have no public count APIs | Marked `unavailable`; re-evaluated when APIs emerge |
| `skills.sh` / `askill.sh` may require different manifest format | Investigate before PR; update validators accordingly |
