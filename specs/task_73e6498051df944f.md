## Summary

The Invest page (`/invest`) currently frames opportunities in **financial spreadsheet language**: “Value gap,” “Est. cost,” and “ROI” with a **bare horizontal progress bar**. That reads as cold optimization and hides the product story: contributors are **nurturing ideas** with real compute and attention.

This spec reframes the page around a **garden metaphor**: each idea is a living plot; **growth potential** is shown as a **sprout → sapling → tree** (or similar staged) visual progression instead of a generic bar. **Numeric signals remain available** (tooltip, secondary row, or expandable “details”) so power users and auditors are not blocked, but the **primary emotional read** is care, growth, and life—not a ledger.

An explicit product question is answered: **how we know the metaphor is working and how proof gets clearer over time** (instrumentation, qualitative checks, and iteration criteria)—see §Measurement & Proof.

---

## Purpose

- **Why**: Align surface copy and visuals with Coherence Network’s narrative (attention → working code; living ecosystem) and reduce “spreadsheet anxiety” on a page meant to invite participation.
- **Who benefits**: First-time visitors deciding whether to stake, returning contributors scanning opportunities, and reviewers who need numbers without them dominating the layout.
- **Failure/cost avoided**: Users bouncing because the page feels like a trading desk; misinterpretation of ROI as a guarantee; accessibility regressions when replacing the bar with illustrative graphics.

## Requirements

### R1 — Primary language: garden, not ledger

- Replace user-visible **row headers** so the default scan does not say “Value gap,” “Est. cost,” or “ROI” as the main nouns.
- Acceptable primary framings (pick one consistent set in implementation): e.g. **Room to grow**, **Water & care (est.)**, **How full the canopy could get** — exact strings are **implementation details** but must read as **garden/ecosystem**, not accounting.
- **Stake** may remain as the action label (already consistent with “planting” metaphor) or be paired with a garden synonym in secondary copy only if usability testing favors it.

### R2 — Growth visual: sprout-to-tree (not a generic bar)

- Replace the **ROI horizontal bar** as the **primary** growth indicator with a **staged visual** (sprout → tree or 3–5 discrete stages), driven by a **deterministic mapping** from existing metrics (see §Data Model).
- The visual must **not be the only encoding**: provide text or `aria-*` equivalent for screen readers (e.g. “Growth stage 2 of 4: sapling”).
- **Numbers stay accessible**: at minimum, a **secondary** pattern—e.g. muted numeric row, “Details” disclosure, or `title`/`aria-describedby` linking to formatted currency and ratio—so `formatUsd` values and a derived multiplier remain discoverable.

### R3 — Hierarchy: feeling first, numbers second

- Layout order: **idea name + stage badge** → **garden growth visual + plain-language caption** → **numeric detail** (collapsible or de-emphasized).
- Sorting may remain ROI-like internally but the **UI must not headline “ROI”** as the first column.

### R4 — Empty, loading, and error paths

- **No ideas**: keep a warm empty state; optional garden-themed copy (“No seedlings yet”).
- **API failure** (empty list from failed fetch): must not crash; message should not blame the user.
- **Ledger unavailable**: existing “Unavailable” balance behavior remains acceptable; copy may soften to garden tone if changed in scope.

### R5 — Non-goals (this spec)

- Does **not** redefine ROI math or portfolio accounting (see spec 157).
- Does **not** require new backend persistence **for MVP**; optional future `growth_stage` on API responses is listed under Known Gaps.

## Data Model (presentation)

**Client-side derivation (MVP)**  
Map each idea to a **growth stage index** (e.g. 0–3 or 1–4) using a **documented formula** in code comments and in this spec, for example:

- Inputs: `manifestation_status`, `confidence`, `value_gap`, `estimated_cost`, `free_energy_score` (already on `IdeaWithScore` in `page.tsx`).
- **Example policy** (implementation may tune constants): low manifestation + low confidence → sprout; mid → sapling; high manifestation or high normalized potential → tree. **Exact thresholds** are implementation choices but must be **monotonic** and **stable** (same inputs → same stage).

**Accessibility**  
Stage must be exposed as text, not color-only.

## Measurement & Proof (open question)

**How we improve the idea, show whether it is working, and make proof clearer over time:**

1. **Instrumentation (quantitative)**  
   - Event hooks (privacy-preserving): e.g. `invest_view`, `invest_growth_detail_expand`, `invest_stake_click` with idea id hash or coarse category only if required by policy.  
   - Funnel: landing `/invest` → scroll depth → click Stake → completion on idea page.

2. **Comparative baselines**  
   - Before/after or A/B: time on page, stake click-through rate, bounce rate. **Definition of “win”**: +X% stake clicks or −Y% bounce with no increase in support tickets about “can’t find numbers.”

3. **Qualitative**  
   - Short intercept copy on `/invest` (dismissible): “Does this page feel more like caring for ideas than a spreadsheet?” Y/N + optional comment (stored per product analytics policy).

4. **Accessibility audit**  
   - WCAG: contrast of new illustrations; screen reader walkthrough for growth stage + numbers.

5. **Iteration gate**  
   - If after N weeks metrics are flat **and** qualitative feedback is negative, revisit **mapping thresholds** or **copy** before changing API.

## Verification Scenarios

Each scenario is written so a **reviewer can run it against production** (`API_URL=https://api.coherencycoin.com` or project default).

### VS1 — Read: Invest page loads ideas from API

- **Setup**: Production API reachable; at least zero ideas allowed.  
- **Action**: `curl -sS "$API_URL/api/ideas?limit=60" -H "Accept: application/json"`  
- **Expected**: HTTP `200`, JSON with `ideas` array (possibly empty). Body parses without error.  
- **Edge**: Invalid host / timeout → Invest page must still render shell + empty/error handling without 500 from Next (browser test).

### VS2 — Full create → read cycle for an idea (API contract)

- **Setup**: Valid `POST /api/ideas` payload allowed by deployment (name, description, etc. per OpenAPI).  
- **Action**:  
  1. `curl -sS -X POST "$API_URL/api/ideas" -H "Content-Type: application/json" -d '{...minimal valid body...}'`  
  2. Note `id` from response.  
  3. `curl -sS "$API_URL/api/ideas/{id}"`  
- **Expected**: `201` on create; `200` on get; retrieved idea matches created `id` and name.  
- **Edge**: POST with missing required fields returns `422` (not `500`). GET unknown id returns `404`.

### VS3 — Browser: garden metaphor visible; numbers secondary

- **Setup**: At least one idea in list (or seed data).  
- **Action**: Open `https://coherencycoin.com/invest` (or local equivalent). Inspect first idea card.  
- **Expected**: Primary labels **do not** read as the old trio “Value gap / Est. cost / ROI” as the **only** prominent headers; a **sprout-to-tree** (or staged) visual is visible; numeric detail is present in secondary position, tooltip, or disclosure.  
- **Edge**: With **no** ideas, empty state shows without broken layout.

### VS4 — Ledger read + bad contributor id (error handling)

- **Setup**: Contributor id optional.  
- **Action**: `curl -sS -o /dev/null -w "%{http_code}" "$API_URL/api/contributions/ledger/not-a-real-contributor-xyz"`  
- **Expected**: Non-`500` response (e.g. `404` or empty body per existing API behavior); Invest page client must handle `!res.ok` without crash.  
- **Edge**: Empty `contributor_id` in UI → balance section prompts for name (existing behavior).

### VS5 — Accessibility: growth stage has text equivalent

- **Setup**: Post-implementation.  
- **Action**: Screen reader or DOM inspection: growth control exposes accessible name/description (not color-only).  
- **Expected**: Each card exposes a text phrase for stage (e.g. “Growth stage: sapling”) in `aria-label`, visible caption, or adjacent text.  
- **Edge**: CSS/images disabled → user still sees stage label and key numbers.

## Risks and Assumptions

- **Risk**: Garden metaphor feels cute to expert users. **Mitigation**: Numbers always reachable in one click or row.  
- **Risk**: Illustrations hurt performance. **Mitigation**: Prefer CSS/SVG inline; lazy-load if heavy.  
- **Assumption**: `GET /api/ideas` fields remain stable; if renamed, mapping must update.
