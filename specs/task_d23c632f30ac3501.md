# Spec: Idea Translation Across Worldviews and Belief Systems

**Task / traceability:** `task_d23c632f30ac3501`  
**See also:** `specs/169-belief-system.md` (contributor belief profiles, resonance), `api/tests/test_concept_translation.py` (verification contract for framing translation)

## Summary

The same idea legitimately reads differently across worldviews: a decentralized network can signal freedom (libertarian), engineering efficiency, institutional risk, or entrepreneurial opportunity. Coherence Network should **reframe ideas through multiple lenses without mutating the canonical idea**—surfacing *bridges* (ontology links, shared vocabulary, resonance scores) so people see how their perspective connects to others. The experience extends to **news and external narrative ingestion** with **explicit point-of-view (POV) filters**, and to **every idea being viewable through any registered worldview lens** (built-in axes plus user- or org-defined templates).

This spec is the product contract for that capability: goals, API/data shape, acceptance criteria, executable verification scenarios (runnable against production), risks, and how we prove the feature is working and improving over time.

## Purpose

- Reduce cross-tribe misunderstanding by making *translation* a first-class, inspectable artifact (not a hidden recommendation).
- Let contributors and readers choose a lens explicitly (including for news), aligning with epistemic transparency.
- Ground translations in the graph (concepts, axes, bridges) rather than opaque model-only paraphrases, so outputs are auditable and comparable across time.

## Requirements

1. **Canonical idea immutability by default** — Translation returns a *projection* (summary, bridges, metadata). The stored idea `name` / `description` change only via normal edit flows, not via “translate.”
2. **Lens catalog** — System exposes a discoverable set of lenses: at minimum the worldview axes aligned with Spec 169 (`scientific`, `spiritual`, `pragmatic`, `holistic`, `relational`, `systemic`) plus **socio-political / role templates** (e.g. `libertarian`, `institutional`, `entrepreneurial`, `engineering`) as named presets mapping to axis weights and vocabulary hints.
3. **Per-idea multi-lens read** — For any existing idea `id`, clients can request `GET /api/ideas/{id}/translate?view=<lens>` and receive a stable JSON shape including `translation_kind`, `view`, `summary`, `lens_description`, `bridging_concepts[]`, and optional `analogous_ideas[]`.
4. **Per-concept cross-lens bridge** — `GET /api/concepts/{id}/translate?from=<lens>&to=<lens>` returns framing between two lenses for ontology nodes (supports curriculum and shared vocabulary work).
5. **News / ingestion POV** — Ingested items (URLs, feeds, articles) can be tagged at ingest with `pov_lens` and optionally re-projected through alternate lenses for the same item id (stored as projections, not overwriting source text).
6. **Contributor alignment (optional input)** — If the caller supplies `contributor_id` or `X-Contributor-Context`, the API may blend lens preset with that contributor’s belief profile (Spec 169) and return `resonance_hint` / `profile_alignment` fields (0.0–1.0).
7. **Validation** — Unknown lens → `422`; missing idea/concept → `404`; invalid combination (e.g. `from == to` on concept translate) → `400`.
8. **Observability** — Each successful translation response includes `projection_id` (or deterministic hash of inputs + idea revision) for caching, analytics, and audit.

## API Contract

### `GET /api/ideas/{idea_id}/translate`

**Query**

- `view` (required): enum — built-in lens id or registered template id.
- `contributor_id` (optional): string — if present and valid, blend with belief profile.
- `locale` (optional): BCP-47 tag — future: localized lens copy; MVP may ignore.

**Response 200**

```json
{
  "idea_id": "decentralized-network-001",
  "view": "libertarian",
  "translation_kind": "concept_framing",
  "lens_description": "Emphasizes individual agency, voluntary coordination, and exit rights.",
  "summary": "Non-empty reframing grounded in ontology bridges (see bridging_concepts).",
  "bridging_concepts": [
    { "id": "autonomy", "name": "Autonomy", "score": 0.91, "axes": ["relational"] }
  ],
  "analogous_ideas": [],
  "projection_id": "sha256:…",
  "spec_ref": "task_d23c632f30ac3501"
}
```

**Response 404** — idea not found.  
**Response 422** — invalid `view`.

### `GET /api/concepts/{concept_id}/translate`

**Query**

- `from`, `to` (required): lens enums, must differ.

**Response 200** — includes `concept_id`, `from_lens`, `to_lens`, `translation_kind`, `summary`, `target_bridging_concepts`, `source_axes` (see tests).

**Response 400** — `from == to`.  
**Response 404** — unknown concept.

### `GET /api/worldviews/lenses` (new)

**Response 200** — list of `{ "id", "label", "description", "axis_weights"?, "category": "epistemic|socioeconomic|role" }` for UI and CLI discovery.

### News / ingestion (phase 2 in same spec family)

### `POST /api/news/items` (or existing ingest hook — align with implementation)

**Body (excerpt)**

```json
{
  "url": "https://example.com/article",
  "title": "Headline",
  "pov_lens": "institutional",
  "raw_excerpt": "…"
}
```

### `GET /api/news/items/{id}/translate?view=<lens>`

Same projection semantics as ideas, keyed to news item id.

*If the codebase uses a different news resource path, the implementing spec SHALL name the exact router module in `Files to Create/Modify`.*

## Data Model

**WorldviewLens** (registry node or table)

- `id` (string, unique)
- `label`, `description`
- `category`: `epistemic` | `socioeconomic` | `role` | `custom`
- `axis_weights` (map string → float, optional) — aligns with Spec 169 axes
- `vocabulary_hints` (string[], optional) — token hints for retrieval/ranking
- `version`, `updated_at`

**IdeaTranslationProjection** (materialized or cache)

- `projection_id` (string)
- `idea_id`, `view`, `idea_content_hash` (hash of canonical fields used)
- `payload` (JSON blob matching API response)
- `created_at`

**NewsItem** (extension)

- `pov_lens` (string)
- optional `projections` map `lens_id → projection_id`

## Acceptance Criteria

1. Discovery: `GET /api/worldviews/lenses` returns 200 with at least the built-in epistemic axes and four role/socioeconomic templates (libertarian, engineering, institutional, entrepreneurial).
2. Idea translate: for an existing idea, `GET /api/ideas/{id}/translate?view=<valid>` returns 200 with `translation_kind == concept_framing`, non-empty `summary`, and `bridging_concepts` as a list of objects with `id`, `name`, `score`.
3. Stability: repeated GET with same idea revision and lens returns the same `projection_id` (or explicitly documented cache TTL if non-deterministic — then document in Risks).
4. News: ingested item with `pov_lens` persists; `GET /api/news/items/{id}/translate?view=other` returns 200 or documented 501 with `Retry-After` only if phase 2 is split — **MVP prefers 200 with projection or clear 404 if news module absent** (implementing engineer must not leave ambiguous 500).
5. All scenarios in **Verification Scenarios** pass against production API base `https://api.coherencycoin.com` when auth headers match deploy policy.

## Verification Scenarios

Scenarios below are the **contract** for reviewers. Replace `{API}` with `https://api.coherencycoin.com` and `{KEY}` with a valid API key if the endpoint requires it.

### Scenario 1 — Lens catalog is public and structured

- **Setup:** None (cold caller).
- **Action:** `curl -sS "{API}/api/worldviews/lenses"`
- **Expected:** HTTP `200`; JSON array with length ≥ 6; each item has `id` and `label`; at least one item has `id` in `scientific`, `spiritual`, `libertarian`, `engineering`, `institutional`, `entrepreneurial` (exact id set may use prefixes — reviewer checks for documented ids in response body).
- **Edge:** `curl -sS -o /dev/null -w "%{http_code}" "{API}/api/worldviews/lenses/typo"` → `404` (if sub-resource not implemented) or `405` — must not be `500`.

### Scenario 2 — Idea translate after create

- **Setup:** `curl -sS -X POST "{API}/api/ideas" -H "Content-Type: application/json" -H "X-API-Key: {KEY}" -d '{"id":"wvt-demo-net","name":"Decentralized mesh","description":"Peer coordination without central broker; incentives and protocol rules align participants.","potential_value":10,"estimated_cost":1,"confidence":0.5,"tags":["network","p2p"]}'`
- **Action:** `curl -sS "{API}/api/ideas/wvt-demo-net/translate?view=libertarian"`
- **Expected:** HTTP `200`; body contains `"idea_id":"wvt-demo-net"`, `"view":"libertarian"`, `"translation_kind":"concept_framing"`, `"summary"` with length > 40, `"bridging_concepts"` is a non-empty JSON array, each element includes `"id"`, `"name"`, and numeric `"score"`.
- **Edge (missing idea):** `curl -sS "{API}/api/ideas/does-not-exist-wvt/translate?view=libertarian"` → HTTP `404` with JSON `detail` string.
- **Edge (bad lens):** `curl -sS "{API}/api/ideas/wvt-demo-net/translate?view=not_a_lens"` → HTTP `422`.

### Scenario 3 — Same idea, two lenses, comparable projections

- **Setup:** Idea `wvt-demo-net` from Scenario 2 exists.
- **Action:**  
  `curl -sS "{API}/api/ideas/wvt-demo-net/translate?view=engineering" > /tmp/wvt_eng.json`  
  `curl -sS "{API}/api/ideas/wvt-demo-net/translate?view=institutional" > /tmp/wvt_inst.json`
- **Expected:** Both HTTP `200`; `jq -r '.summary' /tmp/wvt_eng.json` and `jq -r '.summary' /tmp/wvt_inst.json` are non-empty and **not byte-identical** in typical deployments (allows identical only if idea text is symmetric — then assert `.lens_description` differs).
- **Edge:** Omit `view`: `curl -sS "{API}/api/ideas/wvt-demo-net/translate"` → HTTP `422`.

### Scenario 4 — Ontology concept bridge

- **Setup:** `curl -sS "{API}/api/concepts/activity"` returns `200` (seeded ontology).
- **Action:** `curl -sS "{API}/api/concepts/activity/translate?from=scientific&to=artistic"`
- **Expected:** HTTP `200`; body has `"concept_id":"activity"`, `"from_lens":"scientific"`, `"to_lens":"artistic"`, `"translation_kind":"concept_framing"`, `"summary"` length > 10.
- **Edge:** `curl -sS "{API}/api/concepts/activity/translate?from=scientific&to=scientific"` → HTTP `400`.
- **Edge:** unknown concept → HTTP `404`.

### Scenario 5 — News item POV and alternate projection (when news submodule shipped)

- **Setup:** `POST` a news item with `pov_lens":"institutional"` (exact path per implementation).
- **Action:** `curl -sS "{API}/api/news/items/{id}/translate?view=entrepreneurial"`
- **Expected:** HTTP `200`; response includes `translation_kind`, `summary`, and `projection_id`.
- **Edge:** translate unknown news id → `404`; invalid lens → `422`.

*If news endpoints are not yet deployed, Scenario 5 is marked **deferred** in the implementation PR and MUST be listed under Known Gaps with a target spec slice; Scenarios 1–4 remain blocking.*

## How We Improve the Idea, Know It Works, and Clarify Proof Over Time

1. **Metrics (quantitative)**  
   - `translation_requests_total` and `translation_requests_by_lens` (counter).  
   - `bridge_concept_hit_rate` — fraction of responses where `len(bridging_concepts) > 0`.  
   - `projection_cache_hit_rate` — efficiency and stability signal.  
   - **Cross-lens divergence score** — average cosine distance between embedding vectors of summaries for the same idea across two lenses (optional; document model version).

2. **Human-in-the-loop quality**  
   - Thumbs-up/down on a projection; store on `projection_id`.  
   - Periodic sample review task in the agent backlog: “Does this summary misrepresent the canonical idea?” (binary + comment).

3. **A/B and cohort**  
   - Expose lens default in UI by cohort; measure time-on-idea, follow-up link clicks to `bridging_concepts`, and contributor edits after viewing translations.

4. **Public evidence artifacts**  
   - Live OpenAPI entry for translate endpoints.  
   - Read-only dashboard or `GET /api/worldviews/roi` aggregating counters (pattern from Spec 169 ROI endpoint).  
   - Screenshot of idea detail with lens selector stored in release notes (attested URL).

5. **Iteration loop**  
   - Each month: publish a short “translation quality” appendix in `docs/STATUS.md` or spec Known Gaps: metric values + one user quote + one failure case addressed.

## Out of Scope (this spec slice)

- Training new foundation models solely for paraphrase.  
- Automatic rewriting of user-authored idea text without explicit user action.  
- Legal moderation of controversial lenses (handled under separate trust & safety spec).

## Risks and Assumptions

- **Assumption:** Ontology and concept graph contain enough edges to support non-empty bridges for most ideas; otherwise summaries may feel generic. **Mitigation:** vocabulary_hints on lenses + fallback “sparse graph” template text that still cites uncertainty.  
- **Risk:** Socio-political templates may be perceived as editorial bias. **Mitigation:** label lenses as explicit *perspectives*, show axis weights, link to Spec 169 epistemic framing.  
- **Risk:** Non-deterministic LLM use breaks `projection_id` stability. **Mitigation:** freeze prompt version in response metadata or use deterministic retrieval-only MVP.

## Known Gaps and Follow-up Tasks

- Wire news ingest paths to actual router names once the news module lands; finalize Scenario 5 or defer with date.  
- CLI: `cc idea <id> --lens libertarian` mirroring API.  
- Federation: propagate `projection_id` across instances for comparable audits.

## Failure/Retry Reflection

- **Failure mode:** Translate endpoint timeouts under load.  
- **Blind spot:** Uncached graph walks per request.  
- **Next action:** Add projection cache keyed by `(idea_id, content_hash, lens_id, prompt_version)` and stale-while-revalidate.
