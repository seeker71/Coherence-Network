## Summary

Every contributor needs a single page that answers: *"What have I done? What did I invest in?
Where is my CC? How are my ideas doing?"*

Currently the system records contributions, stakes, tasks, and CC balances — but no personal
view aggregates them. A contributor must query five separate endpoints to reconstruct their
own history. This spec defines the **My Portfolio** page (`/portfolio/me`) and the backing
API endpoint family (`/api/portfolio/me`), which deliver a unified, drill-down personal
ledger.

The metaphor: a **garden where each plant is an idea you helped grow**. The technical
reality: a **full-value ledger with audit trail**, where every CC earned, staked, and
returned is traceable to a contribution event.

---

## Problem Statement

A contributor cannot currently:
- See all their CC earnings in one place or as a chart over time
- View ideas they contributed to alongside their specific contribution type and current idea value
- Track staking ROI — what they staked, when, and what returned
- Review completed tasks (with provider and outcome) without querying the raw task log
- Drill down from an idea to their specific contribution to the CC event that paid them

This information exists in the system. It is not surfaced. This spec surfaces it.

---

## Goal

Deliver a **single-page personal portfolio view** that:

1. Shows linked identities at the top (handle, trust level, GitHub hint, wallet hint)
2. Shows CC balance and earning history as a mini time-series chart
3. Lists ideas the contributor contributed to — with status, contribution type, and current idea value
4. Lists ideas the contributor staked CC on — with amount, timestamp, ROI since staking
5. Lists tasks the contributor completed — with provider, outcome (success/fail/partial), and CC earned
6. Supports drill-down: idea → my contributions → full value lineage chain
7. Supports drill-down: contribution → CC earned → lineage chain it belongs to

## Requirements

### R1 — Authentication: Session Token (TOFU)

- The portfolio is **personal**. Only the authenticated contributor may view their own portfolio.
- Authentication uses the TOFU session token from Spec 168: `Authorization: Bearer <session_token>`.
- `GET /api/portfolio/me` resolves `contributor_id` from the token (via `contributors` table, column `session_token`).
- Unauthenticated requests → **401** `{ "detail": "authentication_required" }`.
- A contributor **cannot** view another contributor's portfolio via this endpoint. Separate admin/public endpoints are out of scope for this spec.
- No new session infrastructure is introduced; the existing TOFU token from Spec 168 is the key.

### R2 — Identity Section

`GET /api/portfolio/me` response includes an `identity` block:

```json
{
  "identity": {
    "contributor_id": "uuid",
    "handle": "alice",
    "trust_level": "tofu",
    "hint_github": "alice-dev",
    "hint_wallet": null,
    "created_at": "2026-03-01T12:00:00Z"
  }
}
```

### R3 — CC Balance and Earning History

Response includes a `cc` block:

```json
{
  "cc": {
    "balance": 420.5,
    "total_earned": 560.0,
    "total_staked": 100.0,
    "total_returned": 80.0,
    "network_share_pct": 0.042,
    "earning_history": [
      { "date": "2026-03-01", "earned": 50.0, "cumulative": 50.0 },
      { "date": "2026-03-15", "earned": 120.0, "cumulative": 170.0 }
    ]
  }
}
```

- `balance` = `total_earned - total_staked + total_returned`
- `earning_history` is daily-bucketed, covering the last 90 days
- Both absolute CC amounts **and** `network_share_pct` are returned — answering the open
  question *absolute or percentage?*: **both are returned; UI defaults to absolute, with a
  toggle for % of network**

### R4 — Ideas Contributed To

Response includes a `contributions` array, one entry per idea the contributor has touched:

```json
{
  "contributions": [
    {
      "idea_id": "graphql-caching",
      "idea_title": "GraphQL Caching Layer",
      "idea_status": "specced",
      "idea_current_value": 72.0,
      "idea_coherence_score": 0.81,
      "my_contribution_type": "code",
      "my_contribution_count": 3,
      "my_cc_earned": 45.0,
      "first_contributed_at": "2026-03-10T08:00:00Z",
      "latest_contributed_at": "2026-03-22T14:30:00Z"
    }
  ]
}
```

- `idea_current_value` is sourced from the grounded metrics (Spec 116)
- `my_contribution_type` is the dominant type: code / review / spec / test / idea
- `my_contribution_count` = number of contribution records for this contributor × idea
- Sorted by `latest_contributed_at` descending by default

### R5 — Stakes (Investments)

Response includes a `stakes` array:

```json
{
  "stakes": [
    {
      "stake_id": "uuid",
      "idea_id": "graphql-caching",
      "idea_title": "GraphQL Caching Layer",
      "idea_status": "testing",
      "amount_staked_cc": 50.0,
      "staked_at": "2026-03-05T09:00:00Z",
      "amount_returned_cc": 35.0,
      "roi_pct": -30.0,
      "roi_status": "partial_return",
      "current_idea_value": 72.0
    }
  ]
}
```

- `roi_pct` = `(amount_returned_cc - amount_staked_cc) / amount_staked_cc * 100`
- `roi_status`: `pending` (no return yet), `partial_return`, `full_return`, `loss`
- Stakes where `amount_returned_cc = 0` and idea is still active → `pending`
- Stakes where idea is `closed` or `archived` with no return → `loss`

### R6 — Completed Tasks

Response includes a `tasks` array:

```json
{
  "tasks": [
    {
      "task_id": "uuid",
      "idea_id": "graphql-caching",
      "idea_title": "GraphQL Caching Layer",
      "task_type": "impl",
      "provider": "claude",
      "model": "claude-sonnet-4-6",
      "outcome": "success",
      "cc_earned": 8.0,
      "completed_at": "2026-03-20T11:00:00Z"
    }
  ]
}
```

- Only tasks with `status = completed` and `contributor_id = me` are included
- `outcome` values: `success`, `partial`, `failed`
- `provider` and `model` pulled from task execution record

### R7 — Drill-Down: Idea Contributions Detail

`GET /api/portfolio/me/ideas/{idea_id}/contributions` returns the contributor's specific
contributions to that idea with full lineage chain:

```json
{
  "idea_id": "graphql-caching",
  "idea_title": "GraphQL Caching Layer",
  "my_contributions": [
    {
      "contribution_id": "uuid",
      "contribution_type": "code",
      "cost_amount": 120.0,
      "coherence_score": 0.82,
      "cc_earned": 15.0,
      "lineage_chain": [
        { "step": 1, "event": "contribution_created", "value_assigned": 15.0 },
        { "step": 2, "event": "spec_linked", "value_delta": 5.0 },
        { "step": 3, "event": "test_passed", "value_delta": 3.0 }
      ],
      "created_at": "2026-03-18T10:00:00Z"
    }
  ]
}
```

### R8 — Drill-Down: Contribution Lineage

`GET /api/portfolio/me/contributions/{contribution_id}` returns a single contribution
with its full CC lineage chain (same structure as R7 `lineage_chain`).

- Returns **404** if the contribution belongs to a different contributor (not just missing)
- The lineage chain is sourced from the `value_lineage` service (Spec 048)

### R9 — Web Page: `/portfolio/me`

The Next.js page at `web/app/portfolio/me/page.tsx` renders the portfolio using
`GET /api/portfolio/me`. Sections:

1. **Identity chip bar** — handle, trust badge, GitHub hint (if set), wallet hint (if set)
2. **CC Overview card** — balance, total earned, mini sparkline chart (90-day history),
   toggle button for % of network
3. **Ideas I Contributed To** — sortable table (idea name, status badge, contribution
   type, my CC earned, current value, last activity date); row click opens drill-down
4. **My Stakes** — table (idea, amount staked, returned, ROI % with color coding,
   status badge); row click opens idea detail page
5. **Tasks I Completed** — table (idea, task type, provider, outcome badge, CC earned,
   date); row click reserved for future task detail page

The page is reachable from the top navigation as "Portfolio" (authenticated only).
Unauthenticated visitors see a prompt to register (Spec 168 onboarding).

### R10 — CC Display: Absolute + Network Share Toggle

Both views available via a single toggle in the CC Overview card:
- Default: absolute CC numbers
- Toggle: percentage of total CC in circulation
  - `network_share_pct = (contributor_balance / total_cc_in_circulation) * 100`
  - Sourced from `GET /api/cc/supply` (total CC minted minus burned)

## API Endpoints

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/api/portfolio/me` | Bearer token | Full portfolio summary |
| `GET` | `/api/portfolio/me/ideas/{idea_id}/contributions` | Bearer token | Contributor's contributions to one idea + lineage |
| `GET` | `/api/portfolio/me/contributions/{contribution_id}` | Bearer token | Single contribution + CC lineage chain |
| `GET` | `/api/cc/supply` | None (public) | Total CC in circulation (minted minus burned) |

## Data Model

No new tables required. All data is served by joining existing tables:

| Data Needed | Source Table / Service |
|-------------|------------------------|
| Contributor identity | `contributors` (handle, trust_level, hints) |
| Session token resolution | `contributors.session_token` (Spec 168) |
| CC balance / history | `cc_transactions` (amount, event_type, created_at) |
| Contributions by contributor | `contributions` (contributor_id, idea_id, type, cc_earned) |
| Idea current value / status | `ideas` + grounded metrics service (Spec 116) |
| Stakes by contributor | `stakes` (contributor_id, idea_id, amount, returned) |
| Tasks by contributor | `tasks` (contributor_id, idea_id, provider, model, outcome, cc_earned) |
| Value lineage chain | `value_lineage_events` (idea_id, contribution_id, step, value_delta) |

**Earning history query** (daily buckets, last 90 days):

```sql
SELECT
  DATE(created_at)  AS date,
  SUM(amount)       AS earned
FROM cc_transactions
WHERE contributor_id = :cid
  AND event_type    = 'earn'
  AND created_at   >= NOW() - INTERVAL '90 days'
GROUP BY DATE(created_at)
ORDER BY date ASC;
```

## Verification Scenarios

### Scenario 1 — Full Portfolio Fetch (happy path)

**Setup**: Contributor `alice` registered via Spec 168 with token `tok_alice`. Alice has:
  - 2 contributions (code + spec) to idea `graphql-caching`
  - 1 stake of 50 CC on `graphql-caching` with 30 CC returned
  - 1 completed task (provider: claude / outcome: success / cc_earned: 8.0)

**Action**:
```bash
curl -s https://api.coherencycoin.com/api/portfolio/me \
  -H "Authorization: Bearer tok_alice"
```

**Expected result**: HTTP 200 with JSON body containing all five top-level keys:
- `identity.handle == "alice"` and `identity.trust_level == "tofu"`
- `cc.balance` is a positive number and `cc.earning_history` is a non-empty array
- `contributions` array contains one entry with `idea_id == "graphql-caching"` and `my_contribution_count == 2`
- `stakes` array contains one entry with `amount_staked_cc == 50.0` and `roi_status == "partial_return"`
- `tasks` array contains one entry with `provider == "claude"` and `outcome == "success"`

**Edge**: No `Authorization` header → **401** `{ "detail": "authentication_required" }`

**Edge**: `Authorization: Bearer bad_token` → **401** `{ "detail": "authentication_required" }`

---

### Scenario 2 — Idea Contribution Drill-Down

**Setup**: Alice has contribution `contrib_abc` (code, 15 CC earned) on `graphql-caching`.

**Action**:
```bash
curl -s https://api.coherencycoin.com/api/portfolio/me/ideas/graphql-caching/contributions \
  -H "Authorization: Bearer tok_alice"
```

**Expected result**: HTTP 200 with:
- `idea_id == "graphql-caching"`
- `my_contributions` is a non-empty array
- First entry has `contribution_id == "contrib_abc"`, `contribution_type == "code"`, `cc_earned == 15.0`
- `lineage_chain` is present (may be empty array if no lineage events recorded yet)

**Edge**: Alice requests contributions for an idea she never touched (e.g., `idea_id: "unknown-idea"`):
→ **200** with `"my_contributions": []` — not 404, because the endpoint is scoped to Alice's view

---

### Scenario 3 — Contribution Ownership Guard (security)

**Setup**: Alice has `contrib_abc`. Bob has separate token `tok_bob`.

**Action** (Bob tries to read Alice's contribution):
```bash
curl -s https://api.coherencycoin.com/api/portfolio/me/contributions/contrib_abc \
  -H "Authorization: Bearer tok_bob"
```

**Expected result**: **404** `{ "detail": "not_found" }` — not 403, to avoid leaking
existence of the contribution ID.

**Then** (Alice reads her own):
```bash
curl -s https://api.coherencycoin.com/api/portfolio/me/contributions/contrib_abc \
  -H "Authorization: Bearer tok_alice"
```

**Expected result**: **200** with `contribution_id == "contrib_abc"` and
`lineage_chain` as an array (empty or populated).

---

### Scenario 4 — CC Supply Endpoint

**Action**:
```bash
curl -s https://api.coherencycoin.com/api/cc/supply
```

**Expected result**: HTTP 200 with:
- `total_minted` is a number >= 0
- `total_burned` is a number >= 0
- `in_circulation` equals `total_minted - total_burned` exactly

**Edge**: No CC transactions have ever occurred →
`{ "total_minted": 0, "total_burned": 0, "in_circulation": 0 }` — not 404 or 500.

---

### Scenario 5 — Earning History Updates After New Contribution

**Setup**: Alice's portfolio shows `cc.total_earned = E` and `contributions` list has
N entries.

**Action**:
```bash
# Step 1: create a new contribution for Alice
curl -s -X POST https://api.coherencycoin.com/api/contributions \
  -H "Content-Type: application/json" \
  -d '{"contributor_id":"<alice_uuid>","asset_id":"another-idea","cost_amount":50}'

# Step 2: re-fetch portfolio
curl -s https://api.coherencycoin.com/api/portfolio/me \
  -H "Authorization: Bearer tok_alice"
```

**Expected result after Step 2**:
- `cc.total_earned` > E (strictly greater — CC was awarded)
- Today's date bucket in `earning_history` has a positive `earned` value
- `contributions` list now contains an entry for `another-idea` (if it was a new idea)

**Edge**: `cost_amount: -1` (negative) → **422** from contributions endpoint.
Portfolio unchanged when re-fetched after the failed request.

## Risks and Assumptions

| Risk | Mitigation |
|------|------------|
| `cc_transactions` table may not exist or have different schema | Graceful fallback: return `earning_history: []` and `balance: 0`; log warning at WARN level. |
| Value lineage chain empty for old contributions | Return `lineage_chain: []` — never error on missing lineage. |
| Performance with large contribution history | Default `limit: 50` on all list sections; cursor-based pagination via `?after=<id>` query param. |
| Identity spoofing via forged token | Tokens are opaque 64-char hex, resolved server-side from DB — cannot be guessed. |
| Negative CC balance due to staking bugs | Return actual computed value as-is; surface it for debugging visibility. |
| No data yet (fresh contributor) | All sections return empty arrays; `balance: 0`. Page must render graceful empty states. |

## Known Gaps and Follow-up Tasks

- **Spec 169 (OAuth upgrade)**: When `trust_level` upgrades to `verified`, portfolio identity section should show verified badge and linked GitHub/wallet. This is purely additive — no data model changes needed in this spec.
- **Public portfolio**: A read-only `GET /api/portfolio/{handle}` for sharing one's portfolio is out of scope here.
- **Cross-portfolio comparison**: Leaderboard (Spec 128) is the venue for comparative views.
- **Pagination**: Initial implementation uses `limit: 50` for all list arrays. Full cursor pagination is deferred.
- **Real-time updates**: Page fetches on load only. WebSocket / SSE push is a future enhancement.
- **Mobile responsiveness**: Tables must collapse gracefully at 375px viewport width without horizontal scroll.
