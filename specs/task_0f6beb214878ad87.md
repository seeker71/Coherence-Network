## Summary

The same idea means different things to different people. The **Belief Systems Translation Idea Lens Engine** is the product capability that lets any stored **idea** (and, in later phases, **news** items) be **viewed through alternative worldviews**—not to replace the source text, but to make cross-perspective sense-making explicit and measurable. This task spec consolidates the **goal**, **contract surfaces** (API, web, optional CLI), **data expectations**, and **proof-of-value** story so implementers and reviewers share one executable contract.

**North star**: Every idea is addressable as a stable node; every worldview lens is a **projection** of that node. News ingestion gains **custom point-of-view filters** so editorial dominance does not silently set the only frame. Success is **observable**: engagement across lenses, reach indices, and time-series metrics—not vanity counts alone.

## Open Question (Addressed Here)

**Question**: How can we improve this idea, show whether it is working yet, and make that proof clearer over time?

**Response**:

1. **Improve the idea (product iteration)**  
   - Treat lenses as first-class registry objects with explicit `archetype_axes` (aligned with contributor belief profiles, spec-169).  
   - Ship **pre-computed** translations for builtin lenses so the feature is visible without a “power user discovers GET” moment.  
   - Add **per-idea reach** (`reach_index`) so authors see whether their idea is bridging perspectives or staying in one silo.

2. **Show whether it is working (short-term)**  
   - Expose **aggregate ROI** via `GET /api/lenses/roi` with counters that move when real users request translations.  
   - Require **`spec_ref: "spec-181"`** (or successor) in translation payloads so clients can assert contract version in tests and monitoring.

3. **Clearer proof over time (medium/long-term)**  
   - **`GET /api/lenses/metrics/timeseries`** (v2) with `window=7d|30d|90d` for trend visibility.  
   - **Cross-lens engagement rate** and (when instrumented) **cross-lens collaboration events** in ROI.  
   - Optional dashboard follow-up: operational visibility without spamming the core read path.

## Requirements

### Functional

- [ ] **Lens registry**: List and retrieve worldview lenses; create custom lenses with validated axis weights.
- [ ] **Idea translation**: For any valid `idea_id` and `lens_id`, return an `IdeaTranslation` consistent with spec-181; cache by `(idea_id, lens_id, source_hash)`.
- [ ] **Bulk read**: `GET /api/ideas/{idea_id}/translations` returns all cached translations for an idea (v2: must include all six builtins once precompute is implemented).
- [ ] **Force regenerate**: `POST` with `force_regenerate` updates cache when source idea changes.
- [ ] **Resonance (optional)**: When `contributor_id` is supplied, return `resonance_delta` in `[-1.0, 1.0]` or `null` if profile missing.
- [ ] **News POV (v2)**: Articles carry optional `dominant_lens_id`; `GET /api/news/{article_id}/translations/{lens_id}` provides lens-shaped framing; feed filtering by lens as specified in v2.
- [ ] **Reach scoring (v2)**: `GET /api/ideas/{idea_id}/lens-reach` exposes `reach_index` and which lenses were viewed.

### Non-functional

- [ ] Invalid `lens_id` / missing idea → **404**, not 500.  
- [ ] Duplicate `POST /api/lenses` with same `lens_id` → **409**.  
- [ ] Axis values outside `[0.0, 1.0]` → **422**.  
- [ ] Six builtin lenses seeded: `libertarian`, `engineer`, `institutionalist`, `entrepreneur`, `spiritual`, `systemic`.

## Data Model (Summary)

Aligned with [`specs/181-belief-systems-translation.md`](./181-belief-systems-translation.md):

- **`WorldviewLens`**: `lens_id`, `name`, `description`, `archetype_axes` (floats in `[0,1]`), optional `framing_template`, `is_builtin`, timestamps.  
- **`IdeaTranslation`**: `idea_id`, `lens_id`, `original_name`, `translated_summary`, `emphasis[]`, `risk_framing`, `opportunity_framing`, optional `resonance_delta`, `cached`, `generated_at`, `source_hash`, `spec_ref`.  
- **News (v2)**: article record extended with `dominant_lens_id` (nullable).

Persistence strategy (PostgreSQL vs in-memory) is implementation detail; behavior and HTTP codes above are normative.

## Verification Scenarios

Each scenario is written for **production** verification with `API=https://api.coherencycoin.com` (replace with staging if needed).

### Scenario 1 — Full lens read path (list → pick → translate)

**Setup**: API is up; at least one idea exists in the system (`GET /api/ideas` returns non-empty).  
**Action**:
```bash
API=https://api.coherencycoin.com
curl -sS "$API/api/lenses" | jq '{total: .total, first: .lenses[0].lens_id}'
IDEA_ID=$(curl -sS "$API/api/ideas?limit=1" | jq -r '.ideas[0].id // .[0].id // empty')
curl -sS "$API/api/ideas/$IDEA_ID/translations/engineer" | jq '{spec_ref, has_summary: (.translated_summary != null and (.translated_summary | length) > 0)}'
```
**Expected**: `total >= 6`; `spec_ref` is `"spec-181"` (or documented successor); `has_summary` is `true`.  
**Edge**: `IDEA_ID` empty → skip sub-steps and file bug (data setup); `translations/bad_lens` → HTTP **404** JSON `detail` present.

### Scenario 2 — Create–read–update cycle (custom lens + translation refresh)

**Setup**: Operator API key available if `POST /api/lenses` requires auth.  
**Action**:
```bash
API=https://api.coherencycoin.com
KEY="${COHERENCE_API_KEY:?set export COHERENCE_API_KEY}"
RID="task-verify-$(date +%s)"
curl -sS -o /tmp/lens_create.json -w "%{http_code}" -X POST "$API/api/lenses" \
  -H "Content-Type: application/json" -H "X-API-Key: $KEY" \
  -d "{\"lens_id\":\"$RID\",\"name\":\"Verify Lens\",\"description\":\"PM spec verify\",\"archetype_axes\":{\"pragmatic\":0.7}}"
# Expect 201
curl -sS "$API/api/lenses/$RID" | jq '.lens_id'
IDEA_ID=$(curl -sS "$API/api/ideas?limit=1" | jq -r '.ideas[0].id // empty')
curl -sS "$API/api/ideas/$IDEA_ID/translations/$RID" | jq '.lens_id'
curl -sS -X POST "$API/api/ideas/$IDEA_ID/translations/$RID" \
  -H "Content-Type: application/json" -d '{"force_regenerate":true}' | jq '.cached, .spec_ref'
```
**Expected**: Create returns **201**; GET lens returns same `lens_id`; translation references `$RID`; POST force returns updated body with `spec_ref` present.  
**Edge**: Repeat `POST /api/lenses` with same `lens_id` → **409**; without valid key (if required) → **401/403** per deployment (document actual code).

### Scenario 3 — ROI and “is anything happening?”

**Setup**: None.  
**Action**:
```bash
curl -sS "https://api.coherencycoin.com/api/lenses/roi" | jq '{total_translations_generated, spec_ref}'
```
**Expected**: HTTP 200; numeric counters non-negative; `spec_ref` matches contract.  
**Edge**: If service cold-start returns zeros, still **200** (not 500); document baseline in ops notes.

### Scenario 4 — Error handling (bad axis + missing resources)

**Setup**: API key if needed for POST.  
**Action**:
```bash
API=https://api.coherencycoin.com
curl -sS -o /tmp/bad.json -w "%{http_code}" -X POST "$API/api/lenses" \
  -H "Content-Type: application/json" -H "X-API-Key: ${COHERENCE_API_KEY}" \
  -d '{"lens_id":"bad-axis","name":"Bad","description":"Bad","archetype_axes":{"x":2.0}}'
cat /tmp/bad.json | jq .
curl -sS -o /tmp/miss.json -w "%{http_code}" "$API/api/ideas/nonexistent-idea-id-000/translations/engineer"
```
**Expected**: First request HTTP **422** with validation detail; second HTTP **404**.  
**Edge**: Malformed JSON body → **422** (FastAPI default) — acceptable if documented.

### Scenario 5 — Web: idea page shows lens UI (smoke)

**Setup**: Browser or `curl -sI` for SSR availability.  
**Action**: Open `https://coherencycoin.com/ideas/<known_idea_id>` (use real ID from API). Confirm a **lens selector** or tab region is present (manual or Playwright: selector for lens panel).  
**Expected**: Page **200**; lens UI visible; switching lens updates translated content without full page error.  
**Edge**: Unknown `idea_id` in URL → app **404** page, not blank **500**.

## Risks and Assumptions

- **Risk**: LLM outputs drift or misrepresent the source idea. **Mitigation**: Original always primary; translations labeled; `source_hash` invalidates cache on edit.  
- **Risk**: Cost/latency from generation. **Mitigation**: Cache; async precompute; rate limits (deployment policy).  
- **Assumption**: Spec-169 belief axes remain the bridge between contributors and lenses.  
- **Assumption**: Production and local dev may differ on API keys for `POST /api/lenses`; scenarios document env var.
