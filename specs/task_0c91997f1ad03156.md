# Spec — Pipeline Data Flow: Code Must Reach `origin/main` Before Phase Advancement

**Task ID:** `task_0c91997f1ad03156`
**Status:** Draft — ready for implementation
**Priority:** Critical — without these fixes, code produced by agents stays on a single node and never reaches production. Phase advancement occurs on phantom work.
**Created:** 2026-03-28

---

## Summary

The agent pipeline (spec → impl → test → code-review → deploy → verify) advances phases automatically when tasks complete. Analysis of the current `agent_runner.py` reveals **seven data flow gaps** where the pipeline advances or discards state before code is safely in `origin/main`. The result: code exists only on the node that produced it, the PR contains a stale base, deploy phases SSH into production for code that was never pushed, and verify phases hallucinate results because no provider can actually reach the running server from a sandbox.

This spec defines the required fix for each of the seven gaps, the API and runner contracts that enforce them, and concrete verification scenarios that prove code reached production before the pipeline moved on.

## Open Questions — Answered

### How can we improve this idea?

| Improvement | Why it matters |
|-------------|---------------|
| Surface `push_confirmed` on the task card in the web UI | Operators can see at a glance whether the code is actually in origin or only local |
| Add a `data_flow_health` endpoint (`GET /api/pipeline/data-flow-health`) | Returns per-phase counts of tasks where push_confirmed=false — alerts before work is lost |
| Record push SHA in task context and verify it at deploy time | Closes the window between push and merge; ensures deploy pulls the expected commit |
| Emit a structured event (`pipeline.phase_advanced`) after each successful push | Allows downstream dashboards and the coherence graph to track lineage end-to-end |

### How can we show whether it is working yet, and make that proof clearer over time?

| Signal | How to read it |
|--------|---------------|
| `context.push_confirmed = true` on every completed impl/test task | Proves that task's code is in origin |
| `context.push_sha` matches `git log origin/main -1 --format=%H` on VPS | Proves deploy pulled the right commit |
| `GET /api/pipeline/data-flow-health` returns `stale_pushes: 0` | Proves no task completed without confirming push |
| Verify phase records `curl_exit_code: 0` with actual HTTP status in context | Proves verify ran a real curl, not a fabricated assertion |
| Phase advancement is gated by `push_confirmed` field, visible in task list API | Proves advancement cannot happen without confirmation |

## Seven Fixes

### Fix 1 — `git fetch origin/main` Before Worktree Creation

**Problem:** `_prepare_pr_branch()` creates a branch from local `origin/main`. If the local ref is stale, the branch diverges from the true upstream; commits in main since the last fetch are not included, and merges can conflict or silently drop changes.

**Required behaviour:**
- Before any worktree or branch is created for an `impl`, `test`, or `code-review` task, the runner **must** call `git fetch origin --prune` and assert `returncode == 0`.
- If the fetch fails (network error, auth error), the task must **not start** — it should fail with `"stale_base: git fetch failed"` and the runner must log the error. The task advances to `failed` so the retry mechanism can pick it up; it must not silently use a stale base.
- This is already partially present in `_prepare_pr_branch()` (line 5478) but is logged as a warning and the function returns `True` even when fetch fails. The fix is: return `False` on fetch failure, and have callers treat `False` as a fatal setup error.

**Files:** `api/scripts/agent_runner.py` — `_prepare_pr_branch()`

---

### Fix 2 — Capture `git diff` After Provider Execution, Not Just Stdout Text

**Problem:** `_extract_partial_work()` in `pipeline_advance_service.py` tries to extract `context.diff_content` but this field is only populated when the runner explicitly captures a diff and stores it. Currently `diff_content` is often empty, causing retries to receive provider output text instead of actual file changes. The `_collect_manifestation_blocks()` helper captures unstaged diff at manifest time, but that diff is never stored in the task's `context` for use by the next phase.

**Required behaviour:**
- After provider execution completes (regardless of pass/fail), the runner **must** run `git diff HEAD --unified=3 --no-color` in the worktree and store the result in `task.context.diff_content`.
- The diff must be stored via `PATCH /api/agent/tasks/{id}` before the task is marked `completed` or `failed`.
- If the diff is empty (no changes), `diff_content` must be set to `""` (empty string) — not omitted — so the next phase can distinguish "provider made no changes" from "diff was never captured".
- Maximum stored size: 64 KB (truncate with a trailing marker `"[truncated]"` if longer).

**Files:** `api/scripts/agent_runner.py`, `api/app/services/pipeline_advance_service.py`

---

### Fix 3 — Push to `origin/main` After Merge, Not Just Locally

**Problem:** After a PR is merged (or after a local `git merge origin/main`), the runner does not always push the merged result back to `origin`. The code exists only on the runner node's local disk. When the runner process restarts, or when a different node picks up the deploy phase, the code is gone.

**Required behaviour:**
- After any `git merge` or `gh pr merge` that results in a local main being ahead of remote, the runner **must** immediately push: `git push origin main`.
- The push must succeed (`returncode == 0`) before the task is marked `completed`.
- On push failure the task must be marked `failed` (not `completed`) and `context.push_error` must contain the stderr output.
- The push result (SHA, timestamp) must be stored in `context.push_sha` and `context.push_confirmed_at`.

**Files:** `api/scripts/agent_runner.py` — PR merge flow (`_merge_pr_and_advance`, `_run_pr_flow`)

---

### Fix 4 — Block Phase Advancement Until Push Succeeds

**Problem:** `pipeline_advance_service.py` creates the next-phase task as soon as the current task reaches `completed`. There is no check for whether `context.push_confirmed` is `true`. A task can be marked `completed` with only local changes, and the deploy phase immediately starts — deploying code that does not exist in the remote repository.

**Required behaviour:**
- `pipeline_advance_service.py` must check `task.context.get("push_confirmed", False)` before advancing from `impl` → `test`, `test` → `code-review`, and `code-review` → `deploy`.
- If `push_confirmed` is `False` or missing for an `impl` or `test` task, the advance must be **blocked**: the task must be moved to `needs_decision` with the reason `"push_not_confirmed: code not in origin"` and a structured message prompting a human to investigate.
- `push_confirmed` is set by the runner after a successful `git push` (Fix 3). It must never be set by a provider — providers run in sandboxes that cannot push to GitHub.
- The `push_confirmed` field must be visible in `GET /api/agent/tasks/{id}` response under `context`.

**Files:** `api/app/services/pipeline_advance_service.py`, `api/app/routers/agent.py` (to expose `context` fields)

---

### Fix 5 — Keep Worktree Until Push Is Confirmed

**Problem:** The runner deletes the worktree (or temp repo path) after task completion to reclaim disk space. If the push has not yet been confirmed, this deletes the only copy of the code. Recovery requires the agent to redo all the work.

**Required behaviour:**
- The runner must not delete the worktree until `context.push_confirmed == True` has been recorded via a successful API PATCH.
- The deletion guard must be a named function `_safe_delete_worktree(repo_path, task_id, context)` that checks `context.get("push_confirmed")` before calling `shutil.rmtree`.
- If deletion is blocked because push was not confirmed, the runner must log: `"task=%s worktree retained: push not confirmed path=%s"`.
- Retained worktrees must be listed in a lightweight registry (`api/logs/retained_worktrees.json`) so the runner can retry the push on restart.

**Files:** `api/scripts/agent_runner.py` — all worktree cleanup paths

---

### Fix 6 — Deploy Phase Is a Runner Action (Has SSH), Not a Provider Action

**Problem:** The `deploy` task type is dispatched to providers (Claude, Codex, Cursor). Providers run in sandboxed environments with no network access to `187.77.152.42`. The deploy "succeeds" because the provider outputs shell commands that look correct, but the actual SSH command was never run. Production is not updated.

**Required behaviour:**
- Tasks with `task_type == "deploy"` must **not** be dispatched to any provider.
- The runner must handle `deploy` tasks natively via a dedicated `_run_deploy_task(task)` function that:
  1. Reads `task.context.push_sha` — the commit that must be on the VPS.
  2. SSHs into `root@187.77.152.42` using `~/.ssh/hostinger-openclaw`.
  3. Runs: `cd /docker/coherence-network/repo && git pull origin main && cd /docker/coherence-network && docker compose build --no-cache api && docker compose up -d api`.
  4. Verifies the deploy by running `curl -s https://api.coherencycoin.com/api/health` and checking for HTTP 200.
  5. Stores `context.deploy_confirmed = True`, `context.deployed_sha`, and `context.deploy_timestamp` via PATCH.
- If any step fails, the task is marked `failed` with a structured error in `context.deploy_error`.
- SSH key path is configurable via `DEPLOY_SSH_KEY` env var, defaulting to `~/.ssh/hostinger-openclaw`.
- VPS address is configurable via `DEPLOY_VPS_HOST` env var, defaulting to `root@187.77.152.42`.

**Files:** `api/scripts/agent_runner.py` — new `_run_deploy_task()`, task dispatch routing

### Fix 7 — Verify Phase Is a Runner Curl, Not a Provider Assertion

**Problem:** The `verify` task type is dispatched to providers. Providers cannot reach `coherencycoin.com` from their sandboxes. They output plausible-looking verification results ("The API returned 200 OK with the expected fields") that are fabricated. Production health is never actually checked.

**Required behaviour:**
- Tasks with `task_type == "verify"` must **not** be dispatched to any provider.
- The runner must handle `verify` tasks natively via a dedicated `_run_verify_task(task)` function that:
  1. Reads the spec's verification scenarios from `task.context.verification_scenarios` (list of `{url, method, expected_status, expected_body_contains}`).
  2. For each scenario, runs `curl -s -o /dev/null -w "%{http_code}" <url>` (or equivalent Python `httpx` call).
  3. Checks actual HTTP status against `expected_status` and actual body against `expected_body_contains`.
  4. Records each result as `{url, expected_status, actual_status, passed: bool, response_preview}` in `context.verify_results`.
  5. Sets `context.verify_passed = True` if all scenarios pass, `False` otherwise.
- The task is marked `completed` only if `verify_passed == True`.
- If any scenario fails, the task is marked `failed` with `context.verify_results` containing the specific failures.
- The runner must not accept provider output as proof of verification — only runner-executed HTTP calls count.

**Files:** `api/scripts/agent_runner.py` — new `_run_verify_task()`, task dispatch routing

## Data Model Changes

### Task Context Fields (additions)

All new fields live under `task.context` (JSONB on PostgreSQL):

| Field | Type | Set by | Meaning |
|-------|------|--------|---------|
| `diff_content` | `string` | Runner | Full `git diff HEAD` output after provider execution, ≤ 64 KB |
| `push_confirmed` | `bool` | Runner | `true` when `git push origin main` succeeded and returncode == 0 |
| `push_sha` | `string` | Runner | Git SHA that was pushed |
| `push_confirmed_at` | `ISO 8601` | Runner | Timestamp of successful push |
| `push_error` | `string` | Runner | stderr from failed push, if any |
| `deploy_confirmed` | `bool` | Runner | `true` when SSH deploy + health-check both succeeded |
| `deployed_sha` | `string` | Runner | Git SHA confirmed running on VPS after deploy |
| `deploy_timestamp` | `ISO 8601` | Runner | Timestamp of confirmed deploy |
| `deploy_error` | `string` | Runner | Error from failed deploy step, if any |
| `verify_passed` | `bool` | Runner | `true` when all verification curl scenarios passed |
| `verify_results` | `list` | Runner | Per-scenario `{url, expected_status, actual_status, passed, response_preview}` |
| `verification_scenarios` | `list` | Spec / PM | List of `{url, method, expected_status, expected_body_contains}` from the spec |

### No schema migration required

`context` is already JSONB. New keys are additive and non-breaking.

## API Changes

### Existing endpoint, new field exposure

`GET /api/agent/tasks/{id}` already returns `context`. No changes required — new context fields are automatically returned.

### New endpoint (optional, Phase 2)

`GET /api/pipeline/data-flow-health`

Returns a summary of the current pipeline's data-flow state:

```json
{
  "stale_pushes": 3,
  "tasks_missing_push_confirmed": [
    {"id": "abc123", "type": "impl", "status": "completed", "created_at": "2026-03-28T04:00:00Z"}
  ],
  "tasks_missing_diff": 1,
  "deploy_tasks_sent_to_providers": 0,
  "verify_tasks_sent_to_providers": 0,
  "last_checked": "2026-03-28T04:30:00Z"
}
```

`stale_pushes > 0` is a pipeline alarm condition.

## Verification Scenarios

### Scenario 1 — `push_confirmed` Is Set Before Phase Advances

**Setup:** A completed `impl` task exists where `context.push_confirmed` is absent (simulating the old behaviour). Pipeline advance service is invoked.

**Action:**
```bash
TASK_ID=$(curl -s $API/api/agent/tasks \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{"direction":"add a comment to api/app/main.py","task_type":"impl","idea_id":"test-data-flow","status":"completed","context":{}}' \
  | jq -r .id)

# Trigger advance (simulated via direct service call or via status PATCH)
curl -s $API/api/agent/tasks/$TASK_ID \
  -X PATCH \
  -H "Content-Type: application/json" \
  -d '{"status":"completed"}'
```

**Expected result:**
- The next phase task is NOT created (no `test` task appears in `GET /api/agent/tasks` for this idea).
- The `impl` task transitions to `needs_decision` with `context.push_not_confirmed: true`.
- `GET /api/agent/tasks/$TASK_ID` returns `context.push_confirmed` absent or false.

**Edge case:** If `push_confirmed` is `true` but `push_sha` is missing, treat as not confirmed — push_sha is proof the runner actually ran `git push`.

### Scenario 2 — Deploy Task Runs SSH, Not a Provider

**Setup:** A deploy task exists for an idea. The runner is processing tasks.

**Action:**
```bash
curl -s $API/api/agent/tasks \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{"direction":"deploy api to VPS","task_type":"deploy","idea_id":"test-data-flow","status":"pending","context":{"push_sha":"abc1234","push_confirmed":true}}'
```

**Expected result:**
- The task is NOT dispatched to any provider (no `executor` field set to `claude`, `codex`, `cursor`).
- The runner logs `"Running deploy task natively via SSH"`.
- After runner processes it: `context.deploy_confirmed == true`, `context.deployed_sha` is a valid git SHA, `context.deploy_timestamp` is a recent ISO timestamp.
- `GET https://api.coherencycoin.com/api/health` returns HTTP 200 (the real VPS is up).

**Edge case:** If SSH key file is missing (`~/.ssh/hostinger-openclaw` not found), the task fails with `context.deploy_error` containing `"SSH key not found"` — not silently skipped.

### Scenario 3 — Verify Task Runs Real Curl Scenarios

**Setup:** A verify task exists with `context.verification_scenarios` populated.

**Action:**
```bash
curl -s $API/api/agent/tasks \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{
    "direction": "verify api health endpoint",
    "task_type": "verify",
    "idea_id": "test-data-flow",
    "status": "pending",
    "context": {
      "verification_scenarios": [
        {"url": "https://api.coherencycoin.com/api/health", "method": "GET", "expected_status": 200, "expected_body_contains": "ok"}
      ]
    }
  }'
```

**Expected result:**
- The task is handled by the runner natively — no provider dispatch.
- `context.verify_results` contains `[{"url": "https://api.coherencycoin.com/api/health", "actual_status": 200, "passed": true, "response_preview": "..."}]`.
- `context.verify_passed == true`.
- Task status is `completed`.

**Edge case:** If `verification_scenarios` is empty or absent, task fails with `context.verify_error: "no verification scenarios defined"` — not silently completed.

### Scenario 4 — Stale Base Is Prevented by Prefetch

**Setup:** The runner's local `origin/main` ref is 10 commits behind the real remote (simulated by manually advancing remote).

**Action:** Trigger an impl task that would create a worktree from the stale base.

**Expected result:**
- `_prepare_pr_branch()` runs `git fetch origin --prune`.
- The worktree is created from the freshly fetched `origin/main`, not the stale one.
- `git log --oneline worktree-branch..origin/main` returns zero lines (no commits missed).

**Edge case:** If `git fetch` fails (e.g., DNS failure), `_prepare_pr_branch()` returns `False` and the task fails with `"stale_base: git fetch failed"` — it does not proceed with stale state.

---

### Scenario 5 — Worktree Retained on Push Failure

**Setup:** An impl task completes with local file changes but `git push origin main` fails (e.g., no network access in this test environment).

**Action:** Observe what happens to the worktree directory and the task status.

**Expected result:**
- The worktree directory still exists on disk.
- `api/logs/retained_worktrees.json` contains an entry for this task and path.
- The task is marked `failed` (not `completed`) with `context.push_error` containing the push stderr.
- `context.push_confirmed` is absent or `false`.

**Edge case:** On runner restart, the retained worktrees registry is read and the runner retries the push for each entry before accepting new tasks. If push succeeds on retry, `push_confirmed` is set and the worktree is then deleted.

## Risks and Assumptions

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| SSH key not available on runner node | Medium | Configurable via `DEPLOY_SSH_KEY` env var; task fails fast with clear error rather than silently skipping |
| `push_confirmed` check breaks existing completed tasks that never had the field | High | Pipeline advance service only enforces the check for tasks created after this spec is deployed; backfill sets `push_confirmed=null` (treated as unknown, not blocked) |
| Runner self-update loop conflicts with active deploy task | Low | Deploy tasks are serialized; runner self-update checks `task_type != "deploy"` before triggering redeploy |
| Provider generates `push_confirmed: true` in its output and injects it into context | Medium | Runner validates: only it can set `push_confirmed`; any value present in provider output that conflicts with runner's own result is overwritten by the runner's PATCH |
| Retained worktrees accumulate and fill disk | Low | Registry caps at 50 entries; entries older than 7 days are pruned; disk usage is logged |

## Known Gaps and Follow-up Tasks

1. **Retained-worktree push retry on startup** — the retry loop is specified here but not fully detailed. A follow-up spec should define the retry cadence and maximum retry count.
2. **`GET /api/pipeline/data-flow-health` endpoint** — marked optional (Phase 2). Should be implemented once the core seven fixes land; gives operators a live signal without querying individual tasks.
3. **Verification scenario authorship** — this spec assumes `context.verification_scenarios` is populated by the spec/PM task. A follow-up should enforce that spec tasks always output at least 3 scenarios in structured form (not just prose).
4. **Multi-node push contention** — if two runner nodes push to `origin/main` concurrently, one will get a non-fast-forward rejection. The retry mechanism handles this, but a follow-up spec should define merge strategy (rebase vs merge commit) for concurrent runners.
5. **Web UI signal** — `push_confirmed` and `deploy_confirmed` should be surfaced as coloured badges on the task card in the web UI (`/tasks/[id]`). Out of scope for this spec.

## Implementation Order

These fixes should land together in a single PR because they are mutually dependent:

1. Fix 1 (fetch) + Fix 5 (retain worktree) — foundation, no existing tests break
2. Fix 2 (diff capture) — additive, no breaking changes
3. Fix 3 (push to origin) + Fix 4 (block advance) — the core gate; tests must cover the blocking behaviour
4. Fix 6 (deploy native) + Fix 7 (verify native) — dispatch routing change; requires runner integration test

All seven fixes must be present before any single fix is considered done — partial implementation creates a false sense of safety (e.g., capturing diffs but not blocking advancement).

## Acceptance Criteria

- [ ] `_prepare_pr_branch()` returns `False` and logs an error when `git fetch` fails; callers treat `False` as fatal.
- [ ] After every provider execution, `context.diff_content` is set in the task (empty string if no changes).
- [ ] After `git push origin main`, `context.push_confirmed = True`, `context.push_sha`, and `context.push_confirmed_at` are stored.
- [ ] `pipeline_advance_service.py` blocks `impl` → `test` and `test` → `code-review` advancement when `push_confirmed` is absent or false.
- [ ] Worktrees are not deleted until `context.push_confirmed = True`; retained paths are written to `api/logs/retained_worktrees.json`.
- [ ] `deploy` tasks are handled by `_run_deploy_task()` in the runner, never dispatched to a provider.
- [ ] `verify` tasks are handled by `_run_verify_task()` in the runner using real HTTP calls, never dispatched to a provider.
- [ ] All five verification scenarios pass against production.
