# Specification: Universal Node and Edge Primitives with Typed Relationships

## Summary

Coherence Network treats **everything as a node**: ideas, concepts, services, contributors, specs, implementations, domains, and future entity types. **Edges are first-class, typed relationships** (for example: inspires, depends-on, implements, contradicts, extends, analogous-to, parent-of). Each node participates in an **Ice / Water / Gas** lifecycle so the system can keep persisted atoms small while still supporting rich, derivable views. This specification defines the **data-layer contract** that makes fractal composition (same pattern at every scale) possible: one graph vocabulary, many projections.

## Purpose

The platform currently spans multiple stores and resource-specific APIs; without a universal node–edge layer, cross-domain reasoning, lineage, and coherence scoring stay fragmented. This spec establishes a single canonical **graph primitive** so contributors, automation, and the web UI can traverse relationships consistently. It prevents duplicate parallel models per entity type, reduces integration cost for new domains, and gives measurable hooks (coverage, degree distributions, lifecycle mix) to prove the model is **actually used**, not only designed.

## Requirements

- [ ] **Universal node envelope**: Every persisted entity exposes a shared node record with `id`, `kind` (discriminator: idea, spec, service, …), `lifecycle` (`ice` | `water` | `gas`), `labels` (optional facets), `metadata` (JSON, size-bounded), and timestamps (`created_at`, `updated_at` UTC ISO 8601).
- [ ] **Typed edges only**: Relationships are stored as directed edges with `id`, `from_node_id`, `to_node_id`, `rel_type` from the approved vocabulary (see Data model), optional `weight` (0.0–1.0 where semantically valid), `metadata` (JSON), and timestamps. No untyped “related” edge in the persisted kernel without an explicit follow-up spec (see Open questions).
- [ ] **Lifecycle semantics**: Ice nodes persist minimal canonical facts; Water nodes are fully queryable materialized records; Gas nodes are ephemeral or purely derived (recomputable) and may be excluded from strict durability guarantees. Transitions must be explicit (`PATCH` or domain workflow), not silent.
- [ ] **Discovery API**: Clients can list valid `rel_type` values and node `kind` values with stable string identifiers (kebab-case in JSON, snake_case in Python as per project mapping).
- [ ] **Graph-safe operations**: Create, read, update (where allowed), and delete nodes and edges with validation (unknown `rel_type` → 422; missing node → 404; duplicate edge identity → 409 where uniqueness is defined).
- [ ] **Observability**: Expose counts or a summary endpoint for nodes by `kind`, edges by `rel_type`, and lifecycle mix so “is this working?” is answerable from production.

## Open Questions (Addressed)

### Minimal edge-type set without overwhelming users

**Answer (v1 kernel):** Adopt a **two-tier model**:

1. **Kernel vocabulary (closed, small):** exactly these eight types, sufficient for most product and engineering graphs:
   - `inspires` — motivational or creative lineage (non-blocking).
   - `depends_on` — hard or soft dependency (blocking vs non-blocking can live in `metadata.dependency_strength`).
   - `implements` — spec/code or intent toward realization.
   - `contradicts` — logical or evidence conflict (use sparingly; may require review workflow later).
   - `extends` — version or evolution (new node extends prior).
   - `analogous_to` — cross-domain similarity without identity merge.
   - `parent_of` — structural containment (tree edges).
   - `relates_to` — **controlled escape hatch** for otherwise unclassified linkage (must include `metadata.rel_note` non-empty in API validation when used).

2. **Extension mechanism (not in kernel count):** Domain-specific **qualified** edges may be added later as `rel_type` namespaced values (e.g. `codex.meta/derives_from`) *only* via spec + migration; never ad hoc in clients.

This keeps the **cognitive surface at eight** while avoiding pressure to misuse `implements` for every unknown relation. If `relates_to` abuse appears in metrics, tighten policy (human review, deprecation).

### How we improve the idea, show whether it is working, and clarify proof over time

**Improvement loop:**

- Instrument **usage metrics** (see Verification): edge diversity, `relates_to` ratio, mean degree per `kind`, subgraph query latency.
- Add **quality gates** in CI: schema validation, migration idempotency, pytest contract tests on the universal API.
- **Product proof:** Explorer UI path shows real nodes/edges from production data; empty graphs trigger a documented “seed or connect” state, not silent 200-with-null.

**Proof clarity over time:**

| Phase | Signal |
|-------|--------|
| Bootstrap | API returns non-empty `edge-types` and `node-kinds`; CRUD tests pass. |
| Adoption | Rising count of non-`relates_to` edges; multiple `kind` values present. |
| Fractal | Cross-kind paths (e.g. idea → spec → implementation) queryable in one call pattern. |

## API Contract (Changes)

Base path prefix: **`/api/graph`** (new router; does not remove existing resource-specific routes in the first phase—those may **project** into universal nodes in a follow-up).

### `GET /api/graph/edge-types`

**Response 200**

```json
{
  "edge_types": [
    {
      "id": "depends_on",
      "label": "Depends on",
      "description": "Dependency between nodes.",
      "directed": true
    }
  ]
}
```

### `GET /api/graph/node-kinds`

**Response 200**

```json
{
  "kinds": [
    { "id": "idea", "description": "Captured idea node." },
    { "id": "spec", "description": "Specification document node." }
  ]
}
```

### `POST /api/graph/nodes`

**Request body**

```json
{
  "kind": "idea",
  "lifecycle": "water",
  "labels": ["fractal"],
  "metadata": { "title": "Universal graph" }
}
```

**Response 201** — returns full node including `id`.

### `GET /api/graph/nodes/{node_id}`

**Response 200** — node envelope. **404** if missing.

### `PATCH /api/graph/nodes/{node_id}`

**Response 200** — updated node. **422** for invalid lifecycle transition or unknown `kind`.

### `DELETE /api/graph/nodes/{node_id}`

**Response 204**. **409** if orphan policy forbids delete while edges exist (policy TBD; default: cascade delete edges in same transaction—document in implementation).

### `POST /api/graph/edges`

**Request body**

```json
{
  "from_node_id": "<uuid>",
  "to_node_id": "<uuid>",
  "rel_type": "implements",
  "weight": 0.85,
  "metadata": {}
}
```

**Response 201**. **422** for unknown `rel_type` or invalid nodes. **409** on duplicate edge if uniqueness `(from_node_id, to_node_id, rel_type)` enforced.

### `GET /api/graph/nodes/{node_id}/edges?direction=out|in|both`

**Response 200** — list of edges with embedded or referenced endpoints.

### `GET /api/graph/summary`

**Response 200**

```json
{
  "node_count": 0,
  "edge_count": 0,
  "nodes_by_kind": {},
  "nodes_by_lifecycle": {},
  "edges_by_rel_type": {}
}
```

## Scenario-Level Verification

### Scenario 1: Full create–read–update cycle for a node

- **Setup:** API running with empty graph partition (or fresh test DB); `API_URL` points to base (e.g. `https://api.coherencycoin.com`).
- **Action:**
  1. `curl -sS -X POST "$API_URL/api/graph/nodes" -H "Content-Type: application/json" -d '{"kind":"idea","lifecycle":"ice","metadata":{"title":"Fractal seed"}}'`
  2. Capture `id` from JSON as `NODE_ID`.
  3. `curl -sS "$API_URL/api/graph/nodes/$NODE_ID"`
  4. `curl -sS -X PATCH "$API_URL/api/graph/nodes/$NODE_ID" -H "Content-Type: application/json" -d '{"lifecycle":"water"}'`
- **Expected result:**
  - Step 1: HTTP **201**, body contains `"kind":"idea"`, `"lifecycle":"ice"`, a non-empty `"id"`.
  - Step 3: HTTP **200**, same `id`, `title` preserved in `metadata`.
  - Step 4: HTTP **200**, `"lifecycle":"water"`, `updated_at` ≥ previous.
- **Edge case:** `PATCH` with invalid lifecycle value (e.g. `"lifecycle":"steam"`) returns **422** with validation detail; node unchanged on re-`GET`.

### Scenario 2: Typed edge creation and listing

- **Setup:** Two nodes exist: `FROM_ID`, `TO_ID` (create via Scenario 1 or seed).
- **Action:**
  1. `curl -sS -X POST "$API_URL/api/graph/edges" -H "Content-Type: application/json" -d "{\"from_node_id\":\"$FROM_ID\",\"to_node_id\":\"$TO_ID\",\"rel_type\":\"implements\",\"metadata\":{}}"`
  2. `curl -sS "$API_URL/api/graph/nodes/$FROM_ID/edges?direction=out"`
- **Expected result:**
  - Step 1: HTTP **201**, body includes `rel_type":"implements"`, `from_node_id`/`to_node_id` match.
  - Step 2: HTTP **200**, JSON list contains the new edge with matching `id`.
- **Edge case:** `rel_type":"not_a_real_type"` returns **422** (unknown type). **Edge case:** `from_node_id` references missing node returns **404** or **422** with explicit “node not found” (spec implementation must document which; prefer **422** with structured error for batch safety).

### Scenario 3: Error handling — missing resource and bad input

- **Setup:** No node with id `00000000-0000-0000-0000-000000000001` (or use a random UUID not present).
- **Action:**
  1. `curl -sS -o /dev/null -w "%{http_code}" "$API_URL/api/graph/nodes/00000000-0000-0000-0000-000000000001"`
  2. `curl -sS -X POST "$API_URL/api/graph/nodes" -H "Content-Type: application/json" -d '{"kind":"","lifecycle":"water"}'`
- **Expected result:**
  - Step 1: HTTP **404** (not 500).
  - Step 2: HTTP **422** with validation error for empty `kind`.
- **Edge case:** Malformed JSON body returns **422** Unprocessable Entity (FastAPI default), not **500**.

### Scenario 4: Summary proves adoption signal

- **Setup:** At least two nodes and one edge from prior scenarios; or production with real data.
- **Action:** `curl -sS "$API_URL/api/graph/summary" | jq .`
- **Expected result:** HTTP **200**; JSON includes numeric `node_count`, `edge_count`, and non-null objects `nodes_by_kind`, `edges_by_rel_type` (may be empty objects only if graph empty).
- **Edge case:** Unauthenticated access follows existing API auth policy (if graph is public-read, 200 without token; if protected, **401** without token—must match global API security spec).

### Scenario 5: Discovery endpoints stable for automation

- **Setup:** None.
- **Action:**
  1. `curl -sS "$API_URL/api/graph/edge-types" | jq '.edge_types[].id'`
  2. `curl -sS "$API_URL/api/graph/node-kinds" | jq '.kinds[].id'`
- **Expected result:** HTTP **200** for both; `edge-types` lists at least the eight kernel types (after implementation); ids are stable strings suitable for codegen.
- **Edge case:** Repeated calls return identical sets (idempotent read).

## Concurrency Behavior

- **Reads:** Safe concurrent reads; summary may be eventually consistent if backed by replica.
- **Writes:** Single-edge uniqueness `(from_node_id, to_node_id, rel_type)` enforced at DB level where possible; last-write-wins on concurrent conflicting PATCH unless ETag added in a later spec.

## Out of Scope

- Full graph visualization (force-directed layout, large-scale rendering) beyond a minimal `/graph` table view.
- Automatic ingestion of legacy resources (ideas, specs) into universal nodes without a dedicated migration spec.
- Fine-grained authorization per edge type (assume global API auth until a security spec extends this).
- Cypher/raw Neo4j query exposure to untrusted clients.

## Risks and Assumptions

- **Risk:** Parallel domain-specific APIs duplicate state vs universal nodes → **Mitigation:** define projection tasks; mark this spec as source of truth for *new* relationships.
- **Assumption:** Neo4j and PostgreSQL remain available in production; if not, in-memory fallback is dev-only and must not pass production verification.
- **Risk:** `relates_to` becomes a junk drawer → **Mitigation:** metrics alert when `relates_to` / total edges > threshold.

## Known Gaps and Follow-up Tasks

- **Follow-up:** Migration spec to project existing `/api/ideas` and specs into `kind=idea` / `kind=spec` nodes without double writes long-term.
- **Follow-up:** UI polish for `/graph` (filters, deep links).
- None blocking initial API implementation if tests and verification scenarios pass in CI.

## Failure / Retry Reflection

- **Failure mode:** Clients send conflicting `kind` strings (singular/plural).
- **Blind spot:** Registry endpoints may drift from validation.
- **Next action:** Single `graph_constants.py` or JSON config loaded by both router and tests.

## Decision Gates

- Confirm default **delete behavior** (cascade vs reject) before first production write traffic.
- Approve **public vs authenticated** read access for `/api/graph/summary` for reviewer scenarios.
