# Spec — Concept Translation (Cross-View Synthesis)

**Task ID:** `task_74074f3dc4c3b246`  
**Living Codex lineage:** `cross-view-synthesis` relationship type (concept graph ↔ resonance edges).  
**Non-goal:** This is **not** machine translation between natural languages. It is **translation of conceptual framework** — reframing an idea’s meaning through another epistemic lens while preserving traceability to the graph.

---

## Summary

Coherence Network already stores ideas and concepts as nodes with edges (including resonance and cross-domain links; see `GET /api/ideas/{idea_id}/concept-resonance`). **Concept translation** extends that: for any idea (or concept), the system produces a **lens-specific view** — scientific, economic, spiritual, artistic, philosophical — by walking the concept graph, following resonance and cross-view relationships, and optionally invoking a constrained LLM pass that must cite **graph-backed concept IDs** in its output.

Users choose a **lens** on the idea detail page; integrators call REST, CLI, or MCP. Over time, **quality and proof** improve via explicit **provenance** (which concepts and edges were used), **user feedback** on lens usefulness, and **deterministic regression tests** on seeded scenarios.

---

## Problem

- The same atomic idea reads differently across worldviews; without a first-class “lens,” the system defaults to one implicit framing.
- Contributors cannot **see** whether a translation is graph-grounded or generic text.
- There is no **single API** to request “this idea as spiritual” vs “this idea as economic,” blocking automation and MCP tooling.

---

## Goals

1. **API:** Expose stable, cache-friendly translation endpoints for ideas and concepts with explicit lens parameters.
2. **Web:** On the idea detail page (`/ideas/[idea_id]`), add a **lens selector** with tabs: **Original | Scientific | Economic | Spiritual | Artistic** (philosophical may map to a dedicated lens or a sub-mode; see Decision Gates).
3. **CLI:** `cc idea <id> --view <lens>` prints the same payload shape as the API (or a human summary with `--json` for machine use).
4. **MCP:** Register `coherence_translate_idea` with arguments `{ idea_id, view, locale? }` returning structured translation + provenance.
5. **Proof over time:** Each response includes **metadata** so operators can tell if the feature is “working” (graph coverage, staleness, model vs heuristic).

---

## Requirements

### Functional

- [ ] `GET /api/ideas/{idea_id}/translate` accepts `view` (required): one of `scientific`, `economic`, `spiritual`, `artistic`, `philosophical` (enum; alias `original` returns 200 with `view: original` and no transformation).
- [ ] `GET /api/concepts/{concept_id}/translate` accepts `from` and `to` (required): same enum set; translates **between lenses** for a single concept node (e.g. `from=scientific&to=artistic`).
- [ ] Response body includes **at minimum**: `id`, `view` (or `from`/`to` for concept path), `summary` (string), `key_points` (array of strings), `cited_concepts` (array of `{ concept_id, role, edge_types[] }`), `confidence` (0.0–1.0), `generated_at` (ISO 8601 UTC), `source` (`graph_heuristic` | `llm_augmented` | `stub`).
- [ ] When the graph has **no** connecting path to concepts in the target lens, return **200** with `summary` explaining insufficient coverage, `confidence` low, and `cited_concepts` empty or partial — **not** HTTP 500.
- [ ] Invalid `view` / `from` / `to` → **422** with explicit enum list in `detail`.
- [ ] Unknown idea or concept → **404**.

### Non-functional

- [ ] Idempotent reads: same `idea_id` + `view` + graph version may be cached; include `graph_version` or `etag` in response when available.
- [ ] Rate limiting: align with existing `/api/ideas` patterns; LLM-backed path must respect automation budgets if reused.

### Out of scope (MVP)

- Full multilingual UI copy (i18n) — only conceptual lens, not locale translation.
- Writing translations back as persistent nodes (optional follow-up: `POST` to freeze a translation as a “water” artifact).

## API Changes

### `GET /api/ideas/{idea_id}/translate`

| Query param | Type   | Required | Description |
|-------------|--------|----------|-------------|
| `view`      | string | yes      | `scientific` \| `economic` \| `spiritual` \| `artistic` \| `philosophical` \| `original` |

**Response 200 (example):**
```json
{
  "idea_id": "example-idea",
  "view": "spiritual",
  "summary": "Reframed summary grounded in cited concepts.",
  "key_points": ["Point one", "Point two"],
  "cited_concepts": [
    { "concept_id": "interdependence", "role": "anchor", "edge_types": ["resonance", "cross_view_synthesis"] }
  ],
  "confidence": 0.72,
  "source": "llm_augmented",
  "generated_at": "2026-03-28T12:00:00Z",
  "graph_version": "abc123"
}
```

**Response 404:** idea not found.  
**Response 422:** invalid `view`.

---

### `GET /api/concepts/{concept_id}/translate`

| Query param | Type   | Required | Description |
|-------------|--------|----------|-------------|
| `from`      | string | yes      | Source lens (enum as above, excluding `original` or treating `original` as neutral) |
| `to`        | string | yes      | Target lens |

**Response 200:** same conceptual fields as ideas, with `concept_id`, `from`, `to` instead of `idea_id`, `view`.

**Response 404:** concept not found.  
**Response 422:** invalid `from`/`to` or `from === to` (reject with clear message).

## Open Question — How We Improve, Show “Working,” and Clarify Proof Over Time

| Mechanism | Purpose |
|-----------|---------|
| **Provenance** (`cited_concepts`, `edge_types`) | Proves *why* a translation looks the way it does; empty citations → visible “not grounded” state. |
| **Confidence** + **source** | Distinguishes heuristic stub vs LLM-augmented vs graph-only; trending confidence on seeded fixtures indicates regression. |
| **`graph_version` / ETag** | Invalidates cache when ontology changes; supports before/after comparisons. |
| **Feedback hooks (follow-up)** | Optional `POST /api/ideas/{id}/translate/feedback` with thumbs + lens — not required for MVP but listed in Known Gaps. |
| **Dashboard / ROI (follow-up)** | `GET /api/ideas/translate/roi` — aggregate: requests per lens, avg confidence, % with ≥1 citation. |
| **Verification tests** | Seeded ideas + concepts in CI; golden JSON snapshots for `view=spiritual` on known fixtures. |

## Verification Scenarios

### Scenario 1 — Full create-read cycle for translation (idea)

**Setup:** Idea `translate-seed-idea` exists via `POST /api/ideas` with body `{"id":"translate-seed-idea","title":"Seed","description":"..."}` (or use existing seed). Graph has at least one linked concept.

**Action:**
```bash
curl -s "$API/api/ideas/translate-seed-idea/translate?view=scientific"
```

**Expected:** HTTP 200, JSON includes `"idea_id":"translate-seed-idea"`, `"view":"scientific"`, non-empty `summary`, `confidence` between 0.0 and 1.0, `generated_at` ISO string, `source` one of allowed enums.

**Edge:** `GET` same URL with `view=invalid_lens` → HTTP 422, `detail` mentions valid enum values.

---

### Scenario 2 — Concept lens-to-lens translation

**Setup:** Concept `entropy` exists (`GET /api/concepts/entropy` returns 200).

**Action:**
```bash
curl -s "$API/api/concepts/entropy/translate?from=scientific&to=spiritual"
```

**Expected:** HTTP 200, JSON includes `concept_id`, `from`, `to`, `summary`, `key_points` (array), `confidence`.

**Edge:** `from=economic&to=economic` → HTTP 422 (same lens).

---

### Scenario 3 — Missing resource (404)

**Setup:** No idea with id `missing-idea-xyz`.

**Action:**
```bash
curl -s -o /dev/null -w "%{http_code}" "$API/api/ideas/missing-idea-xyz/translate?view=spiritual"
```

**Expected:** HTTP 404, body contains `detail` with not-found semantics (not 500).

---

### Scenario 4 — Error handling and thin graph

**Setup:** Idea exists but has zero concept links.

**Action:**
```bash
curl -s "$API/api/ideas/isolated-idea/translate?view=artistic"
```

**Expected:** HTTP 200, `confidence` < 0.3 OR `cited_concepts` empty, `summary` acknowledges limited coverage; **not** HTTP 500.

---

### Scenario 5 — CLI parity

**Setup:** API running locally; `cc` configured.

**Action:**
```bash
cc idea translate-seed-idea --view economic --json
```

**Expected:** stdout parses as JSON matching `GET /api/ideas/translate-seed-idea/translate?view=economic` (same keys).

**Edge:** `cc idea missing-idea-xyz --view spiritual` exits non-zero with stderr containing 404 or API error.

## Risks and Assumptions

- **Risk:** LLM hallucination without citations — **Mitigation:** require `cited_concepts` from graph; reject or downgrade LLM output that invents IDs.
- **Assumption:** ontology includes enough cross-lens edges for demos; otherwise MVP shows honest low-confidence.
- **Risk:** Confusion with i18n — **Mitigation:** name the feature “Concept lens” in UI where appropriate.
