# Spec — Consolidate `/automation`, `/usage`, `/remote-ops` into `/nodes` and `/pipeline`

**Task ID**: `task_b1dc9bcb70271052`  
**Status**: proposed  
**Scope**: minimal — information architecture and web routing only; no optional UX polish.

---

## Summary

Three legacy web routes (`/automation`, `/usage`, `/remote-ops`) overlap with two consolidated surfaces: **`/nodes`** (federation, provider readiness, node-centric operations) and **`/pipeline`** (task queue, execution activity, usage-style execution metrics). This change removes standalone pages for the legacy paths and **permanently redirects** them so bookmarks and external links keep working. Primary navigation exposes **`/nodes`** and **`/pipeline`** only; legacy paths must not appear in the primary nav. Backend routes under `/api/automation/...`, `/api/agent/...`, and federation APIs **stay as-is**; consolidation is a **Next.js / front-end routing** change, not an API redesign.

---

## Requirements

1. **Redirects (301, permanent)**  
   - `/automation` → `/nodes`  
   - `/usage` → `/pipeline`  
   - `/remote-ops` → `/pipeline`  

2. **Destination pages**  
   - `/nodes` and `/pipeline` remain the canonical app routes (`web/app/nodes`, `web/app/pipeline`) and continue to load federation and agent data via existing client fetches.

3. **Navigation**  
   - Site header primary nav includes `/nodes` and `/pipeline` and **excludes** `/automation`, `/usage`, and `/remote-ops`.

4. **API stability**  
   - No removal or renaming of REST paths required for this spec (e.g. `GET /api/automation/usage/readiness`, `GET /api/agent/pipeline-status`, `GET /api/agent/tasks` remain available).

5. **Out of scope (explicit)**  
   - No new API versions, no DB migrations, no redesign of dashboard sections beyond what is needed to merge content and links into the two pages.

---

## API changes

**None** for public contract versioning. Existing endpoints continue to serve the same JSON shapes. Implementation may update **internal** web `fetch` URLs or shared helpers to point at the consolidated pages; that is a web-layer concern.

| Area | Note |
|------|------|
| `/api/automation/*` | Unchanged; pages under `/nodes` may still call these endpoints. |
| `/api/agent/*` | Unchanged; `/pipeline` continues to use pipeline/task endpoints. |
| `/api/federation/nodes` | Unchanged; `/nodes` remains the federation UI surface. |

---

## Data model

**N/A** — no PostgreSQL, Neo4j, or persisted graph schema changes. Redirects and nav are routing/UI only.

---

## Verification criteria

- `next.config.ts` defines `redirects()` with `permanent: true` for all three legacy paths to the targets above (matches `api/tests/test_consolidate_overlapping_pages.py`).
- Production or staging: `curl -sI` on each legacy path returns **301** (or **308**) with `Location` pointing at `/nodes` or `/pipeline` as specified.
- `GET /api/automation/usage/readiness` returns **200** with a JSON object (smoke contract in tests).
- `GET /api/agent/pipeline-status` and `GET /api/agent/tasks?limit=1` return **200** (smoke contract in tests).
- Primary nav snippet in `site_header.tsx` contains `/nodes` and `/pipeline` and does not list legacy paths in `PRIMARY_NAV`.

---

## Risks

| Risk | Mitigation |
|------|------------|
| Bookmarks to old paths | Permanent redirects preserve UX; search engines update over time. |
| Drift vs. older docs/specs that mapped `remote-ops` → `/nodes` | **Tests and this spec** align on `remote-ops` → **`/pipeline`**; update any conflicting narrative when implementing. |
| Long pages after merge | Keep minimal scope: move/link content only; defer tabs and heavy layout work to follow-up specs if needed. |
| Missed hard-coded links | Grep web for `/automation`, `/usage`, `/remote-ops` after implementation and fix or rely on redirects until cleaned. |

## Known gaps and follow-up tasks

- Deep UX (tabs, mobile sticky headers, sparklines) is intentionally **not** in this minimal spec; add only if a separate issue requires it.
- Telegram or inventory strings that still mention `/usage` as a **path** should be updated to `/pipeline` where user-visible (follow-up, not blocking redirect behavior).
