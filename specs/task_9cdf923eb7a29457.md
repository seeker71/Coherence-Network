# Spec: Coherence CLI (`cc`) тАФ two-way bridge between OpenClaw and the network

**Task ID:** `task_9cdf923eb7a29457`  
**Related:** `specs/148-coherence-cli-comprehensive.md` (full CLI product), `specs/156-openclaw-bidirectional-messaging.md` (inbox/session protocol)

## Summary

Define the **Coherence CLI** (`cc`, package `coherence-cli`) as the **primary, stable interface** between any autonomous agent (especially OpenClaw-driven sessions) and the Coherence Network. The CLI is not merely a convenience wrapper: it is the **contractual two-way channel** тАФ **outbound** (agent тЖТ network: ideas, stakes, contributions, heartbeats, diagnostics) and **inbound** (network тЖТ agent: inbox messages, federation events, task visibility).

This spec narrows scope to the **bridge surface** required for OpenClaw integration: `cc ideas`, `cc status`, `cc stake`, `cc contribute`, `cc node` (plus the messaging pair `cc inbox` / `cc msg` that complete the loop). Implementation already lives under `cli/`; this document locks **behavior, API mapping, verification, and proof metrics** so reviewers can run production scenarios and fail closed on ambiguity.

### Open question: How do we improve this idea, show whether it is working, and make proof clearer over time?

| Mechanism | What it proves | How proof sharpens over time |
|-----------|----------------|------------------------------|
| **Versioned bridge spec** | The agentтЖФnetwork contract is explicit and testable. | Each revision adds verification scenarios; regressions are caught by the same curl/cc steps. |
| **CLI тЖФ API traceability** | Every `cc` subcommand maps to documented HTTP routes (no тАЬmagicтАЭ). | `cc meta endpoints` (or OpenAPI) cross-checks drift; CI can diff route lists. |
| **Contribution + idea lineage** | Work is attributed and linked to ideas. | `cc contribute --idea <id>` plus `GET /api/ideas/{id}` shows cumulative evidence on the idea card. |
| **Inbox-first sessions** | Bidirectional messaging is real, not fire-and-forget. | Count of unread messages resolved per session; spec 156 metrics. |
| **Health and status** | The network is reachable and semantically correct. | `cc status` aggregates `GET /api/health` + `GET /api/agent/status-report` freshness; alerts when stale. |
| **Observability hooks** | Long-running tasks emit heartbeats. | `POST /api/agent/tasks/{task_id}/activity` events appear in `GET /api/agent/tasks/{task_id}/stream` (or activity queries). |

## Purpose

- Give OpenClaw (and any headless agent) a **single binary interface** that works in sandboxes, CI, and developer laptops without a browser.
- Encode **deterministic** request/response mapping to public API endpoints so integrators never guess URLs or payloads.
- Make **failure modes** explicit (HTTP 4xx/5xx, empty inbox, stale status-report) so agents can branch reliably.

## Requirements

- [ ] **R1 тАФ Bridge commands** тАФ The CLI MUST expose at minimum: `ideas`, `status`, `stake`, `contribute`, `node` (see Command surface). `inbox` and `msg` MUST remain supported for the return path (align with spec 156).
- [ ] **R2 тАФ API fidelity** тАФ Each command MUST map to the documented endpoints below; no hidden hosts except `COHERENCE_API_URL` / config override (`~/.coherence-network/config.json` per spec 148).
- [ ] **R3 тАФ Non-interactive mode** тАФ `contribute`, `stake`, and message sends MUST support flag-based invocation suitable for OpenClaw scripts (no TTY required).
- [ ] **R4 тАФ Identity** тАФ Commands that attribute economic or social actions (`stake`, `contribute`) MUST resolve `contributor_id` or `provider`+`provider_id` consistently with the API (see `StakeRequest` and contributions record).
- [ ] **R5 тАФ Machine-readable output** тАФ Where feasible, support `--json` (or global JSON mode) for parseable output in agent loops (implementation follow-up if not already uniform).
- [ ] **R6 тАФ Error clarity** тАФ On HTTP errors, print status code and response `detail` when present; exit non-zero.

## Command surface (CLI-facing)

| Command | Purpose | Notes |
|---------|---------|--------|
| `cc ideas` | List/browse ideas (portfolio) | Maps to portfolio list; optional filters as supported by CLI. |
| `cc idea <id>` | Read single idea | Full read after create/list. |
| `cc status` | Network + pipeline health | Aggregates health + optional agent status-report freshness. |
| `cc stake <idea_id> ...` | Stake CC on an idea | Non-interactive flags for amount and identity. |
| `cc contribute ...` | Record contribution | Uses `POST /api/contributions/record`. |
| `cc node` / `cc nodes` | Federation node identity and peers | **Spec requirement:** `cc node` MUST be implemented as an alias of `cc nodes` (or inverse) so task prompts and docs match. |
| `cc inbox` | Read messages for this node | Session-start inbound path (spec 156). |
| `cc msg ...` | Send directed or broadcast message | Outbound social/control path. |

## API changes

**No new API endpoints are required** for this spec to be satisfied; the bridge MUST consume existing routes:

| Capability | HTTP method | Path |
|------------|-------------|------|
| Health | GET | `/api/health` |
| Pipeline / agent status | GET | `/api/agent/status-report` |
| List ideas | GET | `/api/ideas` |
| Create idea | POST | `/api/ideas` |
| Get idea | GET | `/api/ideas/{idea_id}` |
| Stake on idea | POST | `/api/ideas/{idea_id}/stake` |
| Record contribution | POST | `/api/contributions/record` |
| List federation nodes | GET | `/api/federation/nodes` |
| Register / heartbeat node | POST | `/api/federation/nodes`, `/api/federation/nodes/{node_id}/heartbeat` |
| Inbox / send | GET/POST | `/api/federation/nodes/{node_id}/messages` |
| Task activity (runner/agent) | POST | `/api/agent/tasks/{task_id}/activity` |

**`POST /api/agent/tasks/{task_id}/activity` request body (Pydantic):**

```json
{
  "node_id": "",
  "node_name": "",
  "provider": "",
  "event_type": "agent_heartbeat",
  "data": { "step": "string", "files_touched": "string", "progress_pct": 50 }
}
```

Note: Integrations MUST use `event_type`, not `event`.

## Verification Scenarios

### Scenario 1 тАФ Health and status (read path + stale-data edge)

**Setup:** `COHERENCE_API_URL=https://api.coherencycoin.com` (or omit for default).  
**Action:**

```bash
curl -sS -o /dev/null -w "%{http_code}" "$API/api/health"
curl -sS "$API/api/health" | head -c 400
curl -sS "$API/api/agent/status-report" | head -c 800
node cli/bin/cc.mjs status
```

**Expected:**

- First curl prints `200`.
- `/api/health` returns JSON including a recognizable healthy indicator (e.g. `"status":"ok"` or equivalent field present in live schema).
- `/api/agent/status-report` returns HTTP 200 and JSON with `generated_at` or documented fallback fields (per router behavior).
- `cc status` exits 0 and prints human-readable summary without stack trace.

**Edge (missing/invalid API):**

```bash
COHERENCE_API_URL=https://127.0.0.1:9 node cli/bin/cc.mjs status; echo exit:$?
```

**Expected:** Non-zero exit; error message indicates connection failure (not silent success).

---

### Scenario 2 тАФ Full create тЖТ read тЖТ update cycle for ideas (CRUD subset)

**Setup:** No pre-existing id collision тАФ use a fresh `id` string (e.g. `bridge-test-<uuid>`).  
**Action:**

```bash
IDEA_ID="bridge-test-$(date +%s)"
curl -sS -X POST "$API/api/ideas" -H "Content-Type: application/json" -d "{\"id\":\"$IDEA_ID\",\"name\":\"Bridge test\",\"description\":\"CLI bridge verification\",\"potential_value\":1.0,\"estimated_cost\":0.5,\"confidence\":0.5}"
curl -sS "$API/api/ideas/$IDEA_ID" | head -c 600
curl -sS -X PUT "$API/api/ideas/$IDEA_ID/tags" -H "Content-Type: application/json" -d "{\"tags\":[\"openclaw-bridge\"]}"
curl -sS "$API/api/ideas/$IDEA_ID" | head -c 600
```

**Expected:**

- POST returns HTTP 201 with JSON containing `"id":"$IDEA_ID"`.
- GET returns HTTP 200 and the same `id` and `name`.
- PUT tags returns HTTP 200 and tags include `openclaw-bridge` (normalized form acceptable per spec 129).
- Final GET reflects updated tags.

**Edge (duplicate create):**

```bash
curl -sS -o /tmp/out.json -w "%{http_code}" -X POST "$API/api/ideas" -H "Content-Type: application/json" -d "{\"id\":\"$IDEA_ID\",\"name\":\"Dup\",\"description\":\"x\",\"potential_value\":1,\"estimated_cost\":1,\"confidence\":0.5}"
```

**Expected:** HTTP **409** or documented conflict behavior (not 500). If API returns 422, document as current behavior and adjust test with `needs-decision` тАФ **do not** paper over 500s.

### Scenario 3 тАФ Stake and contribute (economic attribution)

**Setup:** Valid `IDEA_ID` from Scenario 2; contributor `contributor_id` set to a test id (e.g. `openclaw-tester`).  
**Action:**

```bash
curl -sS -X POST "$API/api/ideas/$IDEA_ID/stake" -H "Content-Type: application/json" \
  -d "{\"contributor_id\":\"openclaw-tester\",\"amount_cc\":0.25,\"rationale\":\"bridge spec verify\"}"
curl -sS -X POST "$API/api/contributions/record" -H "Content-Type: application/json" \
  -d "{\"contributor_id\":\"openclaw-tester\",\"type\":\"code\",\"amount_cc\":5,\"idea_id\":\"$IDEA_ID\",\"metadata\":{\"description\":\"bridge spec contribution\"}}"
node cli/bin/cc.mjs contribute --type code --cc 5 --idea "$IDEA_ID" --desc "non-interactive contribute"
```

**Expected:**

- Stake returns HTTP 200 and JSON including stake bookkeeping fields (non-empty).
- Contributions record returns HTTP 200/201 (as implemented) with acknowledgment.
- `cc contribute` exits 0 and prints success marker (e.g. checkmark or explicit OK).

**Edge (stake on missing idea):**

```bash
curl -sS -o /dev/null -w "%{http_code}" -X POST "$API/api/ideas/does-not-exist-000/stake" \
  -H "Content-Type: application/json" -d "{\"contributor_id\":\"x\",\"amount_cc\":1}"
```

**Expected:** HTTP **404** (not 500).

### Scenario 4 тАФ Federation messaging (inbound + outbound)

**Setup:** `NODE_ID` configured in CLI config for the sender; recipient node exists or broadcast. Use real node ids from `GET /api/federation/nodes`.  
**Action:**

```bash
curl -sS "$API/api/federation/nodes" | head -c 1200
curl -sS -X POST "$API/api/federation/nodes/$NODE_ID/messages" -H "Content-Type: application/json" \
  -d "{\"from_node\":\"$NODE_ID\",\"to_node\":null,\"type\":\"text\",\"text\":\"bridge ping\"}"
curl -sS "$API/api/federation/nodes/$NODE_ID/messages?unread_only=true&limit=20"
node cli/bin/cc.mjs inbox
node cli/bin/cc.mjs msg broadcast "bridge broadcast test"
```

**Expected:**

- List nodes returns HTTP 200 and a JSON array or object with node records.
- POST message returns HTTP 201 with `id` and `timestamp` fields.
- GET messages returns HTTP 200 with `messages` array (may be empty if filtered).
- `cc inbox` and `cc msg` exit 0 when API succeeds.

**Edge (invalid limit):**

```bash
curl -sS -o /dev/null -w "%{http_code}" "$API/api/federation/nodes/$NODE_ID/messages?limit=9999"
```

**Expected:** HTTP **422** (limit bounds).

### Scenario 5 тАФ Task activity hook (OpenClaw / runner heartbeat)

**Setup:** Known `task_id` (e.g. `task_9cdf923eb7a29457`).  
**Action:**

```bash
curl -sS -X POST "$API/api/agent/tasks/task_9cdf923eb7a29457/activity" \
  -H "Content-Type: application/json" \
  -d "{\"event_type\":\"agent_heartbeat\",\"data\":{\"step\":\"verify\",\"files_touched\":\"specs/task_9cdf923eb7a29457.md\",\"progress_pct\":100}}"
curl -sS "$API/api/agent/tasks/task_9cdf923eb7a29457/stream" | head -c 800
```

**Expected:**

- POST returns HTTP 201 and JSON event record.
- GET stream returns HTTP 200 and includes the new event in the task stream array (ordering: newest may be last тАФ assert containment, not position).

**Edge (malformed body):**

```bash
curl -sS -o /dev/null -w "%{http_code}" -X POST "$API/api/agent/tasks/task_9cdf923eb7a29457/activity" \
  -H "Content-Type: application/json" -d "{}"
```

**Expected:** HTTP **422** (missing `event_type`).

## Out of Scope

- New provider-specific executor logic (belongs in `agent_service_executor.py` per project rules).
- WebSocket push replacement for inbox (Phase 3 in spec 156).
- Changing staking economics or treasury rules.

## Risks and Assumptions

- **Risk:** Public API rate limits or auth requirements for write endpoints may block anonymous integration. **Mitigation:** document required API keys or contributor setup; align with `require_api_key` on sensitive routes.
- **Risk:** `status-report` may be stale if monitor not running; agents might misread liveness. **Mitigation:** surface `fallback_reason` and timestamps in `cc status`.
- **Assumption:** Node 18+ is available where OpenClaw runs.

## Known Gaps and Follow-up Tasks

- Uniform `--json` across all bridge commands (audit each command module).
- Automated production smoke job running Verification Scenarios nightly.
- **Follow-up:** Optional contract test that duplicate `POST /api/ideas` returns **409** with `detail: "Idea already exists"` (matches `ideas.py` router).

## Failure/Retry Reflection

- **Failure mode:** Transient TLS or DNS errors during `cc` runs.  
- **Blind spot:** Agents treating stderr as success.  
- **Next action:** Require non-zero exit codes and structured logs for OpenClaw wrappers.
