## Summary

Every contributor operates from a personal worldview: a set of values, interests, and concept affinities that shape how they engage with ideas. The Belief System makes this worldview explicit, structured, and actionable.

A **BeliefProfile** stores which concepts a contributor resonates with, which epistemic axes they value (scientific rigor vs. spiritual intuition vs. pragmatic utility vs. holistic systems thinking, etc.), and which worldview framing they apply to new ideas. These preferences drive personalized news matching, idea recommendations, and contribution suggestions — surfacing content that genuinely resonates rather than flooding with noise.

Adapted directly from the `UserConceptModule` and `BeliefSystemModule` in the Living Codex origin project. This spec covers the full implementation contract: data model, API, web UI, CLI, and — critically — how we prove the system is actually working over time.

---

## Goal

Build a belief profile system that:
1. Stores per-contributor worldview preferences across 7 canonical axes
2. Allows contributors to explicitly set and update their beliefs via API, web, and CLI
3. Computes resonance scores between a contributor's beliefs and specific ideas
4. Tracks engagement lift from belief-driven recommendations to prove the system works
5. Displays proof-of-working metrics inline so contributors can see their own lift trend

---

## Open Question Addressed: How Do We Prove It's Working?

The key open question: *How can we show that belief profiles are actually working, and make that proof clearer over time?*

**The answer is measurable, compounding, and visible to the contributor themselves — not just admins.**

We define "working" as:
- Contributors with complete belief profiles (`belief_completeness >= 0.7`) receive idea recommendations with measurably higher engagement rates than contributors without profiles (`belief_completeness < 0.3`).
- The gap between these cohorts grows over time as profiles are refined through use.
- Individual contributors can see their own lift trend — not just a snapshot — via a sparkline chart.

Three concrete mechanisms ensure proof accumulates automatically:

### Mechanism 1: `belief_recommendation_events` table
Every recommendation shown to a contributor records its `resonance_score`. Every engagement (click, contribute, credit, comment) records `engaged_at` and `engagement_type`. This raw data is the evidentiary foundation — it cannot be gamed because it is written at recommendation time, not engagement time.

### Mechanism 2: `GET /api/contributors/{id}/beliefs/roi?days=30`
Per-contributor lift endpoint. Returns `engagement_rate`, `baseline_engagement_rate` (network average), `lift` (the delta), and `lift_trend` (weekly values for sparkline). Contributors with < 10 recommendations shown receive `lift=null` with an explanation: `"Need at least 10 recommendations to compute lift"`. This prevents premature conclusions from small samples.

### Mechanism 3: `GET /api/system/beliefs/effectiveness` (admin)
System-wide cohort analysis showing completeness vs. engagement lift, updated daily. Lets platform operators see whether the belief system is adding value at scale.

### Mechanism 4: `BeliefROICard` web component
Displayed inline on the contributor's beliefs page. Shows engagement lift, a time-series sparkline, and a status badge:
- **Green**: lift > 0.05 (working — beliefs are surfacing better content)
- **Red**: lift < -0.05 (needs recalibration — beliefs may be too narrow or misconfigured)
- **Gray**: insufficient data (< 10 recommendations)
- **Yellow**: neutral / early signal

This is visible to the contributor, not just admins. It converts an abstract ML signal into a personal feedback loop the contributor can act on.

## Data Model

### BeliefAxis (enum) — CANONICAL 7 AXES

> **CRITICAL**: These are the EXACT 7 enum values. Do NOT add, rename, or remove any axis. Do NOT use `relational`, `systemic`, or any other name.

```python
class BeliefAxis(str, Enum):
    scientific   = "scientific"    # empirical, evidence-driven, skeptical of unfalsifiable claims
    spiritual    = "spiritual"     # meaning-oriented, transcendent, sacred dimension of experience
    pragmatic    = "pragmatic"     # utility-first, solution-oriented, what works in practice
    holistic     = "holistic"      # systems thinking, interconnectedness, emergent properties
    synthetic    = "synthetic"     # integrative, bridge-builder, draws across worldviews
    critical     = "critical"      # power-aware, deconstructive, questions default narratives
    imaginative  = "imaginative"   # speculative, futures-oriented, comfort with uncertainty
```

### ConceptResonance

```python
class ConceptResonance(BaseModel):
    concept_id: str          # references a concept node in the graph
    concept_name: str        # denormalized for display
    score: float             # 0.0 (no resonance) to 1.0 (full resonance) — field is "score", not "weight"
    updated_at: datetime
```

### BeliefProfile

```python
class BeliefProfile(BaseModel):
    contributor_id: str
    worldview_axes: dict[BeliefAxis, float]   # axis -> 0.0–1.0 value
    concept_resonances: list[ConceptResonance]
    tag_affinities: dict[str, float]          # tag -> 0.0–1.0 score
    primary_worldview: BeliefAxis | None      # dominant axis, derived or explicitly set
    updated_at: datetime
    created_at: datetime
```

### BeliefProfileUpdate (PATCH body)

All fields optional. `worldview_axes` is merged (not replaced), so updating one axis does not erase others.

```python
class BeliefProfileUpdate(BaseModel):
    worldview_axes: dict[str, float] | None = None
    concept_resonances: list[ConceptResonance] | None = None
    tag_affinities: dict[str, float] | None = None
    primary_worldview: BeliefAxis | None = None
```

Validation:
- Axis key must be one of the 7 canonical axes → 422 if not
- Axis value must be in [0.0, 1.0] → 422 if not

### ResonanceResult

> Field is `overall_score` — NOT `resonance_score`, NOT `score`.

```python
class ResonanceResult(BaseModel):
    contributor_id: str
    idea_id: str
    overall_score: float          # 0.0–1.0
    concept_overlap: float        # Jaccard similarity of idea concepts vs contributor top resonances
    worldview_alignment: float    # cosine similarity of idea worldview vector vs contributor axes
    tag_match: float              # proportion of idea's tags in contributor tag_affinities (>= 0.4)
    explanation: list[str]        # e.g. "Strong alignment on 'holistic' axis"
    recommended_action: str | None  # "Contribute", "Follow", or "Skip"
```

Scoring formula:
```
overall_score = (0.4 × concept_overlap) + (0.4 × worldview_alignment) + (0.2 × tag_match)
```

### BeliefROI

```python
class BeliefROI(BaseModel):
    contributor_id: str
    period_days: int
    recommendations_shown: int
    recommendations_engaged: int
    engagement_rate: float
    belief_completeness: float     # 0.0–1.0 — axes filled / 7
    baseline_engagement_rate: float | None
    lift: float | None             # null if < 10 recommendations
    lift_trend: list[float] | None # weekly values for sparkline (null if < 4 weeks)
    insufficient_data_reason: str | None
```

### PostgreSQL Migration

```sql
CREATE TABLE belief_profiles (
    contributor_id   TEXT PRIMARY KEY REFERENCES contributors(id) ON DELETE CASCADE,
    worldview_axes   JSONB NOT NULL DEFAULT '{}',
    concept_resonances JSONB NOT NULL DEFAULT '[]',
    tag_affinities   JSONB NOT NULL DEFAULT '{}',
    primary_worldview TEXT,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_belief_profiles_primary_worldview ON belief_profiles(primary_worldview);

CREATE TABLE belief_recommendation_events (
    id              TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
    contributor_id  TEXT NOT NULL REFERENCES contributors(id) ON DELETE CASCADE,
    idea_id         TEXT NOT NULL,
    resonance_score FLOAT NOT NULL,
    shown_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    engaged_at      TIMESTAMPTZ,
    engagement_type TEXT  -- 'click', 'contribute', 'credit', 'comment'
);

CREATE INDEX idx_belief_rec_events_contributor ON belief_recommendation_events(contributor_id);
CREATE INDEX idx_belief_rec_events_shown_at ON belief_recommendation_events(shown_at);
```

## API Contract

### GET /api/contributors/{id}/beliefs

Returns the current belief profile. If no profile exists yet, returns 200 with empty defaults — NOT 404. Returns 404 only if the contributor does not exist.

**Response 200** (profile exists):
```json
{
  "contributor_id": "alice",
  "worldview_axes": {
    "scientific": 0.8, "holistic": 0.6, "pragmatic": 0.5,
    "spiritual": 0.2, "synthetic": 0.4, "critical": 0.3, "imaginative": 0.7
  },
  "concept_resonances": [
    {"concept_id": "c-emergence", "concept_name": "Emergence", "score": 0.9, "updated_at": "2026-03-28T12:00:00Z"}
  ],
  "tag_affinities": {"ai": 0.9, "graph-theory": 0.7},
  "primary_worldview": "scientific",
  "updated_at": "2026-03-28T12:00:00Z",
  "created_at": "2026-03-01T00:00:00Z"
}
```

**Response 200** (no profile yet — empty defaults, not 404):
```json
{
  "contributor_id": "alice",
  "worldview_axes": {},
  "concept_resonances": [],
  "tag_affinities": {},
  "primary_worldview": null,
  "updated_at": "...",
  "created_at": "..."
}
```

**Response 404**: contributor `id` does not exist

---

### PATCH /api/contributors/{id}/beliefs

Partial update. `worldview_axes` merges into existing (does not replace). Omitted fields unchanged.

**Request body**:
```json
{
  "worldview_axes": {"scientific": 0.9},
  "tag_affinities": {"quantum-computing": 0.8},
  "primary_worldview": "scientific"
}
```

**Response 200**: updated BeliefProfile
**Response 422**: axis value out of range → `{"detail": "Axis value must be between 0.0 and 1.0, got 1.5"}`
**Response 422**: unknown axis → `{"detail": "'astrology' is not a valid BeliefAxis. Valid axes: scientific, spiritual, pragmatic, holistic, synthetic, critical, imaginative"}`
**Response 404**: contributor not found
**Response 403**: contributor can only update their own profile (unless admin)

---

### GET /api/contributors/{id}/beliefs/resonance?idea_id={idea_id}

Computes belief-idea alignment score.

**Response 200**: ResonanceResult (field: `overall_score`)
**Response 200** (empty profile): all scores = 0.0 (valid, not an error)
**Response 404**: contributor or idea not found

---

### GET /api/contributors/{id}/beliefs/roi?days=30

Engagement lift attributable to belief-driven recommendations. Default window: 30 days.

**Response 200**: BeliefROI
- `recommendations_shown < 10` → `lift=null`, `insufficient_data_reason="Need at least 10 recommendations to compute lift"`
- `recommendations_shown >= 10` → `lift = engagement_rate - baseline_engagement_rate`

**Response 404**: contributor not found

---

### GET /api/system/beliefs/effectiveness (admin)

System-wide cohort analysis. Returns aggregate stats comparing completeness cohorts (< 0.3, 0.3–0.7, > 0.7) vs engagement rates.

## Verification Scenarios

These scenarios will be run against production by a reviewer. Each must pass.

### Scenario 1: Create and read a belief profile (full create-read-update cycle)

**Setup**: Contributor `alice` exists in the system. No belief profile exists yet.

**Action**:
```bash
# Step 1: GET returns 200 with empty defaults (not 404)
curl -s $API/api/contributors/alice/beliefs | jq '{worldview_axes, concept_resonances, contributor_id}'
# Expected: {"contributor_id": "alice", "worldview_axes": {}, "concept_resonances": []}

# Step 2: PATCH to set beliefs
curl -s -X PATCH $API/api/contributors/alice/beliefs \
  -H "Content-Type: application/json" \
  -d '{"worldview_axes": {"scientific": 0.8, "holistic": 0.6}, "tag_affinities": {"ai": 0.9}}'
# Expected: HTTP 200, response contains scientific=0.8, holistic=0.6, ai=0.9

# Step 3: GET returns the updated profile
curl -s $API/api/contributors/alice/beliefs | jq '.worldview_axes.scientific'
# Expected: 0.8

# Step 4: PATCH only scientific — holistic must NOT be erased (merge, not replace)
curl -s -X PATCH $API/api/contributors/alice/beliefs \
  -H "Content-Type: application/json" \
  -d '{"worldview_axes": {"scientific": 0.9}}'
curl -s $API/api/contributors/alice/beliefs | jq '.worldview_axes.holistic'
# Expected: 0.6 (unchanged — merge semantics confirmed)
```

## Risks and Assumptions

| Risk | Mitigation |
|------|-----------|
| Low adoption — contributors don't fill in profiles | BeliefROICard shows gray badge with clear CTA. Onboarding flow (Spec 162) can prompt. |
| Resonance algorithm too naive for real personalization | Algorithm is transparent and tunable. Weights (0.4/0.4/0.2) are constants that can be adjusted. Start simple, iterate. |
| `belief_recommendation_events` table not populated (ROI always null) | The recommendation pipeline (Spec 163/170) must write events to this table when serving belief-driven suggestions. This is a cross-spec dependency. |
| Axis semantics are subjective | Axes have explicit one-sentence definitions in the enum. WorldviewSelector cards show these definitions. Contributors see what each axis means before setting values. |
| Empty profile returns wrong status code | Guard-rail explicitly documented: GET returns 200 + empty defaults, never 404 for missing profile. |
| Profile replaced instead of merged on PATCH | Guard-rail: `worldview_axes` must be merged (dict.update semantics), not replaced. Test Scenario 1 Step 4 verifies this. |

## Known Gaps and Follow-up Tasks

1. **Implicit belief updates from interactions** — This spec covers explicit PATCH only. Implicit updates from click patterns, dwell time, and contribution history are a follow-up (Spec 163+).
2. **Neo4j graph layer for belief profiles** — PostgreSQL is source of truth for now. Graph relationships between contributors sharing belief axes could power social features later.
3. **Belief-driven news matching** — The belief profile is the *input* to the recommendation pipeline. The pipeline itself (Spec 163/170) consumes this data; wiring it up is a cross-spec dependency.
4. **Admin effectiveness dashboard** — `GET /api/system/beliefs/effectiveness` is specified but the web UI for admins is deferred.
5. **Belief export/import** — Contributors cannot currently export their profile as JSON or import from another platform. Deferred.
