## Summary

Every contributor operates from a personal worldview: a set of values, interests, and concept affinities that shape how they engage with ideas. The Belief System makes this worldview explicit, structured, and actionable.

A **BeliefProfile** stores which concepts a contributor resonates with, which epistemic axes they value (scientific rigor vs. spiritual intuition vs. pragmatic utility vs. holistic systems thinking, etc.), and which worldview framing they apply to new ideas. These preferences drive personalized news matching, idea recommendations, and contribution suggestions.

Adapted from the UserConceptModule and BeliefSystemModule in the Living Codex origin project. This spec covers the full implementation contract: data model, API, web UI, CLI, and — critically — how we prove the system is working over time.

---

## Goal

Build a belief profile system that:
1. Stores per-contributor worldview preferences across 7 canonical axes
2. Allows contributors to explicitly set and update their beliefs via API, web, and CLI
3. Computes resonance scores between a contributor's beliefs and specific ideas
4. Tracks engagement lift from belief-driven recommendations to prove the system works
5. Displays proof-of-working metrics inline so contributors can see their own lift trend

---

## Open Question: How Do We Prove It Is Working?

The key open question: How can we show that belief profiles are actually working, and make that proof clearer over time?

**The answer is measurable, compounding, and visible to the contributor — not just admins.**

We define "working" as:
- Contributors with complete belief profiles (belief_completeness >= 0.7) receive idea recommendations with measurably higher engagement rates than contributors without profiles (belief_completeness < 0.3).
- The gap between these cohorts grows over time as profiles are refined through use.
- Individual contributors can see their own lift trend via a sparkline chart, not just a static snapshot.

### Mechanism 1: belief_recommendation_events table

Every recommendation shown to a contributor records its resonance_score. Every engagement (click, contribute, credit, comment) records engaged_at and engagement_type. Written at recommendation time — cannot be gamed retroactively.

### Mechanism 2: GET /api/contributors/{id}/beliefs/roi

Per-contributor lift endpoint. Returns engagement_rate, baseline_engagement_rate (network average), lift (the delta), and lift_trend (weekly values for sparkline). Contributors with fewer than 10 recommendations shown receive lift=null with explanation: "Need at least 10 recommendations to compute lift".

### Mechanism 3: GET /api/system/beliefs/effectiveness (admin)

System-wide cohort analysis showing completeness vs. engagement lift, updated daily.

### Mechanism 4: BeliefROICard web component

Displayed inline on the contributor's beliefs page. Shows engagement lift, a time-series sparkline, and a status badge:
- **Green**: lift > 0.05 (working)
- **Red**: lift < -0.05 (needs recalibration)
- **Gray**: insufficient data (fewer than 10 recommendations)
- **Yellow**: neutral / early signal

Visible to contributors. Converts abstract ML signal into a personal feedback loop.

## Data Model

### BeliefAxis (enum) — CANONICAL 7 AXES ONLY

**CRITICAL**: These are the EXACT 7 enum values. Do NOT add, rename, or remove any axis. Never use relational, systemic, or any other unlisted name.

```python
class BeliefAxis(str, Enum):
    scientific   = "scientific"    # empirical, evidence-driven
    spiritual    = "spiritual"     # meaning-oriented, transcendent
    pragmatic    = "pragmatic"     # utility-first, solution-oriented
    holistic     = "holistic"      # systems thinking, interconnectedness
    synthetic    = "synthetic"     # integrative, bridge-builder
    critical     = "critical"      # power-aware, deconstructive
    imaginative  = "imaginative"   # speculative, futures-oriented
```

### ConceptResonance

```python
class ConceptResonance(BaseModel):
    concept_id: str      # references a concept node in the graph
    concept_name: str    # denormalized for display
    score: float         # 0.0 to 1.0 -- field name is "score", NOT "weight", NOT "affinity"
    updated_at: datetime
```

### BeliefProfile

```python
class BeliefProfile(BaseModel):
    contributor_id: str
    worldview_axes: dict[BeliefAxis, float]   # axis -> 0.0 to 1.0
    concept_resonances: list[ConceptResonance]
    tag_affinities: dict[str, float]          # tag -> 0.0 to 1.0
    primary_worldview: BeliefAxis | None
    updated_at: datetime
    created_at: datetime
```

### BeliefProfileUpdate (PATCH body)

All fields optional. worldview_axes merges into existing (not replaced entirely).

```python
class BeliefProfileUpdate(BaseModel):
    worldview_axes: dict[str, float] | None = None
    concept_resonances: list[ConceptResonance] | None = None
    tag_affinities: dict[str, float] | None = None
    primary_worldview: BeliefAxis | None = None
```

Validation rules:
- Axis key must be one of the 7 canonical axes; 422 if not
- Axis value must be in [0.0, 1.0]; 422 if not

### ResonanceResult

Top-level score field is **overall_score** — NOT resonance_score, NOT score.

```python
class ResonanceResult(BaseModel):
    contributor_id: str
    idea_id: str
    overall_score: float         # 0.0 to 1.0
    concept_overlap: float       # Jaccard similarity: idea concepts vs contributor top resonances
    worldview_alignment: float   # cosine similarity: idea worldview vector vs contributor axes
    tag_match: float             # proportion of idea tags in contributor tag_affinities (score >= 0.4)
    explanation: list[str]       # human-readable reasons
    recommended_action: str | None  # "Contribute", "Follow", or "Skip"
```

Scoring formula:
```
overall_score = (0.4 * concept_overlap) + (0.4 * worldview_alignment) + (0.2 * tag_match)
```

### BeliefROI

```python
class BeliefROI(BaseModel):
    contributor_id: str
    period_days: int
    recommendations_shown: int
    recommendations_engaged: int
    engagement_rate: float
    belief_completeness: float      # axes filled / 7
    baseline_engagement_rate: float | None
    lift: float | None              # null if fewer than 10 recommendations
    lift_trend: list[float] | None  # weekly values for sparkline; null if fewer than 4 weeks
    insufficient_data_reason: str | None
```

### PostgreSQL Migration

```sql
CREATE TABLE belief_profiles (
    contributor_id   TEXT PRIMARY KEY REFERENCES contributors(id) ON DELETE CASCADE,
    worldview_axes   JSONB NOT NULL DEFAULT '{}',
    concept_resonances JSONB NOT NULL DEFAULT '[]',
    tag_affinities   JSONB NOT NULL DEFAULT '{}',
    primary_worldview TEXT,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_belief_profiles_primary_worldview ON belief_profiles(primary_worldview);

CREATE TABLE belief_recommendation_events (
    id              TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
    contributor_id  TEXT NOT NULL REFERENCES contributors(id) ON DELETE CASCADE,
    idea_id         TEXT NOT NULL,
    resonance_score FLOAT NOT NULL,
    shown_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    engaged_at      TIMESTAMPTZ,
    engagement_type TEXT
);

CREATE INDEX idx_belief_rec_events_contributor ON belief_recommendation_events(contributor_id);
CREATE INDEX idx_belief_rec_events_shown_at ON belief_recommendation_events(shown_at);
```

## API Contract

### GET /api/contributors/{id}/beliefs

Returns the current belief profile. If no profile exists, returns HTTP 200 with empty defaults — NOT 404.
Returns 404 only if the contributor does not exist.

**Response 200** (profile exists):
```json
{
  "contributor_id": "alice",
  "worldview_axes": {
    "scientific": 0.8, "holistic": 0.6, "pragmatic": 0.5,
    "spiritual": 0.2, "synthetic": 0.4, "critical": 0.3, "imaginative": 0.7
  },
  "concept_resonances": [
    {"concept_id": "c-emergence", "concept_name": "Emergence", "score": 0.9, "updated_at": "2026-03-28T12:00:00Z"}
  ],
  "tag_affinities": {"ai": 0.9, "graph-theory": 0.7},
  "primary_worldview": "scientific",
  "updated_at": "2026-03-28T12:00:00Z",
  "created_at": "2026-03-01T00:00:00Z"
}
```

**Response 200** (no profile yet — empty defaults, not 404):
```json
{
  "contributor_id": "alice",
  "worldview_axes": {},
  "concept_resonances": [],
  "tag_affinities": {},
  "primary_worldview": null,
  "updated_at": "...",
  "created_at": "..."
}
```

**Response 404**: contributor id does not exist.

---

### PATCH /api/contributors/{id}/beliefs

Partial update. worldview_axes is merged into existing (not replaced). Omitted fields unchanged.

**Request body**:
```json
{
  "worldview_axes": {"scientific": 0.9},
  "tag_affinities": {"quantum-computing": 0.8},
  "primary_worldview": "scientific"
}
```

**Responses**:
- 200: updated BeliefProfile
- 422: axis value out of range
- 422: unknown axis name
- 404: contributor not found
- 403: contributor can only update their own profile (unless admin)

---

### GET /api/contributors/{id}/beliefs/resonance?idea_id={idea_id}

Computes how well this contributor's belief profile aligns with a specific idea.

**Responses**:
- 200: ResonanceResult (field: overall_score)
- 200 with all scores = 0.0: valid when contributor has empty belief profile (not an error)
- 404: contributor or idea not found

---

### GET /api/contributors/{id}/beliefs/roi?days=30

Engagement lift attributable to belief-driven recommendations. Default window: 30 days.

**Responses**:
- 200: BeliefROI
  - If recommendations_shown < 10: lift=null, insufficient_data_reason set
  - If recommendations_shown >= 10: lift = engagement_rate - baseline_engagement_rate
- 404: contributor not found

---

### GET /api/system/beliefs/effectiveness (admin only)

System-wide cohort analysis. Aggregate stats comparing completeness cohorts vs engagement rates.

## CLI Commands

Display current contributor's belief profile as a compact table.

```
$ cc beliefs
Worldview Axes
scientific    0.8
holistic      0.6
imaginative   0.7
pragmatic     0.5
synthetic     0.4
critical      0.3
spiritual     0.2

Top Concepts: Emergence (0.9), Graph Theory (0.7)
Top Tags: ai, graph-theory, consciousness
```

### `cc beliefs set <axis> <value>`

Update a single axis value (0.0 to 1.0).

```
$ cc beliefs set holistic 0.8
Updated: holistic -> 0.8
```

Error cases:
- Unknown axis: Error listing 7 valid axes
- Out-of-range: Error about 0.0 to 1.0 range

### `cc beliefs match <idea-id>`

Show resonance between current contributor and an idea. Reads **overall_score** from ResonanceResult.

```
$ cc beliefs match idea-abc123
Resonance: 0.74 / 1.00
  Concept overlap:     0.80
  Worldview alignment: 0.70
  Tag match:           0.65
  Recommended action: Contribute
```

## Implementation Guard-Rails

| Mistake | Correct behavior |
|---------|-----------------|
| Using `weight` for concept affinity | Must be `score` (ConceptResonance field) |
| Using `resonance_score` as top-level result | Must be `overall_score` (ResonanceResult) |
| Using axes `relational` or `systemic` | Only the 7 canonical axes |
| Web page at `/profile/[handle]/beliefs` | Must be at `/contributors/[id]/beliefs` |
| 404 when belief profile does not exist | 200 with empty defaults |
| Router stub with no endpoints | All 4 endpoints must be implemented |
| Service file missing | `beliefs_service.py` must have actual logic |
| Migration file missing | Must include valid `upgrade()` and `downgrade()` |
| BeliefROICard missing from web page | Must be imported and rendered |
| CLI reading wrong field | CLI reads `overall_score` from ResonanceResult |

## Verification Scenarios

The reviewer will RUN these scenarios against production. Each must pass.

### Scenario 1: Full create-read-update cycle

**Setup**: Contributor `alice` exists. No belief profile exists yet.

```bash
API=https://api.coherencycoin.com

# Step 1: GET returns 200 with empty defaults (not 404)
curl -s $API/api/contributors/alice/beliefs
# Expected: HTTP 200, worldview_axes={}, concept_resonances=[]

# Step 2: PATCH to set beliefs
curl -s -X PATCH $API/api/contributors/alice/beliefs \
  -H "Content-Type: application/json" \
  -d '{"worldview_axes": {"scientific": 0.8, "holistic": 0.6}}'
# Expected: HTTP 200, scientific=0.8, holistic=0.6

# Step 3: GET returns updated profile
curl -s $API/api/contributors/alice/beliefs | jq .worldview_axes.scientific
# Expected: 0.8

# Step 4: PATCH only scientific -- holistic must NOT be erased (merge semantics)
curl -s -X PATCH $API/api/contributors/alice/beliefs \
  -H "Content-Type: application/json" \
  -d '{"worldview_axes": {"scientific": 0.9}}'
curl -s $API/api/contributors/alice/beliefs | jq .worldview_axes.holistic
# Expected: 0.6 (unchanged -- confirms merge not replace)
```

## Risks and Assumptions

| Risk | Mitigation |
|------|-----------|
| Low adoption | BeliefROICard shows gray badge with CTA. Onboarding (Spec 162) can prompt. |
| Resonance algorithm too naive | Algorithm is transparent and tunable. Weights (0.4/0.4/0.2) are constants. Start simple, iterate. |
| belief_recommendation_events not populated | Recommendation pipeline (Spec 163/170) must write events. Cross-spec dependency. |
| Axis semantics are subjective | One-sentence definitions per axis shown in WorldviewSelector before contributor sets values. |
| Empty profile returns 404 | Guard-rail: GET returns 200 + empty defaults, never 404 for missing profile. |
| Profile replaced instead of merged on PATCH | Guard-rail: worldview_axes uses merge semantics. Verified by Scenario 1 Step 4. |
