# Spec 160: Real-Time Event Streaming — Cross-Service Pub/Sub

**Spec ID**: task_e4946662cb9deb3a
**Status**: Draft
**Author**: product-manager agent
**Date**: 2026-03-27

---

## Summary

Port and extend the `EventStreamingModule` pattern from Living Codex into Coherence Network as a
first-class, cross-service pub/sub bus. Currently, live activity is fragmented: the federation
bridge sends WebSocket messages only for federation events, the runtime telemetry endpoint is
poll-only, and neither the MCP server nor the CLI (`cc`) can subscribe to a live event stream. This
spec defines a unified `EventStreamingService` with a central in-process broker, a filterable
WebSocket endpoint, and a publish hook that any service can call — so the MCP server, CLI, and web
client all share one coherent live stream.

**Why this exists**: Operators need to see the system breathe. Without a live stream, the UI is
always a snapshot. Without cross-service sharing, the CLI and MCP tools cannot react to events
generated by the API or each other. This spec closes that gap.

---

## Open Question Addressed

**Q: How can we improve this idea, show whether it is working yet, and make that proof clearer over time?**

Proof is embedded at three levels:

1. **`GET /api/events/stream` (WebSocket)** — A connected client receives JSON events in real time.
   Proof it works: `cc events tail` shows events flowing live. The count increments visibly within
   seconds of a task completing, an idea being created, or a contribution being logged.

2. **`GET /api/events/recent` (REST)** — Returns the last N events from the in-memory ring buffer.
   Proof it works: `curl $API/api/events/recent?limit=20` returns a list of typed events with
   timestamps. An empty list after a known action is immediate proof of a bug.

3. **`/events` web page** — A live-updating event log on the web client. Each row shows event type,
   entity kind, entity ID, timestamp, and service origin. New rows animate in at the top.
   Proof it works: browser DevTools → Network → WS shows a live connection and frame traffic.

Over time proof compounds: operators can track event rate (`events/min`) in the dashboard, watch
per-service event counts grow, and detect silent failures when the stream goes quiet.

## Goal

Deliver a cross-service publish/subscribe event bus with:
- A central in-process `EventBus` that any service can publish to.
- A WebSocket endpoint (`/api/events/stream`) with filter parameters (event type, entity kind, entity ID).
- A REST endpoint (`/api/events/recent`) that returns the latest N events from a ring buffer.
- MCP tool `events_subscribe` that streams events to MCP clients.
- CLI command `cc events tail [--type TYPE] [--entity ENTITY] [--id ID]` for terminal observation.
- A `/events` web page showing a live-updating event log.

## Requirements

- [ ] **R1** — `EventBus` singleton: in-process pub/sub broker. `publish(event)` fans out to all
  subscribers whose filters match. Async-safe, no external dependencies (no Redis required for
  MVP).
- [ ] **R2** — `WebSocket /api/events/stream` — accepts optional query params `type`, `entity`,
  `entity_id`. Sends JSON-encoded `StreamEvent` objects as they arrive. Sends a heartbeat ping
  every 25 s if no event has been sent. Closes cleanly on disconnect.
- [ ] **R3** — `GET /api/events/recent` — returns last N events (default 100, max 500) from a
  fixed-size ring buffer. Supports the same `type`, `entity`, `entity_id` filter params.
- [ ] **R4** — Publish hooks in at least 3 existing services: `ideas_service`, `contributions`,
  `agent_service`. Events published when: idea created, contribution recorded, task completed.
- [ ] **R5** — MCP tool `events_subscribe` returns the last 20 events from the ring buffer (for
  snapshot-mode callers; true streaming is WebSocket-only for MCP MVP).
- [ ] **R6** — CLI command `cc events tail` connects via WebSocket and renders events as they
  arrive. Supports `--type`, `--entity`, `--id` filters. `CTRL+C` disconnects cleanly.
- [ ] **R7** — Web page `/events` renders a live event log. New events prepend to the top of the
  list with a subtle slide-in animation. Shows: timestamp, event type badge, entity kind, entity
  ID (linkable if known), service origin, payload summary.
- [ ] **R8** — Ring buffer max size configurable via `AGENT_EVENT_RING_SIZE` env var (default 1000).
- [ ] **R9** — No external queue dependencies for MVP. Redis pub/sub is a follow-up.

## API Contract

### `WebSocket /api/events/stream`

**Query Parameters**
- `type` (optional): Filter by event type, e.g. `idea.created`, `task.completed`.
  Supports glob prefix: `idea.*` matches all idea events.
- `entity` (optional): Filter by entity kind, e.g. `idea`, `task`, `contribution`.
- `entity_id` (optional): Filter by specific entity UUID/slug.
- `token` (optional): Must match `COHERENCE_BRIDGE_TOKEN` env var when set.

**Frame: Server → Client**
```json
{
  "v": 1,
  "schema": "coherence.events.stream.v1",
  "event_type": "idea.created",
  "entity": "idea",
  "entity_id": "abc123",
  "service": "ideas_service",
  "timestamp": "2026-03-27T22:05:00.000Z",
  "payload": { "title": "Real-time streaming", "stage": "backlog" }
}
```

**Frame: Server → Client (heartbeat)**
```json
{ "v": 1, "schema": "coherence.events.stream.v1", "event_type": "heartbeat", "timestamp": "..." }
```

**Close codes**
- `1008 Policy Violation` — invalid or missing token when `COHERENCE_BRIDGE_TOKEN` is set.
- `1001 Going Away` — server shutdown.

---

### `GET /api/events/recent`

**Query Parameters**
- `limit` (int, 1–500, default 100): Number of events to return.
- `type` (optional string): Filter by event type (same as WebSocket).
- `entity` (optional string): Filter by entity kind.
- `entity_id` (optional string): Filter by entity ID.

**Response 200**
```json
[
  {
    "v": 1,
    "schema": "coherence.events.stream.v1",
    "event_type": "idea.created",
    "entity": "idea",
    "entity_id": "abc123",
    "service": "ideas_service",
    "timestamp": "2026-03-27T22:05:00.000Z",
    "payload": { "title": "Real-time streaming" }
  }
]
```

### MCP Tool: `events_subscribe`

**Input schema**
```json
{
  "type": "object",
  "properties": {
    "type": { "type": "string", "description": "Event type filter (optional)" },
    "entity": { "type": "string", "description": "Entity kind filter (optional)" },
    "entity_id": { "type": "string", "description": "Entity ID filter (optional)" },
    "limit": { "type": "integer", "default": 20, "maximum": 100 }
  }
}
```

**Output**: JSON array of the last N matching events from the ring buffer.

## Data Model

```yaml
StreamEvent:
  properties:
    v: { type: integer, const: 1 }
    schema: { type: string, const: "coherence.events.stream.v1" }
    event_type: { type: string, description: "dot-namespaced, e.g. idea.created" }
    entity: { type: string, description: "entity kind: idea | task | contribution | agent | etc." }
    entity_id: { type: string, description: "UUID or slug of the affected entity" }
    service: { type: string, description: "originating service name" }
    timestamp: { type: string, format: "date-time" }
    payload: { type: object, description: "event-specific data; shape varies by event_type" }
  required: [v, schema, event_type, timestamp]

EventBusFilter:
  properties:
    type: { type: string, nullable: true, description: "glob prefix match on event_type" }
    entity: { type: string, nullable: true }
    entity_id: { type: string, nullable: true }
```

**Standard event types**

| Event type | Emitted by | Payload keys |
|---|---|---|
| `idea.created` | `ideas_service` | `id`, `title`, `stage` |
| `idea.updated` | `ideas_service` | `id`, `title`, `stage`, `changed_fields` |
| `task.completed` | `agent_service` | `task_id`, `agent_id`, `duration_s` |
| `task.failed` | `agent_service` | `task_id`, `agent_id`, `error` |
| `contribution.recorded` | `contributions` | `contribution_id`, `contributor_id`, `cc_awarded` |
| `agent.started` | `agent_service` | `agent_id`, `task_id` |
| `heartbeat` | `event_bus` | _(none)_ |

## Verification Scenarios

### Scenario 1 — Idea create triggers a live event

**Setup**: API is running locally. WebSocket client connected at
`ws://localhost:8000/api/events/stream?type=idea.created`.

**Action**:
```bash
# Terminal 1 — connect and watch
wscat -c 'ws://localhost:8000/api/events/stream?type=idea.created'

# Terminal 2 — create an idea
curl -s -X POST http://localhost:8000/api/ideas \
  -H 'Content-Type: application/json' \
  -d '{"title":"Streaming test idea","description":"verify live stream"}'
```

**Expected result**: Within 2 seconds, Terminal 1 receives a JSON frame with
`"event_type": "idea.created"`, `"entity": "idea"`, and the new idea's ID in `entity_id`. The
`payload.title` must equal `"Streaming test idea"`.

**Edge case**: If the wrong `type` filter is applied (`type=task.completed`), no frame is received
for the idea POST. The WebSocket stays open and eventually receives a heartbeat frame.

---

### Scenario 2 — REST ring buffer returns recent events

**Setup**: At least one idea has been created since the API started.

**Action**:
```bash
curl -s http://localhost:8000/api/events/recent?limit=5 | python3 -m json.tool
```

**Expected result**: JSON array with 1–5 objects. Each object has keys `v`, `schema`,
`event_type`, `entity`, `entity_id`, `service`, `timestamp`, `payload`. The most recent event is
the last idea creation. Array length ≤ 5.

**Edge case**: `limit=0` returns HTTP 422 (validation error). `limit=600` returns HTTP 422 (exceeds
max 500). `limit=500` with fewer events returns all available events without error.

---

### Scenario 3 — Filter by entity_id narrows the stream

**Setup**: Two ideas exist — IDs `idea-aaa` and `idea-bbb`. WebSocket connected with
`entity_id=idea-aaa`.

**Action**:
```bash
wscat -c 'ws://localhost:8000/api/events/stream?entity_id=idea-aaa'
# Then update idea-bbb (different entity_id)
curl -X PATCH http://localhost:8000/api/ideas/idea-bbb -d '{"title":"other"}'
# Then update idea-aaa
curl -X PATCH http://localhost:8000/api/ideas/idea-aaa -d '{"title":"updated"}'
```

**Expected result**: The WebSocket receives exactly one frame — for `idea-aaa`. The patch to
`idea-bbb` produces no frame on this connection. The received frame has `entity_id: "idea-aaa"`.

**Edge case**: A non-existent `entity_id` filter (`entity_id=does-not-exist`) keeps the connection
open indefinitely and sends only heartbeats. No error frame is sent.

---

### Scenario 4 — MCP tool returns snapshot

**Setup**: At least 3 events have been published to the bus.

**Action** (via MCP client or test harness):
```json
{ "tool": "events_subscribe", "arguments": { "limit": 3 } }
```

**Expected result**: Response is a JSON array of 3 `StreamEvent` objects. Each has `event_type`,
`timestamp`, and `entity`. Objects are ordered newest-first.

**Edge case**: `limit` omitted → defaults to 20. `limit: 0` returns empty array (no error).
`type: "nonexistent.event"` returns empty array (filter applied, no match — not a 404).

---

### Scenario 5 — CLI tail and disconnect

**Setup**: API running, `cc` CLI available.

**Action**:
```bash
cc events tail --type idea.created &
TAIL_PID=$!
# Create an idea to trigger an event
curl -s -X POST http://localhost:8000/api/ideas \
  -H 'Content-Type: application/json' \
  -d '{"title":"CLI test","description":"cc events tail test"}'
sleep 2
kill $TAIL_PID
echo "exit code: $?"
```

**Expected result**: `cc events tail` prints a formatted line to stdout containing `idea.created`
and the idea title or ID within 2 seconds. After `kill`, the process exits with code 0 or 130
(signal); no unhandled exception is printed.

**Edge case**: API is unreachable → `cc events tail` prints `Connection refused: retrying in 5s`
and retries with exponential backoff. After 3 retries it prints `Giving up.` and exits with code 1.

## Concurrency Behavior

- **EventBus.publish**: uses `asyncio.Lock` for subscriber list mutation; event delivery to
  existing subscribers is lock-free (queue operations are thread-safe).
- **Ring buffer**: protected by a single `asyncio.Lock`; oldest event dropped on overflow.
- **WebSocket connections**: each gets its own `asyncio.Queue(maxsize=200)`. Drop-oldest
  backpressure matches the openclaw bridge pattern.
- **Reconnect**: clients must reconnect with exponential backoff. The server never sends a
  reconnect directive; it simply closes the socket on error.
- **Shutdown**: on API shutdown, all open WebSocket connections are closed with code `1001`
  (Going Away).

## Out of Scope

- Redis or any external message broker (follow-up task).
- Persistent event storage beyond the in-memory ring buffer (follow-up: PostgreSQL event log).
- Event replay / seek-to-timestamp (follow-up).
- ACL / per-user event filtering (follow-up).
- Cross-process pub/sub (multi-worker Uvicorn deployments — follow-up requires Redis).
- Server-Sent Events (SSE) variant (WebSocket-first; SSE is a follow-up for clients that
  can't use WebSocket).

## Risks and Assumptions

- **Risk**: `asyncio.Queue` backpressure only protects individual subscribers. A slow subscriber
  cannot block the bus, but will miss events once its queue overflows. **Mitigation**: log a
  warning on each dropped event; expose a `dropped_events` counter on the health endpoint.
- **Risk**: In-process ring buffer is lost on restart. **Assumption**: acceptable for MVP; the REST
  `/api/events/recent` is a convenience, not an audit log.
- **Assumption**: The API runs as a single Uvicorn worker in production (Docker Compose). If
  multiple workers are added, the in-process bus becomes per-process. Redis pub/sub migration
  would be required.
- **Risk**: Naming collision with existing `runtime_service` events. **Mitigation**: `StreamEvent`
  is a new model; `RuntimeEvent` is unchanged. The two buses are independent for now. A future
  spec can unify them.
- **Risk**: WebSocket connections are long-lived. If Cloudflare/Traefik has a 30 s idle timeout,
  heartbeats at 25 s will keep the connection alive. Verify Traefik timeout config before deploy.

## Known Gaps and Follow-up Tasks

- **Follow-up**: Redis-backed cross-process pub/sub for multi-worker deployments.
- **Follow-up**: Persist events to PostgreSQL for full audit history.
- **Follow-up**: Unify `RuntimeEvent` and `StreamEvent` into a single event model.
- **Follow-up**: SSE fallback for HTTP/1.1-only clients.
- **Follow-up**: Per-client event-rate throttle (DoS protection for public deployment).
- **Follow-up**: `cc events replay --since <timestamp>` for historical browsing.

## Failure / Retry Reflection

| Failure mode | Blind spot | Next action |
|---|---|---|
| WebSocket stays silent after idea POST | Publish hook not wired in `ideas_service` | Add `event_bus.publish(...)` call; run Scenario 1 |
| `/api/events/recent` always returns `[]` | Ring buffer not shared with router | Ensure router imports module-level `_ring_buffer` from `event_bus` |
| CLI crashes on disconnect | `asyncio.CancelledError` not caught | Wrap receive loop in `try/except CancelledError` |
| Heartbeat frames cause frontend crash | Frontend JSON parser sees `event_type: heartbeat` and tries to render entity | Guard rendering on `event_type !== 'heartbeat'` |
