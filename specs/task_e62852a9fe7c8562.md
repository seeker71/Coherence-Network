# Spec: Geolocation awareness — nearby contributors, local ideas, regional news

## Summary

Coherence Network should treat **geography** as a discoverability and collaboration signal: contributors may share **coarse** location on profiles; the platform resolves human-readable places to coordinates using a **GeocodingService** aligned with the **Living Codex** pattern (**OpenCage** primary, **Nominatim** secondary, explicit static fallback); clients filter **ideas**, **news**, and **agent tasks** by proximity; and the **web** surfaces nearby collaborators and regional context. Baseline APIs already implement city-level storage, Haversine **nearby** search for contributors and ideas, **forward geocoding**, **task proximity**, and **keyword-based** local news resonance. This specification defines the **end-state product contract**, **phased delivery**, **observability** so operators can prove the feature works and improves over time, and **verification scenarios** runnable against production.

**Open question — how we improve this idea, show whether it is working, and make proof clearer over time**

1. **Metrics**: Expose `source` (and optionally `latency_ms`) on `/api/geocode/forward` responses (partially present); add **`GET /api/geolocation/health`** aggregating geocode success rate by provider, cache hit ratio, and last-N error samples (no PII). Wire optional **runtime events** or existing logging sinks so dashboards can chart OpenCage vs Nominatim vs fallback over weeks.
2. **Product surfacing**: Web **“nearby”** strip or page shows **count of public contributors** and **ideas** within radius; stale data shows “location not set” rather than silent empty states.
3. **Cohort proof**: Compare **idea engagement** or **task completion** for users who enabled location vs control (privacy-preserving aggregates only); document success criteria in `Verification Scenarios`.
4. **Versioning**: Response headers or JSON field `geo_features_version` (semver) so clients detect when proximity filters or ranking change.

## Purpose

- **Why**: Reduce coordination friction for distributed OSS contributors by making **who is near you** and **what matters locally** visible without exposing precise addresses.
- **Who benefits**: Contributors seeking collaborators, reviewers validating regional pipelines, operators proving SLOs for geocoding providers.
- **What failure/cost it prevents**: Blind reliance on a single geocoder, OSM policy violations, and “empty UI” with no diagnostic when geodata is missing.

## Requirements

- [ ] **R1 — GeocodingService (Living Codex borrow)**: Maintain **OpenCage → Nominatim → static fallback** chain in `api/app/services/geocoding_service.py`; normalized query handling; coordinates rounded to **two decimal places** (~1 km); compliant **Nominatim User-Agent**; no bulk unthrottled calls (rate limit / cache as implemented or extended per follow-up).
- [ ] **R2 — Contributor profiles**: `PATCH` / `GET` / `DELETE` **`/api/contributors/{id}/location`** store and return city-level location; list responses that represent contributors may embed **`geo_location`** (or linked resource) where the portfolio spec allows.
- [ ] **R3 — Nearby collaborators and local ideas**: **`GET /api/nearby`** returns contributors and ideas within **`radius_km`** of **`lat`/`lon`**; privacy rules (`public` / `contributors_only`) respected; distances approximate.
- [ ] **R4 — Regional news**: **`GET /api/news/resonance/local?location=...`** returns news items scored by textual resonance to the location string.
- [ ] **R5 — Tasks by proximity**: **`GET /api/geo/tasks/nearby`** returns agent tasks whose `context` includes geo coordinates or resolvable contributor location within radius.
- [ ] **R6 — Forward geocode**: **`GET /api/geocode/forward?q=...`** resolves free text to lat/lon + `source`; invalid/short `q` → **422**.
- [ ] **R7 — Ideas proximity (target)**: **`GET /api/ideas`** gains optional **`near_lat`**, **`near_lon`**, **`radius_km`** and/or **`location_query`** (resolved via GeocodingService) to filter or rank ideas by contributor geography — **may be a follow-up phase** if not yet merged; this spec lists exact contract below.
- [ ] **R8 — Web (target)**: Dedicated page **`/explore/nearby`** (or equivalent route under `web/app`) lets a user enter a place or use browser geolocation (with consent), calls **`/api/geocode/forward`** then **`/api/nearby`**, and lists nearby contributors + ideas; links to local news — **phased**; minimum viable may be API-only with documented `curl`.
- [ ] **R9 — Proof over time**: Documented **`GET /api/geolocation/health`** (or extension of **`/api/ideas/health`** with `geolocation` subsection) listing provider mix and error budget consumption.

## API Contract (exact endpoints)

### Existing (must remain stable; part of this feature story)

| Method | Path | Role |
|--------|------|------|
| PATCH | `/api/contributors/{contributor_id}/location` | Set coarse contributor location |
| GET | `/api/contributors/{contributor_id}/location` | Read location |
| DELETE | `/api/contributors/{contributor_id}/location` | Remove location |
| GET | `/api/nearby` | Nearby contributors + ideas |
| GET | `/api/news/resonance/local` | Regional news resonance |
| GET | `/api/geocode/forward` | Forward geocode `q` |
| GET | `/api/geo/tasks/nearby` | Agent tasks by proximity |

Query parameters (representative):

- **`/api/nearby`**: `lat` (required), `lon` (required), `radius_km` (default 100), `limit` (default 50).
- **`/api/news/resonance/local`**: `location` (required, min length 2), `limit`.
- **`/api/geocode/forward`**: `q` (required, min length 2).
- **`/api/geo/tasks/nearby`**: `lat`, `lon`, `radius_km`, `limit`.

### Target / follow-up (contract for full product spec)

| Method | Path | Role |
|--------|------|------|
| GET | `/api/ideas` | Optional `near_lat`, `near_lon`, `radius_km`, `location_query` for geographic filter/rank |
| GET | `/api/geolocation/health` | Aggregated geocoder health and provider mix (JSON) |

**Response shapes** (existing patterns): use Pydantic models `NearbyResult`, `LocalNewsResonanceResponse`, `GeocodeForwardResponse`, `NearbyAgentTasksResponse` in `api/app/models/`.

## Data Model

- **Contributor location**: City-level **latitude** / **longitude** (rounded 2 dp), **city** / **region** / **country** strings, **visibility** enum (`public` | `contributors_only` | `private` as implemented).
- **Geocode result**: `{ found, latitude?, longitude?, label?, source, ... }` with `source` in `opencage` | `nominatim` | `fallback`.
- **Idea ↔ geography**: Ideas do not require a pin; proximity is derived from **contributor/author** graph location unless a future **per-idea** geo field is added (out of scope unless specified).
- **Agent task**: `context` may include `geo_lat`, `geo_lon`, and/or `contributor_id` for distance computation.

## Web surfaces

| Route | Purpose |
|-------|---------|
| **`/explore/nearby`** (target) | User enters place or shares coarse browser location; lists nearby contributors and ideas; optional link to news |
| **Contributor profile UI** (target) | Show optional city/region when visibility allows; never show exact coordinates in UI |

CLI: **`cc`** is not required for this feature; reviewers use **`curl`** as in Verification Scenarios. If a future **`cc geo`** or **`cc nearby`** is added, it should wrap the same HTTP API.

## Verification Scenarios (contract — run against production or staging)

### Scenario 1 — Contributor location full cycle (create → read → update → delete)

- **Setup**: Known `contributor_id` for a test contributor (create via existing contributor API if required). No prior location.
- **Action**: `curl -sS -X PATCH "$API/api/contributors/$CID/location" -H "Content-Type: application/json" -d '{"latitude":52.52,"longitude":13.40,"city":"Berlin","visibility":"public"}'`
- **Expected**: HTTP **200**, JSON includes `city` **Berlin**, lat/lon rounded to **two** decimal places (e.g. `52.52`, `13.40`).
- **Then**: `curl -sS "$API/api/contributors/$CID/location"` → **200**, same fields.
- **Then**: `curl -sS -X DELETE "$API/api/contributors/$CID/location"` → **204**.
- **Then**: `curl -sS "$API/api/contributors/$CID/location"` → **404** (no location).
- **Edge**: `PATCH` with latitude **200** → **422** (out of range). Unknown `contributor_id` → **404** on PATCH.

### Scenario 2 — Nearby contributors and ideas (read + distance semantics)

- **Setup**: At least one contributor with **public** location near **52.52, 13.40**; optional seed ideas linked to geography per data model.
- **Action**: `curl -sS "$API/api/nearby?lat=52.52&lon=13.405&radius_km=50&limit=20"`
- **Expected**: HTTP **200**, JSON `contributors` / `ideas` arrays (may be empty if no data), each contributor entry includes **approximate** distance fields as documented (not exact addresses).
- **Edge**: `curl -sS "$API/api/nearby?lat=999&lon=13"` → **422** (invalid lat). Empty result sets still **200**, not **500**.

### Scenario 3 — Forward geocode + regional news (geocoder + news stack)

- **Setup**: API reachable; `q` valid city name.
- **Action**: `curl -sS "$API/api/geocode/forward?q=Berlin"`
- **Expected**: HTTP **200**, JSON `found: true`, numeric `latitude` / `longitude`, `source` one of **`opencage`** | **`nominatim`** | **`fallback`**.
- **Then**: `curl -sS "$API/api/news/resonance/local?location=Berlin&limit=5"`
- **Expected**: HTTP **200**, JSON includes list of news items with **resonance** scores in **[0, 1]**.
- **Edge**: `curl -sS "$API/api/geocode/forward?q=a"` → **422** (query too short).

### Scenario 4 — Agent tasks by proximity (full read path)

- **Setup**: Tasks exist with `context` containing `geo_lat`/`geo_lon` or contributor with location, or empty set acceptable for structural test.
- **Action**: `curl -sS "$API/api/geo/tasks/nearby?lat=52.52&lon=13.405&radius_km=500"`
- **Expected**: HTTP **200**, JSON `tasks` array, `total` equals `len(tasks)`, each task includes **`distance_km`** ≥ 0 when present.
- **Edge**: `curl -sS "$API/api/geo/tasks/nearby?lat=200&lon=0"` → **422**.

### Scenario 5 — Error handling and missing resources

- **Action**: `curl -sS -o /dev/null -w "%{http_code}" "$API/api/contributors/nonexistent-id-000/location"`
- **Expected**: **404** on GET (no location or unknown contributor per implementation).
- **Action**: `curl -sS "$API/api/geocode/forward"` (omit `q`)
- **Expected**: **422** validation error.
- **Edge**: Duplicate **PATCH** with same payload is idempotent (**200**), not **500**.

## Concurrency Behavior

- **Reads**: Safe concurrent reads; geocode and nearby are stateless aside from caches.
- **Writes**: Contributor location updates are **last-write-wins** per contributor id unless ETags are added later.

## Out of Scope (unless a new spec extends)

- Sub-meter GPS accuracy, turn-by-turn, or storing street addresses.
- Self-hosted Nominatim cluster (optional ops follow-up).

## Risks and Assumptions

- **Assumption**: Low-volume use of public Nominatim remains acceptable with User-Agent and throttling; high scale may require OpenCage-only or self-hosted geocoder.
- **Risk**: Provider outage — **Mitigation**: fallback chain + health metrics + cached geocodes.
- **Risk**: Privacy — **Mitigation**: 2-decimal rounding and visibility enums remain mandatory.

## Known Gaps and Follow-up Tasks

- **`GET /api/ideas`** geographic filters — track in implementation PR; not all params may exist on main yet.
- **Web `/explore/nearby`** — UI phase; API can ship first.
- **`GET /api/geolocation/health`** — add when metrics endpoint is implemented.

## Failure/Retry Reflection

- **Failure mode**: OpenCage rate limit during spike.
- **Blind spot**: Uncached repeat queries for the same city hammering Nominatim.
- **Next action**: Increase cache TTL for normalized queries; backoff; surface **503** with `Retry-After` when appropriate.
