## Summary

The same idea means different things to different people. "Decentralized network" reads as
**freedom** to a libertarian, **efficiency** to an engineer, **threat** to an institution,
and **opportunity** to an entrepreneur. The Idea Lens Engine (spec-181) established the
foundational translation mechanism. This spec addresses three specific gaps that must be
solved before the feature can prove its value:

1. **News ingestion integration** — articles ingest with dominant-lens tagging so every
   contributor receives news filtered through their worldview rather than a single editorial
   POV.
2. **Every idea is viewable through any worldview** — the web surface exposes all lenses
   with no gatekeeping; translations are pre-computed at idea creation time for the 6
   builtin lenses.
3. **Proof it is working** — quantified signals exposed via `/api/lenses/roi`, time-series
   metrics via `/api/lenses/metrics/timeseries`, and explicit per-idea reach scoring.

This spec is the implementation contract for those three surfaces. The reviewer will run the
Verification Scenarios section against production to confirm delivery.

## Problem Statement

### Problem 1 — Translations are on-demand only

Spec-181 generates translations on first request (GET). This means a contributor must know
to ask. Most never will. Result: the lens feature exists but is invisible.

**Fix**: Pre-compute all 6 builtin lens translations when a new idea is created (async
background task). Expose `GET /api/ideas/{id}/translations` as a bulk endpoint.

### Problem 2 — News arrives in one framing

The news ingestion pipeline (`api/app/services/news_service.py`, `api/app/routers/news.py`)
stores articles with no lens classification. Contributors receive editorial POV with no
alternative.

**Fix**: Add `dominant_lens_id` tag to news articles at ingest time. Expose
`GET /api/news/{article_id}/translations/{lens_id}` to translate news through any lens.
Opt-in "View through my lens" toggle on the news feed.

### Problem 3 — No time-series visibility

`GET /api/lenses/roi` returns aggregate counters. A counter of 412 tells nothing about
whether the feature is growing, flat, or declining.

**Fix**: Add `GET /api/lenses/metrics/timeseries` with configurable time windows and daily
breakdowns. This is the primary proof-over-time surface.

### Problem 4 — Idea reach is not surfaced

An idea viewed through 6 of 6 lenses has maximum perspective reach. An idea viewed through
only 1 has zero bridging value. This is never shown anywhere.

**Fix**: Add `GET /api/ideas/{id}/lens-reach` returning `reach_index` (0.0–1.0) and the
list of lenses through which the idea has been viewed.

## Requirements

### Core — Lens Engine (spec-181 foundation, must remain working)

- [ ] `GET /api/lenses` returns all registered lenses, `total >= 6` after startup. (HTTP 200)
- [ ] `GET /api/lenses/{lens_id}` returns lens definition or 404.
- [ ] `POST /api/lenses` creates custom lens (201), duplicate returns 409, invalid axis weight returns 422.
- [ ] `GET /api/ideas/{idea_id}/translations/{lens_id}` returns `IdeaTranslation` with `spec_ref: "spec-181"`.
- [ ] `POST /api/ideas/{idea_id}/translations/{lens_id}` with `{"force_regenerate": true}` re-generates and updates cache.
- [ ] `GET /api/lenses/roi` returns non-negative counters and `spec_ref: "spec-181"`.

### New — Bulk translation and pre-computation

- [ ] `GET /api/ideas/{idea_id}/translations` returns all cached translations for the idea (all 6 builtin lenses pre-populated at idea creation).
- [ ] `POST /api/ideas/{idea_id}/translations/precompute` triggers async pre-computation of all 6 builtin lens translations. Returns `{"status": "queued", "lens_count": 6}` (HTTP 202).
- [ ] Pre-computation runs automatically on idea creation (async, non-blocking). Ideas created before this feature can be backfilled via the precompute endpoint.
- [ ] `GET /api/ideas/{idea_id}/translations` returns `{"idea_id": ..., "translations": [...], "total": N, "spec_ref": "spec-181"}`.

### New — Idea lens reach scoring

- [ ] `GET /api/ideas/{idea_id}/lens-reach` returns:
  ```json
  {
    "idea_id": "...",
    "reach_index": 0.67,
    "lenses_viewed": ["libertarian", "engineer", "systemic"],
    "lenses_available": ["libertarian", "engineer", "institutionalist", "entrepreneur", "spiritual", "systemic"],
    "total_views_across_lenses": 34,
    "spec_ref": "spec-181"
  }
  ```
  `reach_index = lenses_viewed.count / lenses_available.count`, range [0.0, 1.0].
- [ ] Missing idea returns 404.

### New — News ingestion POV integration (opt-in MVP)

- [ ] `GET /api/news/{article_id}/translations/{lens_id}` returns an `IdeaTranslation`-shaped response for a news article viewed through the specified lens (uses the same LLM translation layer as ideas).
- [ ] News articles stored by the ingest pipeline include `dominant_lens_id` field (nullable; null means "unclassified").
- [ ] `GET /api/news?lens={lens_id}` filters news articles by their `dominant_lens_id`.
- [ ] Missing article or missing lens returns 404.

### New — Time-series metrics

- [ ] `GET /api/lenses/metrics/timeseries` accepts `?window=7d|30d|90d` query param (default `7d`).
- [ ] Returns daily breakdown:
  ```json
  {
    "window": "7d",
    "from": "2026-03-21",
    "to": "2026-03-28",
    "days": [
      { "date": "2026-03-21", "translations_generated": 12, "unique_ideas": 4, "unique_contributors": 3 },
      ...
    ],
    "totals": { "translations_generated": 84, "unique_ideas": 21, "unique_contributors": 15 },
    "spec_ref": "spec-181"
  }
  ```
- [ ] Invalid `window` value returns 422.
- [ ] When no data exists for a day, that day's counts are 0 (never omitted).

### Web interface

- [ ] `/ideas/{id}` page: "View through another lens" section with IdeaLensPanel (already exists per spec-181); now defaults to displaying the pre-computed translation immediately (no spinner on first load if pre-computed).
- [ ] `/lenses` page: lists all lenses with their archetype_axes, reach stats (most-translated lens highlighted).
- [ ] Lens selector shows `reach_index` badge on each lens tab: "34 views" or "New".

### Validation

- [ ] All 6 builtin lenses seeded at startup: `libertarian`, `engineer`, `institutionalist`, `entrepreneur`, `spiritual`, `systemic`.
- [ ] Axis weights outside [0.0, 1.0] on `POST /api/lenses` return 422.
- [ ] `GET /api/lenses/metrics/timeseries?window=bad` returns 422.
- [ ] Invalid idea_id on any translation endpoint returns 404 (not 500).
- [ ] Invalid lens_id on any translation endpoint returns 404 (not 500).
- [ ] `GET /api/ideas/{id}/translations/precompute` (GET instead of POST) returns 405.

## Data Model

### Existing (unchanged from spec-181)

```python
class WorldviewLens(BaseModel):
    lens_id: str                          # slug: "libertarian"
    name: str
    description: str
    archetype_axes: dict[str, float]      # all values in [0.0, 1.0]
    framing_template: str | None = None
    is_builtin: bool = False
    created_at: datetime
    updated_at: datetime | None = None

class IdeaTranslation(BaseModel):
    idea_id: str
    lens_id: str
    original_name: str
    translated_summary: str
    emphasis: list[str]
    risk_framing: str
    opportunity_framing: str
    resonance_delta: float | None = None  # [-1.0, 1.0]
    cached: bool
    generated_at: datetime
    source_hash: str
    spec_ref: str = "spec-181"
```

### New models

```python
class LensTimeseriesDay(BaseModel):
    date: str                             # ISO date "YYYY-MM-DD"
    translations_generated: int
    unique_ideas: int
    unique_contributors: int

class LensTimeseriesResponse(BaseModel):
    window: str                           # "7d" | "30d" | "90d"
    from_date: str = Field(alias="from")
    to_date: str = Field(alias="to")
    days: list[LensTimeseriesDay]
    totals: LensTimeseriesDay
    spec_ref: str = "spec-181"

class IdeaLensReach(BaseModel):
    idea_id: str
    reach_index: float                    # [0.0, 1.0]
    lenses_viewed: list[str]
    lenses_available: list[str]
    total_views_across_lenses: int
    spec_ref: str = "spec-181"

class BulkTranslationResponse(BaseModel):
    idea_id: str
    translations: list[IdeaTranslation]
    total: int
    spec_ref: str = "spec-181"

class PrecomputeResponse(BaseModel):
    status: str = "queued"
    lens_count: int
    idea_id: str
```

## Risks and Assumptions

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| LLM translation introduces bias or distorts facts | Medium | High | Translations labeled as lens-filtered; original always shown first |
| Pre-computation exhausts LLM budget at scale | Medium | Medium | Pre-compute only for builtin 6 lenses; cap at creation time; rate-limit precompute endpoint |
| Lens framing reinforces tribalism | Low | Medium | Show all lenses side-by-side; no lens is "recommended" over others |
| In-memory ROI counters reset on process restart | High | Low | v1 uses in-memory (acceptable for MVP); persistent counters are a follow-up |
| News `dominant_lens_id` classification is wrong | Medium | Low | Classification is a suggestion, not authoritative; user can override with manual lens view |
| Translation latency on first request | Medium | Medium | Pre-compute at idea creation; surface `202 Accepted` for async paths |

**Key assumption**: Spec-181 implementation (`api/app/routers/lenses.py`, `api/app/services/lens_translation_service.py`, `api/app/models/lens_translation.py`) is already deployed and passes its tests before this spec is implemented. This spec adds endpoints and models without replacing spec-181 code.

## Known Gaps and Follow-up Tasks

- **Persistent timeseries storage**: Current ROI counters are in-memory and reset on restart. A follow-up spec should persist daily counts to PostgreSQL (`lens_timeseries` table) so historical charts survive deploys.
- **News POV classification accuracy**: `infer_dominant_lens()` uses keyword heuristics for MVP. A follow-up spec should use LLM-based classification with confidence score.
- **Contributor lens preference learning**: When a contributor consistently views ideas through `engineer`, the system should surface that lens first. Follow-up: personalized default lens.
- **`/ops/lenses-dashboard`**: A web page with charts showing reach_index distribution, top lenses by day, and cross-lens collaboration events. Follow-up after MVP.
- **CLI `cc lenses translate --idea <id> --lens <name>`**: Follow-up convenience wrapper.
- **Cross-lens collaboration event tracking**: Track when two contributors with different dominant lenses both contribute to the same idea after viewing it through different lenses. This is the strongest proof-of-bridging metric but requires the contribution graph to be live.
- **Force-regenerate authentication**: `POST /api/ideas/{id}/translations/{lens_id}` with `force_regenerate: true` currently open. Should require operator API key (deferred to auth spec).

---

## Out of Scope

- Automatic lens assignment based on contributor browsing history (follow-up).
- Fine-tuned lens-specific LLM models (MVP: prompt engineering only).
- Conflict detection when two lenses give contradictory framings.
- Push notifications for new lens translations on followed ideas.
- Lens versioning (track how translation changes as the idea evolves).
- `GET /api/news/{article_id}/translations` bulk endpoint for news (follow-up).
