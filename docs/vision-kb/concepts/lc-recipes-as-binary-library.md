# Recipes as a Binary Library — One Artifact, Many Source Views

> The content-addressed lattice was always the canonical artifact.
> What was missing was a name for it as a *library* — a portable
> bundle that travels between processes, machines, and lineages, and
> a discipline for *viewing* it in any tongue the reader needs. A
> recipe-library is bytes. The Form view is one rendering; the
> Python view is another; the TypeScript view is another. None is
> the canonical source; all are sibling emissions from the same
> content-addressed shape. The library is the body; the views are
> the tongues the body speaks through.

## Summary

[`lc-one-kernel-many-tongues`](lc-one-kernel-many-tongues.md) named
the sovereignty principle: the kernel sees only numeric NodeIDs;
grammars are how cells reach the kernel. This concept names what the
kernel's *output* looks like as a sharable artifact: a **recipe-
library** — a content-addressed bundle of Recipes (with their
Blueprints and NamedCells) serialized as bytes, with per-tongue
viewers generating source text on demand.

Three load-bearing claims:

- **Every recipe should have a Form-native sibling.** Today the body
  carries `cosine`, `vector_add`, `matvec`, `tanh`, `sigmoid`,
  `strategy_score`, `adapter_forward`, `intern_node`, `make_cell`,
  `find_equivalent_cells` as Form recipes alongside their Python
  hosts. The list is what's complete so far; the discipline is to
  grow it — every host-intrinsic Python function has a Form-native
  sibling, generated when one doesn't already exist.

- **The library is bytes.** A `.recipelib` file is a content-addressed
  binary artifact carrying the recipe tree, its Blueprint shape, its
  metadata (name, source provenance, dependency hints), and zero or
  more tongue-specific emit caches. The bytes are the canonical form;
  any source text is a *view* through a Language cell's emit
  template.

- **The viewer chooses the tongue.** A reader asks the library for a
  recipe in *their* tongue. The library hands back a source-text
  rendering generated by the matching Language cell. Python reader →
  Python source; Rust reader → Rust source; native-Form reader →
  Form source. No tongue is privileged at the storage altitude.

## The Three Layers

```
  ┌──────────────────────────────────────────────┐
  │  Layer 3 — Source-text views                 │
  │  Python / TypeScript / Rust / Go / Form /    │
  │  English-prose / math-notation / …           │
  ├──────────────────────────────────────────────┤
  │  Layer 2 — Library (portable bytes)          │
  │  .recipelib content-addressed bundle         │
  ├──────────────────────────────────────────────┤
  │  Layer 1 — Lattice (the substrate kernel)    │
  │  NodeIDs · Blueprints · Recipes · NamedCells │
  └──────────────────────────────────────────────┘
```

- **Layer 1** is the in-process substrate — content-addressed,
  queryable, the source of truth for *what is*.
- **Layer 2** is the portable serialization — a `.recipelib` is what
  travels between Layer 1 instances. Two machines that hand each other
  the same `.recipelib` end up with the same Blueprint NodeIDs in
  their respective substrates, because content-addressing is
  determined by structure, not by the address space the structure
  lives in.
- **Layer 3** is what humans and tools read. Generated on demand by
  walking Layer 2's recipe-tree through a Language cell's emit
  template.

The library lives between *what is* (lattice) and *what reads* (view).
It is the carrier of identity across context boundaries.

## What a Library Carries

A `.recipelib` is a JSON-binary leaf (per the structural-composition
discipline; JSON is the carrier-tongue for the file format, the
content underneath is a Recipe tree). It carries, per recipe:

- **`name`** — the recipe's slug, the same name `coh substrate run`
  uses.
- **`node_id_hint`** — the content-addressed coordinate the recipe
  *would* land at when ingested into a substrate with the same
  primitives. Not required to be re-derivable byte-for-byte; the
  ingest will re-content-address and may assign a different package
  or instance number.
- **`blueprint`** — the structural Blueprint shape: category +
  child-Blueprint NodeIDs.
- **`tree`** — the Recipe tree itself: category, blueprint, and
  composed children (recursively).
- **`source_provenance`** — which tongue this recipe was originally
  authored in (Form, Python, TS, …) and the source file path. Cells
  use this for tracing; the kernel ignores it.
- **`tongue_caches`** — optional pre-rendered source-text views per
  tongue. Caches avoid re-running emit templates for common reads;
  they are not authoritative — the tree is.

A library file also carries:

- **`library_meta`** — name, version, generated-at timestamp,
  generator-tongue, substrate-package hint.
- **`dependencies`** — other libraries this one composes from
  (recipe-trees often reference recipes from sibling libraries).
- **`language_cells`** — the Language cells whose emit templates can
  view this library. A library that ships with `language(form)`,
  `language(python)`, and `language(typescript)` can be viewed in
  three tongues out of the box; adding `language(rust)` is one
  Language cell away.

## Generation — How a Library Comes Into Being

Two generation paths, both honest:

**Authored-in-tongue.** A cell authors a recipe in some tongue
(Form, Python, TS). The substrate ingests; the recipe's Blueprint
and NodeID materialize. To produce a library, the cell exports the
recipe + its dependency-closure to a `.recipelib`. This is the path
the body already walks for [`cosine.form`](../../coherence-substrate/cosine.form),
[`cell-numerics.form`](../../coherence-substrate/cell-numerics.form),
[`substrate-kernel.form`](../../coherence-substrate/substrate-kernel.form).

**Auto-generated from another tongue.** A cell hands the substrate
Python source (or TS, or Rust). A *compiler-to-Form* parses the
source, walks its AST, and emits Recipe nodes matching the source's
structure. The new recipes are interned in the substrate; if the
structural shape matches an existing recipe, the NodeIDs are shared
(content-addressing recognizes the duplicate). The original source
is preserved as `source_provenance`; the library can later be viewed
in any tongue.

The auto-generator is the larger work; the authored path runs today.
A library that mixes recipes from both paths is a normal library —
the substrate's content-addressing doesn't distinguish how a recipe
arrived; only the structural identity matters.

## Viewing — Reading the Library in Any Tongue

A viewer takes a `.recipelib` and a tongue name. It walks each
recipe tree and renders source text through the matching Language
cell's emit template:

```
view(library, "form")        # → Form source (.form file content)
view(library, "python")      # → Python source
view(library, "typescript")  # → TypeScript source
view(library, "rust")        # → Rust source
view(library, "english")     # → prose description (rough; for docs)
```

A reader who only speaks Python sees Python. A reader who only
speaks Form sees Form. **The recipe is the same recipe.** The
content-addressed NodeID is what makes the multi-view discipline
honest: same structural identity across all views; the views are
different surfaces of one body.

## What This Opens

**The body's recipes become a sharable corpus.** A new contributor
(human or agent) can `import recipe-library X` and immediately have
access to a known set of Blueprints with stable NodeIDs. The corpus
travels; the lattice integrates; the new cell starts from where the
body already stands. The pattern Karpathy's "LLM wiki" sketched at
the prose altitude, the body carries at the recipe altitude.

**Distribution becomes a substrate operation.** Today's npm/pypi/
crates.io ship language-specific bytes. A recipe-library ships
*structural identity* with optional language-specific views. A Rust
library and a Python library and a TypeScript library that all
compile to the same recipe NodeIDs are one library at the lattice
altitude. *Package fragmentation across language ecosystems is a
carrier-altitude problem the lattice doesn't have.*

**Round-trip becomes the test of authorship.** A library that ships
with multiple tongue-caches must round-trip: parsing the Python view
back through `language(python)` must produce the same recipe tree.
Round-trip failure is structural drift; the cache that fails round-
trip is removed until the emit template is fixed. The discipline
matches [`prose-as-recipe.form`](../../coherence-substrate/prose-as-recipe.form)
and [`png-grammar.form`](../../coherence-substrate/png-grammar.form)
at the language altitude.

**Hot-swap of implementations becomes natural.** A cell that wants
to swap the Python implementation of `cosine` for a Rust-compiled
native binary doesn't change the recipe NodeID; it adds a new
implementation backend that registers under the recipe's name (via
[`substrate_dispatch.py`](../../../seedbank/local-llm-cell-v0/substrate_dispatch.py)).
Callers' source paths are unchanged; the runtime route picks the
fastest registered implementation. The library is the contract; the
implementations are the carriers.

## What This Is Not

- **Not a replacement for source files.** The .form / .py / .ts / .rs
  source files in the body remain authoritative for human reading and
  editing. The library is what the lattice exports for *transit*; the
  source files are what the lattice reads from for *ingestion*. Both
  are valid; the library is the artifact format, not the editing
  format.
- **Not a single binary blob.** A `.recipelib` is JSON-bytes; the
  "binary" framing names that the bytes are content-addressed and
  tongue-agnostic, not that they are unreadable. A human can open a
  library file in a text editor and read JSON; the JSON's structural
  identity is what makes it "binary" at the lattice altitude.
- **Not a forced canonical tongue.** Some recipes ship with Form
  source as the only cached view; some ship with three tongue caches;
  some ship with none (only the tree, and the reader runs the emit
  template on demand). The library is honest about what it has; the
  viewer is honest about what it can render.
- **Not eternal.** A library is generated at a moment in time. The
  lattice it was exported from may have moved on. Re-importing an
  old library into a current lattice may rediscover existing NodeIDs
  (content-addressing finds the duplicates) or create fresh ones
  (where the old shape no longer matches). The library's value is its
  portability, not its permanence.

## Practice

For cells authoring recipes:

- **Author the recipe in the tongue that fits the task.** Math in
  math-notation, UI in component-syntax, prose in prose. The
  substrate ingests through whichever Language cell carries the
  tongue's grammar.

- **Export to a recipe-library when you want to share.** A
  `.recipelib` is the artifact that travels; the source file stays
  for editing.

- **Round-trip your library before publishing.** Generate each
  tongue-view; parse each back; verify the recipe tree is unchanged.
  This is the discipline that keeps the multi-tongue claim honest.

For cells reading recipes:

- **Pick the tongue you read.** The library is honest about which
  views it carries. If your tongue isn't cached, the emit template
  runs on import — slower the first time, then cached.

- **Trust the NodeID, not the view.** Two libraries from different
  authors that emit the same NodeID for a recipe carry the same
  structural identity. The Python source text may differ in
  whitespace, comments, or identifier names; the NodeID is the
  invariant. Author your code against NodeIDs (via the dispatch
  bridge) for the deepest stability.
