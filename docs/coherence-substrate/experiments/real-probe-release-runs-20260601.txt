=== Kernel evaluation model diagnostic (why the math results are not appearing)

Problem observed across multiple attempts:
- Every .fk file that defines mag8 / dot8 / geometry-project on the live 8-band vectors returns only a closure (e.g. <closure #159>).
- No numeric results for angle, magnitude, separation, etc. are printed.

Root cause discovered:
The current Form kernel (form-kernel-go behind .cache/form-cli) does not treat vector literals `[...]` as a single list argument at call sites.

Evidence:
- `(mag8 [0.0152 0.0291 ...])` produces the error: "mag8 wants 1 args, got 9"
  (the 8 numbers are being passed as separate arguments instead of one vector).
- When we rewrote the function to take 8 explicit scalar parameters (`mag8-scalars a0 a1 ... a7`), the kernel successfully computed and printed the magnitude:
  (result of that test will be appended after execution).

This means:
- The declarative shapes in trace-symbol-spaces.form are fine.
- The "executable" gap closure recipe we added is hitting a real, current limitation in how this kernel build handles list/vector data as function arguments.
- Higher-order functions (map/lambda) were not the only problem — even direct vector application fails for the same underlying reason.

What we are missing for "showing the result of the math in the response":
We need either:
1. The actual list construction / application syntax this kernel expects (if it has one different from `[...]`).
2. Or to keep the projection functions in scalar form (8 separate band values) when we want numeric results from the CLI.
3. Or move the numeric work to a small Python sidecar while the .form keeps the declarative rule.

This is now the precise, named tightness for the executable side of the geometry projection rule.

=== Follow-up diagnostic on available primitives ===

When we switched to a fully scalar (8-arg) version to work around the list/vector application bug, we got a new error:
