# Spec: SSE Control Channel — Real-Time Steer, Checkpoint, Abort, Ask-Permission for Native Agent CLIs

*Task ID: task_d04f93663743f5e8*
*Format: [specs/TEMPLATE.md](TEMPLATE.md)*

---

## Purpose

Native agent CLIs (claude, codex, cursor agent, gemini) currently execute as black boxes: once
started they run for minutes with no feedback, no ability to redirect, and no mechanism to
pause-and-confirm before a risky action. If an agent goes off-course, the only option is to kill
it and lose all progress.

The **SSE Control Channel** makes every agent execution observable and steerable in real time.
When a runner starts a task, it opens a Server-Sent Events (SSE) subscription to
`GET /api/agent/tasks/{task_id}/control-stream`. The API pushes structured **command events**
into that stream: `checkpoint`, `steer`, `abort`, `ask`, and `report`. The runner (wrapper
around the CLI) processes each event and takes the corresponding action — saving state,
redirecting the prompt, stopping cleanly, or surfacing a permission request — then POSTs a
response back to `POST /api/agent/tasks/{task_id}/control-response`.

This turns every long-running agent task into an **interactive process with a two-way channel**,
enabling Telegram/web users to guide agents mid-flight and enabling the pipeline manager to
recover automatically from drift without sacrificing progress.

## Requirements

### Core Control Commands

- [ ] **R-01** — Runner SSE subscriber: when a task starts execution, the runner thread opens
  `GET /api/agent/tasks/{task_id}/control-stream` and listens for the full task lifetime.
- [ ] **R-02** — `checkpoint` command: runner writes current output/state to
  `.task-{task_id}-checkpoint.json` and POSTs confirmation to `/control-response`.
- [ ] **R-03** — `steer` command: runner appends a steer message to the agent's command file
  (`/tmp/.agent-cmd-{task_id}`) and the agent CLI polls/reads it next iteration.
  POSTs acknowledgement with current step.
- [ ] **R-04** — `abort` command: runner sends SIGTERM to the agent process, waits up to 10 s,
  writes a final checkpoint, updates task status to `aborted`, POSTs confirmation.
- [ ] **R-05** — `ask` command: runner pauses the agent (SIGSTOP or command-file `PAUSE`),
  emits an `ask_pending` activity event with the question text, and holds until a
  `/control-response` with `type=ask_response` arrives; then resumes (SIGCONT or `RESUME`).
- [ ] **R-06** — `report` command: runner reads current `progress_pct` and `current_step`
  from task state and POSTs them as a `status_report` response.

### API Endpoints

- [ ] **R-07** — `GET /api/agent/tasks/{task_id}/control-stream` — SSE endpoint; emits
  `text/event-stream` with `data: <json>\n\n` frames; supports `Last-Event-ID` header for
  resume-after-reconnect.
- [ ] **R-08** — `POST /api/agent/tasks/{task_id}/control-command` — Operators (Telegram bot,
  web UI, pipeline manager) push a command into the channel; the API persists it and emits it
  on the SSE stream.
- [ ] **R-09** — `POST /api/agent/tasks/{task_id}/control-response` — Runner posts the result
  of a command back; stored in task activity log and forwarded to waiting `ask` waiters.
- [ ] **R-10** — `GET /api/agent/tasks/{task_id}/control-history` — Returns the ordered list
  of all control commands and responses for a task (for audit/review).

### Observability

- [ ] **R-11** — Every command and response is logged as a `task_activity` event so the
  existing `/api/tasks/activity` feed and `/api/tasks/{task_id}/stream` include control events.
- [ ] **R-12** — The `GET /api/agent/pipeline-status` response includes a
  `control_capable: bool` flag per active task (true if an SSE subscriber is connected).
- [ ] **R-13** — A `cc control` CLI command wraps the API:
  `cc control <task_id> steer "focus on X"` / `cc control <task_id> checkpoint` /
  `cc control <task_id> abort` / `cc control <task_id> ask "may I delete Y?"`.

### Safety

- [ ] **R-14** — Commands are idempotent for `checkpoint` and `report`; duplicate delivery is
  safe.
- [ ] **R-15** — `abort` after task completion is a no-op (returns 200, `{"ignored": true}`).
- [ ] **R-16** — `ask` times out after 300 s; on timeout the agent resumes (safest default)
  and logs a `ask_timeout` activity event.
- [ ] **R-17** — The SSE stream emits a `keepalive` comment (`: keepalive\n\n`) every 15 s to
  prevent proxy timeouts.

## API Contract

### `GET /api/agent/tasks/{task_id}/control-stream`

Opens an SSE channel. The runner calls this as soon as a task starts executing.

**Response — `200 text/event-stream`**

Each event:
```
id: <sequential int>
event: command
data: {"command_id": "cmd_abc123", "type": "steer", "payload": {"message": "focus on X"}, "issued_at": "2026-03-28T12:00:00Z"}

: keepalive

```

`event` values: `command`, `keepalive`

**Headers sent by client (runner)**:
- `Last-Event-ID: <last seen id>` — reconnect resumes from that point, replaying missed commands

**Notes**:
- Stream stays open for task lifetime; server closes it when task reaches a terminal state.
- Authentication: same API key mechanism as all other endpoints.

---

### `POST /api/agent/tasks/{task_id}/control-command`

Operator pushes a command. Persisted and immediately emitted on the SSE stream.

**Request**
```json
{
  "type": "steer",
  "payload": {
    "message": "Stop writing tests; focus on the implementation first."
  },
  "issued_by": "telegram:user_4321"
}
```

`type` enum: `checkpoint` | `steer` | `abort` | `ask` | `report`

`payload` per type:
- `checkpoint` — `{}` (no payload required)
- `steer` — `{"message": "string (1–2000 chars)"}`
- `abort` — `{"reason": "optional string"}`
- `ask` — `{"question": "string", "timeout_sec": 300}`
- `report` — `{}`

**Response 201**
```json
{
  "command_id": "cmd_abc123",
  "type": "steer",
  "task_id": "task_d04f93663743f5e8",
  "queued_at": "2026-03-28T12:00:00Z",
  "subscriber_connected": true
}
```

`subscriber_connected: false` means no runner is listening; command is queued and will be
delivered when the runner connects or reconnects.

**Response 404** — task not found
**Response 422** — invalid `type` or payload validation failure

---

### `POST /api/agent/tasks/{task_id}/control-response`

Runner posts the outcome of executing a command.

**Request**
```json
{
  "command_id": "cmd_abc123",
  "type": "checkpoint",
  "status": "ok",
  "data": {
    "checkpoint_path": "/tmp/.task-task_abc-checkpoint.json",
    "progress_pct": 62,
    "current_step": "writing api/app/routers/agent_control_routes.py"
  }
}
```

`status` enum: `ok` | `error` | `timeout`

**Response 200**
```json
{"accepted": true, "command_id": "cmd_abc123"}
```

**Response 404** — command_id not found or does not belong to task_id

---

### `GET /api/agent/tasks/{task_id}/control-history`

Returns full audit trail of commands and responses for a task.

**Response 200**
```json
{
  "task_id": "task_d04f93663743f5e8",
  "entries": [
    {
      "command_id": "cmd_abc123",
      "type": "checkpoint",
      "issued_by": "telegram:user_4321",
      "issued_at": "2026-03-28T12:00:00Z",
      "response": {
        "status": "ok",
        "data": {"progress_pct": 62},
        "responded_at": "2026-03-28T12:00:05Z"
      }
    }
  ],
  "total": 1
}
```

**Response 404** — task not found

## Data Model

```yaml
ControlCommand:
  properties:
    command_id:    { type: string, format: "cmd_<uuid4_short>" }
    task_id:       { type: string }
    type:          { type: string, enum: [checkpoint, steer, abort, ask, report] }
    payload:       { type: object }
    issued_by:     { type: string, nullable: true }
    issued_at:     { type: string, format: datetime-iso8601-utc }
    sse_event_id:  { type: integer }          # monotonic; used for Last-Event-ID resume

ControlResponse:
  properties:
    command_id:    { type: string }
    task_id:       { type: string }
    type:          { type: string, enum: [checkpoint, steer, abort, ask, report] }
    status:        { type: string, enum: [ok, error, timeout] }
    data:          { type: object }
    responded_at:  { type: string, format: datetime-iso8601-utc }

ChannelState (in-memory per task):
  properties:
    task_id:        { type: string }
    commands:       { type: array, items: ControlCommand }
    responses:      { type: array, items: ControlResponse }
    next_event_id:  { type: integer }
    subscriber_q:   { type: asyncio.Queue }    # fed by POST /control-command, drained by SSE generator
    ask_waiters:    { type: dict[str, asyncio.Future] }  # keyed by command_id
```

Storage: in-memory `dict[task_id, ChannelState]` within `agent_control_service.py`.
Future: migrate to Redis pub/sub for multi-process deployments.

logged, agent resumes with `data: {"answer": "timeout", "default": "proceed_safely"}`.

## Runner-Side Integration

The runner (the process that shells out to `claude`, `codex`, etc.) wraps execution with a
`ControlChannelListener` thread:

```python

    Background thread that subscribes to GET /api/agent/tasks/{task_id}/control-stream
    and dispatches received commands to the agent subprocess.
    """
    def __init__(self, task_id: str, api_base: str, agent_process: subprocess.Popen,
                 command_file_path: str):
        ...

    def run(self) -> None:
        """Loop: connect → parse SSE events → dispatch → reconnect on drop."""
        ...

    def _dispatch(self, command: dict) -> None:
        cmd_type = command["type"]
        if cmd_type == "checkpoint":
            self._do_checkpoint(command)
        elif cmd_type == "steer":
            self._do_steer(command)
        elif cmd_type == "abort":
            self._do_abort(command)
        elif cmd_type == "ask":
            self._do_ask(command)
        elif cmd_type == "report":
            self._do_report(command)

    def _do_checkpoint(self, command: dict) -> None:
        # Write current task output to checkpoint JSON, POST response
        ...

    def _do_steer(self, command: dict) -> None:
        # Append steer message to command file, agent polls it
        ...

    def _do_abort(self, command: dict) -> None:
        # SIGTERM agent, wait 10s, SIGKILL if still alive, write checkpoint, POST response
        ...

    def _do_ask(self, command: dict) -> None:
        # SIGSTOP agent, POST ask_pending activity event, poll for response with timeout,
        # inject answer into command file, SIGCONT agent
        ...

    def _do_report(self, command: dict) -> None:
        # Read current progress from shared state, POST response immediately
        ...
```

### Command File Protocol

The **command file** (`/tmp/.agent-cmd-{task_id}`) is a newline-delimited JSON file:

```jsonl
{"type": "steer", "message": "Focus on implementation, skip tests for now.", "at": "2026-03-28T12:00:00Z"}
{"type": "resume", "ask_answer": "no", "at": "2026-03-28T12:01:00Z"}
```

Agent CLIs that support a context-injection mechanism (claude via `--context-file`, codex via
prompt injection at next iteration) poll this file every few seconds. CLIs that do not support
polling receive the steer via SIGTERM + restart with amended prompt.

## Concurrency Behavior

- **SSE stream**: one subscriber per task recommended; multiple subscribers receive identical
  event sequence (fan-out via asyncio Queue broadcasting).
- **Push command**: immediately enqueued; SSE generator dequeues and emits; latency < 100 ms
  under normal load.
- **Ask waiter**: `asyncio.Future` registered in `ask_waiters[command_id]`; resolved by
  `post_response()`. Thread-safe with `asyncio.get_event_loop().call_soon_threadsafe`.
- **Write operations**: `ChannelState.commands` and `responses` lists are append-only; reads
  are safe without locking at in-memory MVP scale.

## Improving and Proving This Works Over Time

The open question — _"How can we improve this idea and make proof clearer over time?"_ — is
addressed through three layers:

### 1. Visibility Dashboard

Add a `control_capable` flag to `GET /api/agent/pipeline-status` per active task. Display in
the web UI alongside task progress. This makes it immediately obvious which tasks are
subscribing to the SSE control channel vs. which are still black-box.

### 2. Control Event Metrics

Since every command and response flows through `task_activity`, the existing
`GET /api/agent/metrics` can be extended with:
- `control_commands_issued` per task type per day
- `ask_timeout_rate` — fraction of ask commands that timed out (high = human not available)
- `abort_recovery_rate` — fraction of aborted tasks that were retried and eventually succeeded
- `mean_steer_response_latency_ms` — time from command to runner ack

These metrics quantify whether the control channel is being used and whether it's effective.

### 3. Integration Test Harness

`api/tests/test_agent_control.py` includes a simulated SSE consumer that connects to the test
client's event stream, sends a command, and asserts the event appears within 1 second. This is
CI-runnable proof that the two-way channel works end-to-end without a real agent subprocess.

## Risks and Assumptions

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Agent CLI doesn't poll command file | Medium | Steer falls back to restart with amended prompt; document per-provider behavior |
| SIGSTOP/SIGCONT not portable on Windows agents | Low | Windows runner uses named-pipe pause instead; add platform guard in `_do_ask` |
| SSE stream drops under Cloudflare 100 s proxy timeout | Medium | Keepalive `: keepalive` comment every 15 s; Cloudflare stream timeout configured to 300 s |
| In-memory store lost on API restart | Low (MVP) | Checkpoint file is on disk; runner reconnects and replays from `Last-Event-ID` |
| ask command deadlocks if human never responds | Medium | `timeout_sec` (default 300) + auto-resume with safe default answer |
| Multiple runners connect to same task stream | Low | Fan-out queue ensures both receive; idempotent dispatch on runner side |

## Failure/Retry Reflection

- **Failure mode**: SSE subscriber disconnects mid-task (network drop).
  - **Blind spot**: Runner may miss commands issued during disconnect window.
  - **Next action**: Runner reconnects with `Last-Event-ID` header; API replays all missed
    events; `missed_commands_replayed` counter in activity event confirms replay worked.

- **Failure mode**: `abort` command delivered but agent process already exited.
  - **Blind spot**: `SIGTERM` to a dead PID raises `ProcessLookupError`.
  - **Next action**: Wrap in try/except; check process status before signal; post `ignored`
    response.

- **Failure mode**: Checkpoint write fails (disk full, permission error).
  - **Blind spot**: Checkpoint response posted as `ok` but file doesn't exist.
  - **Next action**: Verify file size > 0 after write; post `error` status with details;
    operator can issue another checkpoint command after resolving disk issue.
