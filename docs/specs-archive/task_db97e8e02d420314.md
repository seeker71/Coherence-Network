## Summary

Four web pages currently expose overlapping data with unclear mental-model boundaries:

| Old page | Primary content |
|---|---|
| `/automation` | Provider readiness, exec stats, federation node list, usage alerts |
| `/usage` | Provider health, daily summary, friction, runtime costs |
| `/remote-ops` | Task queue, pipeline controls, deployment uptime |
| `/nodes` | Federation node list, system metrics, streaks, messaging |

This spec consolidates all four into **two clear pages**:

1. **`/nodes`** — everything about infrastructure: who is online, their health, providers, remote control.
2. **`/pipeline`** — everything about execution: queue, running tasks, completed results, performance.

**This spec goes beyond the structural merge.** It answers the open question:
> *"How can we improve this idea, show whether it is working yet, and make that proof clearer over time?"*

The answer is a **proof layer** built into both pages: explicit freshness chips, trend sparklines, health-delta indicators, and a "consolidation health card" that signals whether the merger is actively reducing confusion or creating a new information desert.

---

## Goals

1. **Remove 3 redundant pages** — `/automation`, `/usage`, `/remote-ops` deleted; 301 redirects installed.
2. **No information loss** — every data element from removed pages is reachable on the new pages.
3. **Cleaner mental model** — Nodes = infrastructure; Pipeline = execution. Users answer their question on one page.
4. **Proof over time** — each page carries observable signals that prove it is live, current, and not hiding data.
5. **Measurable success** — define what "working" looks like so a future reviewer can confirm it without guessing.

---

## Non-Goals

- No new API endpoints required (all data already available).
- No data model or schema changes.
- No design system overhaul — use existing shadcn/ui components and patterns.
- No change to CLI commands (`cc nodes`, `cc pipeline-status` remain unchanged).

## Target Architecture

### `/nodes` — "Infrastructure View"

**Purpose**: Answer "what is the network capable of right now and is it healthy?"

**Section order:**

1. **Network health bar** *(proof: live signal)*
   - Live node counts: `N online / M total`. Color-coded badges: green < 5 min, yellow < 60 min, red > 60 min since last seen.
   - Blocking issues banner (red) if any required provider is unconfigured.
   - Data-freshness chip: `"Updated X min ago"` driven by `generated_at` from `/api/federation/nodes/stats`.

2. **Node cards grid**
   - One card per registered node: hostname, OS icon, status badge, last seen (relative).
   - System metrics: CPU %, memory %, disk %.
   - Git SHA with up-to-date indicator (green checkmark or orange ⚠).
   - Streak summary: `completed / failed / success_rate%`.
   - Provider list tags.
   - Expand → per-provider breakdown (`by_provider` stats from streak).

3. **Provider readiness panel**
   - Table: provider name, configured (✓/✗), required (✓/✗), status badge, missing env vars.
   - Collapse if all required providers are ready (show "All required providers ready ✓").

4. **Provider exec stats** *(summary)*
   - Compact table: provider, success_rate, last_5_rate, avg_duration_s, blocked, needs_attention.
   - "Full details →" link to `/pipeline#provider-performance`.

5. **Remote control panel** *(moved from `/remote-ops`)*
   - Execute token input.
   - Executor selector, task type, model override.
   - Force paid / run as PR thread / auto-merge toggles.
   - "Create Task" and "Pick Up Task" buttons with inline status feedback.

6. **Node messaging** *(unchanged from current `/nodes/MessageForm.tsx`)*

7. **Fleet capabilities summary**
   - Total CPUs, total RAM, GPU-capable nodes, active executors.
   - From `/api/federation/fleet-capabilities`.

**Data sources:**
- `GET /api/federation/nodes`
- `GET /api/federation/nodes/stats`
- `GET /api/automation/usage/readiness`
- `GET /api/providers/stats`
- `GET /api/automation/usage`
- `POST /api/agent/tasks` (remote control)

### `/pipeline` — "Execution View"

**Purpose**: Answer "what is the network executing right now, and how well is it performing?"

**Section order:**

1. **Live task counter bar** *(proof: live signal)*
   - `N pending / M running / K completed today`. Auto-refreshes every 30 s.
   - Freshness chip: last-refreshed timestamp.
   - Trend indicator: ▲ / ▼ / → completed vs same window yesterday (when data available).

2. **Running tasks**
   - Live list: task ID, node, provider, elapsed time, last activity event.
   - "No tasks currently running" placeholder (not a crash).

3. **Pending queue**
   - Paginated table: task ID, type, queued_at, assigned executor.
   - Filter by executor dropdown.

4. **Completed tasks**
   - Paginated reverse-chronological list.
   - Outcome badges: ✓ completed, ✗ failed, ⏱ timed-out.
   - Columns: task ID, provider, duration, completed_at.

5. **Provider performance panel** *(canonical location; `/nodes` links here)*
   - Full table: success_rate, last_5_rate, avg_duration_s, selection_probability, blocked, needs_attention.
   - Alerts section (from `GET /api/providers/stats` `.alerts`).
   - Error breakdown expandable per provider.

6. **Daily summary** *(moved from `/usage`)*
   - Friction score + signals.
   - Quality awareness score.
   - Worker failed events (raw, recoverable, active, recovery streak).
   - Top tools / attention areas table.
   - Host runner breakdown by task type.

7. **Streaks dashboard** *(proof: trend over time)*
   - Aggregated across all nodes: completed today, failed today, network success_rate %.
   - 7-day sparkline trend chart (requires data from node streak history or rolling aggregate).
   - Color delta: green if current rate > last-week average, red if below.

8. **Runtime cost by idea** *(moved from `/usage`)*
   - Paginated table: idea_id, idea title, estimated runtime cost.
   - `page` and `page_size` query params preserved.

9. **View performance** *(moved from `/usage`)*
   - Table: API route, avg response ms, p95, request count.
   - Collapsed by default (power-user section).

**Data sources:**
- `GET /api/agent/pipeline-status`
- `GET /api/providers/stats`
- `GET /api/agent/usage`
- `GET /api/federation/nodes`
- `GET /api/agent/tasks` (paginated)

## Proof-of-Working Layer (Open Question Answer)

The open question is: *"How can we improve this idea, show whether it is working yet, and make that proof clearer over time?"*

### Why existing approaches fail

Before this spec, "is it working?" meant "do the pages load?" That is insufficient. The consolidation could succeed structurally (pages exist, redirects work) while failing functionally (data is stale, sections silently empty, users still confused).

### The Proof Layer

Three mechanisms built into the UI:

#### 1. Data-freshness chips (immediate proof)

Every major section carries a chip: `"Updated N min ago"` or `"No data — last seen Xh ago"`. Source: the `generated_at` or `last_seen_at` timestamps already returned by all API endpoints.

- Green: < 5 min
- Yellow: 5–60 min
- Red: > 60 min

A user can land on `/nodes` and confirm within 2 seconds that the node data is live.

#### 2. Streaks sparkline (proof over time)

The `/pipeline` streaks dashboard includes a 7-day trend sparkline for:
- Network-wide success_rate (%)
- Total tasks completed per day

This is visible evidence that the pipeline is running and improving (or degrading). An improving trend line after the consolidation is deployed is the strongest proof that removing 3 pages did not hide data.

**Data requirement**: The sparkline requires rolling 7-day aggregate. If `GET /api/agent/usage` already returns this, use it. If not, use per-node `streak.last_10` arrays as a proxy (last 10 tasks per node, aggregated). Raise a follow-up idea for a dedicated `GET /api/agent/streak-history?days=7` endpoint.

#### 3. Consolidation health card (structural proof)

A small card rendered on both `/nodes` and `/pipeline` immediately after deployment:

```
Consolidation status:
  ✓ /automation → /nodes (301)
  ✓ /usage → /pipeline (301)
  ✓ /remote-ops → /nodes (301)
  ✓ All sections migrated
  ✓ No broken nav links
Last verified: <timestamp from a build-time check>
```

This card is rendered from a static JSON file (`web/public/consolidation-status.json`) generated during the build or by a post-deploy verification script. It answers "did we actually do this right?" without requiring a human to re-read the spec.

The card can be hidden behind a query param `?show_consolidation=true` so it is not permanently on-screen but is inspectable during verification.

## Verification Scenarios

### Scenario 1 — Redirects are permanent and correct

**Setup**: Production deployment with `next.config.js` redirects in place.

**Action**:
```bash
API=https://coherencycoin.com
curl -sI $API/automation | grep -iE "^(HTTP|location)"
curl -sI $API/usage | grep -iE "^(HTTP|location)"
curl -sI $API/remote-ops | grep -iE "^(HTTP|location)"
```

**Expected result**:
```
HTTP/2 301
location: https://coherencycoin.com/nodes

HTTP/2 301
location: https://coherencycoin.com/pipeline

HTTP/2 301
location: https://coherencycoin.com/nodes
```

**Edge case — query params**:
```bash
curl -sI "$API/automation?force_refresh=true" | grep -iE "^(HTTP|location)"
```
Expected: still HTTP 301, location points to `/nodes` (query param may or may not be forwarded — 301 is the requirement).

**Edge case — does the redirect target load?**:
```bash
curl -sI $API/nodes | grep "^HTTP"
curl -sI $API/pipeline | grep "^HTTP"
```
Expected: `HTTP/2 200` for both.

### Scenario 4 — Remote control dispatch from /nodes works inline

**Setup**: Valid `AGENT_EXECUTE_TOKEN` available. At least one executor registered in `/api/federation/nodes`.

**Action**:
1. Navigate to `https://coherencycoin.com/nodes`
2. Scroll to Remote Control panel
3. Enter valid execute token
4. Select executor from dropdown
5. Enter task type: `"spec"`
6. Click "Create Task"

**Expected result**:
- POST to `https://api.coherencycoin.com/api/agent/tasks` returns HTTP 200 or 201
- UI shows inline feedback: `"Task created: task_<id>"`
- No full-page navigation occurs; panel stays visible with task ID shown

**Edge case — invalid token**:
Enter a wrong token, click "Create Task".
Expected: API returns HTTP 403. Panel shows: `"Error: Execute token invalid (403)"`. Page does not crash. No task is created.

**Edge case — no executor selected**:
Click "Create Task" without selecting executor.
Expected: Client-side validation prevents submission, shows `"Select an executor"` inline message.

## Risks and Assumptions

| Risk | Likelihood | Mitigation |
|---|---|---|
| Users have bookmarked `/automation` or `/usage` | Medium | HTTP 301 permanent redirect — browser caches it, user transparently forwarded. |
| Merged pages become too long / overwhelming | Medium | Collapsible sections + tab sub-nav (defer to UX spec task_5cb9d9ff27681827 for patterns). |
| `NavLinksSection` component (from `/usage`) still imported elsewhere | Low | Grep for `NavLinksSection` before deleting `/usage` — remove all usages before deletion. |
| Streaks sparkline requires data not yet aggregated | Medium | Use `streak.last_10` as proxy until `GET /api/agent/streak-history` endpoint is built. Log a follow-up idea. |
| `/usage` pagination query params (`page`, `page_size`) need migration | Medium | Carry same param names into `/pipeline`. Map 1:1, no API change needed. |
| External tools (CI scripts, monitoring) link to removed routes | Low | Redirects cover this. Document in RUNBOOK. |
| Freshness chips show stale data if `generated_at` field absent from API | Low | Default to `"Freshness unknown"` chip (yellow) — never blank, never crash. |
