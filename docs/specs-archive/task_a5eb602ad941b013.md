# Spec: Value Lineage Visualization

**Task ID**: task_a5eb602ad941b013
**Type**: Feature (Frontend + API extension)
**Status**: Draft

---

## Summary

The value lineage API exists and tracks the full flow from idea → spec → implementation → usage → payout. However, there is **no visual representation** of this flow. A contributor looking at their idea has no way to see how their contribution propagated through the network and generated CC value.

This spec defines a **horizontal flow diagram** embedded on the idea detail page (`/ideas/<idea_id>`) that shows each stage of the lineage with CC amounts, usage events, and contributor attribution. The goal is to make the network's core value proposition tangible and self-evidencing: you can see your idea turn into CC.

### Why this matters

- **Contributor motivation**: Seeing a visual proof that "I wrote this idea → it became a spec → it got implemented → 45 usage events → 3.2 CC payout" is the core reason to participate.
- **Trust signal**: Opaque payout calculations create distrust. A visible flow makes the math legible.
- **Growth mechanic**: Contributors share their lineage diagrams. This is the shareable unit that onboards new contributors.
- **Proof over time**: The diagram updates live — as more usage events arrive, the potential payout grows. This creates a real-time feedback loop that proves the system is working.

---

## Goals

1. A contributor can navigate to `/ideas/<idea_id>` and see a horizontal flow diagram showing their idea's lineage stages.
2. Each stage node shows: stage name, CC invested or potential, usage count (for usage stage), payout amount (for payout stage).
3. Clicking any node shows a contributor breakdown (who contributed at that stage and how much CC).
4. The diagram links to existing API data — no new data model is required, only a new API endpoint that assembles lineage data into a single response.
5. The diagram renders a "not yet" state gracefully when a stage hasn't happened (dimmed node).
6. If no lineage exists for an idea, a CTA is shown explaining how to create one.

### Proof-over-time requirements

- Usage event count must **increment visibly** as new events arrive (revalidation every 90s or on-demand).
- Payout preview amount updates whenever the usage count changes.
- A "last updated" timestamp is shown so contributors know the data is live.
- If `roi_ratio >= 1.0`, show a visual "in profit" indicator on the payout node.

## API Contract

### `GET /api/value-lineage/links/by-idea/{idea_id}`

Looks up the first lineage link whose `idea_id` matches the given `idea_id`.

**Request**
- `idea_id`: string (path param, URL-decoded)

**Response 200**
```json
{
  "id": "lnk_abc123",
  "idea_id": "idea-xyz",
  "spec_id": "spec-789",
  "implementation_refs": ["pr/42"],
  "contributors": {
    "idea": "alice",
    "spec": "bob",
    "implementation": "carol"
  },
  "investments": [
    {"stage": "idea", "contributor": "alice", "energy_units": 2.0, "coherence_score": 0.8, "awareness_score": 0.7, "friction_score": 0.3}
  ],
  "estimated_cost": 11.0,
  "created_at": "2026-03-01T12:00:00Z",
  "updated_at": "2026-03-28T15:00:00Z"
}
```

**Response 404**
```json
{ "detail": "No lineage link found for idea idea-xyz" }
```

## Data Model

No new database tables. The flow endpoint assembles data from existing service functions:
- `value_lineage_service.get_link(lineage_id)` → investments, contributors
- `value_lineage_service.valuation(lineage_id)` → usage event count, ROI
- `value_lineage_service.payout_preview(lineage_id, payout_pool=100.0)` → payout rows

New Pydantic models added to `api/app/models/value_lineage.py`:

```yaml
LineageStageNode:
  properties:
    stage: { type: string }  # "idea" | "spec" | "implementation" | "usage" | "payout"
    present: { type: boolean }
    cc_invested: { type: float, optional: true }
    event_count: { type: int, optional: true }
    measured_value_total: { type: float, optional: true }
    payout_pool: { type: float, optional: true }
    roi_ratio: { type: float, optional: true }
    in_profit: { type: boolean, optional: true }
    contributors: { type: list[LineageStageContributor], optional: true }
    payouts: { type: list[PayoutRow], optional: true }

LineageStageContributor:
  properties:
    role: { type: string }
    contributor: { type: string }
    energy_units: { type: float }
    effective_weight: { type: float }

LineageFlowResponse:
  properties:
    lineage_id: { type: string }
    idea_id: { type: string }
    spec_id: { type: string }
    stages: { type: list[LineageStageNode] }
    estimated_cost: { type: float }
    last_updated: { type: datetime }
```

## Visual Design

```
│     IDEA     │───▶│     SPEC     │───▶│  IMPLEMENTATION  │───▶│     USAGE      │───▶│     PAYOUT      │
│  2.0 CC      │    │  3.0 CC      │    │  8.0 CC          │    │  45 events     │    │  35.0 CC alice  │
│  alice       │    │  bob         │    │  carol           │    │  45.0 measured │    │  25.0 CC bob    │
└──────────────┘    └──────────────┘    └──────────────────┘    └────────────────┘    │  40.0 CC carol  │
                                                                                       │  ✓ In Profit    │
                                                                                       └─────────────────┘
```

- Flow is horizontal (left-to-right), responsive (wraps on mobile).
- Arrow connectors between nodes (SVG or CSS border trick).
- Dimmed/ghost style for stages with `present: false`.
- Clicking a node → Radix UI `Popover` showing contributor table.
- Payout node shows green "In Profit" badge when `roi_ratio >= 1.0`.
- "Last updated: 2 min ago" caption below the diagram.

## Open Questions

### How can we improve this, show whether it is working yet, and make that proof clearer over time?

1. **Progressive disclosure via timestamps**: Show "Idea created 7 days ago → Spec created 5 days ago → Implementation merged 2 days ago → 45 events in last 24h". The time gap between stages tells a story about velocity.

2. **Animated accumulation**: When the page loads, the usage counter increments from 0 to the current count with a count-up animation. This makes "45 events" feel earned rather than static.

3. **Lineage health signal**: Add a `health_score` field to `LineageFlowResponse` that is a composite of: stages_completed / 5, usage_growth_rate, roi_ratio. Show a simple health bar (green/yellow/red). This answers "is this lineage working?" at a glance.

4. **Comparative context**: Show "Top 10% of ideas by usage" or "3x average payout" — percentile-based context makes individual numbers meaningful.

5. **Live update via SSE or polling**: A small WebSocket or polling mechanism (every 30s) that pushes usage event count updates to the diagram without a full page reload. This makes the "live" aspect visible during demos.

6. **Share card**: A `/ideas/<id>/lineage.png` OG image generation endpoint that renders the flow diagram as an image, making the lineage shareable on social media. The flow diagram IS the pitch for the network.

7. **Proof of progress over time (chart)**: Embed a small sparkline below the Usage node showing usage events per day for the last 14 days. Even a flat line is "proof" — a growing line is compelling.

## Risks and Assumptions

- **Assumption**: `idea_id` in `LineageLinkCreate` is stable and matches the `id` field returned by `GET /api/ideas/{id}`. If ideas use a different ID format than what's stored in lineage links, `by-idea` lookup will never find results. Implementation must verify with real idea IDs.
- **Risk**: Multiple lineage links can theoretically share the same `idea_id` (no DB unique constraint). The `by-idea` endpoint should return the most recently created link and document this behavior.
- **Risk**: The `payout_pool=100.0` fixed value in the flow endpoint is arbitrary. If the actual CC in circulation is very different, the payout numbers will be misleading. This should be documented in the UI ("Preview based on 100 CC pool") and the endpoint should accept an optional `?payout_pool=X` query param.
- **Risk**: If the idea page loads lineage data server-side and the lineage API is slow, it will delay the entire page. Fetch lineage data client-side or use React `Suspense` with a fallback skeleton.
- **Assumption**: The web uses Radix UI (via shadcn/ui) — Popover is available without new dependencies.
- **Risk**: No auth required on the flow endpoint (read-only public data). This is intentional — contributor lineage is a public proof of work.

## Known Gaps and Follow-up Tasks

- **Gap**: No endpoint for `GET /api/value-lineage/links/by-spec/{spec_id}` — a spec author can't find their lineage from the spec page. Add in a follow-up spec.
- **Gap**: `GET /api/value-lineage/links/by-idea/{idea_id}` returns only the first matching link. A full-coverage version should return all lineage links for an idea (if an idea has been re-spec'd multiple times). Tracked as follow-up.
- **Gap**: No mobile-optimized layout for the horizontal flow diagram. On narrow screens, the 5-node horizontal layout will overflow. Vertical layout fallback needed for viewports < 640px.
- **Gap**: The `health_score` composite metric described in Open Questions is not in scope for this spec — it requires defining a normalization strategy. Track as `Follow-up task: value-lineage-health-score-spec`.

## Decision Gates

- **Payout pool value**: Should the flow endpoint default to 100 CC or the network's current total CC in circulation? This is a product decision. Default to 100 CC for simplicity; expose `?payout_pool=X` query param so callers can override.
- **Auth on flow endpoint**: Should the `flow` endpoint require an API key? Current recommendation is no — contributor flows are public proofs of work and should be shareable without auth. Escalate via `needs-decision` if this conflicts with privacy requirements.

## Failure/Retry Reflection

- **Failure mode**: `by-idea` endpoint finds no link even though one exists.
  - **Blind spot**: `idea_id` stored in lineage was set by an agent using a shortened/normalized form that doesn't match the canonical idea ID.
  - **Next action**: Log the exact `idea_id` value stored in the DB vs. the lookup value; add a normalization step (lowercase, strip whitespace).

- **Failure mode**: Flow endpoint returns 500 on ideas with zero investments.
  - **Blind spot**: `_compute_global_signals` or weight normalization divides by zero when no investments exist.
  - **Next action**: Add guard in `build_flow` service function; return all stages with `present: false` rather than propagating the error.
