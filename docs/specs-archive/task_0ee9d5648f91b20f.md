## Summary

Coherence Network is a distributed collaboration graph. Knowing *where* contributors are
located unlocks a new class of features: finding collaborators nearby, surfacing ideas
generated in your region, and matching regional news to local contributors. This spec
defines the full **geolocation awareness layer** — from profile enrichment through to
proximity-filtered discovery surfaces — and prescribes the observability hooks needed to
prove the feature is working over time.

The geocoding provider chain (OpenCage → Nominatim → static fallback) is imported from
**Living Codex** and is already implemented. This spec builds the product layer on top of
that foundation.

---

## Motivation

**What problem does this solve?**

Currently the graph treats all contributors and ideas as co-located in a virtual space.
Regional collaboration is invisible:
- A developer in São Paulo cannot discover others in their city.
- An idea tagged with a Brazilian startup ecosystem is not surfaced to contributors who
  follow local startup news.
- Agent tasks triggered by regional news events (e.g. a local hackathon, a city policy
  change) cannot be routed to contributors with local context.

**Why now?**
The geocoding service, contributor location storage, and Haversine proximity engine are
already implemented (see existing code in `api/app/services/geolocation_service.py` and
`api/app/services/geocoding_service.py`). The missing pieces are:

1. A **profile enrichment flow** — automatic geocoding of a city string into lat/lon.
2. **Proximity-filtered idea listing** — `GET /api/ideas` must accept `lat/lon/radius_km`.
3. A **web surface** — the Ideas page and Contributors page need "nearby" filters.
4. **Observability** — dashboards and metrics that prove the feature is active and
   useful over time.


## Requirements

### R1 — Profile enrichment via geocoding

When a contributor provides a city name (or city + country), the system **SHOULD** be
able to resolve it to coordinates automatically. The `/api/geocode/forward` endpoint
already handles this. The caller (web UI or CLI) is responsible for calling
`GET /api/geocode/forward?q=<city>` and then `PATCH /api/contributors/{id}/location`
with the resolved coordinates.

**Acceptance criteria**:
- A contributor can set their location in two steps: forward geocode their city, then
  PATCH their profile.
- The `PATCH` endpoint accepts coordinates already rounded to 2 decimal places.
- Visibility defaults to `contributors_only`.

### R2 — Ideas filtered by geographic proximity

The ideas listing endpoint must accept optional geographic proximity parameters:

```
GET /api/ideas?lat=<float>&lon=<float>&radius_km=<float>
```

When `lat` and `lon` are provided:
- Return only ideas whose `contributor_id` (author) has a stored location within
  `radius_km` of the query point.
- `radius_km` defaults to `100.0` when `lat`/`lon` are provided but `radius_km` is absent.
- Combine with other existing filters (status, tag, etc.) using AND logic.
- Return `X-Geo-Filtered: true` response header when geo filter is active.
- Each idea in the response includes a `contributor_distance_km` field (approximate,
  rounded to 2 decimal places).

**Acceptance criteria**:
- `GET /api/ideas?lat=52.52&lon=13.41&radius_km=50` returns only ideas from contributors
  within 50 km of Berlin.
- When no contributors have location data, returns empty list (not 500).
- `lat` or `lon` provided without the other → **422 Unprocessable Entity**.
- `radius_km=0` → **422** (must be > 0).

### R3 — Nearby collaborators surface (existing endpoint, new query params)

The `GET /api/nearby` endpoint already exists. This spec prescribes:
- Add optional `types` query param accepting `contributors`, `ideas`, or `all`
  (default: `all`). This allows callers to request only contributors or only ideas.
- Ensure the response always includes `total_contributors` and `total_ideas` fields
  (already present in model).

### R4 — Regional news resonance (existing endpoint, clarify behavior)

The `GET /api/news/resonance/local` endpoint already exists. This spec prescribes:
- Document that the `location` query param is matched against article text using
  tokenized keyword overlap (already implemented).
- The `resonance_score` (0.0–1.0) is the fraction of location tokens found in the
  article text.
- No changes to implementation required; this section documents the contract.

### R5 — Geocoding observability endpoint (`GET /api/geo/stats`)

Add a new **read-only** endpoint returning aggregate statistics about location coverage
and geocoding activity:

```json
GET /api/geo/stats

{
  "contributors_with_location": 42,
  "contributors_total": 150,
  "location_coverage_pct": 28.0,
  "visibility_breakdown": {
    "public": 10,
    "contributors_only": 25,
    "private": 7
  },
  "ideas_with_geo_author_pct": 35.3,
  "geocode_sources": {
    "opencage": 18,
    "nominatim": 12,
    "fallback": 4,
    "unknown": 8
  }
}
```

**Fields**:
- `contributors_with_location`: count of contributors who have `geo_location` set.
- `contributors_total`: total contributor nodes in graph.
- `location_coverage_pct`: (with_location / total) × 100, rounded to 1 decimal.
- `visibility_breakdown`: count per visibility enum value.
- `ideas_with_geo_author_pct`: % of ideas whose author has location set.
- `geocode_sources`: count per `source` string from stored `geo_location.source` field
  (requires adding `source` to the stored geo data — see R6).

**Acceptance criteria**:
- Returns 200 with valid JSON.
- When no contributors have location, all counts are 0, coverage is 0.0.
- Does not expose any PII or exact coordinates.

### R6 — Store geocoding source in contributor location

When storing a contributor location (via `PATCH /api/contributors/{id}/location`), the
request MAY include an optional `source` field indicating how the coordinates were
obtained (`opencage`, `nominatim`, `fallback`, `manual`). Defaults to `manual` when
absent.

The `geo_location` blob stored in the graph node gains a `source` field.
The `ContributorLocation` response model gains a `source: Optional[str]` field.

This enables `/api/geo/stats` to show `geocode_sources` breakdown.

### R7 — Latency field in forward geocoding response

Add `latency_ms: Optional[float]` to `GeocodeForwardResponse`. The geocoding service
times each request and populates this field. Useful for diagnosing provider slowness.

## Observability and Proof of Working

This section answers: *"How can we improve this idea, show whether it is working yet,
and make that proof clearer over time?"*

### Immediate proof (day 0)

1. **`GET /api/geo/stats`** — call this to see coverage. If `contributors_with_location > 0`
   and `location_coverage_pct > 0`, the feature is live.
2. **`GET /api/geocode/forward?q=Berlin`** — if `found=true` and `source` is one of
   `opencage|nominatim|fallback`, the geocoding chain is operational.
3. **`GET /api/nearby?lat=52.52&lon=13.41&radius_km=1000`** — if `total_contributors > 0`
   after at least one contributor location is set, the proximity engine is working.

### Short-term proof (week 1–2)

| Metric | Signal | Target |
|--------|--------|--------|
| `location_coverage_pct` | % contributors with location | > 10% within 2 weeks of launch |
| `geocode_sources.fallback` | % resolutions using offline fallback | < 30% (means real providers working) |
| `/api/ideas` with geo filter | Non-zero ideas returned | At least one idea per metro if coverage > 5% |
| `latency_ms` (geocode response) | P95 latency | < 2000 ms (Nominatim can be slow) |

### Long-term proof (month 1+)

- **Contributor discovery funnel**: Track how many contributors view `/nearby` and then
  follow another contributor. This is a session-level event, not yet implemented
  (future work).
- **Idea resonance by region**: Count ideas created within 24h of a regional news item
  with matching location keywords. Proof that local news drives local ideation.
- **Geo coverage growth curve**: Plot `location_coverage_pct` over time. Flat line means
  feature is not being adopted; should grow as onboarding flow includes location prompt.

### How to make proof clearer

1. **Admin dashboard tile**: Surface `GET /api/geo/stats` on the admin/pipeline page.
   A single tile showing "28% contributors located" is instantly readable.
2. **`source` tracking**: The `geocode_sources` breakdown in `/api/geo/stats` shows
   whether OpenCage is configured correctly (if `opencage > 0`) vs. falling back
   to Nominatim or the static table. This is a health signal.
3. **Latency monitoring**: Log `latency_ms` to structured logs. Alert if P95 > 3s
   (indicates Nominatim rate limiting or network issues).
4. **Map integration (future)**: A dot map of contributor locations provides the most
   visceral proof that geolocation is working. Even a static SVG heat map generated
   from `/api/geo/stats` visibility-public contributors would suffice.

## Verification Scenarios

### Scenario 1 — Set contributor location via geocode + PATCH (happy path)

**Setup**: At least one contributor exists (e.g. `contributor_id = "alice"`).

**Action**:
```bash
# Step 1: Geocode a city
curl -s "https://api.coherencycoin.com/api/geocode/forward?q=Berlin%2C+Germany"
# Expected: {"found": true, "latitude": 52.52, "longitude": 13.41, "source": "nominatim", "latency_ms": <float>}

# Step 2: Store location on contributor profile
curl -s -X PATCH "https://api.coherencycoin.com/api/contributors/alice/location" \
  -H "Content-Type: application/json" \
  -d '{"city": "Berlin", "country": "Germany", "latitude": 52.52, "longitude": 13.41, "visibility": "public", "source": "nominatim"}'
# Expected: HTTP 200, response includes contributor_id, city, country, source

# Step 3: Verify stored location
curl -s "https://api.coherencycoin.com/api/contributors/alice/location"
# Expected: {"contributor_id": "alice", "city": "Berlin", "country": "Germany", "latitude": 52.52, "longitude": 13.41, "source": "nominatim"}
```

**Edge cases**:
- `PATCH` with `latitude=200.0` → **422** (out of range).
- `GET /api/contributors/nonexistent/location` → **404**.
- `DELETE /api/contributors/alice/location` → **204**, then `GET` → **404**.

---

### Scenario 2 — Nearby contributors and ideas

**Setup**: Contributor `alice` has location `(52.52, 13.41)` (Berlin). Contributor `bob`
has location `(-33.87, 151.21)` (Sydney). Alice has created at least one idea.

**Action**:
```bash
# Query nearby from Berlin (500 km radius)
curl -s "https://api.coherencycoin.com/api/nearby?lat=52.52&lon=13.41&radius_km=500"
```

**Expected**:
- HTTP 200.
- `contributors` list contains `alice` with `distance_km` < 500.
- `contributors` list does NOT contain `bob` (Sydney is ~16,000 km away).
- `ideas` list contains alice's ideas with `distance_km` populated.
- `total_contributors >= 1`, `total_ideas >= 1`.
- Each contributor object has `contributor_id`, `name`, `city`, `country`, `distance_km`.

**Edge cases**:
- `radius_km=0` → **422**.
- `lat=200` → **422**.
- `GET /api/nearby?lat=52.52&lon=13.41&radius_km=1&types=contributors` → ideas array is
  empty, contributors array may have results.

---

### Scenario 3 — Ideas filtered by proximity

**Setup**: Contributor `alice` (Berlin, 52.52, 13.41) has idea `idea-001`. Contributor
`bob` (Sydney, -33.87, 151.21) has idea `idea-002`.

**Action**:
```bash
# Geo-filtered ideas near Berlin
curl -s "https://api.coherencycoin.com/api/ideas?lat=52.52&lon=13.41&radius_km=1000"
```

**Expected**:
- HTTP 200.
- Response includes `idea-001` with `contributor_distance_km < 1000`.
- Response does NOT include `idea-002` (contributor in Sydney).
- Response header `X-Geo-Filtered: true` is present.

**Edge cases**:
- `lat=52.52` with no `lon` → **422** ("both lat and lon required").
- `radius_km=-5` → **422**.
- No contributors in radius → empty `items` list, HTTP 200 (not 404 or 500).
- Regular idea listing without geo params is unaffected.

---

### Scenario 4 — Geolocation coverage statistics

**Setup**: System has some contributors with and without location data.

**Action**:
```bash
curl -s "https://api.coherencycoin.com/api/geo/stats"
```

**Expected**:
- HTTP 200.
- Response includes `contributors_with_location` (integer ≥ 0).
- `contributors_total` ≥ `contributors_with_location`.
- `location_coverage_pct` = (with_location / total) × 100, rounded to 1 decimal.
  - If total = 0, `location_coverage_pct = 0.0` (no division by zero).
- `visibility_breakdown` object has keys `public`, `contributors_only`, `private`.
- No PII exposed; no individual lat/lon in response.

**Edge cases**:
- System with zero contributors → all counts 0, coverage 0.0, HTTP 200 (not 500).
- Endpoint is GET-only; POST → **405 Method Not Allowed**.

---

### Scenario 5 — Regional news resonance

**Setup**: The system has news articles in the database, at least some mentioning cities.

**Action**:
```bash
curl -s "https://api.coherencycoin.com/api/news/resonance/local?location=Berlin&limit=5"
```

**Expected**:
- HTTP 200.
- `location = "Berlin"` in response.
- Each item has `resonance_score` between 0.0 and 1.0.
- Items sorted by `resonance_score` descending.
- `local_keywords` list is non-empty for items with score > 0.

**Edge cases**:
- `location=` (empty) → **422**.
- `location=XYZNonexistentCity123` → HTTP 200, `items = []`, `total = 0`.
- `limit=0` → **422** (must be ≥ 1).

## Risks and Assumptions

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Nominatim rate limiting (1 req/s) | Medium | Client-side throttle already enforced; add server-side cache for repeated queries |
| OpenCage API key absent in prod | Low | Fallback chain handles it gracefully (`source=nominatim` or `fallback`) |
| O(n) contributor scan for proximity | Medium | Acceptable up to ~10,000 contributors; add PostGIS/spatial index for larger scale |
| Low location coverage → empty nearby results | High at launch | Onboarding prompt, admin nudge via `/api/geo/stats` tile |
| Privacy: contributor location exposed to anyone | Medium | Visibility enforcement already in code; `private` contributors excluded from `/api/nearby` |
| `radius_km` too large → performance | Low | Max 20,000 km enforced at validation layer |

**Assumptions**:
- Graph node `properties` blob is mutable via `graph_service.update_node`.
- The `agent_service.list_tasks` function exists and returns dicts with `context` field.
- Living Codex geocoding chain is stable enough for production use; Nominatim is
  acceptable for moderate volume.

## Known Gaps and Follow-up Tasks

1. **Map visualization**: A geographic heat map of contributors/ideas would be the
   strongest visual proof. Implement as a web component reading from
   `GET /api/geo/stats` + public contributor locations.

2. **Reverse geocoding**: Given lat/lon, resolve to city/country string (useful for
   auto-filling contributor location from browser geolocation API). Not in this spec.

3. **Spatial indexing**: For > 10k contributors, replace O(n) Haversine scan with a
   PostGIS `ST_DWithin` query or R-tree index. Threshold: implement when query p95
   latency exceeds 200ms.

4. **Onboarding flow integration**: Prompt new contributors to share their city during
   signup. The web signup form should call `/api/geocode/forward` as the user types,
   show a confirmation ("Resolving to Berlin, Germany..."), then auto-populate the
   location PATCH payload.

5. **Caching for `/api/geocode/forward`**: Add an in-memory LRU cache (max 1000 entries,
   TTL 24h) to avoid repeat Nominatim calls for the same city string. Not in this spec
   but strongly recommended before public launch.

6. **Ideas geo filter on web**: The `/ideas` web page should surface a "Nearby" toggle
   that uses `navigator.geolocation` (with permission prompt) to auto-fill `lat`/`lon`
   and re-query `/api/ideas` with geo params.

7. **Agent task geo context**: Encourage runners to include `context.geo_lat`/`context.geo_lon`
   when creating regionally-scoped tasks, so `/api/geo/tasks/nearby` stays populated.
