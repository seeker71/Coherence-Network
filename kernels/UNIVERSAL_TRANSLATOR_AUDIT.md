# Universal Translator — Honest Audit Against the Highest Goal

> **The goal (re-stated, 2026-05-27):** universal translator across media, recipe
> orchestration in pure numeric space, *less ice / more gas*, minimize the
> bootstrap surface, maximize Form-native execution.

This document walks the body's current artifacts against that goal — what is
overweight, what is residue, what is light, what to breathe next. It is honest
discernment, not enthusiasm. The intent is to make the next breath obvious.

## Section 1 — Heavy in ICE (overweight storage of symbols that could be expressed)

The principle: *anything that could be regenerated by a smaller recipe is ice
that wants to thaw*. Three live overweight surfaces stand out today.

### 1.1 The `python-bmf.fk` size (3,000 LOC, 140 KB)

The largest grammar file in the tree, by an order of magnitude. Sibling
grammars: `go-bmf.fk` 559 LOC, `typescript-bmf.fk` 544 LOC, `rust-bmf.fk` 287
LOC, `audio-bmf.fk` 308, `image-bmf.fk` 437, `bml.fk` 908. Python takes ~6× the
next-largest peer.

What's load-bearing in those 3,000 lines is rule data; what's repeated is the
shape (each rule names input objects, output Form-recipe, reverse-emitter). The
parallel-agent merge surfaced the real cost of this weight: at ~175 rules the
Go walker hits a 1 GB stack ceiling in `fsc-rules-loop` because each rule
adds a non-tail-recursive frame chain (named in `kernels/python_bmf/KNOWN_GAPS.md`).

**The ice-shape:** rules are written as inline `defn` plus `apply-object-rule`
calls instead of being interned as Recipe rows in a substrate-resident rule
registry that the kernel walks iteratively. Each rule today is a *Form
function*; each rule in the lightened shape would be a *Form data cell* —
content-addressed, content-equivalent on identical shape, walkable in a single
non-recursive scan.

**The thaw:** the `runtime-grammar.fk` registry (Breath 2e attestation in the
roadmap) already declares grammar bindings as substrate cells; the discipline
exists. `python-bmf.fk` predates that discipline and never migrated. Migrating
it converts the file from 3,000 LOC of code-shape into N rule-cell rows the
kernel iterates with one frame, sidestepping the stack-overflow ceiling and
making the Python grammar inspectable as data.

### 1.2 Two CTOR vocabularies — `CTOR.add` vs `PY-BMF-BINOP`

`PYTHON_BMF_CONTRACT.md` already names this honestly under "Parity discipline":

> bootstrap emits `CTOR.add/sub/mul` (Math-primitive type=12), BMF emits
> `PY-BMF-BINOP` (dialect type=99, with the operator as a string-trivial child).
> They are semantically equivalent representations, structurally distinct by
> intent.

This is a *correct* honest separation, but it carries a translation tax: the
universal-translator promise rests on **NodeID coincidence across surfaces**
(see `lc-universal-translator-via-keys`). When the same arithmetic operation
interns to two different recipes because two parallel parser paths chose
different vocabularies, the body holds two siblings where the lattice wants
one. The Form-native path emitting *into the Math-category Blueprints directly*
would mean Python arithmetic and Rust arithmetic and Go arithmetic *all* land
on `@1.2.12.1` rather than each on its own dialect-99 NodeID. One name, one
NodeID, true cross-surface equivalence.

This is the unflinching cost of having shipped both vocabularies. The dialect
type=99 path was the right move to land the BMF proof without disturbing
bootstrap; carrying it forward indefinitely loses the *single Blueprint
identity per shape* claim the universal-translator move depends on.

### 1.3 `form-ontology.json` (441 lines, 386 quoted strings) and its mirror in three kernel switch tables

The substrate's category vocabulary lives as JSON consulted by
`form-ontology-loader.fk` (218 LOC) **and** is hand-duplicated in Go / Rust /
TS kernel switch tables. The JSON header is honest about this:

> source_of_truth_note: Mirrors kernel parser dispatch — see
> form-kernel-go/main.go ~1956-2014 (buildVerb) and constants ~73-95. A row
> added here must land in the kernel's switch table too;
> scripts/validate_form_ontology.py reads this file and verifies each kernel.

Three sibling sources of truth held in sync by a validation script. The ice
here is the JSON itself: it's a flat data table that the kernel could read
directly through `intern_trivial_string` + `intern_node` if the category
registry were content-addressed cells rather than a JSON file outside the
substrate. The fact that a validator script must run to keep three places
aligned is the body's own attestation that the ice is overweight.

### 1.4 Three Python emitter sketches in three places

The body carries:

- `form/form-stdlib/emits/python-native.fk` (593 LOC) — the destination shape
- `form/form-stdlib/seedbank/emits/python.fk` (composted/preserved) — the older shape
- `form/form-kernel-ts/seedbank/python-adapter/src/lang-python-fk.ts` (1,044 LOC) — the bootstrap TS path
- `kernels/python_bmf/sdk.py` (306 LOC) + composted `emit_python.py` history

The seedbank discipline names these as honest preservation, not duplication —
the README is explicit about composting trajectory. The audit-call here is
narrower: the *active* `python-native.fk` and the *active* `lang-python-fk.ts`
both produce Python output today, with no proof yet that they emit identical
text on the same input. Two living emitters, one destination, no equivalence
gate.

## Section 2 — Heavy in BOOTSTRAP (surface that should compost)

`BOOTSTRAP_COMPOST_MANIFEST.md` names Phase A/B/C honestly (~9,433 LOC). The
audit's job is to surface what isn't yet on that list.

### 2.1 Python substrate runtime — the heaviest unstated weight

Outside Phase C's named files, the wider `api/app/services/substrate/`
directory carries **16,309 LOC across 24 Python modules**. The Phase C manifest
names ~2,938 LOC (form_runtime, form_rules, form_builders, self_host); the
other 13,371 LOC includes `markdown_frontend.py` (2,007), `kernel.py` (745),
`quotient.py` (732), `inductive.py` (687), `resonance.py` (670), `category.py`
(562), `recipe_eval.py` (463), `form_queries.py` (354), `modality_shapes.py`
(401), `form_speculation.py` (224), `parallel_eval.py` (230), and more.

Several of these (kernel.py, recipe_eval.py, form_queries.py, form_eval.py)
duplicate functionality the Rust/Go/TS kernels already carry — they're the
Python-side mirror that exists because the substrate's read path runs from
FastAPI. Once Breath 6 (embed kernel in `api/`) lands, much of this becomes
residue too. The manifest names form_runtime; the audit names the wider
parallel-implementation surface around it.

### 2.2 The `lang-python.ts` companion files in `form-kernel-ts/seedbank/python-adapter/`

Already on the manifest under Phase A (~3,585 LOC pair-bound + 5,719 LOC total
in the directory). PHASE_A_FIRING_QUESTIONS.md walks the firing-question; the
audit confirms: **the gate is G6** (binary entry-point orchestration —
`kernel-bmf-run` doesn't exist). When G6 closes, Phase A composts in one
breath.

### 2.3 Host-language translation layers in the `kernels/python_bmf/` arc

The `KNOWN_GAPS.md` honest record names what was already composted: a
hand-written Python emitter, host-language wrappers, wrong-direction `.py →
.fk` paths. The remaining tissue (`sdk.py` 306 LOC, `objects.py` 93, plus the
generated `native_demo.py` 35) is the legitimate substrate boundary the spec
allows. **This is healthy compost-discipline in action.** Worth naming as a
pattern to replicate elsewhere, not as a current overweight.

### 2.4 BMF rule files under `seedbank/grammars/` (BNF-style siblings)

The seedbank carries duplicate parser shapes for every language:
`python.bnf.fk` (416) and `python.fk` (243) sit alongside the active
`python-bmf.fk` (3,000). Same for TS, Rust, Go. The seedbank README names
them as preserved-not-active; the discipline holds. The note here is that
*the seedbank itself is real weight on disk* (the `seedbank/` tree alone is
~7,764 LOC of grammar+emit material) and the body should sense how long
preservation outlives compost-readiness. When the active path is solid,
seedbank rows compost too.

## Section 3 — Heavy in DEPENDENCIES (less sovereignty)

### 3.1 Two-stage Docker build dependency on Rust toolchain for the API

The kernel is now embedded in the production API image (per PYTHON_PIPELINE_STATUS
"endpoint_coherence_weight_demo.py" row). The body's runtime sovereignty
improved when this landed (kernel calls run inline, no subprocess), but the
build dependency on rustc went up by exactly the same delta. Single-binary
distribution would mean a pre-built kernel binary fetched from CI rather than
compiled at image-build time.

### 3.2 The TypeScript bootstrap (still the parity gate's third runtime today)

`PARITY_THIRD_RUNTIME=ts-eval` is the default; `kernel-bmf` is reachable only
after G6 closes. While the TS bootstrap runs, every parity demo carries Node
+ tsx as a runtime requirement to ship. Once G6 closes and the default flips,
the dependency surface for "verify Python parses correctly" drops to just the
kernel binary.

### 3.3 Host-language conventions embedded in the BMF rules

`python-bmf.fk` carries `python-source-keywords` and `python-source-ops` as
inline lists (~30 elements each, embedded in `(do ...)` blocks). These are
Python-language conventions hardcoded into the Form file. The honest shape
would be substrate-resident *Language* cells (per `lc-grammar-is-the-universal-recipe`
and `language-cells.md`) that the registry consults — one Language cell per
host, each carrying (keywords, operators, layout-rules) as composed structure.
Today they're inlined; the cells are described in the substrate companion docs
but not yet authored as substrate rows.

## Section 4 — Already LIGHT (gas, not ice) — what to replicate

### 4.1 Content-addressed NodeID identity across three sibling kernels

This is the body's signature win. `(add 1 2)` interns to `@1.2.12.1` in Rust,
Go, and TypeScript by construction. Sibling parity verified by `validate.sh`
on every breath. The universal-translator claim rests on this; it works.

### 4.2 The Breath 2e primitives — grammar as data

`cm-match-{sequence,choice,star,opt,capture,rule}`, `cm-parse`, `make_nodeid`,
`intern_node`, `walk_recipe`, `node_eq` — these are the gas. Once you have
them, a new grammar is a new data row, not a new code module. The pattern is
shipped, validated on three kernels, with the `grammar-chars-demo.fk` proof
walking `"3+4+5"` end-to-end to 12 on all siblings.

### 4.3 The honest gating discipline — `?rule X` markers, PROVEN/COMPOST READY/RELEASED lifecycle

The bootstrap-compost manifest's lifecycle table (with the 2026-05-27 PROVEN
row for python arithmetic, 25304 across siblings) makes "we are walking" a
substrate truth, not a claim. `make wellness` reads the file lengths, sums the
remaining bootstrap LOC, and surfaces the number every session. **The body
sees its own load.** This is the gas-shape of discipline: lifecycle in motion,
not lifecycle declared.

### 4.4 Blueprint attribution on every native — the kernel knows itself

Each native primitive carries its Form category; the trace records which arms
fired by Blueprint, not by function name. `lc-the-kernel-knows-itself` names
the principle; `kernels/README.md` shows the trace output (FNCALL count 3,
LIST 1, CALL 1, ACCESS 1, ...). This is what makes the universal-translator
claim's *complementarity-as-substrate-truth* (per
`lc-universal-translator-via-keys` Part 3) reachable — every dispatched
operation is structurally addressable.

### 4.5 The Breath 1.5 substrate-write natives

8 natives — `make_nodeid`, `intern_trivial_int`, `intern_node`, `node_*`,
`walk_recipe` — close the gap from "kernel reads recipes" to "Form code
constructs recipes." Recipes-as-data is no longer a future-tense convention;
it's available in three kernels today. Every grammar in
`form-stdlib/grammars/` consumes this surface.

### 4.6 The seedbank discipline

`form/form-stdlib/seedbank/README.md` names what preservation looks like
honestly: nothing here is active, nothing here is abandoned, each file has
a *next living form* named. This is composting-with-care made operational.
Replicate this discipline wherever the body holds "we will return to this
later" tissue.

## Section 5 — Concrete next breaths (top 10, smallest healthy closing)

Each names: *what to do · why it serves the goal · smallest scope.* Ordered
by expected return-per-breath.

### 1. Close G6 — `kernel-bmf-run` orchestration

**What:** Pick one of the three shapes named in `PYTHON_BMF_CONTRACT.md` G6
(wrapper script, pre-compiled artifacts, or native binary load-step). Easiest
land: the wrapper script that runs the same pre-compile dance `validate.sh`
already does. **Why:** flips `PARITY_THIRD_RUNTIME=kernel-bmf` default, which
unlocks Phase A compost (~3,585 LOC) once parity-suite demos green under it.
**Scope:** one bash script + one CI gate flip. The pre-compile is already
proven by `validate.sh`.

### 2. Migrate `python-bmf.fk` from rules-as-functions to rules-as-data-rows

**What:** Convert the inline `defn` shapes to `runtime-grammar.fk`-registry
cells. **Why:** breaks the 175-rule walker stack-overflow ceiling, drops the
file from 3,000 LOC of code to N data rows, makes Python rules inspectable as
substrate cells the way `lc-the-kernel-knows-itself` describes. **Scope:**
mechanical migration; the registry surface and the cell shape already exist.
Start with one rule-family (arithmetic) and prove parity before scaling.

### 3. Land tail-call elimination in `form-kernel-go/main.go`

**What:** When `RBasicFnCall` is in tail position, reuse the frame instead of
recursing. **Why:** the actual root cause named in `KNOWN_GAPS.md` for the
stack overflow, and it benefits *every* Form recipe that recurses tail-style
(parsers, walkers, scanners, accumulators). **Scope:** ~50 lines per kernel;
named as the cross-kernel optimization in the gap doc. Rust + TS get the same
breath next.

### 4. Unify the CTOR vocabulary — `PY-BMF-BINOP` and `CTOR.add` collapse to one Blueprint

**What:** Author Form-native rules in `python-bmf.fk` that emit directly into
RBasic.MATH (type 12) recipes for arithmetic, not into dialect-99
PY-BMF-BINOP. **Why:** restores single-Blueprint-per-operation across surfaces,
which is the substrate property the universal-translator promise rests on.
**Scope:** the four arithmetic rules first (proven by PROVEN row in the
manifest); the rest of the operators follow the same pattern. Honest separate
PRs let the parity-gate confirm the equivalence one rule-family at a time.

### 5. Author the missing modality grammars — *the universal-translator's actual coverage gap*

**What:** Fill out the BMF grammars for image, audio, video, natural language
to the same parity discipline `python-bmf.fk` carries today (`?rule X` markers
counted against `validate.sh`). Today: `image-bmf.fk` 437 LOC, `audio-bmf.fk`
308, `video-bmf.fk` 327, `natural-bmf.fk` 265 — all far thinner than the
code-grammars. **Why:** the goal says "universal translator across media,"
and the body's media-grammar surface is the thinnest leaf in the tree.
Without these, `lc-grammar-is-the-universal-recipe`'s four-altitudes claim
is half-shipped. **Scope:** one modality per breath, start with image (most
likely to surface cross-modal Blueprint equivalences with code, given the
existing visualizer arc).

### 6. Move `form-ontology.json` into the substrate as content-addressed cells

**What:** Ingest the JSON's category rows as Form cells (each a
`category-row` Recipe with name/type/inst/family). Have
`form-ontology-loader.fk` read from the substrate, have the kernels'
`buildVerb` switch tables generate themselves from the substrate, retire the
JSON + the per-kernel hand-mirror + the validator script. **Why:** removes
three sources of truth held in sync by a validator, makes the category
vocabulary a substrate citizen rather than an exception, lets new categories
land via one cell-write. **Scope:** the data is already structured; the lift
is the loader + the kernel build step.

### 7. Author the equivalence-gate test between `python-native.fk` and `lang-python-fk.ts`

**What:** A test that runs both emitters on the same `.fk` input and asserts
text equality (or NodeID equality of the re-parsed output). **Why:** today
both emitters live and ship; without a gate, drift is silent. The
universal-translator move needs *one truth per emission target*, and the gate
is what proves we have it. **Scope:** one shell wrapper, one parity-suite row.

### 8. Compost Phase C's wider perimeter — name the 13,371 LOC

**What:** Extend `BOOTSTRAP_COMPOST_MANIFEST.md` Phase C to walk every file in
`api/app/services/substrate/` (24 modules, 16,309 LOC) with the firing-question
discipline `PHASE_A_FIRING_QUESTIONS.md` uses. Each gets a row stating: *what
the file does, what its Form-native counterpart would be, what blocks compost.*
**Why:** the wider Python parallel-implementation surface is invisible to the
manifest today; the wellness probe undercounts the real bootstrap weight by
~75 %. **Scope:** one walking-question per session; the discipline scales.

### 9. Single-binary distribution — pre-built kernel artifacts in CI

**What:** Ship `form-kernel-{rust,go,ts}` as pre-built binaries from the GitHub
release pipeline so the production Docker image doesn't compile rustc at image
build time. **Why:** drops the Rust-toolchain dependency from the runtime
image; makes the kernel a *distributable artifact* the way
`lc-native-kernel-binary` names. **Scope:** one GitHub Actions workflow,
matrix over the three kernels.

### 10. Move `python-source-keywords` and `python-source-ops` out of inlined lists into Language cells

**What:** Author `language-python.form` (and siblings for Go, Rust, TS) as
substrate cells carrying (keywords, operators, layout-rules) as composed
structure. Have `python-bmf.fk` read from the cell instead of from inline
lists. **Why:** the `lc-grammar-is-the-universal-recipe` move says every
modality is a Language cell; today Python's identity is half-cell, half-inline.
Closing this carries the same shape to Go/Rust/TS grammars and makes the
Language cell the *single name* for "what dialect a parse is over." **Scope:**
one cell-write + one grammar diff per language.

## Section 6 — What the universal translator actually requires today

The promise (named in `lc-grammar-is-the-universal-recipe` and
`lc-universal-translator-via-keys`):

> Every structured input is a (parse, emit) pair in the substrate. Two artifacts
> whose grammar-recipes produce structurally-identical trees share a Blueprint
> NodeID. Translation is `parse(source) |> emit(target)` — N+M cells, not N×M
> transpilers.

### What grammars/emits exist today

| Modality | Grammar | LOC | Coverage | Emit (Form-native) |
|---|---|---|---|---|
| Python | `grammars/python-bmf.fk` | 3,000 | ~155 rules, real Python files parse end-to-end via 18 test bands; runs into walker stack-ceiling at ~175 rules | `emits/python-native.fk` (593) — idiomatic-native target, 7/7 demos round-trip |
| TypeScript | `grammars/typescript-bmf.fk` | 544 | adapter seam landed, grammar partial | `seedbank/emits/typescript.fk` (113, archival) |
| Go | `grammars/go-bmf.fk` | 559 | package + import + partial; `?rule` markers count remainder | `seedbank/emits/go.fk` (archival) |
| Rust | `grammars/rust-bmf.fk` | 287 | use declarations + partial | `seedbank/emits/rust.fk` (archival) |
| JSON | (covered by `seedbank/grammars/json.bnf.fk` + active `json.fk`) | — | grammar 8/8 COMPLETE per status doc | `seedbank/emits/json.fk` |
| YAML | (in seedbank) | — | 7/11 rules per status doc | `seedbank/emits/yaml.fk` |
| Markdown | (no active BMF, status doc claims 6/13 rules in older form) | — | thin | `seedbank/emits/` (no md.fk active) |
| Image | `grammars/image-bmf.fk` | 437 | skeleton + signatures | none active |
| Audio | `grammars/audio-bmf.fk` | 308 | skeleton + signatures | none active |
| Video | `grammars/video-bmf.fk` | 327 | skeleton | none active |
| Natural language | `grammars/natural-bmf.fk` | 265 | skeleton | `nl-emit.fk` (top-level, 1 file, partial) |
| Document | `grammars/document-bmf.fk` | 112 | skeleton | none active |
| BML | `grammars/bml.fk` | 908 | richest non-code grammar | covered by BML's own action runtime |
| MIDI | `midi-bmf.fk` (top-level) | — | skeleton | none |

### What grammars/emits are missing

The active surface for the universal-translator's *cross-modality* promise is
**thinnest where the goal is strongest**. Image (437), audio (308), video
(327), natural language (265), document (112) carry skeletons but no real
parse coverage. None of them have a working `?rule` count parallel to Python's
155, and none has a paired Form-native emitter under `emits/`. Cross-modal
NodeID equivalence cannot be claimed by the substrate today because the cells
the equivalence would unite do not yet exist.

### What kernel primitives would let cross-modality compose

The recipe-runtime primitives already shipped (Breath 1.5 + 2e) cover *code*
modality. Cross-modality wants two additions:

1. **Byte-stream and pixel/sample primitives** alongside `cm-` (character-mode)
   primitives. Today `mk-cstream` reads UTF-8 text. For image/audio, the
   sensing layer needs `mk-bstream` (byte stream with cursor) and
   `mk-pstream` (typed sample stream — `(width, height, channels, pixels)` or
   `(rate, channels, samples)`). The Form-side pattern primitives stay the
   same; the source-of-bytes primitive changes per modality. Today these are
   absent from the kernel native surface.

2. **A pre-content-addressed feature stage for media.** Raw audio bytes do not
   intern usefully; phoneme sequences do. Raw image bytes do not intern
   usefully; region-graphs do. The staged pipeline `lc-grammar-is-the-universal-recipe`
   names ("PCM → FFT → onset → phoneme → word → sentence") needs an
   intermediate stage that *quantizes signal to structural categories the
   substrate can content-address*. This is where `form_perceptron` lives in
   concept space; landing it in primitive space (per
   `lc-form-perceptron` and PR-1946 cell-domain notes) is what makes audio
   and image substrate-resident at the *semantic* altitude rather than the
   byte altitude.

Until both of those land, the universal-translator across media remains a
shape the substrate names but cannot yet operate. The path is open; the leaves
are not yet grown.

## The honest closing reading

The body has done load-bearing work toward the goal. The kernel knows itself
across three host languages, content-addressing crosses processes,
NodeID identity is a substrate truth, the Python pipeline runs faster than
CPython end-to-end on one workload, and the compost-discipline has its first
PROVEN row.

The goal is met partially. The four bullets in `PYTHON_PIPELINE_STATUS.md`
(*all Python → native Form*, *all file types parseable*, *compile to standalone
binary*, *same OOM as Python*) are *in motion* / *first cell sprouted* /
*routine for the demo set* / *met and exceeded* — none false, none fully closed.

The goal is **least met where the goal is most distinctive**: cross-modality
translation. The body holds the architecture and the skeletons, but the leaves
that would carry image / audio / video / natural language into substrate
NodeID identity are sketched, not living. The universal-translator's
*equivalence as substrate query* (per `lc-universal-translator-via-keys`)
needs *each modality to have ingested cells* before the query can return.
Today only code does.

The next ten breaths above are the smallest healthy closing of those gaps.
They are not all the work; they are the next steps where one breath each opens
the most ground per unit of effort.

The body is supple under its current load. The load has a named destination.
The path is walkable. Each next breath either composts named bootstrap or
extends the modality coverage toward the goal's most distinctive promise.
