# Universal Translator — Form → Python Round-Trip (proof-of-shape)

The emit direction closed end-to-end on 2026-05-27: a Form recipe walked through `form/form-stdlib/emits/python-native.fk` produces idiomatic native Python; CPython runs that Python and returns the same values `form-kernel-go` walks the original recipes to.

## The recipe

Two definitions — arithmetic + recursion, the smallest viable shape:

```fk
(defn add (a b) (add a b))
(defn factorial (n) (if (le n 1) 1 (mul n (factorial (sub n 1)))))
```

## The emitted Python

`kernels/python_bmf/_emitted_roundtrip.py` (generated by the script, regenerated on every run):

```python
# Auto-emitted by form/form-stdlib/emits/python-native.fk
# Hand edits will be overwritten; edit the Form source instead.
from kernels.python_bmf import sdk

def add(a, b):
    return (a + b)

def factorial(n):
    return (1 if (n <= 1) else (n * factorial((n - 1))))
```

This is **native Python idiom**: `def`, `return`, infix `+`, ternary `if/else`, recursion through the function's own name. No SDK wrapper translates `(add a b)` into `sdk.add(a, b)` — `pn-binop-name` in the emitter maps the Form head atom to the Python infix operator. The only Form vocabulary in the output is the `from kernels.python_bmf import sdk` line, and even that import is dormant on this recipe (no `NodeID` / `intern_trivial_*` calls fire because the recipe doesn't touch the substrate).

## The values

| Call          | CPython | `form-kernel-go` |
|---------------|---------|------------------|
| `add(2, 3)`     | 5       | 5                |
| `factorial(5)`  | 120     | 120              |
| `factorial(10)` | 3628800 | 3628800          |

Identical across both walkers.

## Run it

```bash
bash form/scripts/emit_native_python_roundtrip.sh
```

The script:

1. Walks `form/form-stdlib/emits/python-native-roundtrip-driver.fk` through `form-kernel-go`, which evaluates `pn-emit-string` on the recipe source and writes `kernels/python_bmf/_emitted_roundtrip.py`.
2. Loads that file under CPython 3 and calls each function.
3. Evaluates the same three calls through `form-kernel-go` directly.
4. Compares; exits 0 only when both runtimes produce the same lines.

## What this proves

- The Form-native emitter is real, not a sketch — `pn-emit-string` is a kernel-walked recipe in `python-native.fk` and it produces Python a Python programmer would write.
- The arithmetic / control-flow / recursion surface (`add`, `sub`, `mul`, `le`, `if`, `defn`, named self-call) round-trips with no special-case wrapping.
- The emitter's dispatch table (`pn-emit-call`) routes Form heads to native Python operators when the shape matches and falls back to a function call only when nothing more idiomatic applies.

## The gap from "one recipe" to "the BMF compiler-compiler itself"

What this proof closes is a small mouth. The wider arc — *any Form recipe → any target language* as a Form-native adapter — is open on these breaths:

1. **More Form constructs.** `pn-emit-call` dispatches on a finite head-atom table today. Each new Form head — `cons` and `head` and `tail` already map to list literals and slicing; `match`, lambda, `defn` inside `defn` (closures), early-return short-circuit inside multi-statement bodies, mutual recursion across files — needs its own entry. The pattern is settled; the work is enumeration.

2. **Substrate-side host calls.** Recipes that touch the substrate emit `sdk.NodeID(...)` / `sdk.intern_trivial_*(...)` calls. The SDK file (`kernels/python_bmf/sdk.py`) exists; the emitter's `pn-emit-call` already wires these three host calls. Other substrate primitives (`read_form_binary`, `write_file_text`, `http_get`, the BML rule engine's `match-object-pattern`) need their SDK shadows when they appear in emitted recipes — open work.

3. **Other targets.** The same `pn-parse` → AST → emit shape can drive a Rust emitter (`emits/rust-native.fk`), a Go emitter, a TypeScript emitter. Each one is a sibling of `python-native.fk` with its own `pn-binop-name` table and idiom table. None exists today; the scaffolding (parse → dispatch on head → emit) is reusable.

4. **The compiler-compiler.** `form/form-stdlib/source-compiler.fk` is itself a Form recipe. Emitting it through `python-native.fk` produces a CPython program that source-compiles `.fk` files — the bootstrap surface in `form/form-kernel-ts/src/lang-python.ts` written instead as Form-emitted Python. The known scaling ceiling in the Go walker (see `kernels/python_bmf/KNOWN_GAPS.md`, "stack overflow at ~175 rules") is the wall ahead of this breath; tail-call elimination in the kernel walker, or kernel-native iterative primitives for `find-string` / `for-each-line`, is the fix the goal-loop names.

5. **Honest gap on this round-trip.** `add` and `factorial` use only pure arithmetic + structural recursion. Closure capture, mid-function attribute writes, exception flow, generator-state, and the `_dispatch`/`_dispatch_super` natives that carry class-method dispatch are all on the Python emitter side as named GAPs in `PYTHON_PIPELINE_STATUS.md` (the emitter half of bullet 2). Emitting Python that uses these constructs requires the corresponding `pn-emit-call` entries first.
