## What was implemented

### 1. `api/app/services/smart_reaper_service.py` (325 lines)
New standalone service with:
- `_is_pid_alive(pid)` — checks process liveness via psutil (with os.kill fallback)
- `_read_tail(path, chars)` — captures partial log output
- `_estimate_partial_pct(output, expected_chars)` — estimates output completeness
- `_parse_runner_online(runner_record, now)` — checks runner lease expiry
- `diagnose_task(task, *, log_dir, runner_map, ...)` — full diagnosis:
  1. Queries runner liveness via lease_expires_at
  2. Checks provider PID liveness
  3. Captures partial output from task log file
  4. Reads .task-checkpoint.md from worktree
  5. Classifies: runner_dead / provider_crashed / slow_provider / unknown
  6. Returns should_reap/should_extend + structured context_patch
- `build_runner_map(runners)` — builds runner_id -> record lookup dict
- `diagnose_batch(tasks, ...)` — batch diagnosis for multiple tasks

### 2. `api/scripts/local_runner.py` — `_reap_stale_tasks` updated
- Fetches runner registry via `/api/agent/runners?include_stale=true`
- Builds runner_map from registry response
- For each stale task: calls `smart_reaper_service.diagnose_task()`
- If `should_extend`: logs and skips (does NOT reap)
- If `should_reap`: writes structured `smart_reap_diagnosis` into task context
- Retry context now includes: `reap_error_class`, `resume_reason` (if partial)
- Fallback to legacy simple diagnosis if service import fails

## Key design decisions
- Smart diagnosis runs BEFORE reaping — alive provider gets extension
- Context patch includes: error_class, runner_alive, provider_pid_alive, partial_pct
- Partial threshold: 20% of expected_output_chars (2000 chars default)
- Slow multiplier: 1.5x max_age_minutes before reaping a slow-but-alive provider
